# 🌈 NURI : Emotion-Driven Digital Memory Archive Platform

> **기억은 사라지지 않습니다. 우리는 아이를 잊지 않습니다.**

NURI는 사랑하는 반려동물의 시간을 사진 몇 장으로 흘려보내지 않기 위해 만든 앱입니다.  
단순히 기록을 쌓는 도구가 아니라, 보호자가 아이의 하루와 감정을 오래 붙잡아 둘 수 있도록 기억을 구조화하는 디지털 아카이브를 지향합니다.

지금의 NURI는 추모만을 위한 앱이 아닙니다.  
함께하는 오늘을 기록하는 앱이면서, 언젠가 돌아보게 될 시간을 위해 감정과 기억을 정리하는 서비스입니다.

---

## 1. Why NURI

NURI는 창업자의 개인적인 상실 경험에서 시작되었습니다.

사랑하는 반려견을 떠나보낸 이후, 남겨진 가족이 슬픔을 정리하지 못한 채 사진과 기억을 흩어진 상태로 붙잡고 있는 모습을 보며 질문이 생겼습니다.

- 왜 우리는 아이와의 시간을 오래 남길 공간을 제대로 갖고 있지 못할까
- 왜 기억은 늘 사진첩과 메신저, 메모 사이에 흩어져 있을까
- 왜 감정은 기록되지 못하고 사라질까

NURI는 이 질문에서 출발한 서비스입니다.

### NURI가 풀고자 하는 문제

1. 반려동물과의 시간을 장기적으로 관리할 공간이 부족합니다.
2. 감정을 정리할 수 있는 기록 구조가 부족합니다.
3. 사진, 메모, 일정이 분리되어 기억의 맥락이 끊깁니다.
4. 추억은 남아도 다시 꺼내보기 쉬운 구조가 부족합니다.
5. 나중에 확장 가능한 감정 데이터 기반이 부족합니다.

### NURI의 핵심 가치

| 핵심 개념 | 설명 |
| --- | --- |
| 기억 구조화 | 사진, 텍스트, 태그, 날짜를 하나의 기록 단위로 묶습니다 |
| 감정 데이터 축적 | 감정/카테고리/활동 데이터를 쌓아 이후 회상과 분석의 기반을 만듭니다 |
| 보호자 중심 UX | 빠르게 기록하고, 자연스럽게 다시 꺼내볼 수 있는 흐름을 우선합니다 |
| 운영 가능한 구조 | 단발성 MVP가 아니라 멀티펫, 날씨, 위젯, 추후 AI/구독까지 확장 가능한 설계를 유지합니다 |

## 3. 서비스 구조

### 3.1 인증 / 온보딩

- Supabase Auth 기반 로그인/세션 유지
- 앱 부팅 시 AsyncStorage hydrate 후 `supabase.auth.getSession()` 동기화
- `profiles.nickname` 기준 닉네임 설정 및 복원
- 로그인 직후 / 앱 재실행 직후 / 로그아웃 직후 상태 전환이 자연스럽게 이어지도록 부트 경로 정리

### 3.2 멀티펫

- `pets[]`, `selectedPetId`를 전역 상태로 관리
- 홈에서 미니 프로필 전환으로 즉시 다른 아이의 데이터로 이동
- 서버 truth 우선 정책으로 stale cache를 줄이는 방향 유지
- 서버가 실제 `0-pet`를 반환하면 캐시를 억지로 유지하지 않고 온보딩 플로우로 정리

### 3.3 기록(Memories)

- 제목, 본문, 날짜, 감정, 태그, 카테고리, 이미지 다중 첨부 지원
- create 후 상세 진입 유지
- edit 후 즉시 반영 유지
- delete 후 복귀 흐름 유지
- local URI 즉시 반영 유지
- 다중 이미지 순서 및 `fraction(1 / n)` 계약 유지

### 3.4 타임라인

- 홈과 역할을 분리한 전체 탐색 화면
- 카테고리 / 월 이동 / 히트맵 / 필터 기반 탐색
- 상세 / 수정 전환 시 하단 탭 구조를 깨지 않도록 Timeline 내부 stack 유지

### 3.5 상세

- 다중 이미지 슬라이더
- 기록 본문 / 메타데이터 / 액션 메뉴
- 수정 / 삭제 / 복귀 흐름 정리
- 날씨와 연결된 기록 맥락 표시 확장 가능 구조 유지

### 3.6 날씨

- 홈 카드와 날씨 상세 간 번들 재사용
- 위치 → 행정동 → 날씨 fetch 체인 최적화
- 실내 활동 추천과 연결
- 캐시/초기 번들/복귀 정책을 보수적으로 정리해 체감 지연을 줄이는 방향 유지

### 3.7 홈

- 보호자가 하루를 빠르게 확인하는 요약 화면
- 오늘의 사진 / 오늘의 기록 / 주간 요약 / 일정 / 최근 활동 / 월간 다이어리
- 멀티펫 전환과 기록 갱신 시 전체 화면이 과하게 흔들리지 않도록 섹션 단위로 렌더 범위 축소

### 5.2 기록 이미지 구조

기록 이미지는 현재 `memory_images`를 중심으로 관리합니다.

#### 현재 원칙

- `memory_images`가 다중 이미지의 실질적인 source of truth입니다.
- `memories.image_url`, `memories.image_urls`는 fallback / 호환 목적의 mirror로 유지합니다.
- 읽기 경로는 `memory_images` 우선, legacy 필드는 fallback으로만 사용합니다.
- 쓰기 경로는 `memory_images`를 먼저 맞춘 뒤 legacy를 동기화합니다.

#### 왜 이렇게 바꿨는가

기존의 단일 `image_url` 중심 구조는 다중 이미지, 썸네일, 재처리, 백필 운영에 한계가 있었습니다.  
지금 구조는 다음을 전제로 합니다.

- 여러 장 이미지를 순서대로 보관할 수 있어야 한다
- 원본과 썸네일을 분리할 수 있어야 한다
- worker가 후처리할 수 있어야 한다
- backfill과 운영 복구가 가능해야 한다

### 5.3 thumbnail worker / backfill 방향

현재 구조는 thumbnail worker와 backfill을 고려한 형태로 정리되어 있습니다.

- 신규 업로드 후 worker가 썸네일을 생성할 수 있는 구조
- 기존 데이터에 대해서도 backfill 가능
- 실패 backlog를 추적하고 재처리 가능한 운영 전제
- 읽기 계층은 썸네일/원본/legacy fallback을 안전하게 해석하는 방향 유지

### 5.4 업로드 큐와 복구 정책

기록 이미지 저장은 단순 fire-and-forget이 아니라, 중단/복귀를 버틸 수 있는 방향으로 정리되어 있습니다.

- upload queue 실행 락 적용
- 동일 task 중복 업로드 방지
- create / edit 저장 직후 즉시 처리 시도
- 실패 시 queue 유지 + 이후 재시도
- 사용자는 조용히 성공처럼 보지 않도록 warning 정책 유지
- 앱 재실행 / foreground 복귀 시 남은 task 복구 가능

### 5.5 delete / update 정합성 기준

- delete는 row/file 불일치 위험이 더 낮은 방향으로 순서를 정리했습니다
- `memory_images`와 legacy 동기화는 partial state를 줄이는 방향으로 보강했습니다
- 제거된 원본/썸네일 파일 cleanup 책임을 서비스 계층으로 모아, 화면별 중복 삭제와 orphan 파일 가능성을 줄였습니다
- 완전한 DB transaction은 아니더라도, 사용자 관점에서 더 안전한 상태를 우선합니다

## 7. 최근 최적화와 안정화 진행 내용

최근 작업은 단순 기능 추가보다, **운영 리스크를 줄이고 실기기 체감을 다듬는 방향**에 집중되어 있습니다.

### 7.1 데이터 정합성 안정화

- 업로드 큐 실행 락 및 중복 task 방지
- create 직후 이미지 저장 복구 구조 정리
- edit 저장 시 silent drop 방지
- delete 정합성 순서 보강
- `memory_images` / legacy 동기화 안정성 보강
- superseded/commit 실패 직전 새로 업로드된 파일 정리와 제거 대상 원본/썸네일 cleanup을 서비스 계층으로 모아 장기 운영 orphan 리스크를 추가로 낮췄습니다

### 7.2 날씨 체감 최적화

- 홈과 상세의 날씨 번들 재사용
- 불필요한 위치/행정동/날씨 재요청 축소
- focus / app active invalidate 정책 보수화
- 상세 진입 직후 체감 지연 완화
- 권한이 이미 `denied/blocked`인 상태에서는 자동 permission 재요청과 live fetch를 반복하지 않도록 정리해, 실사용 환경에서의 지연과 에러 흔들림을 추가로 줄였습니다

### 7.3 부트 / 세션 / 멀티펫 상태 정리

- stale cache보다 서버 truth를 우선하는 방향으로 정리
- `0-pet` 상태와 fetch 실패 상태를 분리
- 펫 전환 직전 prewarm으로 홈 공백 완화
- 첫 진입 splash 구조 유연화

### 7.4 홈 렌더 최적화

- 넓은 selector 구독 축소
- 정적 섹션과 동적 섹션 분리
- 주간 요약 / 최근 활동 / 월간 다이어리 / 오늘의 사진 계산 경계 분리
- `TodayRecordsSection` 슬라이더 상태와 controller를 부모에서 분리
- `오늘의 기록` 슬라이더는 JS 브리지 경로 축소, 인접 카드 우선 이미지 로딩, 자동 슬라이드 규칙 정리까지 1차 최적화를 마쳤지만 저사양 Android와 느린 네트워크 기준 실기기 관찰은 계속 필요합니다

### 7.5 타임라인 / 상세 장기 데이터 리스크 완화

- 타임라인은 `FlashList` extraData와 preload 효과를 더 보수적으로 묶어 entity 변경 시 전체 흔들림과 비포커스 preload 비용을 줄였습니다
- 상세는 missing record hydration 중 곧바로 not-found를 보여주지 않도록 바꾸고, 다중 이미지 signed URL hydration을 1장 우선 → 일부 eager → 나머지 deferred 구조로 나눠 첫 진입 체감 비용을 줄였습니다
- 다만 수백/수천 건 데이터와 저사양 실기기 기준 최종 평가는 여전히 운영 관찰과 release QA가 필요합니다

## 8. 현재 운영 관점에서 중요한 결정사항

### 8.1 서버 truth 우선

- 인증과 펫 목록은 가능한 서버 응답을 truth로 봅니다.
- 캐시는 UX 완충 장치이지, 서버 truth를 덮는 source of truth가 아닙니다.

### 8.2 이미지 저장은 복구 가능해야 함

- 앱이 중간 종료되더라도 이미지 저장 작업은 복구 가능해야 합니다.
- “일단 성공처럼 보이게”보다 “실제로 복구 가능한 구조”를 우선합니다.

### 8.3 리스트/홈/타임라인은 항상 실기기 기준

- 홈은 빠르게 열려야 하고
- 타임라인은 대량 데이터에서 버텨야 하며
- 상세는 이미지와 메타 정보가 자연스럽게 보여야 합니다

### 8.4 fallback은 유지하되, source of truth는 분명하게

- legacy를 급하게 걷어내기보다, 읽기 fallback은 유지합니다
- 대신 새 구조의 truth가 어디인지 문서와 코드에서 분명히 맞춥니다

### 8.5 legacy read fallback 축소 판단 기준

지금 시점의 NURI는 `memory_images`를 중심으로 읽고 쓰는 구조가 맞지만,  
legacy read fallback을 **즉시 제거할 단계는 아직 아닙니다.**

그 이유는 명확합니다.

- create 직후에는 row가 먼저 만들어지고, 이미지 최종 정합은 queue가 이어받는 구간이 남아 있습니다
- 기존 데이터와 운영 중 실패 케이스를 완전히 0으로 보장할 만큼 fallback 관측 기간이 충분히 쌓이지 않았습니다
- `memory_images` read 실패 시 legacy mirror가 실제 안전망으로 동작하는 경로가 아직 남아 있습니다

즉, 지금은 fallback을 “구조적 과도기”가 아니라 “운영 안전망”으로 보고 유지하는 단계입니다.  
다음 판단 기준은 아래와 같습니다.

1. 신규 create/edit/delete 이후 `memory_images` 정합 실패가 충분한 기간 동안 재현되지 않을 것
2. backfill 이후 legacy-only 데이터가 사실상 남아 있지 않을 것
3. 읽기 경로에서 fallback hit이 모니터링 기준으로 거의 사라졌다고 확인될 것
4. queue/worker/복구 경로까지 포함해 중간 종료 케이스가 안정적으로 복구될 것

이 조건이 충족되면, 그때부터는 legacy read fallback을 단계적으로 줄이는 판단이 가능합니다.

## 10. 앞으로 남은 작업 방향

현재 NURI는 “기록 앱이 일단 동작한다” 단계는 넘었습니다.  
이제는 출시 전 기준으로 남은 불안정 경계를 줄이고, 이후 확장 가능성을 해치지 않는 방향이 더 중요합니다.

### 우선순위가 높은 다음 작업

1. 최종 QA와 운영 회귀 점검
2. Crashlytics 실수집 확인
3. 남아 있는 fallback / legacy 축소 판단
4. auth / boot edge case와 타입 우회 / 런타임 shape 리스크 점검
5. 출시 빌드 기준 실기기 검증

### 이후 확장 후보

- Daily Recall 서버 고정 완성
- AI 메시지 / 챗
- 구독 / 결제
- 커뮤니티 / 댓글 / 반응
- 기념일 / 알림 / 회상 자동화

## 11. 개발 원칙

NURI는 다음 원칙을 지키며 개발합니다.

- 보호자 중심 UX를 최우선으로 봅니다.
- 서버와 스토리지 정합성을 가볍게 타협하지 않습니다.
- 실기기 체감 성능이 개선되지 않으면 최적화가 끝난 것이 아닙니다.
- 추억을 다루는 서비스인 만큼, “조용히 실패하는 흐름”을 허용하지 않습니다.
- 임시방편보다 운영 가능한 구조를 우선합니다.

## 12. 한 줄로 정리하면

NURI는 반려동물과의 시간을 사진첩이 아닌 **구조화된 기억**으로 남기기 위한 앱입니다.  
지금의 NURI는 그 기억이 사라지지 않도록, 기록과 이미지와 상태 흐름을 실제 서비스 기준으로 다듬어 가는 단계에 있습니다.
