# NURI 프로젝트 리서치

업데이트: 2026-03-23

## 추가 메모: 2026-03-23 커뮤니티 런타임 오류 원인 정리

- 커뮤니티 리스트 진입 시 `PostCard.tsx`에서 `The result of getSnapshot should be cached to avoid an infinite loop`, `Maximum update depth exceeded`, hook order 경고가 함께 발생했다.
- 실제 원인은 `PostCard`가 리스트 item 컴포넌트 내부에서 직접 Zustand store를 구독하면서, 댓글 preview fetch와 selector 구독이 item 단위로 동시에 돌아간 구조였다.
- 이 구조는 겉보기에는 동작해도, React Native Fabric + React Refresh 환경에서는 item 단위 store 구독이 hook 순서 변경 경고와 snapshot 불안정 문제를 더 쉽게 드러낼 수 있다.
- 특히 `PostCard` 안에서 댓글 cache, 댓글 상태, fetch action을 함께 읽고, passive effect에서 `fetchPostComments(post.id)`를 직접 호출한 점이 런타임 안정성을 떨어뜨렸다.
- 이번 수정에서는 `PostCard`를 store를 모르는 순수 props 컴포넌트로 되돌리고, 댓글 preview fetch 책임을 `CommunityListScreen`으로 올렸다.
- 즉, 리스트 화면이 `commentsByPostId`, `commentsStatusByPostId`, `fetchPostComments`를 한 번만 구독하고, 각 item에는 `latestComment`만 props로 내려주는 구조로 정리했다.
- 이 방식이 더 안전한 이유는 다음과 같다.
  - item 컴포넌트 hook 구성이 단순해진다.
  - selector snapshot 불안정 지점을 리스트 화면 하나로 모을 수 있다.
  - 댓글 preview fetch 책임이 화면 레벨로 올라가서 추적이 쉬워진다.
  - 이후 preview 전용 API나 snapshot 필드를 추가할 때도 교체 범위가 작다.
- 추가로 제목 필드 도입 이후 `communityPostEditor.shared.ts`의 draft 비교 함수가 `undefined.trim()`을 호출할 수 있는 상태였다.
- 이 문제는 old draft / refresh / partially hydrated state에서 `title` 또는 `content`가 비어 있거나 누락됐을 때 바로 런타임 오류로 이어질 수 있었다.
- 이번 수정에서는 draft 비교 함수에서 문자열을 항상 ``${value ?? ''}.trim()`` 형태로 정규화해 nullable-safe 하게 처리했다.
- 중요한 점은, 위 두 문제를 수정한 뒤에도 디바이스에서 기존 `PostCardBase -> useStore`, `current.title.trim()` 스택이 계속 보일 수 있다는 것이다.
- 이 경우는 소스가 아직 안 고쳐진 것이 아니라, React Refresh / Metro가 이전 hook 그래프와 이전 모듈 본문을 붙잡고 있는 상태일 가능성이 높다.
- 특히 hook 순서가 바뀐 컴포넌트(`PostCard`)와 shared 유틸(`communityPostEditor.shared`)을 연속 수정한 뒤에는 Fast Refresh만으로는 안전하게 복구되지 않을 수 있다.
- 따라서 이런 류의 오류는 소스 수정 + `tsc`/`eslint` 검증만으로 끝내지 말고, 반드시 Metro reset + 앱 full reload까지 확인해야 한다.
- 결론적으로 현재 커뮤니티 도메인에서의 실무 기준은 다음과 같다.
  - 리스트 item 컴포넌트 안에서 store를 직접 넓게 구독하지 않는다.
  - preview fetch는 가능하면 화면 레벨에서 제어한다.
  - draft 비교/복원 함수는 nullable-safe 문자열 정규화를 기본으로 둔다.
  - React Refresh 상황에서도 hook 순서가 흔들리지 않도록 item 컴포넌트는 최대한 순수하게 유지한다.
  - RLS policy 안에서 같은 테이블을 다시 직접 조회하면 `42P17 infinite recursion detected in policy`가 날 수 있으므로, self-reference 검증은 `security definer` helper function으로 분리한다.

## 추가 메모: 2026-03-23 커뮤니티 댓글/답글 저장 실패 원인 정리

- 커뮤니티 상세에서 plain 댓글과 답글 모두 `다시 시도해 주세요.` 토스트만 뜨고 저장이 되지 않는 문제가 있었다.
- 처음에는 앱 레이어의 payload shape, nullable 처리, 새 컬럼 미적용 가능성을 의심했지만, 실제 원인은 SQL RLS policy 재귀였다.
- 디버그 로그 기준 실제 에러는 아래였다.
  - `code: 42P17`
  - `message: infinite recursion detected in policy for relation "comments"`
- 원인 policy는 `comments_insert_own`이었다.
- 이 policy가 insert 시점에 다시 `public.comments`를 직접 조회해 `parent_comment_id`, `depth`, parent active 상태를 검증하려고 했고, 그 과정에서 같은 relation의 policy가 다시 평가되면서 재귀가 발생했다.
- 즉, 앱 코드는 정상이어도 DB 정책이 자기 자신을 다시 읽는 구조라면 plain 댓글과 답글 모두 insert 단계에서 막힐 수 있다.
- 해결은 policy 안의 self-reference 검증을 직접 subquery로 두지 않고, `security definer` helper function으로 분리하는 것이었다.
- 이번에 커뮤니티 SQL에는 아래 helper를 추가했다.
  - `public.can_insert_community_comment(target_post_id, target_parent_comment_id)`
- 최종 `comments_insert_own` policy는 이 helper만 호출하도록 바뀌었다.
- 이 방식의 장점은 다음과 같다.
  - policy evaluation recursion을 피할 수 있다.
  - plain 댓글과 reply 검증 로직을 한 함수에 모아 관리할 수 있다.
  - parent comment가 top-level인지, 같은 post인지, active/deleted 상태인지 한 곳에서 검증할 수 있다.
- 추가로 이번 디버깅에서는 Supabase 에러 객체가 plain object 형태로 넘어오는데, 앱 공용 에러 파서가 `Error instanceof`와 string만 읽고 있어 실제 DB 에러가 사용자에게 `다시 시도해 주세요.`로만 보이던 문제도 확인했다.
- 따라서 현재 실무 기준은 아래와 같다.
  - Supabase/PostgREST 에러는 `code/message/details/hint`를 읽을 수 있게 로깅/파싱한다.
  - RLS policy에서 같은 테이블을 직접 다시 조회해야 하면 재귀 가능성을 먼저 의심한다.
- self-reference 검증은 가능하면 `security definer` helper function으로 분리한다.
- 댓글/답글처럼 insert 대상 테이블이 자기 자신을 참조하는 구조는 policy recursion 테스트를 반드시 포함한다.

## 추가 메모: 2026-03-23 커뮤니티 댓글 성능 리팩터링 1차 정리

- 커뮤니티 상세 댓글은 기능적으로는 동작했지만, `CommunityDetailScreen`이 top-level 댓글 렌더, reply 렌더, 더보기 상태, composer 상태를 한 파일 안에서 동시에 처리하고 있어 리렌더링 영향 범위가 넓었다.
- 이번 1차 리팩터링에서는 기능이나 UI를 바꾸지 않고, 렌더 구조와 상태 구독 범위를 줄이는 데만 집중했다.
- 우선 `CommentThreadItem`, `ReplyCommentItem`, `CommentActionRow`를 분리해 top-level thread와 reply row를 화면 밖 컴포넌트로 이동했다.
- `CommunityDetailScreen`은 이제 thread id 목록, footer, composer, modal 같은 orchestration만 담당하고, 실제 댓글 row 렌더는 memoized item 컴포넌트가 담당한다.
- 댓글 파생 계산도 화면 본문에서 직접 하지 않고 `commentHelpers.ts`로 옮겼다.
  - `groupCommentsIntoThreads(comments)`
  - `getVisibleReplies(replyIds, expanded, previewCount)`
- store는 기존 `commentsByPostId` 배열을 유지하되, 렌더 최적화를 위해 아래 캐시를 추가로 가진다.
  - `commentEntitiesById`
  - `topLevelCommentIdsByPostId`
  - `replyCommentIdsByParentId`
- 이 구조의 목적은 screen이 더 이상 댓글 배열 전체를 직접 렌더링 책임으로 들고 있지 않게 만드는 것이다.
- top-level thread item은 자기 comment entity와 자기 reply id 목록만 narrow selector로 읽고, reply item도 자기 entity 1개만 읽는다.
- 좋아요/삭제/추가도 가능한 한 전체 regroup 대신 entity/id list patch로 반영하도록 맞췄다.
  - 댓글 좋아요: 해당 comment entity만 optimistic update
  - 댓글 추가: top-level id list 또는 parent reply id list만 갱신
  - 댓글 삭제: top-level thread 또는 해당 parent reply list만 갱신
- `fetchPostComments`도 이번에 보수적으로 정리했다.
  - 같은 post의 이전 comment entity와 이전 parent reply id key를 먼저 정리한 뒤
  - 최신 fetch 결과로 다시 채우도록 바꿨다.
  - 목적은 stale reply id/key가 남아 다음 렌더에 영향을 주는 것을 막는 것이다.
- 아직 완전한 정규화 store까지 밀어붙이지는 않았다.
- 현재 판단으로는 이번 1차 구조만으로도 아래 효과를 얻을 수 있다.
  - comment draft 변경 시 list item 전체 재렌더 감소
  - reply 더보기 시 해당 thread만 영향
  - 댓글 좋아요/삭제/추가 시 전체 화면 재가공 범위 축소
- 남은 리스크는 아래다.
  - `Best` 배지는 여전히 좋아요 5개 이상 실데이터 검증이 부족하다.
  - Android 실기기 키보드 바/IME toolbar 검증은 아직 별도 확인이 필요하다.
  - 댓글 100개+ / reply 다량 데이터에서의 실제 프레임 드랍은 실기기 체감 QA가 더 필요하다.

## 추가 메모: 2026-03-23 커뮤니티 리스트 성능 리팩터링 1차 계획

- 커뮤니티 리스트는 현재 화면이 `posts` 배열과 댓글 preview 상태를 함께 소비하는 구조라, post 1개 좋아요/수정/삭제가 list 전체 렌더 영향으로 번질 여지가 있다.
- 이번 1차 목표는 store 전면 교체가 아니라 소비 방식을 줄이는 것이다.
- 최종 구조는 아래를 기준으로 잡는다.
  - `CommunityListScreen`: post id list, category/filter, refresh/loadMore, header/footer orchestration만 담당
  - `CommunityPostListItem`: `postId`만 props로 받고, `postsById[postId]`와 해당 `postId`의 latest comment preview만 narrow selector로 읽음
  - `PostCard`: store를 모르는 순수 props 컴포넌트 유지
- 핵심은 screen이 post 객체 배열과 preview 가공을 직접 renderItem에 넘기지 않는 것이다.
- 리스트 data는 `posts.map(post => post.id)` 기준의 `postIds`로 바꾸고, `renderItem`은 `<CommunityPostListItem postId={postId} />`만 반환한다.
- 댓글 preview는 screen에서 전체 `commentsByPostId`를 직접 조합하지 않고, item에서 자기 `postId` 범위만 읽도록 제한한다.
- header/footer는 memoized component 수준으로 유지해 category 전환/refresh/loadMore 시 item 렌더와 같이 흔들리지 않도록 한다.
- FlatList 옵션 튜닝은 구조 분리 후 마지막에만 보수적으로 검토한다.
- 완료 기준은 아래다.
  - post 1개 좋아요 시 해당 card만 반응
  - 댓글 preview 갱신 시 해당 post card만 갱신
  - category 변경/refresh/loadMore 시 header/footer/item 영향 범위가 분리
  - 30개/100개 수준 스크롤에서 기존 대비 체감 흔들림 감소

  - 댓글 상세의 댓글 좋아요
  - 리스트의 게시글 좋아요
- 이 방식은 완전한 서버 단 debounce는 아니지만, 현재 구조를 크게 흔들지 않고도 optimistic UI를 유지하면서 API 중복 호출을 줄이는 1차 방어선으로는 충분하다.

- 기존 레거시 게시글은 title이 없을 수 있으므로, 앱은 `title 미존재`를 허용하고 list/detail에서는 fallback으로 동작해야 한다.

## 1. 작업 배경

NURI는 React Native 기반의 반려동물 기억 및 메모리얼 플랫폼이다.  
다만 현재 실제 코드 기준으로 보면, 단순한 추모 전용 앱은 아니다. 반려동물별 기록, 일정, 홈 회상 허브, 날씨 및 위치 기반 행동 연결, 장소 탐색, 여행 탐색, 가이드 소비, 최소 관리자 운영 기능까지 함께 포함한 생활형 서비스 구조에 가깝다.

현재 개발 방향은 신규 기능을 무작정 넓히는 단계보다, 이미 열린 런타임 도메인을 실제 서비스 수준으로 다듬는 단계에 더 가깝다.  
즉, 현재 병목은 기능 수 부족보다는 다음 항목에 있다.

- 정합성
- 신뢰도 레이어
- 실기기 QA
- 플랫폼 완성도
- 운영 준비도

현재 코드 기준 핵심 도메인은 아래와 같다.

- 앱 부트 및 런타임 조립
- Auth 및 온보딩
- 펫 생성, 펫 프로필 수정, 멀티펫 컨텍스트
- 기록 및 타임라인

## 4. 주요 리스크

### 4.1 타입 / 정합성 리스크

- 기록 도메인은 `memory_images`와 legacy 이미지 필드를 동시에 지원한다.
- 커뮤니티는 현재 `image_url`에서 `image_urls`로 이동 중이다.
- 여러 도메인이 호환성 계층을 유지하고 있어 단기 런타임 안정성은 높지만, 장기적으로는 schema/model drift 리스크를 키운다.

### 4.2 Placeholder 노출 리스크

- Auth는 아직 소셜 로그인, 비밀번호 재설정, 정책 상세 진입을 placeholder로 노출한다.
- More drawer는 건강 기록 리포트와 알림 설정을 준비중 상태로 노출한다.
- Guestbook은 여전히 placeholder 화면이다.

### 4.3 플랫폼 리스크

- Android 일정 알림이 완전히 구현되지 않았다.
- iOS 위치 권한 처리가 서비스급 수준이 아니며 현재는 지나치게 낙관적으로 동작한다.
- Android 우선 개발 기준과 현재 플랫폼 완성도 사이에 간극이 있다.

### 4.4 외부 API 의존 리스크

- 날씨는 Open-Meteo에 의존한다.
- 산책 및 펫동반 장소 탐색은 Kakao Local에 의존한다.
- 반려동물과 여행은 TourAPI에 의존한다.
- 후보 품질과 가용성이 upstream 동작에 따라 흔들릴 수 있다.

### 4.5 Store 결합 리스크

- `AppProviders`가 여러 전역 store, boot logic, cleanup queue, session transition을 한 곳에서 조율한다.
- 현재는 동작하지만, 결합 비용이 이미 무시하기 어려운 수준이며 후속 변경 시 cross-domain 회귀 위험이 있다.

### 4.6 운영도구 부족

- 가이드 admin은 존재하지만, 전체 서비스 운영 도구는 부족하다.
- 장소 검수, 여행 trust moderation, 신고 큐 관리, cross-domain admin tooling이 여전히 비어 있다.

### 4.7 신뢰도 레이어 부족

- 펫동반 장소 trust schema는 존재하지만, 런타임 trust governance는 아직 불완전하다.
- 반려동물과 여행 Trust Layer schema는 존재하지만, 런타임은 아직 candidate-layer 중심이다.
- 따라서 앱은 유용한 결과를 보여줄 수는 있지만, 반려동물 정책 신뢰도를 강하게 보장할 수는 없다.

### 4.8 문서-코드 괴리

- 일부 오래된 커뮤니티 기획 문서는 아직 커뮤니티 미구현 전제를 갖고 있다.
- 현재 런타임 코드는 그 전제를 더 이상 따르지 않는다.
- 다음 작업자가 문서만 읽고 코드를 보지 않으면 잘못된 온보딩이 발생할 수 있다.

## 5. 검토한 선택지 / 판단

### 5.1 지금 무엇을 먼저 고도화해야 하는가

가장 적절한 근거리 우선순위는 아래다.

1. 현재 진행 중인 커뮤니티 정합성 작업을 먼저 마감한다.
2. 반려동물과 여행 Trust Layer 적용 순서와 앱 read 연결 계획을 확정한다.
3. 펫동반 장소 탐색과 반려동물과 여행에 대해 실기기 QA를 수행한다.

### 5.2 왜 이 순서가 맞는가

- 커뮤니티는 이미 현재 worktree에서 활발히 변경 중이므로, 중간 상태로 남겨두면 즉시 회귀 위험이 커진다.
- 반려동물과 여행과 펫동반 장소 탐색은 이미 사용자에게 열려 있지만, 신뢰도와 운영 기반은 약하다.
- 이 두 도메인은 기능 폭보다 신뢰도와 런타임 정합성이 다음 병목이다.

### 5.3 왜 신규 기능 추가보다 정합성 / 신뢰도 / QA가 먼저인가

- 코드베이스는 이미 다도메인 구조다.
- 여러 도메인이 기능적으로는 열려 있지만, 호환성 계층, 외부 후보 데이터, 부분적인 플랫폼 지원에 의존한다.
- 이 상태에서 기능을 더 늘리면 사용자 가치보다 유지보수 비용이 더 빠르게 증가할 가능성이 높다.
- 서비스 기준으로 보면, 현재 가장 가치 있는 작업은 런타임 동작, 저장 데이터, 문서 우선순위, 사용자 신뢰도 사이의 불일치를 줄이는 것이다.

## 6.1 추가 메모: 커뮤니티 댓글 고도화 구현 단계

이번 댓글 고도화는 단순 UI 변경이 아니라, `댓글 UI 리디자인 + 답글 데이터 구조 추가 + 성능/운영 안정화`가 같이 들어가는 작업이다.  
현재 댓글 시스템은 flat comment 기준이므로, 참고 이미지 수준의 서비스형 댓글과 답글 구조를 만들려면 단계적으로 나눠서 진행해야 한다.

### 단계 1. 댓글 UI 리디자인

목표:

- 현재 상세 화면의 댓글 row를 서비스형 대화 리스트 구조로 재배치한다.
- 카드형 느낌을 줄이고, 프로필 / 닉네임 / 날짜 / 본문 / 액션라인 구조를 정리한다.
- 광고는 범위에서 제외한다.

포함 범위:

- `CommunityDetailScreen` 댓글 리스트 레이아웃 개편
- 댓글 composer 레이아웃 개편
- 댓글 count / empty / loading / submit 상태 정리
- 댓글 더보기 버튼 위치 및 영역 정리

아직 하지 않는 것:

- 답글 저장 구조
- 댓글 좋아요
- 베스트 댓글
- 운영 배지

완료 조건:

- 현재 flat comment 구조만으로도 첫 번째 참고 이미지 수준의 서비스형 댓글 화면이 나온다.
- 댓글이 0개, 적은 글, 많은 글에서 레이아웃이 깨지지 않는다.
- Android 기준 키보드와 입력창 가시성이 유지된다.
- Android 실기기에서 키보드가 올라올 때 하단 키보드 바 / IME 영역과 composer가 겹치지 않는다.
- 키보드가 열린 상태에서도 댓글 더보기, 입력창 포커스, 제출 버튼 터치가 막히지 않는다.

### 단계 2. 댓글 데이터 모델 확장

목표:

- 답글이 가능한 1-depth 댓글 구조를 도입한다.
- 원댓글과 답글을 구분할 수 있는 source of truth를 SQL / 타입 / 서비스 / store에 만든다.

핵심 정책:

- 원댓글에는 답글을 여러 개 달 수 있다.
- 답글에도 `답글쓰기`는 가능하게 둘 수 있지만, 저장 구조는 항상 같은 원댓글 기준 `depth 1`로 평탄화한다.
- 즉, `답글의 답글`이 무한 중첩되는 구조는 허용하지 않는다.

권장 SQL 확장:

- `comments.parent_comment_id uuid null`
- `comments.depth smallint not null default 0`
- `comments.reply_count integer not null default 0`
- `comments.like_count integer not null default 0`
- `comment_likes` 테이블 추가

권장 제약:

- `depth in (0, 1)`
- `depth = 0 -> parent_comment_id is null`
- `depth = 1 -> parent_comment_id is not null`
- reply는 반드시 같은 `post_id`의 parent를 참조해야 한다

권장 인덱스:

- `(post_id, parent_comment_id, created_at asc, id asc)`
- `(parent_comment_id, created_at asc, id asc)`
- `(post_id, like_count desc, created_at asc)`
- `(user_id, created_at desc)`

완료 조건:

- 댓글과 답글을 SQL/타입/서비스/store가 모두 같은 기준으로 이해한다.
- 기존 flat comment 글도 마이그레이션 없이 정상 읽기 가능하다.

### 단계 3. 답글 UI 및 입력 흐름 연결

목표:

- 두 번째 참고 이미지 수준의 `답글쓰기`, `답글 더보기`, `parent/reply 구분` UX를 구현한다.

핵심 UX:

- top-level 댓글은 처음 10개 노출
- `댓글 더보기`는 10개 단위
- 각 댓글의 답글은 처음 최신 2개만 노출
- `답글 n개 더보기`로 확장
- composer는 기본 댓글 작성 / 답글 작성 모드를 함께 지원
- reply 작성 중에는 `OO님에게 답글 남기는 중` 상태 표시와 취소 버튼 제공

완료 조건:

- top-level / reply 구분이 시각적으로 명확하다.
- 답글 작성, 답글 취소, 답글 더보기가 모두 자연스럽게 동작한다.
- reply를 눌러도 depth 2 이상으로 저장되지 않는다.

### 단계 4. 댓글 좋아요 및 베스트 댓글 정책 적용

목표:

- 댓글 좋아요를 별도 상호작용으로 추가하고, 베스트 댓글 판단 기준을 적용한다.

초기 정책:

- 베스트 댓글은 `top-level comment`만 대상
- `comment_like_count >= 5`
- `status = active`

이유:

- 절대값이 너무 낮으면 베스트가 남발된다.
- reply까지 베스트 대상으로 넣으면 리스트가 복잡해진다.
- 초기 운영 기준으로는 `원댓글 중심 베스트`가 가장 단순하고 설명 가능하다.

완료 조건:

- 댓글 좋아요 optimistic update / rollback이 안전하게 동작한다.
- 베스트 뱃지가 과도하게 남발되지 않는다.

### 단계 5. 성능 최적화 및 리렌더링 안정화

목표:

- 댓글 수가 많아도 상세 화면 스크롤과 입력 반응이 버벅이지 않게 만든다.
- 댓글 1개 좋아요/삭제/업데이트 시 전체 화면이 흔들리지 않게 만든다.
- composer draft 상태와 댓글 list 상태의 영향 범위를 분리한다.

핵심 전략:

- `CommentThreadItem`, `ReplyCommentItem`로 댓글 렌더를 분리하고 `React.memo`를 기본으로 둔다.
- top-level/reply grouping, visible reply 계산은 pure helper 함수로 분리한다.
- screen은 orchestration만 담당하고, item은 id 또는 최소 props만 받도록 좁힌다.
- 필요 시 flat entity cache + id list 기반 normalized store 구조로 확장한다.

권장 상태 구조:

- `commentEntitiesById`
- `topLevelCommentIdsByPostId`
- `replyCommentIdsByParentId`
- `commentsStatusByPostId`
- `replyExpandedByCommentId`

렌더링 규칙:

- `renderItem` 안에서 reply grouping 계산 금지
- item 내부 store 직접 넓게 구독 금지
- selector 안에서 새 배열 / 새 객체 반환 금지
- item 컴포넌트는 id 또는 최소 primitive props 기반 memo 유지
- 좋아요 / 답글 확장 / 신고 / 삭제는 row 단위로만 갱신되게 설계
- draft 입력으로 screen이 다시 렌더돼도 comment row는 다시 그려지지 않게 유지

구현 순서:

1. `CommentThreadItem` / `ReplyCommentItem` 분리
2. `commentHelpers.ts`로 top-level/reply 파생 계산 분리
3. `CommunityDetailScreen`은 list orchestration만 남기기
4. `communityStore`는 댓글 entity/id list 구조를 도입할지 검토하고, 최소한 업데이트 범위를 row/thread 단위로 제한
5. 마지막에만 FlatList 옵션(`windowSize`, `maxToRenderPerBatch` 등) 조정 검토

완료 조건:

- 댓글이 100개 이상이어도 첫 진입과 스크롤이 급격히 무거워지지 않는다.
- 특정 댓글 상호작용이 전체 댓글 리스트 리렌더로 번지지 않는다.

정적 검증:

로직 검증:

실기기 QA:

완료 조건:

  - 1단계: 댓글 상세 UI 리디자인 진행 중
  - 2단계: reply/comment_like 데이터 계약 및 SQL 설계 확정

    - 답글 저장

    - `답글 n개 더보기`
    - top-level 댓글 삭제 시 reply 동반 제거
  - 현재 QA 미확인:
    - `좋아요 5개 이상` Best 댓글 배지 노출

    - top-level 댓글 첫 노출 10개
    - `댓글 더보기`는 10개 단위
    - reply는 최신 2개 preview 후 `답글 n개 더보기`
    - 베스트 댓글 기준은 `top-level + 좋아요 5개 이상`
    - 답글쓰기 UI는 원댓글/답글 어디서 눌러도 가능하지만 저장은 항상 같은 top-level parent 기준 `depth 1`로 평탄화
    - Android 실기기에서는 reply 모드 composer가 키보드 바/IME 영역과 겹치지 않는지 별도 검증 필수

이번 댓글 고도화는 총 6단계로 나눠서 진행하는 것이 가장 안전하다.

1. 댓글 UI 리디자인
2. 댓글 데이터 모델 확장
3. 답글 UI 및 입력 흐름 연결
4. 댓글 좋아요 및 베스트 댓글 정책 적용
5. 성능 최적화 및 리렌더링 안정화
6. 테스트 및 실기기 QA

즉, 바로 UI부터 덮는 방식이 아니라 `UI -> 데이터 구조 -> 답글 흐름 -> 상호작용 -> 성능 -> QA` 순서로 가는 것이 현재 NURI 코드베이스 기준 최적의 실무안이다.

## 8. 하단 탭에서 기록하기 제거 + 커뮤니티 탭 전환

### 목표

- 하단 탭에서 `RecordCreateTab`을 제거하고 `CommunityTab`을 넣는다.
- 기록하기 화면 자체는 탭 안이 아니라 `RootNavigator`의 독립 스크린 `RecordCreate`로 유지한다.
- 기존 `returnTo` 복귀 흐름은 유지한다.

### 구현 결정

- `AppTabsNavigator`의 탭 순서는 `HomeTab / TimelineTab / CommunityTab / GuestbookTab / MoreTab`로 고정한다.
- `RecordCreateTab`은 탭 파라미터 타입과 `Tab.Screen`에서 완전히 제거한다.
- 기록 작성 복귀 규칙은 탭 파라미터가 아니라 `RecordCreateReturnTo` 공통 타입으로 분리해 재사용한다.
- `RecordCreateScreen`은 `RootNavigator.RecordCreate`를 기준으로 route/navigation 타입을 다시 잡는다.
- 기록 저장 완료 후 타임라인 상세 이동은 탭 내부 direct navigate가 아니라 `AppTabs -> TimelineTab -> RecordDetail` nested navigate로 맞춘다.

### 구현 범위

- `src/navigation/AppTabsNavigator.tsx`
- `src/navigation/RootNavigator.tsx`
- `src/components/navigation/AppNavigationToolbar.tsx`
- `src/screens/Records/RecordCreateScreen.tsx`
- `src/screens/Main/components/LoggedInHome/LoggedInHome.tsx`
- `src/screens/Records/TimelineScreen.tsx`

### 주의점

- `RecordCreateTab` 제거 후에도 홈/타임라인의 기록하기 CTA는 전부 `navigation.navigate('RecordCreate', ...)`로 바뀌어야 한다.
- `RecordCreateScreen`은 이제 탭 스크린이 아니므로 `BottomTabNavigationProp`에 의존하면 안 된다.
- 커뮤니티 탭 추가는 UI 리디자인이 아니라 탭 구조 교체 작업이므로 기존 툴바 색상/레이아웃은 유지해야 한다.

## 9. 커뮤니티 탭 헤더 누락 수정

### 문제

- 하단 탭의 `CommunityTab`은 `CommunityListScreen`을 직접 렌더하고 있었고, 이 경우 루트 스택에서 주입하던 `CommunityStackHeader`를 받지 못했다.
- 결과적으로 탭에서 커뮤니티로 들어가면 상단 헤더가 사라지고, 루트 스택 `CommunityList`로 열었을 때만 헤더가 보이는 구조가 됐다.

### 구현 결정

- 탭 안에서도 커뮤니티는 단일 화면 직접 렌더가 아니라 얇은 스택으로 감싼다.
- `CommunityTabStackNavigator`를 추가하고, 탭의 `CommunityTab`은 이 스택을 렌더한다.
- 이 스택의 첫 화면 `CommunityTabList`는 `CommunityListScreen`을 사용하고, 헤더는 기존 `CommunityStackHeader`를 그대로 재사용한다.

### 이유

- 커뮤니티는 목록/상세/작성/수정 모두 공통 헤더 정책을 이미 갖고 있고, 탭 진입만 예외 처리하면 화면 간 헤더 일관성이 깨진다.
- 탭 안에서도 같은 헤더 스택 규칙을 쓰는 편이 이후 커뮤니티 내부 확장에도 안전하다.

## 10. 추억 상세 뒤로가기 콜백 복구

### 문제

- `RecordDetailScreen` 헤더는 단순 `navigation.goBack()`만 사용하고 있었고, 홈/More에서 진입한 흐름의 복귀 규칙을 따르지 못했다.
- 그 결과 원래 있던 진입 소스별 복귀 동작이 깨졌다.

### 구현 결정

- `TimelineStackParamList.RecordDetail`에 `entrySource?: ScreenEntrySource`를 추가한다.
- 홈/타임라인/기록 저장 직후 상세 진입 시 `entrySource`를 전달한다.
- `RecordDetailScreen`은 `useEntryAwareBackAction`을 사용해:
  - `home`이면 `AppTabs -> HomeTab`
  - `more`이면 `AppTabs -> HomeTab` 이동 후 More drawer 재오픈
  - 그 외는 `goBack()`, 불가하면 `TimelineMain`
  로 처리한다.

### 이유

- `RecordDetail`은 타임라인 내부 스택 화면이지만 실제 진입 경로는 홈/More/작성 완료 후 상세 등 다양하다.
- 따라서 뒤로가기는 스택 히스토리만 신뢰하면 안 되고, 진입 소스 정보를 함께 봐야 한다.

## 11. 소비 화면 화이트 배경 통일

### 목표

- 기록/일정/가이드/탐색 계열 소비 화면의 루트 배경을 `#FFFFFF`로 통일한다.
- 화면 간 배경 톤 차이로 인한 시각적 분절을 줄인다.

### 이번에 반영한 화면

- `RecordDetailScreen`
- `RecordCreateScreen`
- `ScheduleDetailScreen`
- `ScheduleCreateScreen`
- `ScheduleListScreen`
- `GuideListScreen`
- `GuideDetailScreen`
- `IndoorActivityRecommendationsScreen`
- `ActivityGuideScreen`
- `LocationDiscoveryListScreen` / `PetFriendlyPlaceListScreen` 공용 `LocationDiscovery.styles.ts`
- `PetTravelListScreen`
- `GuestbookScreen`은 이미 `#FFFFFF`여서 유지

### 추가 톤 정리

- `RecordDetail` 카드의 상/하단 외곽 보더 제거
- `Timeline` 리스트 게시물 카드 외곽 보더 제거
- `ScheduleCreate` 카드 보더 제거
- `RecordDetail` 헤더는 타임라인과 같은 `좌측 슬롯 + 중앙 타이틀 + 우측 슬롯` 구조로 맞춤
