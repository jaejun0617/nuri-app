# 레거시 이미지 백필 운영 전략

## 목적

이 문서는 기존 `public.memories.image_url / image_urls` 기반 데이터를 `public.memory_images` 구조로 안전하게 이관하기 위한 실제 운영 전략을 정리한다.

이번 백필의 목표는 아래 3가지다.

1. 기존 레거시 이미지 path를 `memory_images.original_path` 로 옮긴다.
2. 새로 들어간 row는 `status='pending'`, `timeline_thumb_path=null` 로 넣어 worker가 썸네일을 채우게 한다.
3. 앱이 이미 쓰고 있는 read/write 전환 구조와 충돌 없이, 점진적으로 legacy fallback 의존도를 낮춘다.

## 현재 전제

- `memory_images` 테이블, 인덱스, RLS, 대표 이미지 뷰가 준비되어 있다.
- worker 메타 컬럼이 준비되어 있다.
- 썸네일 worker는 `pending -> ready/failed` 처리 검증이 끝났다.
- 앱은 이미:
  - 타임라인에서 `timeline_thumb_path` 우선
  - 상세에서 `original_path` 우선
  - create/edit 에서 `memory_images` write
  구조로 동작한다.
- 다만 기존 데이터는 아직 `memory_images` row가 없을 수 있으므로 `memories.image_url / image_urls` fallback 이 남아 있다.

## 백필 대상

백필 대상은 아래 조건을 모두 만족하는 `memories` row다.

- `memory_images` row가 아직 1건도 없는 memory
- `image_urls` 또는 `image_url` 에 실제 storage path가 있는 memory

즉, 이미 신규 create/edit 로 `memory_images`가 생성된 데이터는 다시 건드리지 않는다.

## 배치 단위

권장 배치 크기:

- 초기 검증: 50 ~ 100 memory
- 일반 운영: 200 ~ 500 memory
- 야간/저부하 시간대: 500 ~ 1000 memory

중요:

- 이 배치는 "memory 수" 기준이다.
- 실제 insert row 수는 이미지 개수만큼 늘어난다.
- 다중 이미지가 많은 계정은 같은 batch size여도 row 수가 크게 늘 수 있다.

## 실행 순서

1. `timeline_thumb_path null` 유니크 제약 수정 SQL 실행
2. 백필 전 현황 확인
3. 소량 batch SQL 실행
4. `memory_images` row 생성 확인
5. worker batch 실행
6. `pending/ready/failed` 분포 확인
7. 실패 건 재시도 또는 원본 path 점검
8. 다음 batch 실행
9. 전체 대상이 0이 될 때까지 반복

## 선행 수정이 필요한 이유

초기 migration 에서 `timeline_thumb_path` 에 `unique nulls not distinct` 가 걸려 있으면
`timeline_thumb_path = null` row 가 여러 건 들어가지 못한다.

하지만 worker 설계상 신규/백필 row 는 처음엔 모두 아래 상태로 들어가야 한다.

- `timeline_thumb_path = null`
- `status = 'pending'`

따라서 운영 전에는 반드시 아래 원칙으로 제약을 수정해야 한다.

- null 은 여러 건 허용
- 실제 `timeline_thumb_path` 값이 있는 경우만 unique

## 중복 실행 방지 기준

중복 방지 기준은 `memory_images` 가 없는 `memory_id` 만 집는 것이다.

즉 SQL은 아래 원칙으로 실행한다.

- `not exists (select 1 from memory_images where memory_id = m.id)`
- 같은 memory가 이미 한번이라도 `memory_images`에 들어가 있으면 다시 집지 않음

이 기준이면 SQL을 여러 번 실행해도 이미 처리된 memory는 다시 insert 대상이 되지 않는다.

## 재실행 기준

다음 경우는 같은 SQL을 다시 실행해도 안전하다.

- batch가 중간에 끝났지만 일부만 insert 된 경우
- 일부 memory만 완료되고 나머지가 남은 경우
- worker가 아직 `pending` 을 처리 중인 경우

다만 아래 경우는 별도 점검이 필요하다.

- 한 memory에 `memory_images` row가 일부만 들어간 비정상 상태
- `sort_order` 가 중간에 끊긴 상태
- `original_path` 중복 또는 잘못된 path가 섞인 상태

이 경우는 일반 백필 SQL이 아니라 repair SQL 또는 수동 보정으로 처리하는 게 안전하다.

## 실패 시 복구 전략

### 1. 백필 insert 실패

- SQL Editor 에러 메시지 확인
- 주로 보는 원인:
  - enum/type 미생성
  - unique 충돌
  - `image_urls` 컬럼 유무 차이
  - RLS/권한이 아닌 SQL 실행 문맥 문제

복구 방식:

- 문제 원인을 먼저 정리
- 같은 batch SQL 재실행

### 2. worker 처리 실패

실패 유형:

- `original_not_found`
- `storage_download_failed`
- `storage_upload_failed`
- 기타 transient 실패

복구 방식:

- `last_error_code / last_error_message` 확인
- 원본 path 존재 여부 확인
- 필요한 경우 failed row를 retry 대상으로 다시 풀거나 targeted 실행

## worker 연계 방식

백필 SQL은 `timeline_thumb_path` 를 바로 채우지 않는다.

항상 아래처럼 넣는다.

- `original_path = legacy path`
- `timeline_thumb_path = null`
- `status = 'pending'`

그 다음 worker가 아래를 처리한다.

- 썸네일 생성
- 썸네일 업로드
- `timeline_thumb_path` 저장
- `status='ready'` 전환

즉 백필과 썸네일 생성은 분리된 두 단계다.

## legacy fallback 과도기 운영 기준

현재는 아래 fallback 을 유지한다.

- 타임라인:
  - `timeline_thumb_path` 우선
  - 없으면 `original_path` transform
  - 그것도 없으면 legacy `image_url/image_urls`
- 상세:
  - `memory_images.original_path` 우선
  - 없으면 legacy `image_url/image_urls`

이 fallback 은 아래 조건이 충족될 때 줄일 수 있다.

1. 기존 memory 중 `memory_images` 없는 건이 사실상 0건
2. worker pending backlog 가 안정적으로 관리됨
3. failed row가 운영 허용 범위 이하
4. 신규 create/edit 에서 `memory_images` write가 안정적
5. 타임라인/상세 실기기 검증 완료

## fallback 제거 전 체크리스트

- `memory_images` 없는 memory 개수 확인
- `status='pending'` backlog 확인
- `status='failed'` backlog 확인
- 대표 썸네일 없는 memory 비율 확인
- 타임라인 첫 진입 / 빠른 스크롤 / 상세 진입 검증
- create/edit/delete 회귀 확인
- 로컬 URI 즉시 반영 확인
- 다중 이미지 sort_order 확인

이 체크가 끝나기 전에는 `memories.image_url / image_urls` read fallback 을 제거하지 않는다.

## legacy 축소 순서

1. 기존 데이터 백필 완료
2. worker로 썸네일 생성 안정화
3. pending/failed backlog 안정화
4. 운영 지표 확인
5. 앱에서 legacy read fallback 사용량 점검
6. fallback 제거 후보 릴리즈 준비
7. staging/실기기 회귀 확인
8. 그 다음에만 legacy read fallback 제거

## 이번 단계가 끝나면 기대 상태

- 신규/수정 데이터는 이미 `memory_images`를 직접 사용
- 기존 데이터도 점진적으로 `memory_images`로 수렴
- 타임라인은 원본 이미지 계약 의존도를 계속 낮춤
- 상세는 `original_path` 기준 구조를 일관되게 유지
