# 마이그레이션 운영 기준 통합 완성본

## 1. 문서 목적

이 문서는 NURI 프로젝트에서 `docs/sql/마이그레이션/` 아래에 쌓이는 실제 배포 이력 SQL을 어떻게 보고, 어떻게 추가하고, 어떻게 운영 기준으로 관리할지 정리한 기준 문서다.

핵심은 단순하다.

- `docs/sql/마이그레이션/`은 실제 배포 순서와 이력을 남기는 폴더다.
- 날짜 prefix가 붙은 SQL은 운영 반영 순서와 추적 기준이 된다.
- 도메인 최종본 SQL이나 수동 실행용 릴리즈 SQL은 여기 두지 않는다.

## 2. 현재 역할

현재 `docs/sql/마이그레이션/`은 아래 역할을 가진다.

1. 운영 DB에 반영한 변경 이력 누적
2. 배포 단위 변경 추적
3. 롤포워드 기준 관리
4. 특정 기능이 언제 어떤 SQL로 들어갔는지 확인하는 기준

즉, 이 폴더는 "현재 완성 상태를 설명하는 문서"가 아니라 "실제 변경이 어떤 순서로 들어갔는지 보여주는 운영 이력"이다.

## 3. 현재 포함된 마이그레이션

현재 기준 파일:

- `2026-03-05_메모리-이미지-URL-컬럼-추가.sql`
- `2026-03-06_신규-유저-안전-닉네임-처리.sql`
- `2026-03-06_프로필-닉네임-확정-컬럼-추가.sql`
- `2026-03-06_프로필-닉네임-정책-RPC.sql`
- `2026-03-16_집사-꿀팁-가이드-검색-RPC.sql`

## 4. 현재 구조 해석 기준

### 4.1 타임라인 / 메모리 이미지

- `2026-03-05_메모리-이미지-URL-컬럼-추가.sql`
  - `memories`의 이미지 URL 구조 확장 이력
  - 타임라인 `memory_images` 전환 이전 단계와 연결해서 봐야 한다

### 4.2 프로필 / 닉네임 / 계정 정책

- `2026-03-06_신규-유저-안전-닉네임-처리.sql`
- `2026-03-06_프로필-닉네임-확정-컬럼-추가.sql`
- `2026-03-06_프로필-닉네임-정책-RPC.sql`

위 3개는 `profiles` 안전 닉네임 정책과 RPC, 신규 사용자 처리 흐름을 누적 반영한 이력이다.

### 4.3 집사 꿀팁 가이드 검색

- `2026-03-16_집사-꿀팁-가이드-검색-RPC.sql`
  - 가이드 검색 RPC
  - FTS 보조 구조
  - 인기 검색어 RPC

이 파일은 집사 꿀팁 가이드의 현재 검색 고도화가 운영 DB에 어떻게 반영되는지 보여주는 배포 이력이다.

## 5. 마이그레이션과 도메인 최종본의 차이

헷갈리면 아래 기준으로 구분한다.

- `docs/sql/도메인/`
  - 현재 상태를 한 번에 보는 canonical SQL
- `docs/sql/마이그레이션/`
  - 실제 변경이 누적된 배포 이력

예를 들어 집사 꿀팁 가이드의 구조를 한 번에 보려면 아래를 먼저 본다.

- `docs/sql/도메인/집사-꿀팁-가이드/집사-꿀팁-가이드-운영-스키마-최종.sql`

반대로 검색 RPC가 어떤 배포 단위로 추가됐는지 보려면 아래를 본다.

- `docs/sql/마이그레이션/2026-03-16_집사-꿀팁-가이드-검색-RPC.sql`

## 6. 추가 원칙

새 SQL이 생기면 먼저 아래 셋 중 하나를 구분한다.

1. 현재 상태 최종본인가
2. 실제 배포 이력 migration인가
3. 수동 실행용 릴리즈 SQL인가

마이그레이션에 해당하면 아래 규칙을 따른다.

- 파일명은 날짜 prefix를 유지한다.
- 한 배포 단위 변경을 독립적으로 이해할 수 있어야 한다.
- 이미 적용된 과거 migration 파일은 수정하지 않고, 후속 migration으로 누적한다.
- 도메인 최종본 설명을 대신하려고 장황한 설계를 이 파일에 넣지 않는다.

## 7. 운영 관점에서 볼 때 중요한 것

- 파일 수가 많아져도 "삭제해서 깔끔하게"가 아니라 "순서를 유지하면서 설명 문서로 보조"하는 방식이 맞다.
- 실제 DB 이력은 누적 자산이므로, 도메인 문서처럼 통합 대체해서 없애지 않는다.
- 대신 기준 문서에서 역할과 읽는 순서를 명확히 정리한다.
