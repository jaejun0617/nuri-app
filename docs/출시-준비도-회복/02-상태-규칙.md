# 출시 준비도 회복 상태 규칙

상태: 완료
최종 갱신: 2026-03-30

## 배경

- 문서가 많아질수록 상태 표현이 흔들리면 source of truth 역할을 잃는다.

## 현재 문제

- 완료와 검증 필요, 외부 blocker와 내부 blocker가 같은 표현으로 섞이면 판단 오류가 생긴다.

## 목표

- 상태 용어를 고정하고, 각 액션 문서가 같은 규칙으로 업데이트되게 한다.

## 작업 목록

- 상단 상태 표기 규칙 고정
- 완료 기록 입력 규칙 고정
- 외부 의존 blocker 기록 규칙 고정

## 완료 기준

- 모든 액션 문서 상단에 아래 상태 중 하나가 들어간다.
- 완료 기록 문서는 같은 템플릿을 재사용한다.

## 리스크

- 다른 문서에서 다른 상태 용어를 쓰면 다시 혼선이 생긴다.

## 다음 액션

- 액션 문서를 갱신할 때 상태와 마지막 갱신일을 항상 함께 수정한다.

## 상태

- 완료

## 허용 상태

- `대기`
  - 아직 실행에 들어가지 않았고 선행 분석 또는 외부 조건이 남아 있는 상태
- `진행 중`
  - 이번 턴 또는 현재 세션에서 실제 실행 중인 상태
- `검증 필요`
  - 코드/문서/설정 변경은 들어갔지만, 추가 QA 또는 운영 증적이 있어야 닫히는 상태
- `완료`
  - 이번 액션 범위 안에서 목표와 검증 기준이 모두 충족된 상태
- `보류`
  - 외부 조건 또는 승인 부재로 의도적으로 멈춘 상태
- `범위 밖`
  - 현재 회복 작업의 직접 범위를 벗어나 따로 관리해야 하는 상태

## 문서 상단 규칙

- 모든 액션 문서는 아래 3줄을 맨 위에 둔다.
  - `상태: ...`
  - `최종 갱신: YYYY-MM-DD`
  - `우선순위: P0/P1/P2`

## 완료 기록 규칙

- 하위 작업이 실제로 닫히면 `03-작업-완료-기록.md`에 즉시 기록한다.
- 완료 기록은 액션 전체가 아니라 하위 완료 단위도 남길 수 있다.

## blocker 기록 규칙

- 외부 의존으로 못 닫는 항목은 `왜 지금 못 닫는지`와 `닫는 방법`을 같은 문단에 함께 적는다.
- `미확인`은 실제로 확인하지 못한 경우에만 사용한다.
