## 0. 이 문서의 역할

이 문서는 새 사용자, 새 GPT, 새 Codex 세션이 NURI 프로젝트를 즉시 이어받기 위한 단일 온보딩 문서다.

이 문서 하나로 아래를 바로 파악할 수 있어야 한다.

- NURI가 어떤 서비스인지
- 앱이 어떤 구조로 동작하는지

- 어떤 SQL과 문서가 source of truth인지
- 지금 당장 이어서 해야 할 우선 작업이 무엇인지
- 현재 코드와 문서 사이에 어떤 괴리가 있는지

## 1. 프로젝트 정체성 / 서비스 개요

### 1.1 NURI가 무엇인가

NURI는 React Native 기반 반려동물 생활 관리 앱이다. 현재 코드 기준 핵심 성격은 다음 네 축이 동시에 존재하는 생활형 서비스다.

- 반려동물 단위 기록 관리
- 일정 및 기념일 관리
- 날씨 및 위치 기반 행동 연결
- 장소 탐색, 여행 탐색, 가이드 소비

루트 `README.md`는 정서적 아카이브와 추모 맥락을 강조하지만, 현재 실제 앱은 추모 전용 앱이 아니라 일상 기록, 케어, 탐색, 콘텐츠를 함께 다루는 구조다.

### 1.2 현재 모바일 앱 기준 서비스 목적

현재 모바일 앱이 실제로 수행하는 목적은 아래에 가깝다.

- 보호자가 반려동물별로 기록과 일정을 관리한다.
- 홈에서 최근 행동, 날씨, 일정, 가이드를 한 허브로 소비한다.
- 현재 위치 또는 검색 입력을 기준으로 산책/펫동반 장소를 탐색한다.
- 여행지 후보를 지역/카테고리/키워드 기반으로 찾는다.
- 가이드 콘텐츠를 소비하고, 관리자 권한이 있으면 최소 운영 편집까지 수행한다.

### 1.3 핵심 사용자 가치

- 반려동물별 컨텍스트가 분리된 기록/일정 관리
- 홈에서 최근 생활 데이터를 빠르게 다시 보는 허브 UX
- 날씨와 위치를 실제 행동 추천으로 연결하는 생활 보조
- 외부 데이터 기반 장소/여행 탐색을 앱 문맥에 맞게 정리
- 종/나이/상태를 반영한 가이드 추천과 검색

### 1.4 현재 주요 도메인

- Auth / 온보딩
- Splash / bootstrap
- Home / Main
- Pets
- Records / Timeline
- Schedules / 일정
- Weather
- 우리동네 산책
- 펫동반 장소 탐색
- 반려동물과 여행
- 집사 꿀팁 가이드
- Community
- More / 계정 / 설정
- 관리자 가이드 운영

주의:
- Guestbook은 탭/화면은 있으나 실제 기능 도메인으로 보기 어렵다.
- 멀티펫 구독은 문서만 있고 실제 코드/결제 흐름은 없다.

## 2. 현재 기술 아키텍처 개요

### 2.1 앱 구조

앱 런타임 중심 경로는 아래다.

1. `App.tsx`
2. `KeyboardProvider`
3. `SafeAreaProvider`
4. `AppProviders`
5. `NavigationContainer`
6. `RootNavigator`
7. `Splash` 역할의 `HomeScreen`

`App.tsx`는 Sentry/Crashlytics 초기화, 키보드 컨트롤러, SafeArea, 전역 Provider, 네비게이션 컨테이너, `GlobalToast`만 조립한다. 실제 부트 로직은 `AppProviders`가 담당한다.

### 2.2 navigation 구조

`RootNavigator`는 Native Stack 기반이며, 주요 라우트는 아래다.

- 공통: `Splash`, `AppTabs`
- Auth: `SignIn`, `SignUp`, `NicknameSetup`, `WelcomeTransition`
- Pets: `PetCreate`, `PetManagement`, `PetProfileEdit`, `PetProfileEditDone`
- Records/Timeline: `TimelineMain`, `RecordCreate`, `RecordDetail`, `RecordEdit`
- Schedules: `ScheduleList`, `ScheduleCreate`, `ScheduleDetail`, `ScheduleEdit`
- Weather: `WeatherInsight`, `IndoorActivityRecommendations`, `ActivityGuide`, `WeatherActivityRecord`
- Location Discovery: `WalkSpotList`, `WalkSpotDetail`, `PetFriendlyPlaceList`, `PetFriendlyPlaceDetail`
- PetTravel: `PetTravelList`, `PetTravelDetail`
- Guides: `GuideList`, `GuideDetail`, `GuideAdminList`, `GuideAdminEditor`
- Community: `CommunityList`, `CommunityDetail`, `CommunityCreate`, `CommunityEdit`
- 기타: `EditDone`, `DevTest`

`AppTabsNavigator`는 일반적인 4탭이 아니라 아래 구조다.

- `HomeTab`
- `TimelineTab`
- `RecordCreateTab`
- `GuestbookTab`

실제 하단 툴바는 `AppNavigationToolbar`가 담당하며, More는 탭 화면이 아니라 오버레이 drawer로 열린다. Guestbook도 툴바에 노출되지만 화면은 placeholder다.

### 2.3 store 구조

현재 전역 store는 아래다.

- `authStore`
  - 세션, 로그인 여부, 닉네임, role, profile sync 상태, boot 상태
- `petStore`
  - 펫 목록, 선택 펫, 로컬 캐시 hydrate, 펫 로딩/에러 상태
- `recordStore`
  - 펫별 기록 캐시, `recordsById`, 타임라인 ids, 페이지네이션, 낙관적 반영
- `scheduleStore`
  - 펫별 일정 캐시, 로딩/에러 상태
- `weatherStore`
  - 좌표별 날씨 번들 TTL 캐시
- `communityStore`
  - 커뮤니티 목록, 상세, 댓글, 좋아요, 신고, 캐시 동기화
- `uiStore`
  - 전역 toast, More drawer open 상태

## 3. 앱 첫 시작 플로우

실제 첫 실행 플로우는 `App.tsx`, `AppProviders.tsx`, `src/services/app/boot.ts`, `src/screens/Home/HomeScreen.tsx`, `src/screens/Main/MainScreen.tsx` 기준 아래 순서다.

1. 앱 실행
- `enableScreens(true)` 호출
- `initMonitoring()` 호출
- `KeyboardProvider`, `SafeAreaProvider`, `AppProviders`, `NavigationContainer`, `GlobalToast`가 올라간다.

2. Splash 진입
- `RootNavigator` 초기 라우트는 `Splash`
- `Splash` 화면은 실제로 `src/screens/Home/HomeScreen.tsx`

3. bootstrap 시작
- `AppProviders`가 `hydrateAuth()`, `hydrateSelectedPetId()`를 먼저 수행한다.
- `supabase.auth.getSession()`으로 세션을 읽고, 있으면 `getUser()`로 유효성을 다시 확인한다.
- auth listener를 등록해 `SIGNED_IN`, `SIGNED_OUT`, `INITIAL_SESSION` 등 이벤트를 처리한다.

4. session 확인
- 세션이 없으면 guest 상태를 적용한다.
- 세션이 있으면 `getSessionUserId()` 기준으로 user-scoped load를 시작한다.

5. nickname 확인
- `fetchMyProfile()`로 프로필을 읽는다.
- `profiles.nickname_confirmed`는 DB 구조상 존재하지만 앱의 실제 부트 분기 조건은 `authStore.profile.nickname`과 `profileSyncStatus === 'ready'` 조합이다.
- 닉네임이 비어 있으면 `NicknameSetup`으로 보낸다.

6. pet 존재 여부 확인
- `fetchMyPets()`를 호출한다.
- 펫이 없으면 `PetCreate`로 보낸다.
- 펫이 있으면 `petStore`에 저장하고 선택 펫을 복원한다.

7. 홈 또는 온보딩 진입
- 비로그인: `AppTabs`
- 로그인 + 닉네임 없음: `NicknameSetup`
- 로그인 + 닉네임 있음 + 펫 없음: `PetCreate`
- 로그인 + 닉네임 있음 + 펫 있음: `AppTabs`

8. 주요 가드
- `MainScreen`이 포커스될 때도 로그인 사용자의 닉네임/펫 상태를 다시 확인한다.
- `NicknameSetup` 누락 시 다시 `NicknameSetup`
- 펫이 0마리면 다시 `PetCreate`

9. 탭/드로어 흐름
- `AppTabs` 안에서 Home / Timeline / RecordCreate / Guestbook / More 구조를 쓴다.
- More는 실제 탭 이동이 아니라 `MoreDrawer` 오버레이다.
- 기록 작성은 현재 활성 탭에 따라 `returnTo`를 구성해 복귀한다.

10. 추가 deferred tasks
- 앱 활성화 시 `flushPendingConsentSnapshot()`
- `processPendingMemoryUploads()`
- `flushPendingCommunityImageCleanup()`

## 4. 도메인별 구현 현황

도메인별로 실제 구현, 부분 완료, 미완성, 부족한 점, 현재 작업 중, 다음 우선 작업을 분리한다.

### 4.1 Auth

실제 구현된 것:
- 이메일/비밀번호 로그인
- 이메일/비밀번호 회원가입
- 닉네임 설정
- 닉네임 중복 확인 RPC
- 로그아웃
- 계정 삭제 RPC 연동
- 비밀번호 변경
- 가입 동의 snapshot 저장과 후속 flush

부분 완료 / 구조만 있는 것:
- 이메일 인증이 필요한 경우 SignIn 복귀 흐름은 있음
- 닉네임 draft 복구는 있음
- role 기반 관리자 분기는 있음

미완성:
- 카카오 로그인: 준비 중 Alert
- Google 로그인: 준비 중 Alert
- 비밀번호 찾기: 준비 중 Alert
- 약관/개인정보 상세 문서 화면: 준비 중 Alert

서비스급 기준으로 부족한 점:
- 비밀번호 재설정 실동작 부재
- 소셜 로그인 부재
- 인증 메일 재전송/복구 UX 부재
- 다중 기기 닉네임 변경 정책과 운영 도구 부재

현재 작업 중인 것:
- 별도 인증 작업 흔적은 현재 턴 기준 보이지 않는다.

다음에 바로 이어야 할 것:
- 소셜 로그인보다 먼저, 약관/비밀번호 재설정/인증 실패 UX를 실제 서비스 수준으로 정리하는 것이 더 현실적이다.

### 4.2 Splash / bootstrap

실제 구현된 것:
- 세션 유효성 재검증
- 프로필/펫 hydrate
- user-scoped store clear와 계정 전환 처리
- timeout 기반 부트 보호
- Splash hold 시간 분기

부분 완료 / 구조만 있는 것:
- 캐시가 있을 때 stale 데이터로 먼저 보여주고 서버 fetch를 덮는 흐름
- 부트 실패 시 상태 메시지 저장

미완성:
- 장애 상황을 설명하는 별도 사용자용 재시도 화면 없음
- 부트 지연 원인별 세분화 UX 없음

서비스급 기준으로 부족한 점:
- 네트워크/부분 실패 시 안내가 store 상태 수준에 머문다.
- 부트 에러 telemetry는 있지만 사용자 복구 경로는 약하다.

현재 작업 중인 것:
- 관련 직접 작업 흔적은 없음.

다음에 바로 이어야 할 것:
- 부트 실패/부분 성공 상태를 실기기에서 검증하고, 필요하면 재시도/오프라인 안내를 별도로 만들 것.

### 4.3 Home / Main

실제 구현된 것:
- 게스트 홈 / 로그인 홈 분리

미완성:
- 자체 장소 메타 운영 데이터 적재/검수 UI 없음
- 사용자 북마크/제보/최근 본 장소 기능 없음

- 검증 상태 UI는 있으나 검수 운영 도구가 없다

현재 작업 중인 것:
- project-memory 기준으로 여전히 실기기 QA와 Supabase 장소 메타 SQL 리뷰/적용이 후속 병목이다.

다음에 바로 이어야 할 것:
- Supabase 장소 메타 SQL을 실제 운영 적용 순서로 정리하고, 이후 앱 북마크/검수 최소 기능을 연결해야 한다.

### 4.10 반려동물과 여행

실제 구현된 것:
- More 메뉴 및 라우트 분리
- 검색 입력 + 카테고리 선택 + 리스트 + 상세 + 외부 지도 열기
- TourAPI `KorPetTourService2` 기반 리스트/검색/상세
- 지역 intent resolver와 `areaBasedList2` narrow mode
- `placeType`, `petAllowed`, `travelScore`, `petScore`, `commercialPenalty` 계산

부분 완료 / 구조만 있는 것:
- trust layer 타입과 문서, SQL 초안/최종본이 존재
- 현재 런타임은 candidate-layer 수준

미완성:
- Trust Layer DB 미적용
- admin review 입력 경로 없음
- user_reports 연동 없음
- confirmed 상태 개방 안 됨
- 즐겨찾기/최근 검색/컬렉션 없음

서비스급 기준으로 부족한 점:
- TourAPI raw만으로는 정책 신뢰도를 책임질 수 없다
- 여행지 검색 precision은 규칙 기반 후처리에 많이 의존한다

현재 작업 중인 것:
- project-memory의 현재 최우선 작업이 바로 이 도메인의 Trust Layer SQL 리뷰 후 Supabase 적용 계획 정리다.

다음에 바로 이어야 할 것:
- `pet_travel_places / pet_travel_pet_policies / pet_travel_user_reports` 적용 계획, RLS 검토, 앱 read repository 설계

### 6.4 RLS 방향

공통 초기 구성:
- `profiles`, `pets`, `memories`, `pet_schedules`는 기본적으로 own CRUD
- `posts/comments/likes/reports`는 공개 읽기 + own write 방향으로 시작
- storage는 경로 첫 폴더를 `auth.uid()`와 맞추는 정책

타임라인 메모리 이미지:
- `memory_images`는 `auth.uid() = user_id and owns_pet(pet_id)` 기준 all policy

가이드:
- `pet_care_guides_public_read`
- `pet_care_guides_admin_all`
- `pet_care_guide_events_insert_own`
- `pet_care_guide_events_select_own`
- `pet_care_guide_events_admin_manage`

펫동반 장소 메타:
- 공개 read는 canonical meta/source links/external signals 중심
- bookmarks/user_reports/search_logs는 own 또는 admin 관리 방향
- verification logs는 admin 성격

반려동물과 여행 Trust Layer:
- `pet_travel_places_read_public`
- `pet_travel_places_admin_all`
- `pet_travel_pet_policies_read_public`
- `pet_travel_pet_policies_admin_all`
- `pet_travel_user_reports_*`는 own or admin

## 8. 현재 구현 수준 총평

### 8.1 구현 폭

구현 폭은 넓다. 인증, 펫, 기록, 일정, 날씨, 산책, 펫동반 탐색, 여행, 가이드, 커뮤니티, More, 관리자 가이드까지 코드가 존재한다.

### 8.2 실제 동작 수준

핵심 CRUD와 읽기 흐름은 다수 도메인에서 실제 동작 수준까지 올라와 있다. 특히 인증, 펫, 기록, 일정, 홈, 날씨, 가이드는 서비스 형태를 갖췄다.

### 8.3 서비스급 완성도

서비스급 완성도는 도메인별 편차가 크다.

- 비교적 높음:
  - Auth 기본 흐름
  - Pets
  - Records / Timeline
  - Home
  - Guides
- 중간:
  - Schedules
  - Weather
  - Community
- 낮음 또는 신뢰도 보강 전:
  - 펫동반 장소 탐색
  - 반려동물과 여행
  - Guestbook

### 8.4 운영 가능성

운영 가능성은 일부 도메인에서만 확보됐다.

- 가이드: 최소 운영 경로 있음
- 기록/일정: 사용자 단일 컨텍스트라 운영 부담이 상대적으로 낮음
- 커뮤니티/장소/여행: 운영도구와 신뢰도 레이어가 부족하다

### 8.5 플랫폼 완성도

- iOS:
  - 위치 권한/알림 일부 구현
- Android:
  - 홈 위젯 존재
  - 일정 알림 OS 예약 미구현

플랫폼 완성도는 균일하지 않다.

### 8.6 강점

- 문서와 SQL 자산이 풍부하다
- boot/store/service 구조가 비교적 명확하다
- 이미지, 캐시, fallback, upload queue 같은 운영형 장치가 있다
- 가이드와 타임라인은 설계-코드-SQL 연결이 좋다

### 8.7 약점

- 도메인 수가 많아 우선순위가 흔들리기 쉽다
- 문서와 코드 괴리가 이미 존재한다
- 장소/여행/커뮤니티는 운영 도구와 신뢰도 레이어가 부족하다
- placeholder 메뉴와 플랫폼 기능 불균형이 남아 있다

## 9. 즉시 수정이 필요한 실제 리스크

1. 문서-코드 괴리
- 커뮤니티 관련 문서 중 일부는 미구현 전제인데 실제 코드가 존재한다.
- 새 작업자가 오래된 문서를 그대로 믿으면 잘못된 판단을 한다.

2. More 메뉴의 기능 불일치
- `건강 기록 리포트`, `알림 설정`은 준비 중 토스트만 있다.
- 사용자 기대치와 실제 동작이 어긋난다.

3. Guestbook placeholder 노출
- 하단 툴바에 노출되지만 실제 기능은 임시 문구뿐이다.
- 서비스급 앱 기준으로는 과장된 노출이다.

4. iOS 위치 권한 단순화
- `src/services/location/permission.ts`에서 iOS는 무조건 `granted`
- 실제 권한 거부/제한/차단 상황과 맞지 않는다.

5. Android 일정 알림 미구현
- 권한 확인은 있으나 `upsertScheduleNotification()`은 iOS만 실제 예약
- 사용자에게 일정 알림이 된다고 오해를 줄 수 있다.

6. 커뮤니티 댓글/키보드 UX 안정성 미확정
- 최근 작업 로그가 반복적으로 이 이슈를 다루고 있다.
- 아직 실기기 최종 안정화가 끝난 상태로 보기 어렵다.

7. 펫동반/여행 신뢰도 부족
- 외부 후보와 규칙 기반 후보 정제로는 `confirmed`를 책임질 수 없다.

8. source of truth 혼재
- `memory_images`와 legacy image 필드가 병행
- 장소/여행은 문서상 DB가 있으나 런타임 source of truth는 아직 외부/API 후보층

## 10. 잠재적 리스크

### 10.1 구조적 리스크

- 도메인 수가 많고 store/service가 증가해 새 세션이 우선순위를 놓치기 쉽다.
- 공식 메모리 우선순위와 실제 작업 트리가 어긋나면 세션마다 엇갈린 작업이 누적될 수 있다.

### 10.2 실기기 UX 리스크

- 커뮤니티 상세 키보드
- 작성/수정 폼의 스크롤과 키보드 회피
- 위치/날씨 권한 실패 UX
- Android 일정 알림 기대치

### 10.3 외부 API 의존 리스크

- Kakao Local 품질/정책 의존
- TourAPI 응답 품질과 오탐
- Open-Meteo 지역 오차

### 10.4 상태/store 결합도 리스크

- `AppProviders`가 auth, pet, record, schedule, community, monitoring, consent, upload queue까지 많이 알고 있다.
- boot 로직이 계속 커지면 회귀 범위가 커진다.

### 10.5 신뢰도 데이터 레이어 부재

- 펫동반 장소와 반려동물과 여행 모두 장기적으로 Trust Layer가 핵심인데, 런타임 적용 전이다.
- 이 상태가 길어질수록 UI 신뢰도가 문서보다 뒤처진다.

### 10.6 운영/검수 도구 부족

- 가이드 외 도메인에 관리자 UI가 거의 없다.
- 커뮤니티, 장소, 여행은 운영 강도가 높은데 도구가 없다.

### 10.7 문서와 실제 코드 괴리 가능성

- 이미 커뮤니티에서 발생했다.
- 앞으로도 project-memory, operations 문서, 도메인 문서가 함께 갱신되지 않으면 같은 문제가 반복된다.

### 10.8 placeholder/메뉴 과장 리스크

- Guestbook 탭
- 건강 기록 리포트
- 알림 설정
- 일부 산책/기념일 표현

### 10.9 커뮤니티 장기 운영 리스크

- 신고, 스팸, 제재, 운영 로그, 백필
- 다중 이미지로 확장될수록 스토리지와 정합성 부담 증가

### 10.10 여행/장소 장기 운영 리스크

- 외부 source가 바뀌면 후보층 품질이 크게 흔들릴 수 있다.
- 검수와 사용자 제보가 없으면 `possible` 이상의 신뢰도 확장이 막힌다.

- 반려동물과 여행 Trust Layer SQL을 실제 Supabase 적용 순서로 검토하고, 앱에서 어떤 read 모델로 붙일지 설계하는 것

### 12.6 현재 프로젝트 작업 스타일 / workflow 기준

- 문서 먼저 읽고 코드 수정
- 실제 코드와 문서가 다르면 반드시 괴리를 문서에 남길 것
- 추측 금지
- 기능 추가 전 source of truth 재확인
- 작업 후 project-memory 갱신
- 한글 폴더명/파일명 규칙 유지
