# 5차 최종 기획서

## 1. 4차 대비 핵심 개선 사항

### 1-1. 5차의 핵심 방향

5차의 목적은 기능을 더 늘리는 것이 아니다.
이미 넓게 구현된 기능 축을 배포 가능한 서비스로 전환하기 위해, 4차에서 드러난 구조적 위험을 먼저 닫는다.

우선순위는 아래 6가지로 고정한다.

1. 서비스 계약 먼저 마감
   - 약관, 개인정보처리방침, 위치 고지, 커뮤니티 운영정책, 삭제 정책을 실제 화면과 연결한다.
   - 동의 UI와 실제 문서가 분리된 현재 상태를 종료한다.
2. 인증을 가입 가능에서 복구 가능 상태로 승격
   - 이메일 로그인만 되는 상태에서 멈추지 않는다.
   - 비밀번호 재설정, 계정 복구, 탈퇴 범위 증적까지 포함해 최소 운영 가능 인증으로 올린다.
3. 공개 도메인보다 방어선을 먼저 구축
   - 커뮤니티, 펫동반 장소, 여행 탐색은 기능 확장보다 abuse 방어선, 데이터 신뢰도, 운영 도구가 우선이다.
4. 후보 데이터와 검증 데이터의 경계를 명시적으로 분리
   - Kakao/TourAPI raw 후보 데이터는 Candidate Layer
   - 내부 검수/정제/운영 승인 데이터는 Trust Layer
   - 사용자 저장/신고/북마크는 User Layer
5. 앱 책임 축소, 운영 책임 분리
   - 앱이 외부 API 호출, 후보 정규화, 운영 판단까지 떠안는 구조를 더 키우지 않는다.
   - 운영 판단은 웹 CMS, 운영 도구, 문서, SQL, 백엔드 계약으로 이동한다.
6. 문서와 코드의 source of truth 체계 고정
   - 프로젝트 사실은 실제 코드가 최우선이다.
   - 작업 행동 규칙은 `docs/engineering/advanced-codex-workflow.md`를 따른다.
   - 현재 상태와 다음 작업은 `docs/project-memory/*`에서 관리한다.
   - 과거 낙관 문서는 참고용으로만 취급한다.

### 1-2. Root Cause 요약

- Root Cause A. 기능 확장 속도가 계약/운영/보안 설계를 앞질렀다.
- Root Cause B. 앱이 운영 판단까지 떠안는 구조가 커졌다.
- Root Cause C. 후보 데이터와 확정 데이터가 분리되지 않았다.
- Root Cause D. 공개 도메인 방어선이 부족하다.
- Root Cause E. 환경/배포/외부 API 통제가 약하다.
- Root Cause F. 문서와 실제 상태가 일부 어긋난다.

### 1-3. 5차에서 실제로 바뀌는 것

- 기능 중심 개발에서 계약/운영 중심 개발로 전환한다.
- 앱 단독 판단 대신 앱 + 문서 + SQL + 운영 도구 계약으로 전환한다.
- 후보 데이터 노출 중심에서 신뢰도 계층 분리로 전환한다.
- 새 기능 우선이 아니라 배포 차단 리스크 우선 제거로 전환한다.

## 2. 최종 비즈니스 로직 및 시스템 아키텍처

### 2-1. 제품 정체성

NURI는 반려동물별 기록, 일정, 탐색, 정보 소비, 공개 커뮤니티를 통합한 생활형 모바일 앱이다.
메모리얼 감성은 브랜드 톤과 핵심 기록 경험에 유지하되, 운영 기준은 생활 관리 서비스 수준으로 설계한다.

### 2-2. 도메인 계층 구조

- Layer 1. Core Identity & Account
  - Auth, Profile, Nickname, Pet context, Consents, Account deletion
- Layer 2. Private Life Domain
  - Memory / Timeline, Schedule, Weather-assisted action context, Pet profile / theme / multi-pet context
- Layer 3. Public / Semi-Public Domain
  - Community, Guides, Guestbook placeholder
- Layer 4. Discovery / Travel Domain
  - Location discovery, Pet-friendly places, Pet travel
- Layer 5. Operations & Trust
  - Policy documents, Admin / CMS / moderation tools, Monitoring / QA / release, Source of truth docs, Data trust / verification, Audit / recovery / cleanup

### 2-3. 인증/가입/동의 로직

- 회원가입
  - 이메일/비밀번호 입력
  - 필수 동의: 서비스 이용약관, 개인정보처리방침
  - 선택 동의: 마케팅 수신
  - 실제 정책 문서 화면 또는 외부 URL로 이동 가능해야 한다.
  - 가입 완료 후 이메일 인증 필요 여부를 반영한다.
  - 재진입 시 닉네임/펫 온보딩 분기를 유지한다.
- 로그인
  - 이메일/비밀번호 로그인
  - 세션 재검증
  - 프로필 hydrate
  - 닉네임/펫 존재 여부에 따라 후속 라우트 분기
- 비밀번호 재설정
  - 이메일 입력
  - Supabase reset flow 사용
  - 메일 링크 또는 앱 deep link로 복귀
  - 새 비밀번호 설정 화면 진입
  - 성공 후 로그인 또는 세션 재생성 정책 적용
- 계정 삭제
  - 즉시 삭제와 비동기 정리 정책을 명확히 고지한다.
  - 삭제 범위는 문서화한다.
  - 삭제 요청 성공과 실제 정리 완료를 분리할 수 있다.
  - 삭제 실패 시 부분 실패 사실과 재시도/문의 경로를 제공한다.
- 동의 snapshot
  - 동의 종류, 문서 버전, 동의 시각을 저장한다.
  - `src/services/legal/consents.ts`는 UI 보조가 아니라 계약 레이어의 일부로 취급한다.
- 현재 상태 메모
  - 정책 문서 source of truth와 초안 구조, 동의 snapshot 저장은 구현됐다.
  - 비밀번호 재설정 request/recovery/form 흐름도 구현됐다.
  - 남은 리스크는 최종 정책 문서 공개, Supabase redirect 허용 확인, 실기기 메일 복귀 QA, 계정 삭제 운영 증적 확보다.

### 2-4. Candidate / Trust / User Layer

- Candidate Layer
  - 출처: Kakao Local, TourAPI raw, 외부 검색 후보
  - 특징: 빠르게 노출 가능하지만 정확도 보장 불가
  - UI: `후보`, `확인 필요`, `실제 방문 전 재확인 권장`
- Trust Layer
  - 출처: 내부 SQL, 검수 이력, source link, external signals, user report review, verification logs
  - 특징: 운영 검수와 정책 메타를 read-only로 반영하되, 이번 단계에서는 `confirmed`를 열지 않는다.
  - UI: `검수 반영`, `확인 필요`, `업데이트 기준일 표시`
- User Layer
  - 범위: bookmarks, recent searches, user reports, personal filters, per-user visibility state
  - 특징: 개인화 데이터이며 Trust Layer와 분리 저장
  - 삭제/복구 정책에 포함해야 한다.
- 현재 연결 상태
  - `펫동반 장소 탐색`은 `Kakao candidate + pet_place_service_meta / pet_place_source_links / pet_place_external_signals optional overlay` 구조다. trust row만으로 리스트를 여는 trust-first read path는 아직 없지만, `publicTrust` 기준 `후보 / 확인 필요 / 검수 반영` 라벨과 기준일 노출은 실제 런타임에 연결됐다.
  - 산책은 공통 `locationDiscovery` 레이어를 사용하며, 위치 편의성과 공개 신뢰도를 같은 축으로 보지 않도록 `현재 위치 기반 후보` 성격의 보수 라벨을 적용했다.
  - `반려동물과 여행`은 TourAPI candidate를 기본으로 두고 `pet_travel_places / pet_travel_pet_policies`를 read-only로 읽어 public label을 재결정한다. `pet_travel_user_reports`는 personal layer own-state로만 연결되고 public trust 승격 경로는 열지 않는다.
  - user layer는 `pet_place_bookmarks / pet_place_user_reports / pet_place_search_logs / pet_travel_user_reports`를 public trust와 분리된 실제 read/write 경로로 연결한다. 다만 canonical id가 없는 후보에는 personal write path를 열지 않는다.
  - `trust_reviewed`는 `admin review + 유효한 기준일 존재 + 90일 이내 freshness + evidence 존재 + source conflict 없음 + onsite check 불필요`를 모두 만족할 때만 유지한다.
  - 위 조건 중 하나라도 비면 `needs_verification`으로 강등하고, trust row가 전혀 없거나 후보 신호만 남으면 `candidate`를 유지한다.

### 2-5. 데이터 노출 금지 규칙

- Candidate Layer 데이터만으로 `confirmed` 노출 금지

- 후보, 검수, 개인화 레이어를 같은 store shape로 뭉개지 않음

### 2-7. 정책/법무/인증/삭제/문서 정합성 우선 원칙

- 정책 문서 없이 회원가입 UX만 다듬지 않는다.
- 비밀번호 재설정 없는 인증을 운영 가능 상태로 보지 않는다.
- 삭제 범위 증적 없이 즉시 완전 삭제 문구를 쓰지 않는다.
- 문서와 코드가 어긋나면 프로젝트 사실은 코드 기준으로 문서를 갱신한다.
- 행동 규칙은 `advanced-codex-workflow.md`를 우선한다.

### 2-8. 환경/배포 원칙

- 환경은 dev / staging / prod로 분리한다.
- 환경별로 Supabase project / env, runtime config, monitoring environment, external API keys, release channel / branch, store/test distribution을 분리한다.
- native 변경과 OTA 변경은 분리한다.
- 정책/DB/RPC 변경 포함 릴리즈는 앱 코드 릴리즈와 동일 작업으로 취급한다.
- release checklist와 실기기 QA 증적 없는 배포는 금지한다.

## 3. 5차 실행 방식

### 3-1. 문서 구조 원칙

1. 구현 전에 문서가 먼저다.
2. 단계별 작업은 하나씩만 수행한다.
3. 한 작업이 끝나면 반드시 완료 기록을 남긴다.
4. 완료 기록 없이 다음 작업으로 넘어가지 마라.
5. 후속 태스크를 이번 턴에 미리 구현하지 마라.
6. 문서와 코드가 어긋나면 코드 기준으로 문서를 갱신하되, 행동 규칙은 workflow 기준으로 유지한다.

### 3-2. 다음 단계 운영 순서

1. 문서 기준 태스크 확정
2. 해당 태스크만 착수 보고
3. 코드 분석
4. 최소 수정
5. 검증
6. 문서 갱신
7. 완료 기록 작성
8. 다음 태스크로 이동

## 4. 5차 완료 정의

- 해결됨
  - Task 1 정책 문서 source of truth와 가입 동의 구조 정리
  - Task 2 삭제 추적형 계약과 cleanup queue 도입
  - Task 3 비밀번호 재설정 구현
  - Task 4 커뮤니티 abuse 최소 방어선의 repo 기준 구현, remote blocker 해소, linked project 재검증, 운영 기준 종료 정리
  - 문서와 코드의 source of truth 체계 재고정
- 부분 해결 / 운영 증적 필요
  - 최종 정책 문서 공개와 실제 링크/화면 연결
  - 계정 삭제 cleanup worker 또는 수동 재처리 증적
  - 비밀번호 재설정 redirect 허용과 실기기 복귀 QA
- 미해결 / 다음 작업 대상
  - stale trust와 source conflict를 어떤 기준으로 강등하고 설명할지 운영 기준 보강
  - 외부 API 키/환경 분리/장기 운영 도구 정리

## 6. 현재 즉시 실행 대상

### Task 6 커뮤니티 write-path 콘텐츠 정책 하드닝 준비

- 목표
  - linked project remote까지 포함해 커뮤니티 비속어/금칙어 정책의 실제 source of truth를 잠근다.
  - profile nickname path와 community post/comment write-path를 분리해서, 다음 구현을 실제 연결 작업 기준으로 다시 고정한다.
  - Task 5 physical-device QA는 release blocker 유지 상태로만 두고, 이번 engineering slice의 직접 목표에서는 분리한다.
- 현재 사실
  - linked project remote `grmekesqoydylqmyvfke`에는 `blocked_nicknames`, `blocked_nickname_patterns`, `enforce_profile_nickname_policy()`, `trg_profiles_nickname_policy`가 실제 운영 중이다.
  - 그러나 커뮤니티 post/comment write-path에는 repo와 remote 모두 community-specific blocked-term guard가 없다.
  - `guard_community_post_insert()`와 `guard_community_comment_insert()`는 insert rate limit, dedupe, trim만 처리하고 update path는 보호하지 않는다.
  - local repo의 닉네임 정책 SQL은 remote보다 뒤처져 있어 repo/remote drift가 있다.
  - `view_count`는 앱 타입/UI placeholder만 있고 linked project `public.posts` schema에도 없다.
  - physical-device QA와 Task 4 row-level 운영 캡처는 release blocker registry로 그대로 남아 있다.
- 관련 근거
  - `src/services/supabase/community.ts`
  - `src/store/communityStore.ts`
  - `src/screens/Community/*`
  - `src/types/community.ts`
  - `docs/sql/커뮤니티/커뮤니티-운영-최종.sql`
  - `docs/sql/인증-계정/프로필-닉네임정책-현재로컬미러.sql`
  - `docs/project-memory/현재-프로젝트-상태.md`
- 주의
  - repo에 없다고 바로 `없음` 확정하지 않는다. remote source of truth를 먼저 본다.
  - profile nickname policy를 community write-path와 혼동하지 않는다.

- 현재 기준의 다음 실작업은 release 전 실기기 QA와 운영 캡처 보강이다.
- Task 4는 완료로 유지하고, release 직전 운영 캡처와 SQL/콘솔 SOP 보강만 병렬 메모로 남긴다.
- 정책 URL과 최종 법무 문안은 임의 확정하지 않는다.
- 실제 문서 없는 상태에서 UI만 확장하거나, 운영 증적 없는 상태에서 “완료”라고 올리지 않는다.
- moderation 없는 커뮤니티 기능 확장, candidate 데이터를 confirmed처럼 취급하는 변경은 계속 금지한다.
- 4차 문서는 참고용으로만 보고, 현재 판단은 5차 기획서와 project-memory를 우선한다.
- stale/conflict 기준이 잠겼더라도 admin review write path와 실기기 QA 없이 `confirmed` 개방 검토로 바로 넘어가지 않는다.
- 물리 실기기 QA에서 실제 혼동이 재현될 때만 다음 턴 최소 UI 조정을 연다. 재현이 없으면 다음도 운영 캡처와 장기 운영 리스크 정리 턴으로 유지한다.
