## 1. 지금 앱이 당장 배포되면 위험한 점

- `[부분 해결 / 운영 증적 필요]` 회원가입은 더 이상 placeholder alert만 있는 상태는 아니다. `src/services/legal/documents.ts`와 `docs/policies/*` 초안, 동의 snapshot 구조는 생겼지만 실제 최종 문서 URL과 앱 내 전체 문서 연결은 아직 미확정이다.
- `[부분 해결 / 운영 증적 필요]` 비밀번호 재설정은 이제 UI-only가 아니라 request/recovery/form 흐름까지 구현됐다. 다만 Supabase redirect 허용, 메일 앱 복귀, 실기기 QA 증적은 아직 남아 있다.
- `[해결됨]` 커뮤니티는 더 이상 무방비 상태가 아니다. 서버측 rate limit, duplicate 방지, auto-hide, moderation queue/action log, image cleanup_pending 계약이 repo 기준으로 반영됐고, remote blocker 해소와 linked project 재검증까지 끝나 Task 4는 종료 상태다. release 직전 SQL/콘솔 캡처와 SOP 보강은 운영 메모로만 남긴다.
- `[미해결 / 다음 작업 대상]` 외부 API 키가 앱 클라이언트에 들어가는 구조는 그대로라 유출과 남용 가능성은 계속 열린 상태다.
- `[미해결 / 다음 작업 대상]` 장소/여행 탐색은 candidate 결과는 보여주지만, 사용자에게 Trust Layer 수준의 최종 보장을 주는 운영 구조는 아직 없다.
- `[미해결 / 다음 작업 대상]` dev/staging/prod 환경 분리와 출시 채널 정리는 아직 문서/실환경 모두 충분히 닫히지 않았다.
- `[부분 해결 / 운영 증적 필요]` 탈퇴 RPC는 이제 추적형 계약으로 바뀌었지만, 전체 확장 도메인과 cleanup worker 실동작을 포함한 운영 증적은 아직 부족하다.

## 2. 지금 구조에서 장기 운영 시 무너질 수 있는 점

### 2.1 운영 도구 부재

- `[부분 해결 / 운영 증적 필요]` 커뮤니티는 `community_moderation_queue / actions / reporter_flags / image_assets`로 최소 운영 데이터 저장소가 생겼다. 하지만 전용 운영 UI와 수동 처리 SOP는 아직 없다.
- `[미해결 / 다음 작업 대상]` 장소/여행 데이터 검수 도구는 여전히 없다.
- `[미해결 / 다음 작업 대상]` 구독/결제 운영 화면은 아직 범위 밖이며, entitlement/원본 결제 상태를 보는 구조도 없다.

### 2.2 앱 책임 과다

- `[미해결 / 다음 작업 대상]` 클라이언트가 외부 API 호출, 후보 정규화, 일부 신뢰도 표현을 계속 많이 떠안고 있다.
- `[부분 해결 / 운영 증적 필요]` Candidate / Trust / User Layer 구분은 문서와 타입에 잡혔지만, 장소/여행 런타임은 아직 candidate 중심이다.
- `[미해결 / 다음 작업 대상]` `AppProviders`가 bootstrap, 동의 flush, queue 복구를 넓게 담당하는 구조는 그대로라 부팅 책임이 여전히 무겁다.

### 2.3 문서와 실환경의 간극

- `[부분 해결 / 운영 증적 필요]` source of truth 위계는 5차 문서와 project-memory로 다시 고정됐지만, staging/prod 실제 반영 여부는 여전히 사람 확인이 필요하다.
- `[해결됨]` 과거 문서의 낙관적 표현을 현재 source of truth처럼 쓰는 문제는 이번 정합화 작업으로 낮췄다.
- `[부분 해결 / 운영 증적 필요]` SQL 최종본과 release checklist는 정리됐지만, 실환경 적용 증적과 dirty worktree 영향은 계속 구분해서 봐야 한다.

## 3. 데이터 신뢰도 / 운영도구 / 커뮤니티 / 위치 / 외부 API / 결제 / 인증에서 가장 위험한 점

### 3.1 데이터 신뢰도

- `[미해결 / 다음 작업 대상]` Kakao Local만으로 반려동물 동반 가능 여부를 확정할 수 없다는 리스크는 그대로다.
- `[미해결 / 다음 작업 대상]` TourAPI raw만으로 `confirmed`를 열 수 없다는 리스크도 그대로다.
- `[부분 해결 / 운영 증적 필요]` 커뮤니티는 post/comment author snapshot 보강으로 일부 live fallback 리스크를 줄였지만, 공개 post snapshot이 없는 신규 작성자까지 100% 보장하는 서버 계약은 아직 아니다.
- `[부분 해결 / 운영 증적 필요]` 탈퇴, soft delete, image cleanup, upload queue는 계약과 테이블이 생겼지만 장기 운영에서 완전히 일치하는지 실증은 아직 부족하다.

### 3.2 운영도구

- `[부분 해결 / 운영 증적 필요]` 수동 복구와 수동 제재의 근거 테이블은 일부 생겼지만, 운영자가 쉽게 조회·수정하는 도구는 없다.
- `[미해결 / 다음 작업 대상]` 관리자 기능은 여전히 앱 내부 브릿지 수준에 머무른다.
- `[미해결 / 다음 작업 대상]` 운영자용 분리 채널과 권한 감사 체계는 아직 없다.

### 3.3 커뮤니티

- `[해결됨]` abuse 방어선이 가장 약하다는 과거 진단은 더 이상 현재 사실이 아니다. 서버측 rate limit, dedupe, auto-hide, queue/action log까지 실제 linked project 재검증을 마쳤다.
- `[해결됨]` 이미지 업로드 공개 도메인에 moderation queue가 없던 문제도 더 이상 현재 blocker가 아니다. upload, `community_image_assets`, hidden 이미지 비노출, `cleanup_pending` 전환이 remote에서 다시 확인됐다.
- `[부분 해결 / 운영 증적 필요]` 악성 사용자 대응 기준은 최소 계약과 release checklist까지는 생겼지만, 운영 정책 문안과 실제 처리 절차는 아직 미완성이다.
- `[미해결 / 다음 작업 대상]` 전용 관리자 UI 없이 SQL/콘솔 중심으로만 운영해야 하는 점은 장기 운영 리스크로 남아 있다.

### 3.4 위치

- `[부분 해결 / 운영 증적 필요]` 위치 freshness와 stale 표시는 정리됐지만, 정책 문서와 사용자 고지는 아직 최종 문안/링크가 없다.
- `[부분 해결 / 운영 증적 필요]` 현재 위치 실패, stale 위치, 잘못된 추천이 신뢰를 해칠 수 있다는 리스크는 실기기 QA가 더 필요하다.
- `[미해결 / 다음 작업 대상]` 위치 기반 기능과 장소 신뢰도 레이어가 장기적으로 더 명확히 분리돼야 한다.

## 11. 가장 보수적인 최종 결론

지금 NURI는 더 이상 단순 데모 수준만은 아니다.
Task 1~4 기준으로 계약과 방어선은 실제 코드/SQL에 많이 내려왔다.

하지만 아직 “실서비스급 장기 운영 가능한 앱”이라고 단정하기에는 이르다.

남아 있는 가장 위험한 착시는 아래다.

- 구현이 들어갔으니 운영 준비도 끝난 것처럼 보는 착시
- SQL 최종본이 있으니 staging/prod에도 같을 것이라고 보는 착시
- queue/action log 테이블이 있으니 운영 도구도 있는 것처럼 보는 착시
- candidate 데이터를 보여주고 있으니 trust layer도 곧바로 열 수 있다고 보는 착시

현재 결론은 보수적으로 유지한다.

- 지금부터 더 중요한 일은 새 기능 확장이 아니라 운영 증적, 정책 최종화, 환경 분리, 장기 운영 도구 설계다.
- 구현 완료와 운영 완료를 같은 말처럼 쓰지 않는 것이 장기 운영 리스크를 줄이는 첫 단계다.
