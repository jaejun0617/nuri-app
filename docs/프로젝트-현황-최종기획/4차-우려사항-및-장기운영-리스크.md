# NURI 4차 우려사항 및 장기운영 리스크

작성일: 2026-03-23  
작성 기준:
- 실제 코드
- `docs/` 전체
- `docs/sql/` 전체
- `project-memory`

이 문서는 총괄 문서보다 더 보수적이고 더 냉정하게 쓴다.  
중복을 피하지 않는다. “실제로 지금 배포하면 어디서 사고가 날 수 있는가”를 중심으로 정리한다.

## 1. 지금 앱이 당장 배포되면 위험한 점

- 회원가입은 받지만 실제 약관/개인정보처리방침 문서 연결이 없다.
- 비밀번호 찾기, 계정 복구, 소셜 로그인은 UI만 있고 실제 구현이 없다.
- 커뮤니티는 글/댓글/신고까지 열려 있지만 rate limit, spam 방어, 신고 처리 도구가 없다.
- 외부 API 키가 앱 클라이언트에 들어가는 구조라 유출과 남용을 전제로 봐야 한다.
- 펫동반 장소와 여행 탐색은 후보 데이터는 보이지만, 사용자에게 “믿어도 되는 정보”인지 최종 보장하기 어렵다.
- 환경 분리와 출시 채널 분리가 정리되지 않아 dev/test/prod 혼선 가능성이 높다.
- 탈퇴 RPC는 존재하지만 실제 삭제 범위가 전체 확장 도메인까지 완전히 닫혔다고 공표하기 어렵다.

## 2. 지금 구조에서 장기 운영 시 무너질 수 있는 점

### 2.1 운영 도구 부재

- 신고가 쌓여도 누가 어떻게 처리하는지 도구와 프로세스가 없다.
- 콘텐츠/댓글/사용자 제재 이력 관리 도구가 없다.
- 장소/여행 데이터 검수 도구가 없다.
- 구독/결제가 붙으면 entitlement와 원본 결제 상태를 볼 운영 화면이 없다.

### 2.2 앱 책임 과다

- 클라이언트가 외부 API 호출, 후보 정규화, 일부 신뢰도 표현까지 다 떠안고 있다.
- 모바일 앱이 운영 판단까지 대신하면 릴리즈마다 위험이 커진다.
- AppProviders가 bootstrap, 동의 flush, 업로드 큐 복구, cleanup까지 동시에 처리해 부팅 책임이 무겁다.

### 2.3 문서와 실환경의 간극

- SQL 최종본이 있어도 실제 DB 적용 여부가 항상 보장되진 않는다.
- 과거 문서의 낙관적 진행률은 잘못된 기대를 만들 수 있다.
- source of truth가 분산되어 새 작업자가 잘못된 문서부터 읽을 위험이 있다.

## 3. 데이터 신뢰도 / 운영도구 / 커뮤니티 / 위치 / 외부 API / 결제 / 인증에서 가장 위험한 점

### 3.1 데이터 신뢰도

- Kakao Local만으로 반려동물 동반 가능 여부를 확정할 수 없다.
- TourAPI raw 문구만으로 반려동물 동반 `confirmed`를 열 수 없다.
- 커뮤니티의 펫 메타는 아직 snapshot 완결 구조가 아니라 live fallback 영향이 남아 있다.
- 탈퇴, soft delete, image cleanup, upload queue가 장기 운영에서 완벽히 일치하는지 검증 부족하다.

### 3.2 운영도구

- 수동 복구, 수동 제재, 수동 검수, 감사 로그 조회 도구가 사실상 없다.
- 관리자 기능이 앱 내부 브릿지 수준에 머무른다.
- 운영자용 분리 채널과 권한 감사 체계가 없다.

### 3.3 커뮤니티

- abuse 방어선이 가장 약하다.
- 이미지 업로드가 가능한 공개 도메인인데 anti-spam과 moderation queue가 없다.
- 악성 사용자 대응 기준이 문서화되지 않았다.
- 신고 누적 시 자동 숨김, 임시 차단, 재검토 흐름이 없다.

### 3.4 위치

- 위치 사용은 개인 정보 이슈인데 정책 문서와 사용자 고지가 부족하다.
- 현재 위치 실패, stale 위치, 잘못된 위치 기반 추천이 사용자 신뢰를 해칠 수 있다.
- 위치 기반 기능이 실제 장소 신뢰성과 섞여 보여질 위험이 있다.

### 3.5 외부 API

- 키 유출과 quota 고갈 시 앱 탐색 기능이 한 번에 약해진다.
- 외부 응답 스키마 변경, 품질 저하, 장애 시 운영 대응 문서가 없다.
- 비용 통제 전략이 문서화되지 않았다.

### 3.6 결제

- 지금은 미구현이지만, 바로 붙이면 가장 위험하다.
- entitlement 없이 `pets` 수만 보고 과금 차단을 걸면 데이터 볼모 UX가 된다.
- 서버 검증 없이 앱만으로 처리하면 즉시 부정 사용과 정합성 문제가 생긴다.

### 3.7 인증

- 이메일 로그인만 있는 것은 문제가 아니다.
- 문제는 복구, 재설정, 정책 문서, OAuth 인프라 없이 인증을 확장하려는 유혹이다.
- 현재 상태에서 소셜 로그인을 바로 붙이면 계정 연결/중복/탈퇴 정책이 먼저 무너질 가능성이 크다.

## 4. 개인정보 / 정책 / 법적 관점에서 준비 안 된 점

- 회원가입 UI에 동의 체크는 있으나 실제 법적 문서 연결이 없다.
- 개인정보처리방침 URL, 서비스 이용약관 URL, 커뮤니티 운영정책 URL이 없다.
- 위치정보 사용에 대한 서비스 차원의 고지와 보관 정책이 없다.
- 이미지/게시글/댓글 보관 기간과 삭제 처리 기준이 없다.
- 탈퇴 후 백업/로그/에러리포트에 남는 정보 범위가 정리되지 않았다.
- 제3자 API와 모니터링 도구의 개인정보 처리 고지가 없다.
- 향후 결제를 붙일 경우 필요한 환불/복원/해지/청약철회 정책이 전혀 준비되지 않았다.

## 5. 보안상 가장 취약할 수 있는 구조

### 5.1 클라이언트 내 외부 API 키

- `runtime.ts` 방식은 저장소 밖이어도 앱 번들 내부 키 노출 문제를 해결하지 못한다.
- Kakao REST API 키, TourAPI 키는 추출 가능성을 전제로 봐야 한다.

### 5.2 AsyncStorage 중심 세션/로컬 상태

- RN에서 흔한 방식이지만, 장기 운영 기준으로는 보수적이지 않다.
- 세션, 동의 snapshot, draft, upload queue가 모두 로컬 평문 저장소 계열에 모인다.

### 5.3 커뮤니티 도메인

- 공개 텍스트 + 이미지 + 신고 + 댓글 구조인데 abuse 방어가 약하다.
- authenticated image select 정책과 실제 공개 범위가 장기적으로 어긋날 수 있다.
- rate limit이 없으면 가장 먼저 공격받기 쉬운 표면이다.

### 5.4 로그/모니터링

- Sentry/Crashlytics에 email과 user id를 넣고 있다.
- extras/tags를 잘못 쓰면 signed URL, 위치, 본문 일부가 섞일 수 있다.

## 6. 사고 났을 때 대응이 어려운 부분

- 잘못된 장소/여행 데이터로 사용자 불만이 들어왔을 때 정정 도구 부족
- 커뮤니티 신고 폭주 시 처리 큐와 운영 기준 부재
- 외부 API 키 남용 시 차단/교체/runbook 부재
- 계정 삭제 누락 데이터 발견 시 어떤 범위를 재삭제해야 하는지 문서 부족
- signed URL 또는 이미지 orphan 파일 누적 시 정리 체계 부족
- 운영 환경 장애 시 어떤 화면/도메인을 먼저 셧다운하거나 완화할지 기준 부족

## 7. 지금보다 더 보수적으로 바꿔야 하는 설계

- 외부 API는 가능하면 서버 프록시나 배치 적재 중심으로 이동
- 장소/여행의 `confirmed`는 더 늦게 열어야 함
- 커뮤니티는 기능 추가보다 moderation과 rate limit부터
- 계정삭제는 “삭제 요청 수락”과 “비동기 정리 완료”를 분리해도 됨
- 민감한 로컬 저장은 범위를 더 줄여야 함
- 관리자 기능은 앱 내 브릿지보다 별도 운영 도구로 이관하는 방향이 안전함

## 8. 장기 운영 전에 반드시 닫아야 하는 리스크

1. 실제 정책 문서와 회원가입 연결
2. 비밀번호 재설정과 계정 복구 최소 경로
3. 탈퇴 삭제 범위 재검증
4. 커뮤니티 abuse 방어선
5. 외부 API 키/호출량 통제
6. dev/staging/prod 환경 분리
7. release signing / store metadata / 문의 채널 / privacy URL 정리
8. 실기기 QA 증적 확보

이 8개 중 다수가 열려 있으면 “정식 서비스 출시”라고 보기 어렵다.

## 9. 사장 / PO / 개발자 관점 경고문

### 사장 관점

- 기능이 많아 보여도, 정책과 운영 도구 없이 배포하면 작은 이슈가 신뢰 하락으로 바로 번진다.
- 특히 커뮤니티와 위치 기반 정보는 사고 비용이 크다.

### PO 관점

- 지금 우선순위는 새 기능 추가가 아니다.
- 사용자가 기대하는 계약을 문서, 정책, 계정 흐름으로 먼저 맞춰야 한다.
- Guestbook 같은 placeholder는 제품 구조를 오히려 흐린다.

### 개발자 관점

- 코드가 돌아가는 것과 서비스가 안전한 것은 다르다.
- 이번 프로젝트는 DB 문서와 앱 런타임 사이의 차이를 항상 의식해야 한다.
- OAuth, 결제, 커뮤니티 운영, 장소 신뢰도는 “UI부터 열면” 나중에 더 비싸게 갚는다.

## 11. 가장 보수적인 최종 결론

지금 NURI는 “데모 가능한 다기능 앱”으로는 충분히 의미가 있다.  
하지만 “실서비스급 장기 운영 가능한 앱”으로는 아직 방어선이 얇다.

가장 위험한 착시는 아래다.

- 기능이 많으니 거의 끝난 것처럼 보이는 착시
- SQL 문서가 있으니 운영 준비도 된 것처럼 보이는 착시
- 커뮤니티가 동작하니 서비스형 운영도 가능할 것 같은 착시
- 멀티펫이 되니 구독만 붙이면 될 것 같은 착시

실제는 반대다.

- 지금부터 더 중요한 일은 기능 추가가 아니라 계약과 방어선 마감이다.
- 인증, 정책, 삭제, 보안, 운영도구, 환경 분리가 정리되기 전까지는 낙관 금지다.
