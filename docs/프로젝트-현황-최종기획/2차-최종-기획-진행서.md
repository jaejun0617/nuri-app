## 0. 이 문서의 역할

이 문서는 NURI 프로젝트를 새 GPT, 새 작업자, 새 세션이 즉시 이어받기 위한 단일 기준 문서다. 목표는 아래 다섯 가지다.

- NURI가 어떤 서비스인지 한 번에 이해할 수 있어야 한다.
- 앱 구조, 문서 구조, SQL 구조를 동시에 파악할 수 있어야 한다.
- 무엇이 실제 구현 완료이고 무엇이 부분 완료인지 구분할 수 있어야 한다.
- 운영 리스크와 잠재 리스크를 분리해서 볼 수 있어야 한다.
- 다음 작업 우선순위를 이 문서만 보고 바로 이어갈 수 있어야 한다.

문서 작성 원칙은 아래를 따른다.

- 추측하지 않는다.
- 실제 코드, 실제 SQL, 실제 문서에 있는 것만 적는다.
- mock, seed, fallback, placeholder는 구현 완료로 쓰지 않는다.
- 문서상 계획과 현재 런타임 구현을 분리해서 적는다.
- 문서와 코드가 어긋날 수 있는 지점은 리스크로 명시한다.

## 1. 프로젝트 정체성 / 서비스 개요

### 1.1 NURI가 무엇인가

NURI는 React Native 기반 반려동물 생활 관리 앱이다. 현재 코드 기준 이 앱은 단일 기능 앱이 아니라, 반려동물 중심의 생활 기록, 일정, 날씨 가이드, 위치 탐색, 장소 탐색, 여행 탐색, 콘텐츠 가이드를 한 앱 안에 묶는 구조다.

핵심 사용자 대상은 반려동물 보호자이며, 앱이 제공하려는 실질적 가치는 아래와 같다.

- 반려동물 단위로 일상 기록과 일정을 관리한다.
- 현재 위치와 날씨를 실제 생활 행동으로 연결한다.
- 산책, 장소 탐색, 여행 탐색처럼 외부 데이터를 앱 문맥으로 정리해 보여준다.
- 가이드와 홈 추천을 통해 반려동물 상태와 생활 맥락에 맞는 정보를 노출한다.

### 1.2 현재 앱이 다루는 주요 도메인

실제 코드 기준 주요 도메인은 아래다.

- Auth / 온보딩
- Splash / bootstrap
- Home / 메인 허브
- Pets
- Records / Timeline
- Schedules / Anniversaries 성격의 일정 관리
- Weather / 실내 활동 추천
- 산책 위치기반 탐색
- 펫동반 장소 탐색
- 반려동물과 여행
- 집사 꿀팁 가이드
- More / 계정 / 설정 성격 메뉴
- 관리자 가이드 운영
- 모니터링 / 장애 추적

### 1.3 현재 서비스 상태 한 줄 정의

현재 NURI는 "다양한 생활형 도메인이 실제 코드와 SQL로 이미 올라와 있는 앱"이지만, "도메인별 운영 완성도와 신뢰도가 아직 균일하지 않은 상태"에 가깝다.

### 2.3 store 구조

  - 펫 목록, 선택 펫, 펫 refresh
- `recordStore`
  - 펫별 기록 캐시, 로컬 upsert, refresh, 페이지네이션

현재 구조는 "전역 store + 화면 state + AsyncStorage + 원격 refresh"가 함께 쓰인다. 모든 도메인이 동일한 패턴으로 통일돼 있지는 않다.

기본 모니터링 구조는 있으나, 도메인별 운영 알림 체계까지는 코드상 드러나지 않는다.

- 위젯 동기화는 Android 전용이다.

### 2.9 코드와 문서의 관계

이 프로젝트는 코드와 문서가 모두 적극적으로 유지되는 구조다.

- 코드가 현재 런타임 truth다.
- SQL은 데이터 구조 truth다.
- 도메인 문서는 방향과 운영 조건을 정리한다.
- project-memory는 최근 작업 흐름을 보조 기억한다.

즉 신규 작업자는 코드만 보거나 문서만 보면 안 되고, 둘을 함께 봐야 한다.

## 3. 앱 첫 시작 플로우

### 3.1 부트스트랩 시작

`App.tsx`는 앱 시작 시 아래를 수행한다.

- `initMonitoring()`
- provider 트리 렌더
- 네비게이션 컨테이너 등록

`AppProviders`는 실제 bootstrap 중심 계층이다.

- 현재 세션 조회
- auth listener 등록
- user-scoped state 재로딩 여부 판단
- auth / pet / record / schedule / weather 관련 상태 refresh
- monitoring user 설정

### 3.2 Splash

현재 Splash 역할은 `src/screens/Home/HomeScreen.tsx`가 맡는다.

- 브랜딩 화면을 잠시 보여준다.
- bootstrap 결과를 기다린다.
- 다음 목적지에 따라 splash hold 시간을 다르게 둔다.

`src/services/app/boot.ts` 기준 분기:

- 비로그인 -> `AppTabs`
- 로그인 + nickname 미완료 -> `NicknameSetup`
- 로그인 + nickname 완료 + 펫 없음 -> `PetCreate`
- 로그인 + nickname 완료 + 펫 있음 -> `AppTabs`

### 3.3 세션 / 프로필 / 닉네임 / 펫 확인

실제 확인 순서는 아래 축으로 이해하면 된다.

1. 세션 존재 여부
2. `profiles.nickname_confirmed`
3. 펫 개수
4. 홈 진입 허용 여부

추가로 `MainScreen`에서도 focus 시 재검증이 들어간다. 즉 최초 bootstrap만 믿지 않고 홈 복귀 시점에서도 닉네임과 펫 상태를 다시 확인한다.

### 3.4 온보딩 완료 플로우

현재 온보딩은 아래 흐름이다.

- SignUp
- NicknameSetup
- PetCreate
- `WelcomeTransition`
- `AppTabs > HomeTab`

`WelcomeTransitionScreen`은 실제로 존재하며, 닉네임 설정 직후 짧은 환영 애니메이션을 보여주고 홈으로 reset 이동한다.

### 3.5 게스트 플로우

게스트도 `AppTabs` 진입은 가능하다. 다만 홈은 `GuestHome` 위주 CTA 화면이고, 핵심 관리 기능은 로그인/펫 생성 전제다.

### 3.6 진입 가드 요약

현재 코드상 핵심 가드:

- 닉네임 미완료면 `NicknameSetup`
- 펫 없으면 `PetCreate`
- 관리자 가이드 운영은 role guard
- 위치 기반 기능은 권한 실패 시 fallback/검색 허용
- 계정 삭제와 로그아웃은 local state clear 후 진행

## 5. 도메인별 구현 현황

아래 평가는 실제 코드와 문서 기준으로만 정리한다.

### 5.1 Auth

실제 구현된 것:

- 이메일/비밀번호 회원가입
- 이메일/비밀번호 로그인
- 세션 영속
- 로그아웃
- 회원탈퇴 RPC 호출 구조
- 닉네임 설정 / 중복 확인 RPC
- 약관/개인정보/마케팅 동의 로컬 저장
- 세션 준비 후 `user_consent_history` flush 구조

부분 완료 / 구조만 있는 것:

- 약관/개인정보 상세는 placeholder alert
- 소셜 로그인 버튼은 placeholder alert
- 비밀번호 찾기 액션은 placeholder alert

부족한 것:

- 실제 약관 상세 소비 화면
- 비밀번호 재설정 플로우
- 소셜 로그인

서비스급 리스크:

- placeholder 액션이 사용자에게 노출된다.
- 동의 flush는 `user_consent_history` 테이블이 실제 적용되지 않았을 때 오류 가능성이 있다.

다음 단계:

- legal 상세 화면
- 비밀번호 재설정
- 소셜 로그인 여부 명확화

### 5.2 Splash / bootstrap

실제 구현된 것:

- 세션 조회
- auth listener
- bootstrap route resolve
- user-scoped state reload 기준 분리
- splash hold 시간 차등

부분 완료 / 구조만 있는 것:

- timeout 유틸은 있으나 장애 단계별 세밀한 복구 UX는 약하다.

부족한 것:

- bootstrap 실패 시 사용자에게 구조적 진단 정보를 주지 않는다.

서비스급 리스크:

- auth/profile/pet/store 정합이 시작 시점 품질을 좌우한다.

다음 단계:

- bootstrap 예외 시나리오 QA 보강

### 5.3 Home / Main

실제 구현된 것:

- 게스트 홈 / 로그인 홈 분리
- 펫 테마 팔레트 적용
- 오늘의 사진 선택
- 시간대 메시지
- 주간 요약
- 일정 / 기록 / 가이드 / 날씨 연결
- Android 위젯 스냅샷 동기화

부분 완료 / 구조만 있는 것:

- 홈 추천의 일부는 다른 도메인의 fallback/seed 결과를 소비한다.

부족한 것:

- 홈이 하나의 큰 허브 컴포넌트에 많이 몰려 있다.

서비스급 리스크:

- 여러 도메인 상태가 홈에서 결합되어 장애 전파면적이 크다.

현재 도메인 문서는 아래처럼 구성된다.

- 산책 위치기반 기능
- 산책 조회 신뢰성 개선
- 펫동반 장소 탐색
- 반려동물과 여행
- 집사 꿀팁 가이드
- 집사 꿀팁 가이드 웹 CMS
- 타임라인 메모리 이미지

### 7.4 운영 문서 구조

운영 문서는 아래를 중심으로 본다.

- 프로젝트 전체 진행현황
- 공용 릴리즈 묶음 운영 기준
- 마이그레이션 운영 기준
- 릴리즈 QA 체크리스트

### 7.5 문서와 코드가 어긋날 수 있는 대표 지점

- 웹 CMS는 문서가 있으나 현재 저장소에는 앱 코드만 있다.
- 여행 / 펫동반 장소 탐색은 Trust Layer 문서가 런타임보다 앞서 있다.
- 산책 문서의 일부 표현은 실제 앱 구현보다 넓게 읽힐 수 있다.
- 가이드는 운영 row와 seed fallback 설명을 잘못 읽으면 현재 상태를 과대평가할 수 있다.

## 8. 현재 구현 수준 총평

### 8.1 구현 폭

구현 폭은 넓다. 단일 기록 앱 수준을 넘어, 반려동물 생활 관리 전반을 여러 도메인으로 나눠 실제 화면과 서비스로 올려둔 상태다.

### 8.2 실제 동작 수준

실동작 강도가 높은 영역:

- 인증 기본 흐름
- 펫 생성/수정
- 기록 CRUD
- 일정 CRUD
- 홈 허브

실동작은 있으나 운영 신뢰도 보강이 필요한 영역:

- 날씨
- 산책
- 펫동반 장소 탐색
- 반려동물과 여행
- 가이드 운영 전환

### 8.3 서비스급 완성도

보수적으로 보면 "기능이 많은 앱"이지 "운영 준비가 균일하게 끝난 앱"은 아니다. 특히 외부 API 의존 도메인과 운영 메타가 필요한 도메인에서 편차가 크다.

### 8.4 강점

- 코드, SQL, 문서가 모두 존재한다.
- Supabase 중심 구조가 일관적이다.
- 새로운 GPT가 작업에 들어갈 근거 자료가 충분하다.

### 8.5 약점

- 도메인별 완성도 편차
- placeholder가 일부 사용자 액션에 남아 있음
- 플랫폼 간 기능 불균형
- 외부 API 품질 의존
- 홈 허브 결합도

## 9. 즉시 수정이 필요한 리스크

### 9.1 iOS 위치 권한 처리

`src/services/location/permission.ts`는 iOS에서 실제 권한 확인 없이 `granted`를 반환한다. 위치/날씨/산책/장소 탐색 모두에 직접 영향이 있다.

### 9.2 Android 일정 알림 미구현

Android는 일정 알림이 저장은 되지만 실제 기기 알림 스케줄링은 구현되지 않았다.

### 9.3 인증 placeholder

로그인/회원가입 화면의 소셜 로그인, 비밀번호 찾기, 약관/개인정보 상세는 placeholder alert 상태다.

### 9.4 외부 API 키 하드코딩

`src/config/runtime.ts`에 Kakao REST 키와 Tour API 서비스 키가 상수로 들어 있다.

### 9.5 홈 허브 결합도

홈은 여러 도메인을 한 컴포넌트에서 결합해 장애 파급면적이 넓다.

## 10. 잠재적 리스크

### 10.1 구조적 리스크

- 전역 store, 화면 state, AsyncStorage, 원격 refresh가 혼재한다.
- 대형 화면 파일이 여럿 존재한다.

### 10.2 데이터 신뢰 리스크

- 펫동반 장소 탐색과 여행은 Trust Layer가 필요하다.
- 외부 후보와 확정 데이터는 동일하지 않다.

### 10.3 플랫폼 UX 리스크

- Android/iOS 기능 차이가 알림과 권한 처리에서 존재한다.
- 키보드/스크롤/포커스 UX 민감도가 높은 화면이 많다.

### 10.4 운영 확장 리스크

- 앱 관리자와 향후 웹 CMS의 역할이 충돌할 수 있다.
- seed fallback이 장기 운영에서 혼선을 줄 수 있다.

### 10.5 문서-코드 불일치 리스크

- 문서가 더 앞서 있는 도메인에서는 신규 작업자가 실제 구현 정도를 과대평가할 수 있다.

## 16. 최종 결론

NURI는 문서만 많은 프로젝트가 아니라 실제 앱 코드와 실제 SQL이 넓게 올라와 있는 프로젝트다. 다만 신규 작업자는 "구현 폭"에 먼저 압도되기보다 "현재 런타임에서 실제로 끝난 것과, 문서/SQL이 먼저 준비된 것"을 구분하는 시야가 먼저 필요하다. 이 프로젝트의 핵심은 새 기능을 많이 붙이는 것보다, 이미 열린 도메인들을 서비스급 신뢰도와 운영성 수준으로 끌어올리는 데 있다.
