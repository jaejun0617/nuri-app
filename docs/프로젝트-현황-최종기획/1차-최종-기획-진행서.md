# Nuri 1차 최종 기획 진행서

작성 기준: 2026-03-21 현재 `/nuri` 실제 코드 기준 전수 확인  
작성 원칙:
- 문서, 기억, 이전 대화가 아니라 현재 코드에 있는 내용만 근거로 판단
- 화면만 있거나 구조만 있는 경우 구현 완료로 판정하지 않음
- 서비스급 기준으로 냉정하고 보수적으로 평가
- MVP 수준, 검증용 수준, 서비스급 수준을 분리해서 서술

## 1. 프로젝트 전체 개요

현재 Nuri는 단일 기능 앱이 아니라 아래 축이 동시에 존재하는 반려동물 생활관리 앱 구조로 구현되어 있다.

- 인증: 이메일 회원가입, 로그인, 닉네임 설정, 로그아웃, 계정 삭제, 비밀번호 변경
- 반려동물: 펫 생성, 상세 프로필 수정, 멀티펫 선택/전환
- 홈: 로그인 상태 기준 홈 대시보드, 위젯 스냅샷, 최근 기록/일정/가이드 연결
- 기록: 생성, 수정, 상세, 삭제, 타임라인, 이미지 업로드 큐
- 일정: 생성, 수정, 상세, 삭제, 완료 처리, 반복/리마인더 구조
- 날씨: 현재 위치 기반 날씨/대기질/활동 가이드
- 장소 탐색: 주변 산책 장소, 펫동반 장소 검색/상세
- 반려동물과 여행: 지역/카테고리 기반 여행지 탐색/상세
- 가이드: 일반 사용자용 가이드 목록/상세, 관리자용 가이드 관리 화면
- More/설정: 전체 메뉴, 닉네임 변경, 비밀번호 변경, 로그아웃, 계정 삭제

다만 현재 코드는 "기능 영역이 넓게 펼쳐진 상태"이지, 모든 영역이 서비스급 완성도에 도달한 상태는 아니다.  
특히 실제 운영 기준으로 보면 아래 성격이 혼재한다.

- 비교적 구현이 깊은 영역: 인증 기본 흐름, 펫 생성/수정, 기록, 타임라인, 일정 CRUD, 날씨, 가이드, 홈 통합
- 부분 완성 영역: 장소 탐색, 반려동물과 여행, 홈 위젯/운영 연동, 관리자 기능 범위
- 미완성 영역: 소셜 로그인, 비밀번호 재설정, 약관 문서 진입, 건강 리포트, 커뮤니티, 게스트북, 산책 동선 추천, 운영용 신뢰 레이어

요약하면 현재 프로젝트는 "생활관리형 반려동물 앱의 주요 뼈대와 다수의 실제 화면/로직은 존재하지만, 서비스급 배포 전 단계에서 검증용 기능과 운영 미완성 기능이 섞여 있는 상태"로 판단된다.

## 2. 앱 첫 시작 플로우

앱 최초 실행부터 실제 코드 기준 흐름은 아래와 같다.

1. `App.tsx` 실행
- `GestureHandlerRootView`, `SafeAreaProvider`, `AppProviders`, `NavigationContainer`, `GlobalToast`가 마운트된다.
- Sentry/Crashlytics 초기화 코드가 포함되어 있다.

2. `AppProviders` 부트스트랩 시작
- 인증 상태, 프로필 상태, 펫 상태, 로컬 캐시를 초기화한다.
- `supabase.auth.getSession()`으로 세션 존재 여부를 확인한다.
- 세션이 있으면 `getUser()` 재검증을 수행한다.
- 세션이 유효하면 `fetchMyProfile`, `fetchMyPets`를 호출해 프로필과 펫 목록을 불러온다.
- 선택된 펫 로컬 상태와 펫 캐시를 hydrate 한다.
- 앱이 활성화될 때 미처리 동의서 flush, 메모리 이미지 업로드 큐 처리도 연결되어 있다.

3. `RootNavigator`의 시작 화면은 `Splash`
- 실제 `Splash` 라우트는 `src/screens/Home/HomeScreen.tsx`가 담당한다.
- 이 화면은 별도 부트 로직을 기다린 뒤 다음 라우트로 `reset` 이동한다.

4. `resolveBootRoute` 기준 분기
- 비로그인 상태면 `AppTabs`로 이동한다.
- 로그인 상태인데 닉네임이 없으면 `NicknameSetup`으로 이동한다.
- 로그인 상태이고 닉네임은 있지만 펫이 0마리면 `PetCreate`로 이동한다.
- 로그인 상태이고 닉네임과 펫이 모두 있으면 `AppTabs`로 이동한다.

5. `AppTabs` 진입 후 홈 분기
- 홈 탭(`MainScreen`)은 guest/logged-in 화면을 분리해서 렌더링한다.
- 로그인 상태 홈에서는 포커스 시점에 다시 닉네임/펫 상태를 확인하고, 누락 시 `NicknameSetup` 또는 `PetCreate`로 강제 이동한다.

즉 현재 첫 진입 흐름은 다음과 같이 정리된다.

- 앱 실행
- 부트스트랩
- 세션 확인
- 프로필/닉네임 확인
- 펫 목록 확인
- 비로그인: 게스트 홈
- 로그인 + 닉네임 미설정: 닉네임 설정
- 로그인 + 닉네임 설정 완료 + 펫 없음: 펫 생성
- 로그인 + 닉네임 + 펫 존재: 홈 탭 진입

이 흐름은 코드상 명확하게 존재하며, 현재 앱 전체의 핵심 진입 가드 역할을 한다.

- 앱 시작 / Splash / bootstrap
- Auth
- 회원가입 / 로그인 / 닉네임 설정
- 펫 등록 / 펫 수정 / 멀티펫 선택
- 홈
- 날씨
- 산책/주변 장소 탐색
- 펫동반 장소 탐색
- 반려동물과 여행
- 기록 생성 / 수정 / 상세 / 목록
- 타임라인
- 일정 / 기념일
- 가이드
- 관리자 기능 일부
- More / 설정 / 전체메뉴
- 공통 UI / 네비게이션 / store / services / storage

단, 위 목록은 "도메인 코드가 존재한다"는 뜻이고, 모두 서비스급 완성이라는 뜻은 아니다.

## 4. 각 도메인별 상세 구현 현황

### 4.1 앱 시작 / Splash / bootstrap

#### 구현 완료
- 앱 시작 시 전역 Provider, NavigationContainer, GlobalToast 연결
- 세션 확인, 유저 재검증, 프로필 조회, 펫 조회 부트스트랩
- Splash 대기 후 부트 결과에 따라 라우트 리셋 이동
- 로그인/닉네임/펫 존재 여부에 따른 진입 가드

#### 부분 완료 / 구조만 있음 / 검증 필요
- 부트 중 네트워크 실패, Supabase 일시 장애, 부분 데이터 실패 상황에 대한 실기기 QA 확인이 필요
- 로컬 캐시 hydrate와 서버 실데이터 불일치 시나리오에 대한 장기 검증이 필요

#### 미완료 / 다음 작업 필요
- 부트 실패 사유를 사용자에게 명확히 설명하는 장애 대응 UX는 충분하지 않음

### 4.2 Auth / 회원가입 / 로그인 / 닉네임 설정

#### 구현 완료
- 이메일/비밀번호 회원가입 화면 존재
- 이메일/비밀번호 로그인 화면 존재
- 닉네임 설정 화면 존재
- 닉네임 중복 확인 RPC 연동 존재
- 회원가입 시 약관/개인정보/마케팅 동의 상태 로컬 저장 및 서버 flush 구조 존재
- 로그아웃, 계정 삭제, 비밀번호 변경 기능 존재
- 세션 기반 부트스트랩 및 로그인 후 Splash 재진입 구조 존재

#### 부분 완료 / 구조만 있음 / 검증 필요
- 이메일 인증 필요 시 SignIn으로 보내는 흐름은 있으나, 인증 완료 복귀 UX는 강하지 않음
- 닉네임 초안 복구는 구현되어 있으나, 다중 디바이스/세션 충돌 정책은 없음
- 닉네임 변경은 More 화면에서 로컬 제한 정책을 사용하고 있어 서버 강제 정책과 일치 여부 검증이 필요

#### 미완료 / 다음 작업 필요
- 소셜 로그인 버튼은 Alert placeholder 상태
- 비밀번호 재설정은 Alert placeholder 상태
- 약관/개인정보 문서 상세 진입은 Alert placeholder 상태
- 인증 운영 기준으로 보면 계정 복구, 인증 메일 재전송, 로그인 실패 세분화 UX가 부족함

### 4.3 펫 등록 / 펫 수정 / 멀티펫

#### 구현 완료
- 펫 생성 화면 존재
- 펫 생성 시 이름, 종, 성별, 중성화, 생일, 입양일, 체중, 품종, 성격/취향성 필드, 추모 여부 등 다수 필드 입력 가능
- 펫 프로필 이미지 업로드 구조 존재
- 펫 수정 화면 존재
- 멀티펫 목록 저장 및 선택 펫 전환 store 구조 존재
- 펫 생성 완료 후 환영 전환 화면 존재
- 펫 미등록 시 `PetCreate` 강제 진입 가드 존재

#### 부분 완료 / 구조만 있음 / 검증 필요
- 멀티펫 선택/전환은 가능하지만, 대량 펫 보유 상황 UX와 QA는 확인 필요
- 펫 이름 변경 횟수 제한은 로컬 기반 처리라 디바이스 변경 시 일관성 보장이 약함
- 펫 아바타 업로드와 수정 흐름은 존재하지만 실스토리지 운영 검증이 더 필요

#### 미완료 / 다음 작업 필요
- 펫 삭제 기능이 코드상 확인되지 않음
- 펫 권한/공유 구조, 보호자 다계정 구조가 없음
- 서비스급 기준으로는 펫 데이터 정합성 복구 수단과 운영용 수정 도구가 없음

### 4.4 홈

#### 구현 완료
- 게스트 홈과 로그인 홈 분리 구현
- 로그인 홈에서 선택 펫 기준 최근 기록/일정/가이드 진입 제공
- 홈에서 펫 전환, 기록 상세, 기록 작성, 일정 목록/생성, 펫 수정, 날씨, 가이드 상세 이동 가능
- 주간 요약 계산, 오늘 사진 회상, 홈 위젯 스냅샷 생성 구조 존재

#### 부분 완료 / 구조만 있음 / 검증 필요
- 홈은 여러 도메인을 묶는 허브로서 구조는 크지만, 각 카드의 실운영 데이터 신뢰도와 빈 상태 UX는 도메인별 검증이 더 필요
- 안드로이드 위젯 브리지 연동은 구조가 있으나 실기기 안정성 확인이 더 필요

#### 미완료 / 다음 작업 필요
- 홈 자체의 관리자/운영 지표, 장애 fallback, 개인화 품질 검증이 부족함

### 4.5 날씨

#### 구현 완료
- 현재 위치 기반 날씨/대기질 조회 구조 존재
- Open-Meteo 연동 존재
- 위치 라벨 계산, district reverse geocode, 캐시/미리보기 fallback 존재
- 날씨 인사이트 화면 존재
- 날씨 기반 실내 활동 추천 화면 존재
- 활동 가이드 상세 화면 존재
- 실내 활동 후 기록 생성 화면 존재

#### 부분 완료 / 구조만 있음 / 검증 필요
- 위치 조회 실패 시 preview/fallback 처리가 있으나 지역 정확도와 실기기 체감 품질 검증이 필요
- iOS 위치 권한 처리가 실제 OS 권한 요청이 아니라 단순 granted 반환 형태라 운영 기준으로는 검증 부족

#### 미완료 / 다음 작업 필요
- 권한 거부/제한/정밀 위치 실패 케이스를 서비스급으로 다듬을 필요가 있음
- 날씨 결과 품질 모니터링 및 지역 오차 대응 수단이 없음

### 4.6 산책 / 주변 장소 탐색

#### 구현 완료
- 주변 산책 장소 목록 화면 존재
- 검색, 현재 위치 기반 추천, 정렬, 상세 화면 진입 구조 존재
- Kakao Local 기반 검색 서비스 구현 존재

#### 부분 완료 / 구조만 있음 / 검증 필요
- 현재 구현은 "주변 산책 장소/후보 탐색"에 가깝다
- 코드상 산책 경로 계산, 동선 추천, 내비게이션 엔진은 확인되지 않았다
- 현재 메뉴 문구가 실제 구현 범위를 다소 크게 표현하고 있어 사용자 기대치와 충돌 위험이 있다

#### 미완료 / 다음 작업 필요
- 산책 동선 추천 기능은 실제 코드 기준 미완료
- 즐겨찾기, 최근 본 장소, 사용자 제보, 운영 검수 흐름이 없음

### 4.7 펫동반 장소 탐색

#### 구현 완료
- 펫동반 장소 목록/상세 화면 존재
- 검색, 정렬, 상세, 외부 지도 열기 구조 존재
- Kakao Local 검색 결과와 Supabase 메타 테이블 병합 시도 구조 존재
- 검증 상태(`admin-verified`, `keyword-inferred` 등) 표현 모델이 존재

#### 부분 완료 / 구조만 있음 / 검증 필요
- 메타 테이블 조회 실패 시 Kakao 결과만으로 동작하도록 설계되어 있다
- 즉, 운영용 장소 메타 DB가 없어도 앱은 돌아가지만, 펫동반 정책 정확도는 낮아질 수 있다
- 서비스 메타 테이블이 실제 운영 데이터로 충분히 채워져 있는지는 코드만으로 확인할 수 없다

#### 미완료 / 다음 작업 필요
- 신뢰 레이어 DB 운영 체계, 사용자 신고, 관리자 검수 UI가 없다
- 북마크/저장/리뷰/방문기록 기능이 없다
- 서비스급 기준으로는 "실제 반려동물 동반 가능 여부"를 강하게 보증하기 어렵다

### 4.8 반려동물과 여행

#### 구현 완료
- 여행지 목록 화면 존재
- 지역/카테고리/검색 기반 필터 구조 존재
- TourAPI 기반 리스트/상세 조회 구현 존재
- 상세 화면, 지도 미리보기, 외부 지도 열기 구조 존재
- 장소 유형, 반려동물 허용 상태 후보 모델이 존재

#### 부분 완료 / 구조만 있음 / 검증 필요
- 현재 런타임은 TourAPI 후보 레이어 중심이다
- trust layer 관련 타입/모델은 정의되어 있으나, 실제 운영 데이터와 관리자 워크플로는 코드상 완성되지 않았다
- 반려동물 허용 여부는 `possible` 또는 `check-required` 수준 추정이 많고, 강한 검증 소스는 제한적이다

#### 미완료 / 다음 작업 필요
- 신뢰 레이어 DB, 사용자 제보, 운영 검수, 최종 확정 상태 관리가 필요
- 즐겨찾기/저장/여행 컬렉션/공유 기능이 없다
- 서비스급 배포 기준으로는 데이터 신뢰도 보강 없이는 보수적 운영이 필요하다

## 8. 서비스급 기준 주요 리스크

### 8.1 사용자 흐름 완성도 리스크
- 회원가입 이후 이메일 인증/복귀 흐름이 충분히 매끈하지 않다
- 닉네임/펫 생성 이후 핵심 온보딩은 있으나, 예외 케이스 복구 UX는 약하다

### 8.2 데이터 신뢰도 리스크
- 펫동반 장소와 여행 데이터는 현재 후보 탐색 수준이 강하다
- "반려동물 동반 가능"을 서비스가 강하게 보증하기 어렵다

### 8.3 플랫폼 완성도 리스크
- 일정 알림이 iOS 중심이며 Android는 미완성이다
- 위치 권한 처리도 iOS에서 서비스급 완성도가 낮다

### 8.4 운영 리스크
- 관리자 기능이 가이드 CMS에 편중되어 있다
- 장소, 여행, 신고, 검수, 제휴, 데이터 정정 같은 운영 기능이 없다

### 8.5 상태 관리 리스크
- auth/pet/record/schedule와 업로드 큐, signed URL 캐시가 얽혀 있어 회귀 리스크가 있다
- 낙관적 업데이트와 지연 업로드가 많아 실기기 QA 의존도가 높다

### 8.6 타입 안정성 리스크
- 현재 `yarn tsc --noEmit` 기준 타입 에러가 1건 존재한다
- `src/screens/Pets/PetCreateScreen.tsx:928`에서 `draft.breed`의 `string | undefined` 값이 `setBreed`에 바로 전달되고 있다
- 즉, 현재 코드베이스는 타입체크 완전 통과 상태가 아니다

### 8.7 배포 가능성 리스크
- 지금 상태는 내부 검증 또는 제한적 베타 수준으로는 접근 가능하나, 서비스급 일반 배포 기준으로는 운영/신뢰/알림/설정/관리도구가 부족하다

## 9. 기술 부채 / 구조 부채

- `MoreScreen`과 실제 사용 중인 `MoreDrawerContent` 공존
- 기록 이미지 구조에서 legacy 필드와 신규 테이블 병행
- 장소/여행 도메인에서 후보 탐색 로직과 향후 신뢰 레이어 구조가 함께 존재해 모델 복잡도 증가
- 홈이 많은 도메인을 한 화면에 통합해 결합도가 높다
- 닉네임 변경/펫 이름 변경 제한 일부가 로컬 정책 기반이라 서버 기준과 분리될 위험이 있다
- iOS/Android 기능 완성도 차이가 존재한다
- 운영도구보다 사용자 기능 화면이 먼저 넓어져 운영 부채가 누적되고 있다
- 모니터링 설정은 존재하지만 운영 장애 분류 체계까지 정리된 상태로 보이지 않는다

## 13. 최종 판단

현재 Nuri는 "앱의 뼈대만 있는 초기 상태"는 아니다.  
인증, 펫, 기록, 일정, 날씨, 가이드, 홈 통합 등은 실제로 상당 부분 구현되어 있다.

하지만 동시에 "서비스급 완성"으로 보기도 어렵다.  
이유는 아래와 같다.

- 미완성 기능이 사용자 노출 메뉴 안에 아직 남아 있다
- 장소/여행 도메인의 데이터 신뢰 레이어가 약하다
- Android 일정 알림이 미완성이다
- 운영/검수/admin 도구가 부족하다
- 타입체크가 아직 완전히 깨끗하지 않다

따라서 현재 상태는 아래처럼 보는 것이 가장 보수적이다.

- 인증/펫/기록/일정/날씨/가이드: 기능 구현이 상당히 진행된 베타 전 단계
- 장소 탐색/여행: 실사용 탐색은 가능하지만 운영 신뢰도 보강 전 단계
- 관리자/운영/설정: 서비스급 운영 준비 미완성
- 전체 프로젝트 수준: "기능 폭은 넓지만, 서비스 배포 기준 마감과 운영 체계 보강이 아직 필요한 상태"

## 14. 즉시 수정이 필요한 실제 코드 리스크

### 14.1 타입체크 실패
- 현재 `yarn tsc --noEmit`가 실패한다
- `src/screens/Pets/PetCreateScreen.tsx`에서 `draft.breed`의 `string | undefined` 값을 `setBreed`에 직접 전달하고 있다
- 이는 단일 오류 1건으로 보이지만, 기본 타입 안정성 기준을 아직 만족하지 못한다는 뜻이다

### 14.2 Android 일정 알림 미완성
- 일정 CRUD는 구현되어 있지만 `src/services/schedules/notifications.ts` 기준 실제 OS 알림 예약은 iOS만 지원한다
- Android는 권한 확인/요청만 있고 `upsertScheduleNotification`은 `unsupported`를 반환한다
- 일정 기능을 사용자에게 제공하면서 플랫폼별 체감 기능이 달라지는 상태다

### 14.3 사용자 노출 메뉴와 실제 구현 범위 불일치
- 게스트북 탭은 실제 하단 탭에 노출되지만 현재 placeholder 화면이다
- More 메뉴의 건강 기록 리포트, 커뮤니티, 알림 설정은 `soon` placeholder다
- More 메뉴의 `우리동네 산책 / 산책 동선 추천`은 실제 코드상 장소 탐색이며 동선 추천 엔진은 확인되지 않았다
- More 메뉴의 `테마 설정`은 별도 테마 설정 화면이 아니라 펫 프로필 화면으로 연결된다
- 즉, 현재는 사용자에게 노출되는 메뉴 중 일부가 기능 명칭과 실제 동작이 불일치한다

## 15. 잠재적 리스크

### 15.1 앱 시작 / 상태 동기화 리스크
- 부트스트랩이 세션, 프로필, 펫, 로컬 캐시 hydrate를 한 번에 다루기 때문에 부분 실패 시 회복 경로가 복잡하다
- `AppProviders`에서 인증/펫/기록/일정/동의서 flush/이미지 업로드 큐까지 연결되어 있어 부트 시점 결합도가 높다
- 캐시가 오래됐거나 로컬 상태가 남아 있을 때 일시적으로 잘못된 선택 펫이나 이전 데이터가 보일 가능성을 완전히 배제하기 어렵다

### 15.2 네비게이션 정합성 리스크
- `Splash`가 실제 전용 Splash 컴포넌트가 아니라 `HomeScreen`을 사용한다
- `MoreTab`은 실제 화면 이동이 아니라 오버레이 드로어를 여는 특수 동작이다
- 장소/여행 리스트 화면에서 뒤로가기 시 드로어를 다시 여는 커스텀 처리가 들어가 있다
- 이런 구조는 일반적인 스택/탭 기대와 다르므로, Android 하드웨어 뒤로가기와 포커스 복귀에서 회귀 가능성이 있다

### 15.3 실기기 UX 리스크
- iOS 위치 권한 계층은 실제 권한 상태 조회/요청이 아니라 즉시 `granted`를 반환한다
- 따라서 iOS에서 사용자가 위치 권한을 거부했거나 제한한 상황의 UX가 실제 동작과 어긋날 수 있다
- 홈, 기록, 일정, 날씨, 장소탐색이 모두 실기기 네트워크/권한 상태에 민감해 QA 범위가 넓다

### 15.4 기록/업로드 상태 관리 리스크
- 기록은 낙관적 생성/수정과 이미지 업로드 큐를 함께 사용한다
- 업로드 지연, superseded edit plan, signed URL 캐시, legacy 이미지 필드 호환이 한 도메인에 함께 존재한다
- 구조 자체는 강하지만 회귀 테스트가 부족하면 "텍스트는 저장됐는데 이미지가 늦게 반영되는 상태", "수정 중 이미지 계획 충돌" 같은 문제가 발생할 수 있다

### 15.5 일정 도메인 리스크
- 일정 레코드에는 반복, 리마인더, 외부 캘린더 관련 필드가 있으나 실제 런타임 완성도는 iOS 로컬 알림 중심이다
- Android 사용자 경험은 저장은 되지만 기기 알림이 오지 않는 상태가 될 수 있다
- 서비스 설명과 실기능이 어긋나면 사용자가 일정 기능을 신뢰하지 않게 될 수 있다

### 15.6 위치/날씨/외부 API 의존 리스크
- 날씨는 Open-Meteo, 위치 라벨은 Kakao reverse geocode, 장소탐색은 Kakao Local, 여행은 TourAPI에 의존한다
- 외부 API 장애, 응답 품질 저하, 할당량 이슈, 지역 편차가 발생하면 앱 핵심 기능 품질이 크게 흔들릴 수 있다
- 현재 구조는 fallback이 있으나, 데이터 품질을 운영 기준으로 검증하거나 보정하는 내부 레이어는 아직 약하다

### 15.7 펫동반 장소 탐색 리스크
- `placeMeta` 조회가 실패해도 Kakao 검색 결과만으로 계속 동작하게 설계되어 있다
- 가용성은 확보되지만, 실제 반려동물 동반 가능 여부를 잘못 추정할 위험이 있다
- 운영 메타 DB, 신고, 관리자 검수, 정정 수단이 없으면 잘못된 장소 정보가 장기간 노출될 수 있다

### 15.8 반려동물과 여행 리스크
- travel 도메인은 trust layer 타입과 계획은 존재하지만 런타임은 TourAPI 후보 탐색 비중이 크다
- 현재 라벨링은 "동반 확인"보다 "가능성 있음/확인 필요" 성격이 강하다
- 운영 검수 없이 사용자에게 확정 정보처럼 받아들여질 가능성이 있어, 실제 서비스 노출 문구와 정책 설계가 중요하다

### 15.9 관리자/운영도구 부재 리스크
- 현재 확인 가능한 관리자 기능은 가이드 CMS에 사실상 한정된다
- 장소 검수, 여행 검수, 신고 처리, 데이터 정정, 운영 대시보드, 장애 대응 툴이 없다
- 서비스 확장보다 운영도구가 늦게 따라가면 품질 관리가 어려워진다

### 15.10 사용자 계정/프로필 정책 리스크
- 닉네임 변경 제한, 펫 이름 변경 횟수 제한 일부가 로컬 저장 정책에 의존한다
- 기기 교체, 앱 재설치, 다중 디바이스 사용 시 서버 기준 정책과 어긋날 수 있다
- 서비스 정책으로 강제해야 하는 규칙이 로컬에 남아 있으면 악용이나 혼선 가능성이 있다

### 15.11 구조 부채 리스크
- `MoreScreen`과 `MoreDrawerContent`가 공존하고, 일부 메뉴는 실제 기능보다 앞서 노출되어 있다
- 홈은 여러 도메인을 통합하는 허브라 변경 영향 범위가 크다
- 도메인 수가 빠르게 늘었는데 전역 네비게이션, 상태, 운영 구조 정리는 아직 따라오지 못한 상태다

### 15.12 배포 리스크
- 현재 상태는 기능 검증과 내부 베타를 위한 폭넓은 구현 상태로는 의미가 있다
- 그러나 일반 서비스 배포 관점에서는 타입 안정성, Android 알림, 운영도구, 데이터 신뢰도, placeholder 노출 문제를 먼저 정리해야 한다
