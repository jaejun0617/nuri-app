## 1. 프로젝트 정체성 / 서비스 개요

### 1.1 NURI가 무엇인가

NURI는 현재 코드 기준으로 단순 추모 앱이 아니라, 반려동물별 생활 관리와 기록을 중심으로 확장된 생활형 모바일 서비스다.

현재 앱이 실제로 제공하는 축은 아래다.

- 반려동물 프로필 관리와 멀티펫 컨텍스트
- 기록/타임라인/이미지 관리
- 일정/기념일/행동 기록
- 날씨 기반 행동 가이드
- 위치 기반 산책 및 펫동반 장소 탐색
- 반려동물과 여행 탐색
- 집사 꿀팁 가이드 소비 및 관리자 브릿지 운영
- 커뮤니티 읽기/쓰기/댓글/신고

README와 일부 홈 문구는 감성 메모리얼 플랫폼 정체성을 유지한다. 하지만 실제 제품 구조는 감성 메모리얼 위에 생활 관리, 일정, 탐색, 가이드, 커뮤니티가 올라간 복합 서비스다.

### 1.2 현재 서비스 진화 방향

초기에는 기록/추억/반려동물 중심 구조가 강했다. 현재는 아래처럼 진화했다.

- 홈이 단순 인사 화면이 아니라 생활 허브가 됐다.
- 기록은 `memory_images`와 업로드 복구 큐를 가진 운영형 구조로 확장됐다.
- 산책/펫동반/여행은 외부 API 기반 탐색 구조로 확장됐다.
- 가이드는 읽기 전용을 넘어서 검색, 개인화, 관리자 브릿지, 이벤트 적재까지 들어갔다.
- 커뮤니티는 목록/상세/작성/수정/댓글/신고까지 열렸지만 아직 운영 도구와 정책 완성도가 부족하다.

### 1.3 핵심 사용자 가치

- 반려동물별로 홈, 기록, 일정 컨텍스트를 분리해 관리할 수 있다.
- 홈에서 최근 기록, 일정, 날씨, 가이드를 한 번에 소비할 수 있다.
- 현재 위치와 검색을 기준으로 생활권 탐색을 할 수 있다.
- 여행 후보를 반려동물 동반 관점에서 탐색할 수 있다.
- 개인 기록과 공개 커뮤니티를 분리해 사용할 수 있다.

## 2. 현재 기술 아키텍처 총정리

### 2.1 앱 런타임 구조

- 엔트리: `App.tsx`
- 글로벌 provider: `src/app/providers/AppProviders.tsx`
- 부트 결정 로직: `src/services/app/boot.ts`
- 루트 네비게이션: `src/navigation/RootNavigator.tsx`
- 메인 탭: `src/navigation/AppTabsNavigator.tsx`

실제 구조는 `App.tsx -> AppProviders -> NavigationContainer -> RootNavigator`다.  
`App.tsx`는 모니터링 초기화와 provider 조립만 하고, 세션/프로필/펫 bootstrap은 `AppProviders`가 맡는다.

### 2.2 navigation 구조

- Splash: `HomeScreen`
- Auth: `SignIn`, `SignUp`, `NicknameSetup`, `WelcomeTransition`
- Main tabs: `HomeTab`, `TimelineTab`, `CommunityTab`, `GuestbookTab`, `MoreTab`
- 기록/일정/날씨/가이드/장소탐색/여행/커뮤니티는 모두 root stack에 연결
- More는 실제 탭 화면이 아니라 drawer overlay

중요:
- `GuestbookTab`은 실제 placeholder 화면이다.
- `CommunityTab`은 별도 stack으로 분리되어 공통 `CommunityStackHeader`를 사용한다.
- `RecordCreate`는 독립 stack screen이다. 탭 안의 폼이 아니다.

### 2.3 store 구조

- `authStore`: 세션, 로그인 상태, role, 닉네임, boot 상태
- `petStore`: 펫 목록, 선택 펫, hydrate, 로컬 캐시
- `recordStore`: 기록 엔티티, timeline ids, refresh, optimistic update
- `scheduleStore`: 일정 캐시
- `weatherStore`: 좌표별 날씨 TTL 캐시
- `communityStore`: 목록/상세/댓글/entity cache/like/report
- `uiStore`: toast, More drawer 상태

상태 관리의 핵심은 Zustand다. 도메인별 스토어 분리는 되어 있지만, `AppProviders`와 일부 화면이 여러 store를 동시에 만지는 구간이 있어 결합도가 낮다고 보긴 어렵다.

### 2.4 service 구조

- `services/supabase/*`: 인증, 프로필, 펫, 기록, 일정, 커뮤니티, 가이드, 스토리지
- `services/location/*`: 권한, 현재 위치, 행정동
- `services/locationDiscovery/*`: Kakao Local 기반 산책/펫동반 공통 레이어
- `services/petTravel/*`: TourAPI 기반 여행 탐색
- `services/weather/*`: Open-Meteo
- `services/guides/*`: 가이드 읽기/검색/개인화/이벤트
- `services/legal/consents.ts`: 약관/개인정보/마케팅 동의 snapshot
- `services/monitoring/*`: Sentry/Crashlytics
- `services/local/*`: draft, 최근 검색, 업로드 큐, cleanup queue

### 2.5 외부 API 구조

실제 코드에서 확인되는 외부 의존:

- Supabase Auth / Postgres / Storage / RPC
- Kakao Local REST API
- TourAPI `KorPetTourService2`
- Open-Meteo forecast / air-quality
- Sentry
- Firebase Crashlytics

중요:
- `src/config/runtime.ts`는 저장소에 없고 `runtime.example.ts`만 있다.
- Kakao/TourAPI 키는 클라이언트 런타임 주입 구조다.
- 서버 프록시나 API gateway 계층은 없다.

## 3. 앱 시작부터 끝까지의 주요 사용자 플로우

### 3.1 앱 시작

1. 앱 실행
2. `App.tsx`에서 `enableScreens`, `initMonitoring` 실행
3. `AppProviders`에서 auth/pet hydrate, 세션 검증, 프로필/펫 fetch
4. `HomeScreen(Splash)`가 boot 완료와 최소 splash 시간을 기다림
5. `resolveBootRoute()`로 다음 진입 라우트 결정

### 3.2 bootstrap / session 확인 / 가드

`AppProviders`가 하는 일:

- AsyncStorage 기반 auth profile hydrate
- Supabase `getSession`, `getUser`로 세션 유효성 확인
- 프로필 조회
- 펫 목록 hydrate/fetch
- 선택 펫 scoped bootstrap
- 동의 snapshot flush
- memory image 업로드 큐 복구
- community 이미지 cleanup queue flush

### 3.3 닉네임 / 펫 여부에 따른 분기

- 로그아웃 상태: `AppTabs`
- 로그인 + 닉네임 미확정: `NicknameSetup`
- 로그인 + 닉네임 완료 + 펫 0마리: `PetCreate`
- 로그인 + 닉네임 완료 + 펫 존재: `AppTabs`

### 3.4 홈 / 탭 / 드로어

- `HomeTab`: `MainScreen`
  - 로그인 상태면 `LoggedInHome`
  - 비로그인이면 `GuestHome`
- `TimelineTab`: 기록 중심 stack
- `CommunityTab`: 커뮤니티 stack
- `GuestbookTab`: placeholder
- `MoreTab`: 실제 화면 이동이 아니라 More drawer open

### 3.5 주요 도메인 이동 흐름

- 기록 작성: 홈/타임라인/날씨 CTA -> `RecordCreate`
- 기록 상세/수정: 타임라인/홈 최근 기록 -> `RecordDetail` -> `RecordEdit`
- 일정: 홈/More -> `ScheduleList` -> `ScheduleDetail/Edit/Create`
- 산책/펫동반 장소: 홈/More -> 리스트 -> 상세 -> 외부 지도
- 여행: More -> `PetTravelList` -> `PetTravelDetail` -> 외부 지도
- 가이드: 홈 추천/More -> `GuideList` -> `GuideDetail`
- 커뮤니티: 탭 -> 목록 -> 상세 -> 작성/수정/댓글/신고
- 설정/계정관리: More drawer -> 닉네임 변경, 비밀번호 변경, 테마 변경, 로그아웃, 회원탈퇴

### 3.6 설정 / 계정관리 흐름

실제 가능한 것:

- 닉네임 변경
- 비밀번호 변경
- 로그아웃
- 회원탈퇴 RPC 호출
- 펫 테마 변경

실제 없는 것:

- 비밀번호 찾기 실구현
- 소셜 로그인
- 이용약관/개인정보처리방침 실제 문서 연결
- 계정 복구 플로우
- 고객센터/문의 채널

## 4. 전체 프로젝트 진행 내역 정리

### 4.1 앱 초기 구조

- 구현된 것: RN 앱 기본 구조, theme/provider, root stack, toolbar, More drawer
- 부분 구현: 일부 화면 스타일/헤더 일관성
- 부족한 것: 환경 분리, 빌드 채널 분리, deep link/OAuth 구조
- 현재 상태: 기본 구조는 안정적이나 운영 환경 구조는 미완성

### 4.2 인증 / 온보딩

- 구현된 것: 이메일 로그인/회원가입, 닉네임 확정, 펫 생성 가드, 동의 snapshot 저장/flush, 회원탈퇴 RPC 연동
- 부분 구현: 이메일 인증 경고, 닉네임 정책 강화, 비밀번호 변경
- 부족한 것: 비밀번호 재설정, 소셜 로그인, 약관/개인정보처리방침 실제 연결, 계정 복구, 탈퇴 후 재가입 정책
- 현재 상태: 이메일 기반 기본 인증만 실사용 가능

### 4.3 펫 등록 / 멀티펫

- 구현된 것: 펫 생성, 수정, 선택, 테마, 멀티펫 관리 화면
- 부분 구현: 프로필 관리 UX와 일부 테마/메모리얼 연계
- 부족한 것: 멀티펫 과금/entitlement/결제/제한 정책
- 현재 상태: 멀티펫은 기능상 가능하지만 수익화/권한 구조는 전혀 없음

### 4.4 기록 / 타임라인

- 구현된 것: 기록 CRUD, 카테고리/태그/감정, 타임라인, 이미지 갤러리, `memory_images`, signed URL 캐시, 업로드 큐 복구
- 부분 구현: 썸네일 worker 운영 검증, 대량 데이터 실기기 QA
- 부족한 것: 장기 운영 백필/정리 배치, 관측 대시보드, 운영 도구
- 현재 상태: 핵심 도메인 중 가장 구현이 깊은 편

### 4.5 일정 / 기념일

- 구현된 것: 일정 CRUD, 홈 연동, 색/아이콘/카테고리 매핑
- 부분 구현: 알림 정책 일부, 프레젠테이션 정리
- 부족한 것: 푸시/로컬 알림 실서비스 운용, 반복 일정 고도화, 백그라운드 신뢰성
- 현재 상태: 사용 가능하지만 운영형 일정 서비스로 보긴 이르다

### 4.6 홈 / 날씨

- 구현된 것: 로그인/게스트 홈 분기, 최근 기록, 주간 요약, 위젯 스냅샷, Open-Meteo 기반 날씨/대기질 카드
- 부분 구현: 날씨 행동 기록 연동
- 부족한 것: 외부 API 장애 fallback 고도화, 위젯/날씨 실기기 장기 검증
- 현재 상태: UX는 풍부하지만 운영 관찰 포인트가 부족하다

### 4.7 산책 / 위치 탐색

- 구현된 것: 현재 위치 기반 산책/검색, 행정동, 지도 미리보기, 외부 지도 열기
- 부분 구현: placeMeta canonical 보강

## 6. 리팩토링 필요 영역 총정리

- `AppProviders`는 auth/pet/record/schedule/community/local queue까지 동시에 다뤄 결합도가 높다.
- `AsyncStorage` 기반 로컬 상태와 서버 source of truth 경계가 명확하지 않은 구간이 있다.
- `runtime.ts`, `supabase/config.ts`, `monitoring/config.ts`가 저장소 바깥 로컬 파일이라 환경 분리 정책 문서화가 필요하다.
- More drawer가 계정관리, 도메인 진입, 임시 운영 기능을 함께 안고 있어 책임이 넓다.
- 커뮤니티는 store/entity cache와 화면 로직이 아직 복잡하며, snapshot/모더레이션 구조가 덜 닫혔다.
- `GuestbookTab` placeholder가 실제 탭으로 남아 있다.
- 장소/여행 탐색은 외부 API 후보 레이어와 내부 canonical 메타 레이어가 완전히 분리되지 않았다.
- 문서상 준비된 SQL과 앱 런타임 연결 정도가 서로 다르다.
- 업로드 큐, signed URL cache, cleanup queue는 유용하지만 장기 운영 관측 포인트와 재처리 도구가 없다.
- 공통 헤더/폼/액션 버튼/에러 표시 규칙이 도메인별로 완전히 통일되지는 않았다.
- 대량 데이터 관점에서 홈/타임라인/커뮤니티/가이드 검색은 추가 성능 QA가 필요하다.

이 장은 4차 시점의 총괄 리스크 기록이다.

### 7.5 성능 리스크

- 이미지 signed URL 생성/캐시 만료

- 스택 헤더 전환 깜빡임

### 7.7 플랫폼 차이 리스크

- OAuth/deep link 미구현
- Android/iOS 권한, 알림, 스토리지, 서명키 운영 차이 미정리
- iOS 홈 위젯/정책/스토어 제출 요건 미정리

### 7.8 데이터 신뢰도 리스크

- 커뮤니티 pet meta live 참조 fallback
- 장소/여행 도메인 후보 데이터 신뢰도 한계
- 탈퇴/삭제 RPC가 실제 모든 연관 데이터 정리 범위를 보장하는지 추가 검증 필요

- project-memory의 우선순위와 실제 dirty worktree가 어긋날 수 있음

## 8. 보안 관점 총정리

### 8.1 Supabase Auth 보안

- 현재는 이메일/비밀번호 기반 Auth만 실구현
- `supabase-js` RN client는 `AsyncStorage`에 세션을 영구 저장
- `getSession` 뒤 `getUser` 재검증을 한 번 더 하는 점은 긍정적
- 하지만 세션 토큰 저장소가 하드닝된 secure storage가 아니라는 점은 리스크다

### 8.2 세션 처리 / 토큰 저장 / 민감정보

- 세션은 Supabase client가 `AsyncStorage`에 저장
- `authStore`는 닉네임/role만 별도 저장
- 동의 snapshot, 업로드 queue, cleanup queue도 `AsyncStorage` 사용
- 현재 구조는 편의성은 좋지만, 탈옥/루팅 기기나 디바이스 탈취 관점에서는 보수적이지 않다

### 8.3 AsyncStorage 사용 리스크

- 세션 persistence
- profile summary
- onboarding/record draft
- consent snapshot
- memory upload queue
- community cleanup queue

위 항목 중 민감도 차이가 큰데 동일 저장소에 섞여 있다.  
특히 세션과 개인정보 관련 최소화 원칙 재검토가 필요하다.

### 8.4 API 키 / runtime.ts / config 관리

- Kakao REST API 키, TourAPI 서비스 키가 앱 런타임에서 직접 사용된다.
- `runtime.ts`는 저장소 밖 로컬 파일이지만, 결국 앱 번들에 포함되는 구조다.
- 모바일 앱에 포함된 키는 추출을 전제로 봐야 한다.
- 이 구조는 호출량 남용, quota 고갈, 키 유출 리스크가 있다.

### 8.5 RLS 정책 상태와 부족한 점

좋은 점:
- 핵심 테이블에 RLS가 넓게 적용됨
- 닉네임 availability는 security definer RPC로 우회
- 커뮤니티 댓글 self-reference 검증은 recursion 회피 함수로 보강

부족한 점:
- 실제 운영 DB에 어떤 SQL이 적용됐는지 환경별 증적이 없다
- 장소/여행 Trust Layer는 문서/SQL 준비 대비 앱 연결이 낮다
- 커뮤니티 이미지 bucket은 authenticated select라, 노출 범위 재검토가 필요하다

### 8.6 Storage 업로드 보안

- `memory-images`는 private + signed URL
- `pet-profiles`는 public bucket 구조
- `community-images`는 authenticated select 정책
- `guide-images`는 public bucket

리스크:
- 경로 추측, signed URL 로그 노출, 만료 후 재요청 비용
- public bucket은 프로필/가이드 이미지 노출 통제에 약함
- community image는 인증 유저 전체가 읽을 수 있어 장기적으로 공개 범위와 일치하는지 재검토 필요

### 8.7 악성 입력 / 링크 / WebView / XSS 유사 리스크

- RN이라 웹 XSS와 동일하지는 않다
- 그래도 아래 방어가 필요하다
  - 커뮤니티 악성 텍스트/욕설/스팸 필터
  - 외부 링크 열기 전 whitelist/confirm
  - WebView 사용 시 script/redirect 제어
  - 이미지/본문 길이 제한과 서버 측 validation

현재는 커뮤니티 신고는 있으나 anti-abuse 체계는 부족하다.

### 8.8 abuse / spam / bot / rate limit

- 현재 코드/SQL 기준 rate limit, bot 방어, flood control은 없다
- 커뮤니티, 닉네임 확인 RPC, 이미지 업로드, 댓글 작성은 abuse surface다
- 출시 전 최소 서버측 호출 제한과 신고/숨김 자동화가 필요하다

### 8.9 관리자 기능 노출 리스크

- `profiles.role` 기반 관리자 가드가 있다
- 하지만 운영 화면은 앱 안에 일부 존재하고, 전용 admin 백오피스 수준은 아니다
- 앱 번들 내부에 관리자 진입 UI가 있는 구조는 role 검증 실패 시 위험이 커진다

### 8.10 로그 / 에러 리포팅 민감정보

- Sentry와 Crashlytics에 user id, email을 설정한다
- `captureMonitoringMessage`는 extras/tags를 그대로 보낼 수 있다
- signed URL, policy payload, 위치, 게시글 본문이 실수로 섞이지 않도록 로깅 규칙이 필요하다

### 8.11 계정 삭제 / 데이터 삭제 / 복구 정책

- 현재 `delete_my_account()` RPC는 `user_consent_history`, `memories`, `pet_schedules`, `pets`, `profiles`, `auth.users` 삭제를 처리
- 커뮤니티, 가이드 이벤트, 스토리지 파일, 추가 확장 테이블 정리 범위는 이 문서 기준으로 명확히 보장되지 않는다
- 따라서 “완전 삭제” 정책을 외부에 공표하기에는 아직 근거가 부족하다

### 8.12 개인정보 최소 수집 원칙

- 현재 수집 필드가 아주 많지는 않다
- 하지만 반려동물 프로필, 위치, 이미지, 기록, 커뮤니티 콘텐츠는 모두 개인 식별 가능성을 가진다
- 위치 기반 기능을 쓰는 이상 최소 수집/최소 보관/목적 고지가 필요하다

### 8.13 서비스 운영 중 꼭 필요한 보안 체크리스트

- Auth 세션 저장 하드닝 재검토
- 외부 API 키 프록시화 또는 호출 제한
- 커뮤니티 rate limit / spam 방어
- signed URL 로그 차단
- admin role 감사 로그
- 탈퇴 RPC 삭제 범위 재검증
- storage orphan cleanup 배치
- privacy/logging data minimization

## 9. 개인정보 / 정책 / 법적 준비

현재 실제 앱에는 회원가입 화면에서 약관/개인정보 동의 UI가 있지만, 실제 문서 연결은 placeholder alert다. 이 상태로는 실서비스 출시 불가로 판단한다.

출시 전 최소 필요 문서:

- 개인정보처리방침
- 서비스 이용약관
- 커뮤니티 운영정책
- 신고/차단/제재 기준
- 위치정보 또는 위치기반 기능 고지
- 계정 삭제/보관 정책
- 제3자 API 사용 고지

추가로 필요한 정책 포인트:

- 반려동물 정보와 사용자 정보의 구분
- 이미지, 게시글, 댓글 보관 기간
- 계정 삭제 시 즉시 삭제/유예 삭제/백업 보관 여부
- 날씨/위치/장소 탐색 기능의 위치 권한 안내
- 제3자 API(Kakao, TourAPI, Open-Meteo, Sentry, Firebase) 처리 고지
- 결제/구독 도입 시 환불/해지/복원 정책 문서

현재 상태 판단:

- 약관/개인정보처리방침: 미완성
- 커뮤니티 운영정책: 미완성
- 신고/제재 기준 공개문서: 미완성
- 위치정보 고지: OS permission 문구만 있고 서비스 정책 문서는 없음

## 10. 인증 전략 총정리

### 10.1 현재 구조

- 이메일 회원가입/로그인만 실구현
- 이메일 인증이 켜진 경우 session null 흐름 처리 존재
- 닉네임 확정과 펫 생성이 후속 온보딩

### 10.2 Google / Kakao 로그인 도입 시 고려사항

- 앱 스킴 / universal link / redirect URI 설계 필요
- iOS/Android deep link 보안 검토 필요
- 기존 이메일 계정과 소셜 계정 병합 정책 필요
- 중복 계정 생성 방지 정책 필요
- nickname_confirmed / pet onboarding 재진입 정책 필요

### 10.3 중복 / 연결 문제

필수 정책:

- 같은 이메일의 이메일 계정과 Google 계정을 합칠지
- Kakao는 이메일 제공 불완전 케이스를 어떻게 처리할지
- 소셜 계정 추가 연결 기능을 줄지
- 탈퇴 후 재가입 시 기존 닉네임/데이터 정책을 어떻게 할지

### 10.4 비밀번호 찾기 / 이메일 인증 / 계정 복구

- 현재 비밀번호 찾기 실구현 없음
- 이메일 인증 후 재진입만 일부 처리
- 계정 복구/문의 채널 없음

### 10.5 가장 현실적인 추천 전략

1단계:
- 이메일 로그인 유지
- 비밀번호 재설정 먼저 구현
- 실제 약관/개인정보처리방침 연결
- deep link/OAuth 기반 인프라 정리

2단계:
- Google 로그인 도입

3단계:
- Kakao 로그인 도입

이유:
- Google이 상대적으로 OAuth 표준 흐름과 문서화가 쉽고, 앱 배포 시 검증 경로를 만들기 수월하다.
- Kakao는 한국 사용자 친화성은 높지만 계정 연결/이메일 정책을 별도 설계해야 한다.

결론:
- 지금 당장 “소셜 로그인부터” 붙이는 순서는 위험하다.
- 먼저 이메일 인증/비밀번호 재설정/정책 문서/OAuth redirect 설계를 정리해야 한다.

## 11. 결제 / 구독 시스템 전략

### 11.1 현재 상태

- 결제 미구현
- 멀티펫 구독은 문서만 있음
- 인앱결제 SDK, receipt 검증, entitlement store 없음

### 11.2 구독 구조 권장

- 플랫폼 원본: App Store / Google Play receipt
- 서버 검증: 필수
- 앱 권한 source of truth: 별도 entitlement 테이블
- 앱은 `canAddMorePets` 같은 파생 상태만 사용

### 11.3 왜 앱만으로 처리하면 안 되는가

- 영수증 위변조 위험
- grace / revoke / refund / restore 처리 불가
- 기기 간 동기화 불가
- 운영자 수동 보정 불가

### 11.4 인앱결제 정책 포인트

- iOS/Android 모두 앱 내 디지털 기능은 인앱결제 원칙 준수 필요
- 무료 체험, 해지, 복구, 환불 대응 정책 필요
- 구독 상태는 서버 검증 후 Supabase에 저장

### 11.5 장기 권장 아키텍처

- `user_subscriptions`: 원본 결제 상태
- `user_entitlements`: 앱 권한 상태
- `billing_events`: 감사 로그

결론:
- 결제를 붙이기 전에는 멀티펫 제한 UX를 열지 않는 것이 안전하다.
- 과금 기능을 붙일 경우 앱보다 서버/정책/운영부터 정리해야 한다.

## 16. 결론

NURI는 “앱이 돌아가는가” 기준으로는 많은 도메인이 열려 있다.  
하지만 “안전하게 배포하고 오래 운영할 수 있는가” 기준으로는 아직 중간 단계다.

가장 중요한 사실은 아래다.

- 핵심 UX는 많이 만들어졌지만 운영 방어선이 부족하다.
- 인증/정책/보안/배포 준비가 기능 구현 속도를 따라오지 못했다.
- 외부 API 의존 도메인은 신뢰도 레이어를 아직 운영 수준으로 닫지 못했다.
- 커뮤니티는 기능보다 운영 리스크가 더 큰 상태다.

즉 다음 단계는 기능 추가가 아니라, 서비스 기준의 방어선과 운영 기반을 먼저 닫는 것이다.
