# 타임라인 메모리 이미지 통합 완성본

## 1. 문서 목적

이 문서는 NURI의 타임라인 이미지 시스템을 현재 코드와 운영 기준으로 다시 정리한 통합 문서다.

핵심 목표:

1. `memories.image_url` 중심 구조에서 `memory_images` 중심 구조로 바뀐 흐름을 정리한다.
2. 읽기/쓰기 전환, thumbnail/original 구조, preload 최적화, fallback 운영 기준을 한 번에 설명한다.
3. worker/backfill/운영 관찰 문서를 현재 구조 기준으로 하나의 참고 문서로 통합한다.
4. 남은 리스크와 다음 확장 포인트를 분명하게 남긴다.

이 문서가 기준 문서이며, 기존 설계/초안/운영 보조 문서는 보관 자료로만 본다.

## 2. 현재 상태 요약

### 2.1 완료 범위

- `memory_images`를 다중 이미지 source of truth로 채택
- 읽기 경로를 `memory_images` 우선으로 전환
- 쓰기 경로를 `memory_images` 우선 + legacy mirror 동기화로 전환
- 타임라인 썸네일 variant 해석과 preload 구조 정리
- create/edit/delete 정합성 보강
- upload queue 기반 복구 구조 반영
- fallback 운영 관찰 이벤트 정리
- worker/backfill용 SQL과 운영 가이드 정리

### 2.2 아직 남은 범위

- 운영 환경에서 fallback hit 장기 분포 관찰
- 외부 worker/edge runtime의 장기 안정성 검증
- legacy read fallback 단계적 축소 여부 판단
- orphan cleanup와 failed backlog 장기 운영 자동화

즉, 현재 구조는 `앱 런타임 전환은 완료`, `운영 리스크 축소와 장기 관찰은 진행 중`인 상태다.

## 3. 구조 변화 이력

### 3.1 초기 구조

초기에는 `public.memories.image_url / image_urls`가 사실상 단일 source of truth였다.

문제점:

- 다중 이미지 정렬과 재처리에 약함
- 타임라인 전용 썸네일을 별도 자산으로 다루기 어려움
- 원본/썸네일/재시도/백필을 운영하기 어려움
- delete/edit 시 파일 정리와 DB 정합이 화면별로 흩어지기 쉬움

### 3.2 전환 목표

전환 목표는 아래였다.

1. 이미지 메타를 `memories` 본문과 분리한다.
2. `original_path`와 `timeline_thumb_path`를 분리한다.
3. 타임라인은 가벼운 썸네일 경로를 우선 사용한다.
4. 상세는 원본 이미지를 기준으로 본다.
5. worker/backfill이 가능한 구조로 만든다.

### 3.3 현재 구조

현재는 `memory_images`가 실질적인 source of truth다.

다만 즉시 legacy를 제거하지는 않았다.

이유:

- 기존 데이터 백필 과도기
- create 직후 queue 정합 지연 구간
- runtime read failure에 대한 안전망 필요

즉, 지금 단계는 `truth는 memory_images`, `legacy는 운영 안전망`이다.

## 4. 현재 데이터 구조

### 4.1 핵심 테이블

핵심 테이블은 `public.memory_images`다.

주요 필드:

- `memory_id`
- `user_id`
- `pet_id`
- `sort_order`
- `original_path`
- `timeline_thumb_path`
- `status`
- `retry_count`
- `last_error_code`
- `last_error_message`
- `last_processed_at`
- `processing_started_at`
- `created_at`
- `updated_at`

### 4.2 variant 구조

현재 앱은 두 가지 variant를 명시적으로 쓴다.

- `original`
- `timeline-thumb`

이 구조는 `src/services/supabase/storageMemories.ts`와 `src/services/supabase/memories.ts`에서 공통으로 해석한다.

정책:

- 타임라인 목록: `timeline_thumb_path`가 준비돼 있으면 `timeline-thumb`
- 상세/원본 보기: `original_path`
- 썸네일이 아직 없으면 원본 path를 fallback으로 사용

### 4.3 대표 이미지 구조

타임라인 대표 이미지는 `sort_order`가 가장 작은 이미지 기준으로 잡는다.

정책:

- `memory_timeline_primary_images` 뷰 또는 같은 의미의 앱 해석 로직 사용
- 썸네일이 ready면 썸네일 우선
- 아니면 원본 path fallback

## 5. 읽기 구조 변화

### 5.1 현재 읽기 경로

현재 `src/services/supabase/memories.ts` 기준 읽기 흐름은 아래다.

1. `memories` row 조회
2. `memory_images`를 `memory_id` 기준으로 조회
3. `original_path`, `timeline_thumb_path`, `status`를 해석
4. 타임라인용 대표 path와 상세용 imagePaths를 분리
5. `memory_images` 조회 실패 또는 row 부재 시 legacy fallback 이유를 기록

### 5.2 fallback 이유 분리

현재 운영 관찰을 위해 fallback 이유를 분리한다.

- `schema_fallback`
- `runtime_read_failure`
- `recent_create_gap`
- `legacy_only_row`

이 이벤트는 아래 메시지로 모니터링한다.

- `memory_images_read_fallback`
- `memory_images_legacy_read_used`

핵심 효과:

- fallback이 발생했는지뿐 아니라 왜 발생했는지 구분할 수 있다.
- release에서 fallback 축소 여부를 정량적으로 판단할 수 있다.

## 6. 쓰기 구조 변화

### 6.1 create/edit

현재 create/edit는 `memory_images` 중심으로 정합을 맞춘다.

정책:

- 새 이미지 업로드는 storage path 기준으로 관리
- `memory_images` row를 우선 맞춘다
- legacy `image_url/image_urls`는 mirror로만 유지한다
- queue가 남은 정합 작업과 복구를 이어받는다

### 6.2 delete

delete는 사용자 관점 정합성과 orphan 파일 최소화를 우선한다.

정책:

- DB row와 storage file cleanup 책임을 서비스 계층에 모은다
- 화면별 중복 삭제를 줄인다
- 원본과 썸네일 cleanup을 같이 고려한다

## 7. worker / backfill 구조

### 7.1 worker 역할

worker는 앱 코드 안에 있지 않고, 외부 Edge Function 또는 별도 worker 런타임으로 분리하는 방향이다.

worker 책임:

1. `pending` 또는 재시도 가능한 `failed` row를 잡는다.
2. `original_path` 원본을 다운로드한다.
3. `timeline_thumb_path`를 생성한다.
4. 성공 시 `ready`, 실패 시 `failed`로 상태를 반영한다.
5. `retry_count`, `last_error_*`, `processing_started_at`로 운영 가능성을 보강한다.

### 7.2 backfill 역할

backfill은 기존 `memories.image_url / image_urls` 데이터를 `memory_images.original_path`로 옮기는 작업이다.

정책:

- `memory_images` row가 아직 없는 memory만 대상
- 새 row는 `status='pending'`, `timeline_thumb_path=null`
- worker가 후속으로 썸네일을 채운다
- 여러 번 실행해도 이미 처리된 memory는 다시 집지 않도록 설계

### 7.3 현재 운영 관점

지금 기준으로는 아래가 정리돼 있다.

- base schema
- null unique 제약 수정 방향
- worker 메타 컬럼
- backfill SQL
- worker 검증용 SQL
- fallback 운영 관찰 체크리스트

## 8. preload / 성능 최적화

### 8.1 signed URL 생성 원칙

현재 원칙은 `리스트 fetch 단계에서 signed URL을 만들지 않는다`이다.

이유:

- 타임라인과 홈 리스트에서 JS thread spike를 줄이기 위함
- 스크롤 성능을 보존하기 위함

대신 아래 방식으로 간다.

- 렌더 계층에서 필요 시 `getMemoryImageSignedUrlCached`
- idle 타이밍 prefetch
- variant별 signed URL 캐시

### 8.2 타임라인 최적화

현재 타임라인은 아래 기준으로 최적화돼 있다.

- view window 기반 이미지 로딩
- immediate/deferred preload 분리
- `timeline-thumb` variant 정확 해석
- `FlashList`와 preload effect를 보수적으로 묶어 entity 변경 시 흔들림 축소

### 8.3 홈 오늘의 기록 최적화

홈의 오늘 기록 슬라이더도 같은 방향으로 정리돼 있다.

- immediate preload 우선
- deferred preload는 idle callback 기반
- 인접 카드 중심 preload
- preload 경로와 실제 렌더 경로를 `OptimizedImage` 기준으로 최대한 일치

## 9. fallback / 운영 관찰 포인트

현재 fallback은 제거 대상이 아니라 운영 안전망이다.

판단 기준:

1. `memory_images_read_fallback`가 장기적으로 안정적일 것
2. `memory_images_legacy_read_used` 중 `legacy_only_row`, `recent_create_gap`가 줄어들 것
3. `memory_upload_queue_recovered` 분포가 과도하지 않을 것
4. failed backlog가 장기 잔존하지 않을 것
5. 기존 legacy-only memory가 사실상 사라질 것

즉, 지금 목표는 fallback 제거가 아니라 `운영 관찰을 통해 제거 가능 조건을 모으는 것`이다.

## 10. 현재 완료 범위

### 10.1 완료로 보는 항목

- `memory_images` 중심 읽기/쓰기 구조
- 타임라인 thumb/original variant 분리
- preload와 signed URL 캐시 구조
- queue 기반 복구
- fallback 이유 관찰 이벤트
- worker/backfill SQL과 운영 절차 문서화

### 10.2 아직 리스크가 남는 항목

- release 환경 장기 분포 확인 전 fallback 제거 불가
- 외부 worker runtime의 처리량/timeout/재시도 정책은 운영 검증 필요
- orphan cleanup 장기 자동화 여지
- failed/stuck backlog 회수 자동화 여지

- legacy row repair SQL 정리
