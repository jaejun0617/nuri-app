# 집사 꿀팁 가이드 통합 완성본

## 1. 문서 목적

이 문서는 NURI의 `집사 꿀팁 가이드` 도메인을 지금 시점 기준으로 한 번에 이해할 수 있도록 정리한 통합 문서다.

목표는 아래 네 가지다.

1. 챕터 1, 챕터 2-1, 챕터 2-2 진행 내역을 현재 코드 기준으로 다시 정리한다.
2. 홈 추천, 목록, 상세, 검색, 운영 경로를 하나의 도메인 구조로 묶어 설명한다.
3. Supabase 스키마와 앱 서비스 레이어 계약을 같은 관점에서 정리한다.
4. 남은 확장 포인트와 현재 완료 범위를 분명하게 남긴다.

이 문서가 기준 문서이며, 과거 챕터 문서는 설계/작업 이력 보관 용도로만 본다.

## 2. 운영 방향 정정

집사 꿀팁 가이드의 장기 운영 기준은 아래로 고정한다.

- 최종 운영: 웹 관리자 페이지(CMS)
- 사용자 소비 채널: NURI 모바일 앱
- 데이터 기준 저장소: Supabase
- 보조 수단: SQL
- 현재 앱 관리자 화면: 임시/중간 운영 수단

즉, 현재 앱 안의 관리자 화면은 초기 검증과 브릿지 운영에는 의미가 있지만, 장기적인 최종 운영 방식은 아니다.

왜 웹 CMS가 최종 방향인지:

- 긴 텍스트 작성과 수정이 훨씬 쉽다.
- 태그 / 카테고리 / 상태 / 우선순위 관리가 쉽다.
- 이미지 업로드와 미리보기가 모바일보다 안정적이다.
- 목록 / 검색 / 필터 / 정렬 / 대량 관리가 쉽다.
- 운영자가 실제로 PC 환경에서 관리하는 편이 훨씬 효율적이다.
- 데이터가 많아질수록 앱 관리자 화면보다 웹 CMS가 훨씬 안정적이다.

## 3. 현재 상태 요약

### 3.1 완료 범위

- 챕터 1 완료
  - 홈 추천 2건, 목록, 상세, 로컬 seed fallback, 회전 정책 도입
- 챕터 2-1 완료
  - `pets.species_group`, `species_detail_key`, `species_display_name` 정식화
  - Supabase `pet_care_guides`, `pet_care_guide_events` 구조 반영
  - 공개 조회 서비스와 운영 서비스 분리
- 챕터 2-2 1차 완료
  - `profiles.role` 동기화
  - 관리자 전용 `가이드 운영` 진입
  - 운영 목록 / 등록 / 수정 / 상태 변경
  - 앱 관리자 화면 기준 최소 CMS 브릿지 흐름 확보
- 챕터 2 검색 고도화 단계 구현 완료
  - 서버 검색 RPC 계약 추가
  - 최근 검색어 / 인기 검색어 구조 추가
  - 앱은 RPC 우선, 실패 시 클라이언트 검색으로 폴백
  - 개발/테스트 환경에서 공개 row 0건이면 seed fallback으로 목록/검색/상세 가시성 복구

### 2.2 아직 남은 범위

- 추천/운영 리포트 고도화
- 관리자 이미지 업로드 경로
- 웹 관리자 페이지(CMS) 구축
- 서버 검색 RPC migration 실제 운영 배포

즉, 지금의 집사 꿀팁 가이드는 `브릿지 운영 수단 + 사용자 탐색 가능한 검색 구조`까지 열려 있는 상태다.

## 4. 전체 단계 구분

집사 꿀팁 가이드는 현재 프로젝트 기준으로 아래 `5단계`로 본다.

### 3.1 1단계: 도메인 도입

완료 기준:

- 홈 추천 영역이 별도 가이드 도메인으로 분리됨
- 목록 / 상세 화면이 연결됨
- 로컬 seed 기반으로 최소 콘텐츠 흐름이 동작함
- species / age policy 기반 기본 개인화가 동작함

대응 챕터:

- 챕터 1

### 3.2 2단계: 운영 데이터 구조 정식화

완료 기준:

- `pet_care_guides`, `pet_care_guide_events`가 Supabase에 정식화됨
- `pets.species_group`, `species_detail_key`, `species_display_name` 구조가 정리됨
- 공개 조회 서비스와 운영 CRUD 서비스가 분리됨
- 이벤트 적재와 검색용 필드가 기본 구조로 반영됨

대응 챕터:

- 챕터 2-1

### 4.3 3단계: 앱 관리자 브릿지 개방

완료 기준:

- `profiles.role` 기준 관리자 진입이 열림
- 운영 목록 / 등록 / 수정 / 초안 / 발행 / 보관 / 비활성화가 가능함
- 앱 안에서 최소 CMS 흐름을 브릿지 용도로 검증할 수 있음

대응 챕터:

- 챕터 2-2 1차

### 3.4 4단계: 검색 및 테스트 가시성 고도화

완료 기준:

- 서버 검색 RPC / FTS 계약이 정리됨
- 최근 검색어 / 인기 검색어가 동작함
- 검색 실패 시 클라이언트 폴백이 유지됨
- 공개 row 0건이어도 개발/테스트 환경에서 목록/검색/상세를 검증할 수 있음
- QA용 공개 seed 기준이 정리됨

대응 챕터:

- 챕터 2-2 2차

### 4.5 5단계: 운영 완성 단계

완료 기준:

- 운영 DB에 공개 row와 검색 RPC migration이 실제 반영됨
- 웹 관리자 페이지(CMS)가 정식 운영 도구로 구축됨
- 이미지 업로드 경로가 웹 CMS 기준으로 정식화됨
- 추천/운영 리포트가 웹 운영 도구 기준으로 붙음
- 운영 데이터와 테스트 데이터 구분 기준이 정착됨

대응 챕터:

- 챕터 3

### 4.6 현재 위치

현재는 `4단계 완료` 상태로 본다.

의미:

- 사용자용 탐색 구조는 열렸고
- 앱 관리자 브릿지도 열렸고
- 개발/테스트 기준 공개 가이드 검증 흐름도 복구됐다

아직 남은 것은 `운영 완성 단계`다.

즉, 완성까지는 `1단계`가 남아 있다.

## 4-1. 앱 챕터 3 상세 설계

집사 꿀팁 가이드의 앱 기준 다음 단계는 `앱 챕터 3`으로 정리한다.

앱 챕터 3은 아래 세 갈래로 나눈다.

### 4-1.1 챕터 3-1: 펫 기반 추천 고도화

목표:

- 펫 등록 정보를 추천 로직과 강하게 연결한다.
- 홈 `우리 아이를 위한 추천 팁`을 실제 개인화 추천에 가깝게 올린다.
- 종 / 세부 종 / 나이 / memorial 상태를 추천 분기 기준으로 정리한다.

완료 기준:

- 대표 종과 세부 종 입력 구조가 추천 계약과 연결된다.
- `birth_date` 기준 월령 계산이 추천에 반영된다.
- `death_date` 기준 memorial 분기가 반영된다.
- 공통 팁과 종 특화 팁이 함께 섞인다.
- 홈 추천 / 목록 정렬 / 검색 컨텍스트가 같은 분기 기준을 사용한다.

현재 구현 상태:

- 펫 등록/수정 화면의 대표 종 선택을 `강아지 / 고양이 / 햄스터 / 토끼 / 거북이 / 새 / 물고기 / 기타`로 확장했다.
- 대표 종별 quick chip + 직접 입력 구조를 열어 세부 종/품종 데이터를 더 안정적으로 받도록 정리했다.
- 저장은 기존 `species_group / species_detail_key / species_display_name` 계약을 유지한다.
- 추천 점수는 `세부 종 키워드 > 대표 종 키워드 > species_group > common` 순서로 반영한다.
- 홈 추천과 가이드 목록 정렬이 같은 개인화 점수 기준을 공유한다.
- `death_date`가 있는 memorial 펫은 홈 케어 추천을 숨기고 추억/기록 중심 안내 상태로 분기한다.

### 4-1.2 챕터 3-2: 추천 운영 규칙 고도화

목표:

- 운영 우선순위와 시즌성, 반복 노출 완화를 추천 엔진에 더 깊게 연결한다.

완료 기준:

- `priority / sort_order / rotation_weight` 가중치 재조정
- 같은 팁 반복 노출 완화
- 시즌성/운영 캠페인 팁 반영
- 공통 팁과 종 특화 팁 비율 정리

현재 구현 상태:

- 개인화 점수에 `priority`, `sort_order`, `rotation_weight` 운영 가중치를 다시 반영했다.
- `tags / search_keywords / title / summary` 기준 계절성 키워드를 읽어 현재 시즌과 맞는 가이드를 가산한다.
- 현재 시즌과 맞지 않는 seasonal 가이드는 감점해 홈 추천이 과하게 시즌을 벗어나지 않도록 조정했다.
- 최근 노출된 가이드 자체뿐 아니라 최근 노출된 카테고리에도 페널티를 걸어 반복 체감도를 낮췄다.
- 홈 추천 2건은 가능한 한 `종 특화 팁 1건 + 공통 팁 1건` 조합을 우선으로 뽑도록 섞기 규칙을 추가했다.

### 4-1.3 챕터 3-3: 운영/분석 기준 정리

목표:

- 추천이 실제로 잘 소비되는지 판단할 기준을 만든다.

완료 기준:

- 추천 노출 / 클릭 / 상세 이벤트 기준 정리
- 어떤 추천 조합이 잘 소비되는지 보는 리포트 기준 정리
- 운영 데이터와 테스트 데이터 분리 전략 정착

현재 구현 상태:

- `home_impression / list_click / detail_view` 이벤트는 계속 유지한다.
- 홈/목록/상세 모두 같은 공통 메타데이터 규격으로 이벤트를 적재한다.
- 메타데이터에는 `strategyVersion`, `source`, `searchSource`, `resultRank`, `guideCategory`, `guideTargetSpecies`, `guideAgePolicyType`, `guidePriority`, `guideSortOrder`, `guideRotationWeight`, `petRepresentativeSpecies`, `petRepresentativeLabel`, `petSpeciesDisplayName`, `isMemorialPet`, `seasonBucket`를 남긴다.
- 이후 운영자는 이벤트 테이블만으로도 추천 전략 버전별 반응, 검색 source별 반응, 종/세부 종/계절별 반응을 기본 집계할 수 있다.
- 리포트/점검용 SQL canonical 문서도 함께 추가했다.
- 최종 QA 체크리스트 canonical 문서를 추가해 출시 직전 확인 기준을 분리했다.

## 4-2. 펫 등록 UX 설계

추천 품질을 올리기 위해 펫 등록 화면은 아래 구조를 기준으로 정리한다.

### 4-2.1 대표 종 1차 선택

대표 종은 처음부터 넓게 노출한다.

- 강아지
- 고양이
- 햄스터
- 토끼
- 거북이
- 새
- 물고기
- 기타

이유:

- 사용자가 처음부터 `기타`로 밀려나는 경험을 줄인다.
- 추천 seed와 홈 추천 로직을 바로 종 기준으로 연결하기 쉽다.
- 현재 많이 키우는 종을 처음부터 빠르게 선택할 수 있다.

### 4-2.2 세부 종 입력 방식

대표 종 선택 뒤에는 세부 종 입력을 받는다.

- 강아지 / 고양이:
  - 대표 품종 추천 칩
  - 직접 입력 유지
  - 장기적으로는 검색형 셀렉터로 확장
- 햄스터 / 토끼 / 거북이 / 새 / 물고기:
  - 자주 쓰는 세부 종 칩
  - 없으면 직접 입력
- 기타:
  - 직접 입력 중심
  - 예시 placeholder 제공

현재 앱 단계에서는 `칩 + 직접 입력` 구조를 먼저 연다.

### 4-2.3 저장 계약

DB 스키마는 기존 계약을 유지한다.

- `species_group`
  - `dog`
  - `cat`
  - `other`
- `species_detail_key`
  - 세부 종/품종용 정규화 키
- `species_display_name`
  - 화면 표기 이름

즉, UI는 대표 종을 넓게 보이지만 저장은 기존 `dog / cat / other + detail` 구조를 유지한다.

예:

- 햄스터 선택
  - `species_group = other`
  - `species_detail_key = hamster`
  - `species_display_name = 햄스터`
- 토끼 선택
  - `species_group = other`
  - `species_detail_key = rabbit`
  - `species_display_name = 토끼`
- 기타에서 친칠라 직접 입력
  - `species_group = other`
  - `species_detail_key = chinchilla` 또는 정규화된 직접 입력값
  - `species_display_name = 친칠라`

## 4-3. 추천 규칙 상세

### 4-3.1 종/세부 종 분기

추천 우선순위는 아래 순서를 따른다.

1. `species_detail_key`가 명확하면 세부 종 맞춤 팁 우선
2. 없거나 부족하면 `species_group` 기준 팁
3. 마지막으로 `common` 팁 혼합

예:

- 강아지: 강아지 전용 + 공통
- 고양이: 고양이 전용 + 공통
- 햄스터: 햄스터 전용 + 공통
- 거북이: 거북이 전용 + 공통
- 토끼: 토끼 전용 + 공통
- 새 / 물고기 / 기타: 세부 종 또는 기타 공통 팁 중심

### 4-3.2 나이/생애주기 분기

나이 계산은 `birth_date`를 우선 사용한다.

- 어린 개체
  - 이갈이
  - 사회화
  - 예방접종 후 관찰
- 성체
  - 체중관리
  - 행동 안정화
  - 환경 관리
- 노령
  - 관절
  - 회복 속도
  - 식사 변화
  - 통증/컨디션 관찰

`birth_date`가 없으면:

- 공통 팁 중심으로 fallback
- `adoption_date`는 보조 힌트로만 보고 과한 생애주기 추정은 하지 않는다

### 4-3.3 공통 팁 병행

종 특화 팁만 노출하지 않는다.

- 수분 관리
- 환경 점검
- 계절성 관리
- 기본 건강 체크

이런 `common` 팁을 항상 일부 섞어야 추천이 한쪽 주제로만 쏠리지 않는다.

### 4-3.4 memorial 분기

`death_date`가 존재하면 일반 케어 추천은 기본 숨김 또는 대체가 맞다.

권장 정책:

- 홈 `우리 아이를 위한 추천 팁`의 기본 케어 카드 숨김
- 대신 아래 중 하나로 대체
  - 함께한 시간을 돌아보는 기록 팁

## 9. 검색 구조

### 8.1 현재 앱 검색 흐름

현재 목록 검색은 아래 순서로 동작한다.

1. 검색창 입력
2. `usePetCareGuideSearch` 훅 실행
3. RPC `search_pet_care_guides` 호출 시도
4. 실패 시 현재 catalog 대상으로 클라이언트 검색 폴백

즉, 앱은 이미 서버 검색을 우선 사용하도록 준비됐고, migration이 아직 운영 DB에 반영되지 않아도 UX가 깨지지 않게 설계되어 있다.

### 8.2 최근 검색어

최근 검색어는 `AsyncStorage`에 저장한다.

정책:

- 최대 개수 제한
- 중복 제거
- 최신 검색어가 앞으로 오도록 정렬
- 기기 로컬 저장

핵심 파일:

- `src/services/local/guideSearch.ts`
- `src/hooks/useRecentPetCareGuideSearches.ts`

### 8.3 인기 검색어

인기 검색어는 아래 순서를 따른다.

1. RPC `get_pet_care_guide_popular_searches` 호출
2. 실패 시 공개 카탈로그의 `searchKeywords/tags/speciesKeywords/category` 기반 추정 키워드 사용

즉, 운영 배포 전에도 빈 화면이 되지 않는다.

### 8.4 클라이언트 검색 폴백

클라이언트 폴백 검색은 여전히 유지한다.

이유:

- RPC 배포 이전 개발 환경 보장
- 서버 장애 시 최소 검색 UX 유지
- seed fallback과 동일한 계약 유지

다만 기준 구조는 이미 `서버 검색 우선`이다.

### 8.5 개발/테스트 공개 seed 전략

이번 이슈를 통해 확인된 점은 아래와 같다.

- 최근 검색어 / 인기 검색어가 정상이어도 공개 가이드 row가 0건이면 목록은 empty가 될 수 있다.
- 인기 검색어는 이벤트 기반으로도 노출될 수 있기 때문에 목록 가시성과 항상 일치하지 않는다.
- 개발/테스트 환경에서는 공개 row 0건 성공 응답일 때 seed fallback이 필요하다.

현재 정책:

- 운영 환경에서는 실제 공개 row 기준으로만 목록을 본다.
- 실제 공개 row가 0건이면 환경과 관계없이 로컬 seed fallback을 사용한다.
- DB 기준 QA가 필요할 때는 `개발-테스트용-공개-가이드-seed.sql`을 별도로 넣는다.
- 앱 카탈로그 상태는 `remote / local-seed / remote-empty` 소스로 구분한다.
- 즉 QA 시점에는 지금 보고 있는 목록이 운영 row인지, 개발 seed인지 코드 상태로도 구분 가능해야 한다.
- 개발 환경에서는 홈 추천과 가이드 목록에 현재 source를 작게 표시해 QA가 즉시 판단할 수 있게 한다.

중요:

- 현재 단계에서는 `실제 공개 데이터가 0건이면 seed 데이터가 보이도록` 정책을 고정한다.
- 즉 빈 화면보다 테스트 가능한 상태를 우선한다.
- 운영 공개 row가 준비되면 source는 다시 `remote`로 전환된다.

source 의미:

- `remote`
  - Supabase 공개 row를 정상 조회해서 그 결과를 그대로 쓰는 상태
- `local-seed`
  - 공개 row가 0건이거나 원격 조회가 실패해 로컬 seed를 대신 쓰는 상태
- `remote-empty`
  - 원격 조회 자체는 성공했지만 공개 조건을 만족한 가이드 row가 0건인 상태
  - 현재 정책상 일반 노출 상태라기보다 진단용 상태에 가깝다.

seed 식별 규칙:

- 현재 로컬 seed 가이드는 `guide-` prefix id를 사용한다.
- 개발 환경에서 source가 `local-seed`일 때 홈 카드와 목록 카드에 `테스트 seed` 배지를 표시한다.
- 즉 QA 시점에는 source 배지와 카드 배지를 함께 보고 운영 row와 seed를 바로 구분할 수 있다.

seed -> remote 전환 QA 기준:

- 운영 공개 row가 준비되면 source는 `local-seed`에서 `remote`로 전환되어야 한다.
- 홈 추천 회전 캐시는 `petId`만이 아니라 `species / species_detail_key / species_display_name / birth_date / death_date`까지 포함한 컨텍스트 기준으로 분기한다.
- 즉 같은 펫이라도 품종/세부 종/상태를 수정하면 홈 추천 카드도 바로 다시 계산되어야 한다.
- QA 시점에는 아래를 함께 확인한다.
  - 더보기 목록이 선택 펫 종 + 공통 팁으로 보이는지
  - 홈 추천 카드도 같은 펫 컨텍스트 변경을 즉시 반영하는지
  - source 배지가 `remote`로 전환되는지
- 운영 전환용 canonical 체크리스트는 별도 문서로 관리한다.
  - `docs/domains/집사-꿀팁-가이드/집사-꿀팁-가이드-운영-전환-체크리스트.md`

## 10. 개인화와 추천 구조

현재 홈 추천과 목록 정렬은 아래 우선순위를 따른다.

1. `isActive = true`
2. `status = published`
3. `publishedAt <= now`
4. 더보기 기본 목록은 현재 펫의 `species_group` 또는 `common`만 노출
5. `agePolicy` 일치
6. `priority`, `sortOrder`
7. rotation seed
8. 최근 노출 최소화

이 구조의 장점:

- 운영자가 우선순위를 직접 제어할 수 있다.
- 펫 개인화가 과도하게 흔들리지 않는다.
- 검색/추천/운영이 같은 메타 필드를 공유한다.

## 11. 운영/관리자 확장성

현재 앱 운영 화면은 브릿지 단계의 최소 CMS에 해당한다.

이미 열린 범위:

- 관리자 전용 진입
- 운영 목록
- 검색/상태 필터
- 등록/수정
- 발행/보관/비활성화

남은 확장 방향:

- 웹 CMS 목록 / 상세 / 등록 / 수정 / 삭제 / 비활성화
- 웹 CMS 이미지 업로드 / 미리보기
- 웹 CMS 태그 / 카테고리 / 우선순위 관리
- 웹 CMS 검색 운영 / 추천 운영 / 성과 리포트
- 운영자용 계절/캠페인 배치 관리
- 앱 관리자 화면의 브릿지 축소 또는 긴급 운영 전환

웹 CMS가 최종 방향인 이유:

- 모바일보다 긴 텍스트 입력과 검수가 훨씬 쉽다.
- 대량 데이터 필터 / 정렬 / 검색 / 수정이 효율적이다.
- 이미지 업로드와 미리보기를 안정적으로 붙이기 쉽다.
- 운영자 역할상 PC 관리가 더 자연스럽고 안정적이다.
