# 펫동반 장소 탐색 구현 상세

## 1. 왜 이 기능이 필요한가

NURI의 위치 기반 기능은 더보기 메뉴 안의 부가 기능이 아니라, 보호자가 지금 행동으로 이어질 수 있는 실제 서비스 기능이어야 한다.

이번 턴의 목표는 기존 `펫동반 음식점 찾기`를 단순 메뉴 수준이 아니라 아래 흐름이 가능한 `펫동반 장소 탐색` 도메인으로 여는 것이다.

- 현재 GPS 기반 주변 추천
- 직접 검색
- 리스트/상세
- 지도 미리보기
- 외부 지도 열기
- 후보 데이터와 검증 상태 분리

핵심 철학은 산책 기능과 동일하다.

- 복잡한 지역 브라우징을 다시 만들지 않는다.
- 기본 진입은 `현재 GPS 기준 주변 추천`이다.
- 검색은 현재 위치가 없어도 허용한다.
- 거리 기준은 항상 현재 위치 기준이다.
- stale 위치를 current처럼 보이게 하지 않는다.

## 2. 현재 구조 진단

현재 앱에서 이 기능은 아래 레이어 위에 올라가 있다.

- 메뉴/진입

- 네비게이션
  - `src/navigation/RootNavigator.tsx`
- 위치 공통 레이어
  - `src/hooks/useCurrentLocation.ts`
  - `src/hooks/useDistrict.ts`
  - `src/services/location/currentPosition.ts`
- 장소 탐색 공통 레이어
  - `src/services/locationDiscovery/provider.ts`
  - `src/services/locationDiscovery/kakaoLocal.ts`
  - `src/services/locationDiscovery/service.ts`
  - `src/services/locationDiscovery/placeMeta.ts`
  - `src/services/locationDiscovery/maps.ts`
  - `src/hooks/useLocationDiscovery.ts`
  - `src/screens/LocationDiscovery/*`

진단 결과:

- 산책 기능은 이미 `현재 GPS 기반 주변 추천 + 직접 검색` 구조로 정리되어 있다.
- 펫동반 기능도 같은 공통 레이어를 쓰지만, 이전 상태는 사실상 `외부 검색 결과 얇은 래퍼`에 가까웠다.
- 이번 턴에서 `provider raw -> normalize -> service meta merge -> UI` 경계를 실제 코드로 분리했다.
- 메뉴 명칭은 사용자 친화적으로 유지하되, 내부 도메인은 `펫동반 장소 탐색`으로 넓게 잡는 것이 맞다.

즉, 이번 턴부터는 단순 음식점 메뉴가 아니라 향후 카페/식당/실내/야외/반려동물 동반 가능 장소로 확장 가능한 구조를 기준으로 본다.

## 3. 현재 위치 기반 추천 + 직접 검색 단일 정책

조회 정책은 아래로 고정한다.

- 기본 진입
  - 현재 GPS 좌표가 fresh일 때 주변 추천을 호출한다.
  - 펫동반 장소는 `애견동반 카페`, `애견동반 식당`, `반려동물 동반`, `애견카페`, `반려견 운동장` 묶음으로 후보를 수집한다.
- 직접 검색
  - 검색은 submit 기준으로만 실행한다.
  - 현재 위치가 있으면 주변 검색으로 동작한다.
  - 현재 위치가 없으면 텍스트 검색으로 동작한다.
- 거리
  - 거리 기준은 항상 현재 위치 기준이다.
  - 현재 위치가 없으면 `distanceMeters = null`이고 UI는 `거리 확인 중`으로 표시한다.

재도입하지 않는 구조:

- selected district
- city scope
- district chips
- `고양시 전체`
- 복잡한 지역 브라우징

## 4. 지도/API/provider 구조

현재 구조는 아래처럼 유지한다.

- provider
  - `kakaoLocalSearchProvider`
- normalize/service
  - `searchLocationDiscovery`
  - `toNormalizedPetFriendlyCandidate`
  - `mergePetFriendlyCandidate`
- service meta 경계
  - `loadPetFriendlyPlaceServiceMeta`
- 지도/외부 링크

원칙:

- Kakao raw 응답은 화면에서 직접 참조하지 않는다.

- 지도 provider는 `maps.ts` 경계 뒤로 숨긴다.
- 향후 Supabase 메타/검수/북마크가 붙어도 UI가 raw provider를 직접 알지 않도록 유지한다.

## 6. 정렬 정책

정렬은 산책과 동일하게 `서비스 추천 정렬`을 기본으로 두고, 화면에서만 override를 허용한다.

- 기본: 추천순
- 선택: 가까운순
- 선택: 먼순

펫동반 추천순은 아래 우선순위를 사용한다.

- 카페
- 식당
- 야외 공간
- 실내 공간
- 일반 장소
- 같은 그룹 안에서는 펫 키워드 근거가 있는 후보 우선
- 마지막 tie-breaker는 거리

- `KeyboardAvoidingView`

- 리스트 대량 데이터까지 가정한 기본 렌더링 안정성을 유지한다.

## 10. 위치 freshness 정책 재사용 원칙

펫동반 장소 탐색은 산책/날씨와 같은 공통 freshness 정책을 그대로 쓴다.

- `useCurrentLocation`
- `LOCATION_STALE_AFTER_MS`
- `LOCATION_AUTO_REFRESH_INTERVAL_MS`
- `isFreshLocationCoordinates`
- `getLocationAgeMs`

문구 정책도 동일하다.

- fresh: `현재 위치 기준`
- stale: `최근 확인 위치 기준`
- refreshing: `새 위치 확인 중`

즉 stale 위치를 current처럼 포장하지 않는다.

## 11. Supabase 자체 장소 DB 존재 여부

이번 턴에서 아래 두 축으로 확인했다.

1. 로컬 코드/문서/SQL 검색
2. 실제 Supabase REST 후보 테이블 조회

확인 대상:

- `pet_friendly_places`
- `place_categories`
- `place_tags`
- `place_photos`
- `place_reviews`
- `place_bookmarks`
- `place_search_logs`

확인 결과:

- 로컬 코드/문서/SQL 기준
  - 장소 전용 Supabase 서비스 파일 없음
  - 장소 전용 SQL/마이그레이션 없음
  - 현재 앱 코드도 이 테이블들을 읽지 않음
- 실제 Supabase REST 조회 기준
  - 전부 `Could not find the table ... in the schema cache` 404 응답

결론:

- 현재 프로젝트에는 `자체 펫동반 장소 DB가 아직 없다.`
- 현재 운영 데이터는 외부 API 후보 검색이다.
- `loadPetFriendlyPlaceServiceMeta`는 미래 메타 병합 경계만 먼저 열어둔 no-op 레이어다.

## 12. 현재 데이터 전략 판단

현재 프로젝트 기준 가장 현실적인 단계는 `B. 외부 API + Supabase 메타 혼합`이다.

현재 시점:

- 실제 운영 후보 수집은 Kakao Local이 담당한다.
- 자체 장소 DB는 아직 없다.
- 따라서 단기 운영은 사실상 `A. 외부 API 전용`이다.

하지만 구조 판단은 아래가 맞다.

- 단기
  - 외부 API 기반 현재 위치 추천 + 직접 검색
  - 후보/검증 상태 분리
- 중기
  - Supabase에 북마크, 최근 본 장소, 운영 검수 메타, 사용자 제보, 검색 로그 적재
- 장기
  - 운영 검수된 장소 DB를 service truth로 승격
  - 외부 API는 후보 수집/보조 레이어로 사용

즉 현재 구현은 `단기 운영은 외부 API`, `구조는 혼합형 준비` 상태다.

## 13. 카카오 후보 데이터와 서비스 메타 분리 방식

이번 턴에서 가장 중요한 분리 원칙은 아래다.

- Kakao Local
  - 장소 후보 수집 레이어
  - 좌표/주소/카테고리/링크 제공
  - 펫동반 확정 truth가 아님
- NURI 서비스 메타
  - 향후 Supabase에 저장될 검수/제보/운영 메타
  - 펫동반 여부, 운영 안내, 검증 근거, 북마크, 추천 보강 담당

코드 구조:

- `kakaoLocal.ts`
  - raw provider 호출
- `service.ts`
  - raw 문서 normalize
  - `mergePetFriendlyCandidate`로 서비스 메타 병합
- `placeMeta.ts`
  - 메타 로딩 경계
  - 현재는 no-op이지만, 다음 단계에서 `pet_place_source_links -> pet_place_service_meta -> pet_place_external_signals` join을 붙일 지점

즉 Kakao 응답만으로 `펫동반 가능` 확정 뱃지를 띄우지 않도록 상태 모델을 분리했다.

## 14. 검증 상태 모델과 UI 표현 원칙

내부 상태 모델:

- `unknown`
- `keyword-inferred`
- `user-reported`
- `admin-verified`
- `rejected`

산책 전용 추천 상태:

- `service-ranked`

UI 표현 원칙:

- `admin-verified`
  - `펫동반 가능`
- `keyword-inferred`
  - `펫동반 가능성 있음`
- `user-reported`
  - `사용자 제보 기반`
- `unknown`
  - `현장 확인 필요`
- `rejected`
  - `동반 불가 제보`

세부 규칙:

- 카카오 후보만 있는 상태는 `admin-verified`처럼 보이면 안 된다.
- 상세에서는 `검증 상태`, `검증 근거`, `출처`, `펫동반 안내`를 함께 보여준다.
- 리스트에서도 검증 상태 배지를 항상 같이 노출한다.
- 현재 Supabase 메타가 없으므로 대부분의 펫동반 후보는 `unknown` 또는 `keyword-inferred`로 표시된다.

## 15. Kakao 외 신뢰도 보강 후보 API 검토

현재 기준으로 의미 있는 보강 후보는 아래 세 축이다.

- Kakao Local
  - 국내 장소 후보 수집 1차
- Google Places API
  - 장소 속성 기반 보조 검증 신호
- 한국관광공사 / 공공데이터포털 반려동물 동반 여행 데이터
  - 관광/야외/문화시설 검증 보강

### Google Places API

공식 Places API 문서 기준으로 `outdoorSeating`, `goodForChildren`, `allowsDogs` 같은 속성이 존재한다.

판단:

- 장점
  - 단순 키워드보다 더 구조화된 보조 속성을 줄 수 있다.
  - `allowsDogs`는 펫동반 보조 신호로 직접적이다.
  - `outdoorSeating`는 테라스/야외 좌석 판단 보조 신호로 유용하다.
- 한계
  - 이것도 최종 확정 데이터가 아니다.
  - Places는 FieldMask와 SKU를 신중하게 써야 하고, 과한 필드 요청은 비용 리스크가 있다.
  - 국내 커버리지는 좋지만 Kakao 대비 한글 상호/지역 체감은 혼합 검증이 필요하다.

결론:

- 현재 NURI에서는 `후보 수집`보다 `보조 검증` 역할이 더 맞다.
- 특히 Kakao 후보를 먼저 잡고, 일부 상세 검증이 필요한 후보에만 Google Places를 보조로 붙이는 구조가 현실적이다.

### 한국관광공사 / 공공데이터포털 반려동물 동반 여행 데이터

공공데이터포털 기준으로 `한국관광공사_반려동물_동반여행_서비스` 오픈 API가 존재하고, 무료 트래픽과 운영계정 확장 여지가 있다. 또한 지자체 파일데이터로 `반려동물 동반 가능 문화시설 위치 데이터`, `반려동물 동반 관광지 정보 리스트` 같은 공개 데이터가 있다.

판단:

- 장점
  - 반려동물 동반 가능 여부와 이용 조건을 직접적으로 담는 경우가 있다.
  - 관광지, 문화시설, 야외 공간 검증 보강에는 Kakao보다 신뢰도가 높을 수 있다.
  - 비용 측면이 상대적으로 낮다.
- 한계
  - 카페/식당 전국 단위 커버리지는 제한적이다.
  - 지역 편차가 크고, 최신성도 공급기관마다 다르다.
  - 상호/주소 정합성 보정이 필요하다.

결론:

- `후보 수집 메인`보다는 `야외/관광/문화시설 보강`에 적합하다.
- 펫동반 카페/식당 전반의 source of truth로 쓰기엔 부족하다.

## 16. 각 외부 소스의 역할 판단

- Kakao Local
  - 후보 수집용: 적합
  - 보조 검증용: 제한적
  - source of truth 가능 여부: 부적합
- Google Places API
  - 후보 수집용: 가능하지만 현재 프로젝트에선 비효율
  - 보조 검증용: 적합
  - source of truth 가능 여부: 부적합
- 한국관광공사 / 공공데이터포털
  - 후보 수집용: 일부만 가능
  - 보조 검증용: 관광/야외/문화시설에 적합
  - source of truth 가능 여부: 부적합
- Supabase 서비스 메타
  - 후보 수집용: 초기에는 부적합
  - 보조 검증용: 적합
  - source of truth 가능 여부: 장기적으로 적합

현재 프로젝트 기준 최선의 조합:

- Kakao Local = 국내 장소 후보 수집 1차
- Google Places = 필요 후보에 한정한 속성 보강
- 관광공사/공공데이터 = 관광/야외/문화시설 보강
- Supabase = 최종 검증/운영 메타 source of truth

## 17. Supabase 장소 메타 스키마 설계 방향

현재는 자체 장소 원본 DB를 바로 만들지 않고, 외부 후보를 보강하는 메타 레이어부터 여는 것이 맞다.

권장 핵심 구조:

- `pet_place_service_meta`
  - canonical 메타 본체
  - `id uuid`
  - `verification_status text`
  - `source_type text`
  - `primary_source_provider text null`
  - `primary_source_place_id text null`
  - `pet_policy_text text null`
  - `operating_status_label text null`
  - `admin_note text null`
  - `user_report_count integer`
  - `bookmarked_count integer`
- `pet_place_source_links`
  - 외부 provider 매핑
  - `provider text`
  - `provider_place_id text`
  - `pet_place_meta_id uuid`
- `pet_place_external_signals`
  - Google Places / 관광공사 / 공공데이터 보조 신호
  - `signal_provider text`
  - `signal_key text`
  - `signal_value_boolean boolean null`
  - `signal_value_text text null`
  - `signal_score integer`

권장 방향:

- 내부 PK는 `pet_place_service_meta.id uuid`
- 외부 매핑 키는 `pet_place_source_links.provider + pet_place_source_links.provider_place_id`
- `provider + provider_place_id`에는 unique 제약을 둔다
- 북마크/제보/검수 이력은 별도 테이블로 분리하고 집계값만 canonical 메타에 반영한다
- Google Places / 관광공사 / 공공데이터 신호는 `pet_place_external_signals`에서 정규화한다

상세 초안은 `docs/sql/도메인/펫동반-장소-탐색/펫동반-장소-메타-스키마-초안.md`에 정리한다.
실행 가능한 최종 운영 SQL은 `docs/sql/도메인/펫동반-장소-탐색/펫동반-장소-메타-운영-스키마-최종.sql`에 둔다.

## 18. 외부 원본 ID와 내부 메타 매핑 전략

Kakao place id를 내부 PK로 바로 쓰는 것은 적절하지 않다.

이유:

- provider 종속성이 너무 강하다.
- 향후 Google Places / TourAPI를 병행하면 충돌과 이관 리스크가 생긴다.
- 내부 운영 이력과 외부 원본 식별자는 수명이 다르다.

권장 구조:

- 내부 엔터티 PK = `pet_place_service_meta.id`
- 외부 매핑 식별자 = `pet_place_source_links.provider + provider_place_id`

즉 현재 `placeMeta.ts`는 추후 source link lookup key 기준 조회로 이어지게 맞춰 두었다.

## 19. 검증 상태 모델과 DB 저장 구조 판단

`verification_status`는 enum보다 `text + check constraint`가 현재 프로젝트엔 더 안전하다.

이유:

- 초기 단계에서 상태명이 추가/조정될 가능성이 아직 있다.
- Supabase 운영에서 enum 변경은 마이그레이션 부담이 더 크다.
- `check constraint`면 안전성을 유지하면서도 진화 비용이 낮다.

현재 권장 체크 값:

- `unknown`
- `keyword-inferred`
- `user-reported`
- `admin-verified`
- `rejected`

`source_type`도 동일하게 `text + check constraint`를 권장한다.

## 20. 실기기 QA 체크리스트

- 키보드가 열린 상태에서 검색 submit이 자연스러운지
- 첫 결과 카드가 키보드/예측 바에 어색하게 잘리지 않는지
- stale/fresh/새 위치 확인 중 문구가 과장 없이 이해되는지
- 현재 위치가 없을 때 검색 UX가 자연스러운지
- 외부 지도 deep link가 iOS/Android에서 모두 열리는지
- 리스트에서 상세 진입 후 복귀 스크롤 체감이 자연스러운지
- 상세에서 연관 장소 더보기 시 끊김이 없는지
- 검증 상태 라벨이 `확정`처럼 과장되어 보이지 않는지

## 21. 실제 서비스급 SQL 반영 범위

이번 턴에서는 아래를 포함한 실제 서비스급 SQL을 별도 문서로 작성했다.

- `pet_place_service_meta`
- `pet_place_source_links`
- `pet_place_external_signals`
- `pet_place_bookmarks`
- `pet_place_user_reports`
- `pet_place_verification_logs`
- `pet_place_search_logs`
- `provider + provider_place_id` unique 전략
- `verification_status`, `source_type` check constraint
- `updated_at` 자동 갱신 trigger
- 집계 카운터 sync trigger
- 최소 인덱스 전략
- RLS 정책

즉 이번 턴의 SQL은 초안 메모가 아니라, 현재 프로젝트 기준으로 바로 검토 가능한 운영 스키마다.

## 22. 반려동물과 여행 메뉴 확장 방향

장기적으로는 `펫동반 장소 탐색`과 `반려동물과 여행`을 분리하는 것이 맞다.

- `펫동반 장소 탐색`
  - 현재 위치 기반 일상 생활권 탐색
  - Kakao Local 중심
- `반려동물과 여행`
  - 여행/관광 목적 탐색
  - 관광공사 반려동물 동반여행 서비스와 공공데이터 중심

이번 턴에서는 실제 앱 메뉴를 늘리지 않았고, 대신 초기 설계를 `docs/domains/반려동물과-여행/반려동물과-여행-초기-설계.md`에 남겼다.
