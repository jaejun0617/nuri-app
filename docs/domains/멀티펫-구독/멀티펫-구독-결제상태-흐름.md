# 멀티펫 구독 결제상태 흐름

## 1. 문서 목적

이 문서는 멀티펫 구독의 결제 상태가 앱과 서버에서 어떻게 움직여야 하는지 흐름 중심으로 정리한다.

## 2. 현재 실제 상태

현재 프로젝트에는 결제 상태 흐름이 없다.

즉 아래 상태들은 아직 코드에 없다.

- 구매 중
- 활성
- grace
- 만료
- revoke
- 복원 중

## 3. 권장 상태 집합

### 앱 표기 상태

- `unknown`
- `free`
- `active`
- `grace`
- `expired`
- `restoring`
- `error`

### 서버 원본 상태

- `active`
- `grace`
- `expired`
- `revoked`

## 4. 초기 로딩 흐름

1. 로그인
2. 서버에서 entitlement 조회
3. 앱에서 `canAddMorePets` 계산
4. 멀티펫 등록 버튼/차단 UX 반영

중요:

앱 시작 직후 entitlement가 아직 없으면 `unknown` 상태를 먼저 가지는 것이 맞다.

바로 `free`로 가정하면 결제 복원 유저에게 잘못된 차단이 걸릴 수 있다.

## 5. 구매 성공 흐름

1. 사용자가 구독 시작
2. 플랫폼 결제 완료
3. 서버 검증
4. `user_subscriptions` 업데이트
5. `user_entitlements` 업데이트
6. 앱 entitlement 재조회
7. `active` 반영
8. `아이 등록하기` 재시도 가능

## 6. 복원 흐름

1. 사용자가 복원 버튼 탭
2. 앱 상태 `restoring`
3. 플랫폼 복원 요청
4. 서버 검증/동기화
5. entitlement 재조회
6. 성공 시 `active`, 실패 시 `error` 또는 `free`

## 7. 만료 흐름

1. 서버가 만료 확인
2. `user_subscriptions.status = expired`
3. `user_entitlements.status = inactive`
4. 앱 재조회
5. 새 펫 등록 CTA 차단

중요:

이 흐름에서도 기존 펫 데이터는 그대로 유지해야 한다.

## 8. grace 흐름

권장:

- grace는 일반 free와 다르게 다룬다
- 사용자는 여전히 멀티펫 사용 가능하되, 갱신 이슈를 안내할 수 있다

이유:

결제 갱신 문제는 바로 만료와 같은 UX로 다루면 혼란이 크다.

## 9. 앱에서 필요한 최소 파생값

- `isSubscriptionReady`
- `canAddMorePets`
- `currentPetCount`
- `maxPetCount`
- `needsUpgrade`
- `isRestoring`

## 10. 현재 시점 권장 결론

결제 상태 흐름은 단순 boolean 하나로 끝내면 안 된다.

현재 NURI에서 권장되는 최소 상태 흐름은:

- `unknown`
- `active`
- `grace`
- `expired`
- `restoring`
- `error`

이다.

이 흐름을 먼저 문서로 고정한 뒤 구현에 들어가는 것이 맞다.
