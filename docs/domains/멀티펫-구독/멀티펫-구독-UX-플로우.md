# 멀티펫 구독 UX 플로우

## 1. 문서 목적

이 문서는 NURI에서 `멀티펫 구독`을 붙일 때 사용자가 실제로 보게 될 화면 흐름과 차단/전환 UX를 정리한다.

현재 실제 앱 구조 기준으로, 가장 먼저 연결될 화면은 아래다.

- `PetManagementScreen`
- `PetCreateScreen`
- More > `아이들 프로필 관리`

## 2. 현재 실제 UX 상태

현재 실제 앱에서는:

- `아이 등록하기`를 누르면 바로 등록 화면으로 이동한다
- 여러 마리를 이미 등록할 수 있다
- 현재 선택된 펫 기준으로 홈/기록/일정이 바뀐다

즉, 구독 차단 UX는 아직 없다.

## 3. 1차 UX 원칙

- 사용자가 갑자기 막혔다고 느끼지 않게 한다
- 이미 있는 멀티펫 가치와 연결해서 설명한다
- 결제 유도보다 “왜 필요한지”를 먼저 보여준다
- 등록 차단은 2중 가드로 처리한다

## 4. 핵심 UX 시나리오

### 시나리오 A. 무료 유저, 첫 펫 등록

- 허용
- 기존 흐름 그대로 유지

### 시나리오 B. 무료 유저, 두 번째 펫 등록 시도

진입:

- `PetManagementScreen`에서 `아이 등록하기`

권장 흐름:

1. 등록 화면으로 바로 가지 않음
2. 업그레이드 설명 모달 또는 전용 화면 노출
3. 사용자가
   - `구독 알아보기`
   - `나중에 할게요`
   중 하나 선택

### 시나리오 C. 우회 진입

예시:

- deep link
- 레거시 경로
- 내부 테스트용 흐름

권장:

- `PetCreateScreen` 저장 직전에도 다시 entitlement 체크
- 권한이 없으면 저장 차단 후 구독 유도

즉:

- 1차 가드: 진입 전
- 2차 가드: 저장 전

## 5. 업그레이드 유도 화면 권장 구조

### 제목

- `멀티펫 관리가 필요해요`

### 본문

- `두 마리 이상 함께 관리하려면 멀티펫 구독이 필요해요.`
- `아이별 홈, 기록, 일정을 각각 관리할 수 있어요.`

### 가격 영역

- `월 990원`

### CTA

- `구독 시작하기`
- `나중에 할게요`

## 6. 왜 이렇게 설명해야 하는가

잘못된 설명:

- `아이를 더 추가하려면 결제하세요`

더 나은 설명:

- `여러 아이를 각각 관리하는 기능은 멀티펫 구독으로 제공돼요`

이유:

- 현재 NURI는 아이별 독립 컨텍스트 관리가 실제 가치다
- 단순 슬롯 판매처럼 보이면 사용자가 납득하기 어렵다

## 7. 현재 선택 펫과의 관계

구독 UX는 현재 선택 펫 흐름을 깨면 안 된다.

즉:

- 기존 1마리 사용자는 지금 흐름이 그대로 유지돼야 한다
- 업그레이드가 필요하다고 해서 현재 선택 펫 컨텍스트가 흔들리면 안 된다

## 8. 만료 후 UX 정책

권장:

- 기존 등록된 펫은 계속 보이게 한다
- 기존 펫 전환도 유지 가능하게 한다
- 새 펫 등록만 막는다

화면 메시지 예시:

- `멀티펫 구독이 만료되어 새 아이 등록은 잠시 제한돼 있어요.`

중요:

기존 데이터 접근을 갑자기 막는 UX는 피한다.

## 9. 복원 UX

필수:

- 결제 복원 버튼
- 복원 중 로딩
- 복원 성공/실패 안내

현재 추천 위치:

- 구독 소개 화면 하단
- 설정/더보기 내부 결제 관리 영역

## 10. 현재 시점 권장 결론

멀티펫 구독 UX는 강한 결제 유도보다,
현재 NURI가 이미 제공하는 `아이별 독립 관리 가치`를 사용자에게 자연스럽게 설명하는 방향이 맞다.

1차 구현은:

- 등록 전 가드
- 저장 전 가드
- 업그레이드 소개 화면
- 복원 버튼

이 4개를 최소 기준으로 잡는 것이 맞다.
