# 멀티펫 구독 상세내용

## 1. 문서 목적

이 문서는 `멀티펫 구독 구현기획서`를 실제 구현 단위까지 내려서 정리한 상세 설계 문서다.

이번 문서의 목표는 새 작업자가 이 문서 하나만 읽고도 아래를 바로 이해하는 것이다.

- 현재 앱 구조에서 어디에 구독 로직이 붙는지
- 어떤 DB/상태 모델이 필요한지
- 어떤 UX로 차단/업그레이드가 일어나야 하는지
- 어떤 리스크를 먼저 막아야 하는지

## 2. 현재 실제 구조 기준 연결 포인트

### 펫 목록/선택 상태

- `src/store/petStore.ts`

현재 역할:

- `pets`
- `selectedPetId`
- `setPets`
- `upsertPet`
- `selectPet`

해석:

멀티펫 구독은 결국 `펫 수 제한`과 연결된다.
따라서 구독 상태 판단은 펫 생성/선택 흐름과 직접 맞물린다.

### 펫 생성 흐름

- `src/screens/Pets/PetCreateScreen.tsx`

현재 역할:

- 펫 등록 폼
- 이미지 업로드
- `createPet` 호출
- 생성 후 `fetchMyPets -> setPets`

구독 연결 포인트:

- 등록 화면 진입 전
- 등록 화면 저장 직전

### 멀티펫 관리 허브

- `src/screens/Pets/PetManagementScreen.tsx`

현재 역할:

- 등록된 아이 목록
- 현재 선택 상태 표시
- `선택`
- `프로필 수정`
- `아이 등록하기`

구독 연결 포인트:

- `아이 등록하기` CTA
- 향후 업그레이드 유도 진입

### 더보기 진입

- `src/screens/More/MoreDrawerContent.tsx`

현재 역할:

- `아이들 프로필 관리` 진입 허브

구독 연결 포인트:

- 멀티펫 구독 소개 진입을 붙일 수 있는 잠재 위치

## 3. 현재 코드에 없는 것

실제 코드 기준으로 아래는 없다.

- 결제 SDK
- 결제 상태 store
- entitlement 판단 서비스
- 구독 상품 메타
- 구매/복원/만료 처리
- 서버 DB의 구독 상태 테이블
- 운영자용 구독 상태 확인 화면

즉 이번 기능은 완전히 새 챕터다.

## 4. 권장 아키텍처

### 앱 레이어

- `store/subscriptionStore.ts`
- `services/subscription/*`
- `screens/Subscription/*`

현재 실제 프로젝트에는 없고, 나중에 새로 추가해야 할 레이어다.

### 권장 상태 모델

최소 필요 상태:

- `status`
  - `free`
  - `active`
  - `grace`
  - `expired`
  - `unknown`
- `canAddMorePets`
- `maxPetCount`
- `currentPetCount`
- `productId`
- `expiresAt`
- `lastValidatedAt`

중요:

앱 로컬 계산만으로 최종 판단하면 안 된다.
최종 source of truth는 서버/DB여야 한다.

## 5. 권장 DB/운영 모델

현재 SQL에는 멀티펫 구독용 스키마가 없다.

따라서 최소한 아래 정도는 필요하다.

### 권장 테이블

#### `user_subscriptions`

- `id`
- `user_id`
- `provider`
  - `app_store`
  - `play_store`
- `product_id`
- `status`
  - `active`
  - `grace`
  - `expired`
  - `revoked`
- `started_at`
- `expires_at`
- `original_transaction_id`
- `latest_transaction_id`
- `validated_at`
- `created_at`
- `updated_at`

#### `user_entitlements`

- `user_id`
- `feature_key`
  - 예: `multi_pet`
- `status`
  - `active`
  - `inactive`
- `source`
  - `subscription`
  - `admin`
  - `legacy`
- `max_pet_count`
- `effective_from`
- `effective_until`
- `updated_at`

#### `billing_events`

- provider webhook 또는 검증 로그용
- 복원/만료/해지 이력 점검용

중요:

초기에는 `user_subscriptions`와 `user_entitlements` 2축만으로도 시작 가능하다.
하지만 장기 운영을 생각하면 `billing_events`까지 있어야 사고 분석이 가능하다.

## 6. 권장 권한 판단 규칙

### 앱에서의 판단

앱은 아래를 보여주는 수준까지만 한다.

- 현재 플랜 상태
- 멀티펫 등록 가능 여부
- 업그레이드 필요 여부

### 서버에서의 판단

최종 등록 가능 여부는 서버 기준이어야 한다.

예:

- 현재 펫 수
- entitlement 상태
- 최대 허용 펫 수

권장 규칙:

- 무료: `max_pet_count = 1`
- 구독 활성: `max_pet_count >= 2`

## 7. UX 설계

### 진입 1: 멀티펫 관리 화면

현재 버튼:

- `아이 등록하기`

무료 사용자가 이미 1마리를 가지고 있으면:

- 바로 등록 화면으로 보내지 않는다
- 먼저 업그레이드 유도 화면 또는 모달을 띄운다

### 진입 2: 펫 등록 화면 저장 직전

안전장치로 한 번 더 막아야 한다.

이유:

- deep link
- 예전 경로
- 내부 예외 흐름

때문에 화면 진입 전만 막으면 불완전하다.

따라서 2중 가드가 맞다.

- 1차: 진입 전 가드
- 2차: 저장 직전 가드

### 권장 문구 톤

- 제목: `멀티펫 관리가 필요해요`
- 본문: `두 마리 이상 함께 관리하려면 멀티펫 구독이 필요해요.`
- 버튼:
  - `구독 알아보기`
  - `나중에 할게요`

## 8. 구독 해지/만료 정책

이건 반드시 구현 전 확정해야 한다.

### 추천 정책

- 기존 등록 데이터 삭제 금지
- 읽기 유지
- 새 등록 차단

### 이유

- 데이터 접근을 갑자기 막으면 CS 비용이 커진다
- 기록/일정/추억 데이터는 사용자 자산 성격이 강하다

### 현재 추천 해석

만료 후에도:

- 기존 펫 목록 조회 가능
- 기존 펫 전환 가능
- 새 펫 등록만 차단

이 정책이 현재 NURI 구조와 가장 잘 맞는다.

## 9. 렌더링/성능 관점 주의점

구독 도입은 결제만의 문제가 아니다.

현재 멀티펫 관리 허브는 이미 성능 기준으로 설계돼 있다.

- `FlatList`
- memo 카드
- 전역 `petStore`

구독 가드가 들어올 때 아래를 피해야 한다.

- 카드마다 구독 상태 판단 API 호출
- `usePetStore()` 전체 구독
- 버튼 렌더마다 무거운 구독 상태 계산

권장:

- 구독 상태는 별도 store에서 한 번만 들고
- 화면은 selector로 최소 구독
- CTA 클릭 시점에만 분기

## 10. 장기 운영 리스크

### 1. 로컬 상태만 믿는 구조

위험:

- 재설치 후 꼬임
- 기기 간 불일치

### 2. 결제는 되었는데 권한 반영이 늦는 구조

위험:

- 구매 직후 등록 차단
- 환불/CS 발생

### 3. 만료 정책 미정

위험:

- 이미 등록된 펫을 어떻게 다룰지 혼선

### 4. 운영자 확인 수단 부족

위험:

- 특정 유저 entitlement 상태를 바로 확인 못 함

## 11. 권장 구현 단계

### 0단계

정책 문서 확정

- 무료/유료 경계
- 만료/복원 정책
- 가격/상품명

### 1단계

DB/서버 entitlement 설계

- `user_subscriptions`
- `user_entitlements`

### 2단계

앱 구독 상태 레이어

- subscription store
- entitlement fetch
- 로딩/오류 상태

### 3단계

등록 차단 UX

- `PetManagementScreen`
- `PetCreateScreen`

### 4단계

실제 결제 연동

- 구매
- 복원
- 만료 반영

### 5단계

운영/예외 처리

- 검증 실패
- grace period
- revoke

## 12. 현재 시점 결론

현재 NURI는 멀티펫 구독을 붙일 가치와 구조를 이미 어느 정도 갖고 있다.
하지만 실제 결제 기능을 붙이려면 아직 아래가 필요하다.

- 구독 상태 source of truth
- 서버 entitlement
- 등록 차단 UX
- 복원/만료 정책

즉 지금은 바로 결제 SDK부터 붙일 단계가 아니라,
정책/DB/권한 모델을 먼저 확정해야 하는 단계다.
