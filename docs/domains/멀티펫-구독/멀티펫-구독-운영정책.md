# 멀티펫 구독 운영정책

## 1. 문서 목적

이 문서는 `멀티펫 구독`이 실제 운영에 들어갔을 때 CS, 만료, 복원, 예외 케이스를 어떻게 다룰지 보수적으로 정리한 문서다.

## 2. 현재 실제 상태

현재 프로젝트에는 운영정책을 코드로 강제하는 구조가 없다.
따라서 구현 전에 운영정책을 먼저 고정해야 한다.

## 3. 핵심 운영 원칙

- 사용자 데이터는 함부로 숨기거나 삭제하지 않는다
- entitlement만 바꾸고, 기존 펫/기록 데이터는 유지한다
- 결제 오류와 권한 반영 오류를 구분해서 본다
- 운영자는 특정 유저 상태를 빠르게 확인할 수 있어야 한다

## 4. 무료 → 유료 전환 정책

- 유료 전환 즉시 멀티펫 등록 허용
- 기존 UI 새로고침 또는 재진입 시 즉시 반영

## 5. 유료 → 만료 전환 정책

권장:

- 기존 펫 목록 유지
- 기존 펫 선택/조회 유지
- 새 펫 등록만 제한

이유:

- 데이터 볼모 UX를 피해야 한다
- 구독 만료가 데이터 접근 차단으로 보이면 서비스 신뢰가 크게 떨어진다

## 6. 복원 정책

복원은 필수다.

상황:

- 기기 변경
- 앱 재설치
- 로그인 재진입

복원 실패 시:

- 바로 entitlement를 free로 확정하지 말고
- `확인 중` 또는 `다시 시도` 상태를 둘 필요가 있다

## 7. CS 관점에서 꼭 필요한 것

운영자는 아래를 빠르게 확인할 수 있어야 한다.

- 현재 구독 상태
- 마지막 검증 시간
- 만료 시간
- product id
- 복원 여부
- 최근 billing event

즉 운영용 최소 조회 도구가 없으면 서비스급 운영이 어렵다.

## 8. 예외 케이스

### legacy 사용자

개발/테스트/이벤트 등으로 이미 여러 마리를 가진 사용자

권장:

- `legacy` entitlement source로 예외 허용

### admin 수동 부여

필요 시 운영자가 예외적으로 멀티펫 권한을 줄 수 있어야 한다.

권장:

- `user_entitlements.source = admin`

## 9. 정책상 지금 당장 하면 안 되는 것

- 만료 즉시 기존 펫 숨김
- 만료 즉시 현재 선택 펫 강제 초기화
- 유저가 이미 입력한 기록/일정 접근 차단
- 클라이언트 로컬 판단만으로 최종 권한 결정

## 10. 현재 시점 권장 결론

운영정책은 `권한 차단`보다 `데이터 보호`를 우선해야 한다.

현재 NURI에 가장 맞는 방향은:

- 기존 데이터 유지
- 읽기 유지
- 새 등록만 차단
- entitlement 중심 운영

이다.
