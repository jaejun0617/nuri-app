# 멀티펫 구독 구현기획서

## 1. 문서 목적

이 문서는 NURI 앱의 현재 실제 구조를 기준으로 `멀티펫 구독` 기능을 서비스급으로 도입하기 위한 1차 구현 기획서다.

이번 문서는 아이디어 메모가 아니라 아래를 바로 판단할 수 있게 만드는 기준 문서다.

- 현재 코드에서 이미 가능한 것
- 아직 비어 있는 것
- 구독 기능을 어디서부터 붙여야 하는지
- 무료/유료 경계를 어떻게 잡아야 하는지
- 결제와 권한을 어떤 순서로 구현해야 하는지

중요:

- 이 문서는 실제 현재 코드/문서 기준으로 작성한다.
- 현재 프로젝트에는 멀티펫 관리 기능은 있으나, 구독 결제 기능은 아직 없다.
- 따라서 멀티펫 기능 구현 완료를 구독 기능 완료로 해석하면 안 된다.

## 2. 현재 실제 상태 요약

### 이미 구현된 것

- 여러 마리 반려동물 등록 가능
- 현재 선택된 반려동물 기준 컨텍스트 유지
  - 홈
  - 타임라인
  - 일정
  - 일부 날씨/가이드/더보기 흐름
- 멀티펫 관리 허브 화면 존재
  - `src/screens/Pets/PetManagementScreen.tsx`
- 펫 생성/수정/선택 전환 흐름 존재
  - `src/screens/Pets/PetCreateScreen.tsx`
  - `src/screens/Pets/PetProfileEditScreen.tsx`
  - `src/store/petStore.ts`

### 아직 구현되지 않은 것

- 인앱결제 SDK 연결
- 구독 상품 정의
- 무료/유료 슬롯 제한 로직
- 구독 상태를 저장하는 서버/DB 레이어
- 결제 복원
- 구독 만료/해지 처리
- 관리자/운영자용 구독 상태 점검 도구

### 현재 해석

지금 NURI는 `멀티펫 기능 기반`은 이미 존재한다.
하지만 `구독 monetization layer`는 아직 없다.

## 3. 왜 멀티펫을 구독으로 보는가

현재 NURI는 반려동물 1마리씩 독립 컨텍스트로 본다.

- 현재 선택된 아이가 바뀌면
  - 홈
  - 기록
  - 일정
  - 일부 추천/정보 흐름
  가 해당 아이 기준으로 바뀐다.

즉 멀티펫은 단순 슬롯 추가가 아니라, `아이별 독립 관리 가치`를 가진다.

따라서 과금 포인트를 아래처럼 해석하는 것이 맞다.

- 잘못된 해석:
  - 펫을 한 마리 더 추가하는 기능만 유료
- 더 적절한 해석:
  - 여러 아이를 각자 독립된 홈/기록/일정 컨텍스트로 관리하는 기능이 유료

## 4. 1차 제품 방향

### 기본 정책

- 기본 1마리 무료
- 2마리 이상 등록/유지는 구독

### 현재 논의 중 가격

- 월 990원

### 현재 가격에 대한 보수적 판단

- 장점
  - 진입 장벽이 낮다
  - 초기 전환 테스트에 부담이 적다
- 주의점
  - 단순히 `펫 한 마리 더 등록`만으로는 매달 결제를 설득하기 약하다
  - 구독은 결제 복원, 환불, 해지, 만료, 고객 기대치까지 같이 생긴다

### 권장 해석

월 990원은 테스트 가격으로는 가능하다.
하지만 상품 설명은 `멀티펫 슬롯`이 아니라 `멀티펫 관리`로 가는 것이 맞다.

예시 설명 방향:

- 여러 아이를 각각 기록하고 관리하는 기능
- 아이별 홈/타임라인/일정 기준 전환
- 멀티펫 관리 허브

## 5. 무료/유료 경계 제안

### 무료

- 펫 1마리 등록
- 현재처럼 홈/기록/일정 사용
- 프로필 수정/테마 변경

### 유료

- 2마리 이상 등록
- 멀티펫 관리 허브 사용
- 현재 선택 펫 전환 기반 다중 관리

### 나중에 확장 가능한 유료 묶음

- 가족 공유
- 펫별 위젯
- 펫별 백업/복원
- 고급 리포트
- 추후 멀티펫 알림 관리 확장

중요:

1차 구현에서는 유료 혜택을 과하게 벌리지 않는다.
현재 이미 있는 실제 가치와 가장 가까운 멀티펫 관리만 우선 과금 대상으로 둔다.

## 6. 현재 코드 기준 도입 방식

### 현재 source of truth

- 펫 전역 상태: `src/store/petStore.ts`
- 펫 생성: `src/screens/Pets/PetCreateScreen.tsx`
- 펫 수정: `src/screens/Pets/PetProfileEditScreen.tsx`
- 멀티펫 관리 허브: `src/screens/Pets/PetManagementScreen.tsx`
- 더보기 진입 허브: `src/screens/More/MoreDrawerContent.tsx`

### 구독이 붙을 핵심 지점

- `PetCreateScreen` 진입 전 또는 저장 직전
- `PetManagementScreen`의 `아이 등록하기` CTA
- More의 `아이들 프로필 관리` 흐름

### 1차 권장 위치

가장 먼저 막아야 하는 지점은 `펫 생성`이다.

즉:

- 무료 사용자가 이미 1마리를 가지고 있으면
- `아이 등록하기` 진입 또는 저장 시점에서
- 구독 필요 화면으로 분기

## 7. 구현 우선순위

### 1단계

구독 정책 문서/상태 모델 정의

- 무료/유료 경계 확정
- 만료 시 정책 확정
- 복원 정책 확정
- 안내 문구 확정

### 2단계

권한 모델 설계

- 현재 유저가 멀티펫 등록 가능한지 판단하는 entitlement 모델
- 앱 로컬 임시 판단과 서버 source of truth 분리

### 3단계

결제 연동

- iOS/Android 상품
- 구매
- 복원
- 해지/만료 반영

### 4단계

멀티펫 등록 차단/업그레이드 UX

- 아이 등록하기 CTA
- 결제 유도 모달/화면
- 성공 후 복귀

### 5단계

운영 도구 및 예외 처리

- 고객 문의 대응
- entitlement 상태 점검
- 만료/복원 꼬임 대응

## 8. 반드시 필요한 정책 결정

아래는 구현 전에 확정해야 한다.

### 무료 사용자가 이미 여러 마리를 가진 경우

가능한 시나리오:

- 기존 테스트 유저
- 개발 중 직접 생성
- 이벤트/운영 조치

정책이 필요하다.

- 기존 등록분은 유지하고 새 등록만 차단할지
- 1마리만 활성화하고 나머지를 잠글지
- 읽기만 허용할지

현재 추천:

- 기존 등록 펫은 읽기 유지
- 새 등록만 차단
- 구독 유도는 등록/전환 확장 시점에만 노출

### 구독 만료 후 정책

반드시 정해야 한다.

- 기존 펫 삭제 금지
- 기존 기록 삭제 금지
- 읽기 유지 여부
- 선택 전환 가능 여부

현재 추천:

- 이미 생성된 펫/기록은 삭제하지 않는다
- 읽기는 유지한다
- 새 펫 등록만 차단한다

이유:

만료 즉시 펫 데이터 접근을 막으면 사용자 반발과 운영 비용이 크게 늘어난다.

## 9. 현재 기준 가장 큰 리스크

- 결제는 아직 코드 기반이 전혀 없다
- 현재 Supabase DB에는 구독 상태 source of truth가 없다
- App Store / Google Play 상태를 앱만 보고 판단하면 꼬일 수 있다
- 무료/유료 경계가 `펫 생성 시점` 외 다른 진입과 충돌할 수 있다
- 만료/복원/재설치에서 entitlement 정합성이 흔들릴 수 있다

## 10. 결론

현재 NURI는 멀티펫 구독을 붙이기 좋은 기반은 이미 갖고 있다.
하지만 결제와 권한 레이어는 아직 시작하지 않았다.

따라서 다음 구현은 반드시 아래 순서로 가야 한다.

1. 정책 결정
2. entitlement 모델 설계
3. 결제 연동
4. 등록 차단/업그레이드 UX
5. 복원/만료/운영 대응

지금 당장 코드로 들어가기 전에, 상세 설계와 DB/운영 기준을 먼저 고정하는 것이 맞다.
