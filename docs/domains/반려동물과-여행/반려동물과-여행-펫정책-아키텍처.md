# 반려동물과 여행 펫정책 아키텍처

## 1. 이번 턴의 핵심 결정

`반려동물과 여행`은 더 이상 TourAPI 단독 결과를 곧바로 `펫 동반 가능`로 해석하지 않는다.

실서비스 기준의 최종 구조는 아래 3개 레이어 조합을 전제로 한다.

- 장소 검색 API
  - Kakao Local API
  - Naver Local API
  - Google Places API
- 펫 동반 여부 데이터 레이어
  - TourAPI KorPetTourService2
  - 자체 `pet_policies` DB
- 신뢰도 보강 레이어
  - 유저 제보
  - 관리자 검수
  - 운영 메타

핵심 한 줄:

`장소 검색 API + TourAPI + 자체 DB` 조합이 필수다.

## 2. 4단계 서비스 아키텍처

### Raw Layer

- 장소 검색 API raw 응답
- TourAPI raw 응답
- 현재 앱 코드에서는 TourAPI raw가 실제 입력 데이터다

### Candidate Layer

- `placeType` 분류
- `travelScore / petScore / commercialPenalty` 계산
- 점수는 모두 `0~1` 정규화
- 최종 점수는 `travel` 중심 가중치로 계산

### Trust Layer

- `pet_policies`
- `user_reports`
- 관리자 검수 메타
- TourAPI raw만으로는 `confirmed` 금지

### Aggregation Layer

- Raw + Candidate + Trust 결과를 합쳐 최종 `petAllowed` 결정
- UI는 보수적으로만 노출
  - `confirmed`
  - `possible`
  - `check-required`

## 3. placeType / petAllowed 분리 원칙

- `placeType`은 장소 타입만 분류한다
- `petAllowed`는 정책 신뢰도만 판단한다
- 숙소/캠핑/목장/농원은 여행 후보 점수는 줄 수 있어도, 강한 정책 근거가 없으면 `현장 확인 필요`를 유지한다
- 펫 관련 정규식은 `possible` 보조 신호까지로만 사용하고, 확정 판단에는 쓰지 않는다

## 4. 점수 구조

현재 앱의 Candidate Layer는 아래 구조를 사용한다.

- `travelScore`
  - 여행지 성격과 검색 적합도
- `petScore`
  - TourAPI raw 펫 신호 또는 이후 DB 검증 신호
- `commercialPenalty`
  - 상업시설/브랜드 점포/쇼핑 오염 감점

운영 원칙:

- 세 점수는 모두 `0~1` 정규화
- `travelScore`가 메인 축
- `petScore`는 `travelScore`를 넘지 못하게 제한
- `commercialPenalty`는 강한 감점 축

## 5. 지역 intent 일반화 원칙

지역명 검색은 특정 예시 지역 패치가 아니라 전국 공통 resolver로 다룬다.

현재 방향:

- 전국 시/도 areaCode 메타 유지
- 전국 공통 지역 별칭/행정명/생활권 별칭을 하나의 region catalog로 관리
- 지역명 단독 검색과 구체 장소 검색을 분리
- region-only query일 때만 `areaBasedList2` narrow mode 사용
- local 지역은 `areaCode + address/title anchor`로 다시 좁힌다

즉 `강원도`, `제주`, `강릉`, `일산`은 예시일 뿐이고, resolver 구조는 전국 공통으로 동작해야 한다.

## 6. DB 확장 방향

이번 턴부터 Trust Layer DB 초안을 실제로 연다.

- `places`
  - canonical place 정보
  - 외부 provider 매핑의 anchor
- `pet_policies`
  - 정책 상태
  - 근거
  - 검수 상태
- `user_reports`
  - 유저 제보/신고
  - 최신성 보강

이번 턴 기준:

- `places`는 내부 uuid PK를 canonical anchor로 사용한다.
- 외부 원본 매핑 전용 `source_links`는 다음 단계로 미루고, 현재는 `primary_source + primary_source_place_id`로 bootstrap한다.
- `pet_policies`는 source별 정책 row를 누적 저장하고, confidence와 evidence를 함께 저장한다.
- `user_reports`는 단순 카운트가 아니라 Trust Layer 근거 원본으로 저장한다.

`confirmed`는 장기적으로 `pet_policies + user_reports + admin-review`가 합쳐졌을 때만 노출하는 상태로 본다.

즉 아래는 계속 금지다.

- TourAPI raw만으로 `confirmed`
- 정규식만으로 `confirmed`
