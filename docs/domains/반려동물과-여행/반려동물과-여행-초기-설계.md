# 반려동물과 여행 초기 설계

## 1. 도메인 목적

`반려동물과 여행`은 현재 위치 근처에서 바로 갈 곳을 찾는 메뉴가 아니다.

이 도메인의 1차 목적은 아래 사용자 시나리오를 먼저 지원하는 것이다.

- 여행지 미리 탐색
- 숙소 / 관광지 / 야외공간 후보 탐색
- 지역별 반려동물 동반 가능 여행 정보 확인

즉 사용자는 "지금 근처 어디 가지?"보다 "이번 주말에 어디로 갈까?"를 먼저 해결하려고 들어온다.

## 2. 펫동반 장소 탐색과의 차이

### 펫동반 장소 탐색

- 일상 생활권
- 현재 위치 기반
- 빠른 주변 탐색
- Kakao Local 중심
- 카페 / 식당 / 실내 공간 / 야외 공간

### 반려동물과 여행

- 여행 / 관광 목적
- 지역 기반
- 미리 비교하는 탐색
- 한국관광공사 반려동물 동반여행 서비스 중심
- 관광지 / 숙소 / 체험 / 야외 여행지

결론적으로 두 메뉴는 이름만 비슷할 뿐 사용자 의도, 데이터 소스, 정렬 기준, 캐시 전략이 다르다.

## 3. 왜 1차에서 GPS 기반이 필수가 아닌가

여행 메뉴는 일상 장소 탐색과 달리 아래 특성이 강하다.

- 방문 지역이 현재 위치와 다를 수 있다
- 출발 전에 후보를 저장하지 않아도 미리 비교해야 한다
- 거리보다 지역과 카테고리 선택이 먼저다
- 관광 데이터는 지역 단위 분류와 컨텐츠 분류가 더 중요하다

따라서 1차에서는 `현재 위치 -> 주변 추천`보다 `검색 입력 -> 카테고리 선택 -> 리스트` 구조가 더 맞다.

GPS 기반 주변 여행지 추천은 2차 확장으로 분리한다.

## 4. 1차 UX 구조

이번 턴 기준 1차 구조는 아래로 고정한다.

- 검색 입력
- 카테고리 선택
- 리스트
- 상세 진입
- 외부 지도 열기

추가 판단:

- 입력어에서 지역명이 잡히면 내부 지역 메타를 통해 `areaCode/sigunguCode`로 변환한다
- 지역명이 정확히 잡히지 않으면 전국 TourAPI 응답을 받아 앱 내부 키워드 필터로 좁힌다
- 고급 필터는 1차 범위 아님
- 현재 위치 / 거리 표시는 1차 범위 아님

## 5. 메뉴 / 라우트 구조

앱 라우트는 아래처럼 분리한다.

- `MoreDrawer`
  - `반려동물과 여행` 메뉴 추가
- `RootNavigator`
  - `PetTravelList`
  - `PetTravelDetail`

구조 원칙:

- `펫동반 장소 탐색` 라우트와 분리한다
- `locationDiscovery` 도메인에 억지로 끼워 넣지 않는다
- `PetTravel` 전용 타입 / 카탈로그 / 화면 레이어를 둔다

## 6. 현재 앱 아키텍처 안에서 공통화와 분리 기준

### 공통화 가능한 것

- 화면 기본 레이아웃
- 외부 지도 열기 유틸
- 정적 지도 미리보기 유틸
- 공통 텍스트 / 화면 래퍼 컴포넌트

### 분리해야 하는 것

- provider
- normalize
- 리스트 필터 구조
- 카테고리 체계
- 상세 노출 정보
- 추천 / 정렬 기준

정리하면, 지도 열기 같은 얇은 유틸은 공유해도 되지만 여행 후보 수집과 정규화는 별도 도메인으로 유지하는 편이 안전하다.

## 7. 한국관광공사 반려동물 동반여행 서비스 API 활용 방향

### 왜 이 API가 여행 메뉴에 맞는가

2026년 3월 19일 확인 기준 공공데이터포털의 `한국관광공사_반려동물_동반여행_서비스`는 최신 수정일이 2026년 2월 26일이며, 전국 관광지, 숙소, 음식점, 쇼핑시설 등 반려동물 동반 여행 정보를 제공하고 지역기반 조회와 위치기반 조회를 함께 제공한다.

여행 메뉴 1차에서는 이 중 `지역기반 조회` 성격이 핵심에 맞다.

이유:

- 생활권 장소보다 관광 / 숙소 / 여행지 데이터에 맞춰져 있다
- 지역 코드 기반 탐색과 카테고리 분류가 자연스럽다
- 반려동물 동반 조건, 유의사항, 이용 가능 시설 같은 여행 문맥 필드가 중요하다

### 우선 봐야 하는 핵심 필드

1차에서 우선 봐야 하는 필드는 아래다.

- 지역 파라미터
  - `areaCode`
  - `sigunguCode`
- 카테고리 파라미터
  - `contentTypeId`
- 식별 / 기본 정보
  - `contentId`
  - `title`
  - `addr1`
  - `tel`
- 좌표 / 지도
  - `mapx`
  - `mapy`
- 대표 이미지
  - `firstimage`
  - `firstimage2`
- 상세 정보 계열
  - 반려동물 동반 조건
  - 반려동물 이용 가능 시설
  - 유의사항
  - 운영 정보

파라미터 운영상 우선 후보:

- `areaCode`
- `sigunguCode`
- `contentTypeId`
- `pageNo`
- `numOfRows`
- 정렬 관련 파라미터
- 공통 TourAPI 인증 / 포맷 파라미터

### 응답 구조에서 필요한 판단

여행 메뉴 1차에서는 아래 응답 구조만 우선 안정화하면 된다.

- 리스트용 요약 필드
  - 이름
  - 주소
  - 카테고리
  - 좌표
  - 대표 이미지
  - 반려동물 동반 핵심 안내 1줄
- 상세용 확장 필드
  - 운영 시간
  - 이용 가능 시설
  - 동반 조건
  - 유의사항
  - 연락처

### Kakao 중심 일상 탐색과의 역할 차이

- Kakao Local
  - 생활권 주변 후보 수집
  - 현재 위치 기준 즉시 탐색에 강함
- 관광공사 반려동물 동반여행 서비스
  - 지역 기반 여행 후보 탐색
  - 관광 / 숙소 / 체험 / 여행지 문맥에 강함

따라서 여행 메뉴 1차 핵심 후보 소스는 관광공사 API가 맞고, Kakao는 보조 좌표 확인이나 지도 연계가 필요할 때만 제한적으로 검토한다.

### 1차 데이터 소스 전략

현재 구현 판단:

- 1차 UI 구조는 이미 열려 있다
- 리스트 조회는 지역 기반 TourAPI 호출로 연결한다
- 상세 조회는 `detailCommon` + `detailIntro` 계열 응답을 함께 읽어 정규화한다
- 지역 코드는 앱 카탈로그에서 검색 입력어를 TourAPI 파라미터로 내부 매핑한다
- 런타임에서는 `TOUR_API_SERVICE_KEY`가 필요하다

운영 전략 판단:

- 앱 1차 오픈 직후에는 제한적 실시간 호출보다 구조 검증이 먼저다
- 중기적으로는 배치 적재 + 캐시 + Supabase 메타 병합이 더 적합하다
- 따라서 장기 source of truth는 Supabase가 가져가되, 여행 후보 원본 소스는 관광공사 API가 맡는다

## 8. 1차 범위

- MoreDrawer 메뉴 추가
- RootNavigator 라우트 추가
- 여행 도메인 전용 타입 / 카탈로그 구조 시작
- 검색 입력 / 카테고리 선택 / 리스트 / 상세 / 외부 지도 열기 화면 구조 오픈
- 관광공사 API를 핵심 후보 소스로 문서와 구조에 반영

## 9. 2차 확장 범위

아래는 이번 턴에서 하지 않는다.

- GPS 기반 주변 여행지 추천
- 현재 위치 기준 거리 표시
- 길찾기 / 이동 동선 추천
- Google Places 보강
- Supabase 여행 메타 레이어
- 즐겨찾기 / 최근 본 여행지
- 여행 일정 / 코스 기능
- 고급 필터
- 자유 검색 중심 UX

## 10. 현재 구현 메모

이번 턴에서는 `PetTravel` 전용 provider/service를 추가해 지역 / 카테고리 기반 리스트와 상세를 실제 TourAPI 응답 기준으로 연결한다.

추가 source of truth:

- 펫 허용 신뢰도와 장기 서비스 구조는 `반려동물과-여행-펫정책-아키텍처.md`를 우선 기준으로 본다.

구현 메모:

- 리스트 조회
  - `areaCode`
  - `sigunguCode`
  - `contentTypeId`
  - `pageNo`
  - `numOfRows`
- 상세 조회
  - `contentId`
  - `contentTypeId`
  - `detailCommon`
  - `detailIntro`
- 응답 정규화
  - 리스트: 이름 / 주소 / 좌표 / 연락처 / 반려동물 안내 / 대표 이미지
  - 상세: overview / 운영 정보 / 휴무 / 주차 / 반려동물 동반 조건 / 이용 가능 시설
