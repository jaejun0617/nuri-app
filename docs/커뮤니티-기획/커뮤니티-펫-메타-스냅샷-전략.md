# 커뮤니티 펫 메타 스냅샷 전략

## 목적

커뮤니티 게시글에 연결된 반려동물 정보가 `live pet profile` 변경에 따라 뒤늦게 바뀌는 리스크를 줄이기 위해,
게시글 작성 시점 기준의 펫 메타를 어떻게 저장하고 노출할지 기준을 고정한다.

## 현재 상태

- 현재 커뮤니티 게시글은 `posts.pet_id`만 저장한다.
- 목록/상세 화면은 `pet_id`를 기준으로 현재 `pets` 테이블을 다시 읽어 와서
  `petName`, `petSpecies`를 화면용으로 조합한다.
- 작성 화면의 `나이 함께 표시`는 현재 작성 UX와 draft 복원에만 반영된다.
- 즉, 현재 구조는 `live pet profile 참조`다.

## 현재 구조의 리스크

1. 반려동물 이름/품종/대표 종이 바뀌면 과거 게시글의 메타도 같이 바뀐다.
2. 반려동물 나이를 게시글별로 보일지 말지 결정해도, 현재는 영구 저장되지 않는다.
3. 추후 반려동물 프로필 삭제/비공개/추모 정책이 바뀌면 과거 게시글 해석이 흔들릴 수 있다.
4. 커뮤니티는 공개 도메인이라, 타임라인처럼 live 참조만 믿고 가면 과거 글의 의미가 뒤틀릴 수 있다.

## 최종 권장 방향

커뮤니티는 `pet_id live 연결`과 `post snapshot`을 함께 유지한다.

- `pet_id`
  - 현재 어떤 아이와 연결된 글인지 추적하는 참조값
  - 향후 필터, 내 아이 글 모아보기, 상세 진입 연결에 사용
- snapshot 필드
  - 작성 시점의 이름/품종/나이/프로필 이미지 정보를 고정 저장
  - 목록/상세 기본 노출은 이 snapshot을 우선 사용

즉:

- 관계 추적은 `pet_id`
- 사용자 노출 메타는 `snapshot`

이 구조가 장기 운영 기준으로 가장 안전하다.

## 권장 snapshot 필드

`posts` 보강 컬럼 후보:

- `pet_snapshot_name text`
- `pet_snapshot_species text`
- `pet_snapshot_age_label text`
- `pet_snapshot_avatar_path text`
- `show_pet_age boolean not null default true`

주의:

- `pet_snapshot_avatar_path`는 signed URL이 아니라 storage path 저장 기준으로 간다.
- 실제 렌더 시 signed URL 해석은 앱 레이어가 담당한다.
- `show_pet_age = false`면 age label이 있어도 UI에서 숨긴다.

## 작성 시 저장 규칙

게시글 작성 시:

1. `pet_id` 선택 여부 확인
2. 선택된 펫이 있으면 현재 store 기준으로 아래 snapshot을 만든다
   - 이름
   - 품종 또는 대표 종
   - 나이 라벨
   - 프로필 이미지 path
3. `show_pet_age` 상태를 함께 저장한다
4. 게시글 본문과 동일 트랜잭션 흐름에서 저장한다

## 수정 시 규칙

게시글 수정 시:

- 본문/카테고리만 수정하면 snapshot은 유지
- 반려동물 연결을 다른 아이로 바꾸면 snapshot 전부 새 아이 기준으로 재생성
- `나이 함께 표시` 토글만 바꾸면 `show_pet_age`만 수정

## 목록/상세 렌더 규칙

우선순위:

1. snapshot 필드가 있으면 snapshot 사용
2. snapshot이 비어 있고 `pet_id`가 있으면 live pet fallback 사용
3. 둘 다 없으면 반려동물 메타 미노출

이렇게 해야:

- 신규 게시글은 안정적으로 고정 메타 표시
- 기존 게시글은 마이그레이션 전까지 live fallback 유지

## 이미지 draft 복원 정책

작성 화면 draft의 첨부 이미지는 로컬 URI를 저장하므로, 앱 완전 재시작/OS 정리 후 복원이 실패할 수 있다.

현재 권장 UX:

- 본문/카테고리/반려동물 선택/나이 토글은 우선 복원
- 이미지 URI가 유효하지 않으면 이미지만 제거
- 사용자에게 경고 모달 또는 토스트로 다시 선택을 안내

즉, draft 이미지 복원은 `best effort`로 간다.

## 구현 단계 권장

1. SQL 보강
2. community 타입 보강
3. service create/update payload 보강
4. 작성/수정 화면 snapshot 생성 로직 연결
5. 목록/상세 렌더를 snapshot 우선 구조로 전환
6. 기존 게시글 fallback 정책 검증

## 이번 단계에서 하지 않는 것

- 과거 게시글 일괄 snapshot backfill
- 펫 프로필 변경 시 게시글 snapshot 자동 갱신
- 반려동물 공개 범위 정책과 커뮤니티 노출 정책 완전 연동

## 결론

커뮤니티는 공개 도메인이므로, 반려동물 메타를 live 참조만으로 유지하면 장기 운영에서 의미가 흔들린다.
따라서 `pet_id + snapshot 병행 구조`를 다음 단계의 공식 기준으로 채택한다.
