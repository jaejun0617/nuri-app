# NURI 커뮤니티 실서비스급 아키텍처 기획서

작성일: 2026-03-21  
작성 기준: 실제 앱 코드, 실제 docs, 실제 SQL, `docs/project-memory`, `docs/engineering/codex-workflow.md`

## 0. 문서 목적

이 문서는 NURI 프로젝트에 커뮤니티 도메인을 도입할 경우, 현재 코드베이스와 SQL 자산을 기준으로 어떤 범위가 현실적인지, 어떤 구조가 안전한지, 어떤 리스크를 먼저 막아야 하는지를 정리한 실서비스급 기획/아키텍처 문서다.

이 문서는 아이디어 브레인스토밍 문서가 아니다. 아래를 판단하기 위한 기준 문서다.

- NURI에 커뮤니티를 지금 넣는 것이 맞는가
- 넣는다면 어떤 범위가 가장 안전한가
- 현재 SQL 자산을 어디까지 재사용할 수 있는가
- 실서비스 운영, 타입 안전성, Supabase RLS, moderation 비용을 감당 가능한가

이 문서는 프로젝트 내부 공식 기준 문서다. 외부 기획서의 화면 흐름과 표 구성은 참고할 수 있지만, 현재 구현 상태와 운영 준비 수준 판단은 이 문서를 source of truth로 본다.

## 1. 전제와 현재 상태 요약

현재 NURI 앱은 반려동물 기록/일정/위치기반/가이드 중심 앱이다. 실제 코드 기준으로는 Auth, Pets, Records, Schedules, Weather, 산책, 펫동반 장소 탐색, 반려동물과 여행, Guides 도메인이 구현돼 있다.

커뮤니티와 직접 연결된 앱 런타임 코드는 현재 확인되지 않았다.

- `src/screens`, `src/navigation`, `src/services`, `src/store`에서 커뮤니티 전용 화면/스토어/서비스는 없다.
- `docs/project-memory/최근-작업-로그.md` 기준으로 More 메뉴에서 `커뮤니티` 노출은 오히려 제거된 상태다.
- 반면 SQL 초기 구성에는 `posts`, `comments`, `likes`, `reports`, `notifications`, `audit_logs`가 존재한다.
- `notification_type` enum에는 `community_comment`, `community_like`가 포함돼 있다.

즉 현재 상태는 아래로 판단하는 것이 정확하다.

- 앱 런타임 기준 커뮤니티: 미구현
- DB 초기 자산 기준 커뮤니티 씨앗: 존재
- 운영/모더레이션/UI/서비스 레이어: 미구현

따라서 이번 문서의 핵심은 “기존 SQL 씨앗을 그대로 믿고 바로 커뮤니티를 여는 것”이 아니라, “현재 자산을 어디까지 재사용할 수 있고 무엇을 재설계해야 하는지”를 보수적으로 구분하는 것이다.

## 2. 커뮤니티 도메인 도입 목적

### 2.1 왜 필요한가

NURI의 현재 핵심 구조는 기록, 케어, 생활형 탐색, 가이드다. 이 구조에서 커뮤니티는 아래 세 가지 역할을 가질 수 있다.

- 기록형 앱의 체류시간을 늘리는 소셜 보조 레이어
- 가이드/생활 팁/실사용 경험을 사용자 생성 콘텐츠로 확장하는 참여 레이어
- 반려동물 양육 경험을 공유하는 신뢰 형성 레이어

### 2.2 왜 지금은 조심해야 하는가

현재 프로젝트 전체 진행 방향은 `운영 완성도`, `실기기 QA`, `위치기반/가이드/타임라인 마감`에 가깝다. 즉 커뮤니티는 자연스러운 확장 도메인이지만, 현재 앱의 최우선 과제는 아니다.

커뮤니티는 다른 도메인보다 운영 비용이 훨씬 크다.

- moderation
- 신고 처리
- 스팸/도배 대응
- 알림 확장
- 개인정보/반려동물 정보 노출 통제
- 운영자 도구

즉 커뮤니티는 “기능 한두 개 추가”가 아니라 “운영 가능한 서비스 단위”로 봐야 한다.

### 2.3 NURI에 맞는 커뮤니티의 정체성

NURI에 가장 자연스러운 커뮤니티는 범용 SNS가 아니라 `반려동물 생활형 커뮤니티`다. 시작 범위는 아래가 적절하다.

- 짧은 생활 공유형 게시글
- 반려동물과 연결 가능한 작성 구조
- 이미지 1~4장 수준의 가벼운 첨부
- 댓글
- 좋아요
- 신고

반대로 초기부터 과한 범위는 맞지 않다.

- 익명 커뮤니티
- 실시간 채팅
- 대형 피드 추천 알고리즘
- 대댓글 무제한 트리
- 그룹/오픈채팅/팔로우 그래프
- 영상 중심 피드

### 2.4 타임라인 vs 커뮤니티 분리 원칙

타임라인과 커뮤니티는 카드 UI 일부를 재사용할 수 있어도, 데이터 모델과 운영 모델은 분리해야 한다.

| 구분 | 타임라인 | 커뮤니티 |
| --- | --- | --- |
| 관계 구조 | 나 + 내 반려동물 중심 1:N | 보호자 간 N:N |
| 데이터 성격 | 개인 기록, 회고, 보관 | 공개 발화, 반응, 신고 |
| 주 저장소 | `memories` 계열 | 별도 `posts/comments/reports` 계열 |
| 핵심 UX | 닫힘, 정적, 개인 히스토리 | 열림, 발견, 상호작용 |
| 핵심 리스크 | 기록 유실, 개인 데이터 정합성 | 스팸, 악용, 신고, moderation |
| 상태 관리 | `recordStore` 중심 | 별도 `communityStore` 필요 |
| 운영 도구 | 개인 데이터 복구/정합성 | 신고 큐, 숨김/복구, 제재 |

즉 타임라인 기록이 커뮤니티로 자동 노출되면 안 되고, store/service/RLS도 별도 레이어로 다뤄야 한다.

## 3. 현재 NURI 구조와의 접점 분석

### 3.1 Auth / Profile / Role

현재 Auth 구조는 커뮤니티와 자연스럽게 연결될 수 있다.

- `authStore`가 세션, 로그인 상태, 프로필, role을 이미 들고 있다.
- `profiles.nickname`과 `nickname_confirmed` 흐름이 있어 작성자 표시 이름 기반은 이미 있다.
- `profiles.role`이 존재하고, 가이드 관리자에서 `admin`, `super_admin` guard를 실제로 사용하고 있다.

따라서 커뮤니티에서도 아래는 재사용 가능하다.

- 로그인 상태 가드
- 작성자 식별
- 관리자 role 해석

### 3.2 Pets

현재 `posts.pet_id`가 SQL에 이미 존재한다. 앱도 멀티펫 구조와 선택 펫 컨텍스트가 있다. 이는 커뮤니티 게시글에 반려동물 연결을 옵션으로 붙이기 좋은 상태다.

다만 주의가 필요하다.

- 게시글과 펫 연결은 “선택 사항”이어야 한다.
- 펫의 민감 정보 노출 범위를 따로 정의해야 한다.
- 게시글에서 펫 공개 시 어떤 필드만 노출할지 별도 정책이 필요하다.

### 3.3 Records / Timeline

기록과 커뮤니티는 성격이 다르다.

- `memories`는 개인/가정 단위 기록 데이터
- 커뮤니티 게시글은 공개 소통 데이터

둘을 한 테이블로 섞으면 안 된다.

연결 가능한 지점은 아래 정도다.

- 기록에서 커뮤니티 공유로 내보내기
- 커뮤니티 글 작성 시 최근 기록을 참고하는 UX
- 기록 이미지 재사용 여부

하지만 저장소와 정책은 분리해야 한다.

### 3.4 Home / More / Guides / Notifications

현재 구조상 커뮤니티 진입 위치 후보는 세 가지다.

- 하단 탭 신규 진입
- More 드로어 진입
- 홈 카드 진입

현재 NURI 구조를 기준으로 가장 안전한 초기 위치는 `More 드로어 진입`이다.

이유:

- 현재 하단 탭은 Home / Timeline / 중앙 기록 / More 구조로 안정화돼 있다.
- 새로운 탭 추가는 하단 구조와 기록 작성 FAB, 툴바 복귀 흐름까지 다시 흔든다.
- 커뮤니티는 초기에는 핵심 작동축보다 확장 도메인이다.

홈 연동은 2차가 맞다.

- 인기 글
- 최근 반응
- 가이드와 연결된 커뮤니티 카드

알림은 장기적으로 연결 가능하지만, 초기에 강하게 열면 운영 부담이 커진다.

### 3.5 현재 코드 패턴 재사용 가능성

재사용 가능한 패턴:

- Auth / role guard
- RootNavigator + AppTabs + More 허브 구조
- Zustand store 패턴
- Supabase service 패턴
- GlobalToast, branded error, AsyncStorage draft 패턴
- GuideAdminListScreen 수준의 관리자 권한 차단 UX

재사용하기 어려운 부분:

- 현재 SQL `posts/comments/likes/reports`를 그대로 서비스급 커뮤니티에 올리는 것
- 현재 notifications를 커뮤니티 알림까지 즉시 확장하는 것
- audit_logs를 moderation 운영 로그로 바로 보는 것

## 4. 기존 SQL 자산 재사용성 평가

### 4.1 `posts`

현재 정의:

- `user_id`
- `pet_id`
- `visibility`
- `content`
- `image_url`
- `created_at`
- `updated_at`

장점:

- 작성자와 펫 연결 구조가 있다.
- `visibility` enum이 있다.
- `updated_at` trigger가 있다.

한계:

- 제목 없음
- 카테고리 없음
- 상태(status) 없음
- moderation 상태 없음
- soft delete 없음
- denormalized counter 없음
- 이미지가 `image_url` 단일 필드
- 검색용 컬럼/tsvector 없음
- 공개 범위가 `public/private` 2단계뿐

평가:

- `posts`는 MVP 씨앗으로는 재사용 가능하다.
- 그러나 실서비스 커뮤니티 테이블로는 그대로 부족하다.
- 결론: `부분 재사용 또는 확장 재정의 필요`

권고:

- 기존 `posts`를 직접 쓰더라도 컬럼 추가와 정책 재정의가 필수다.
- 안전하게는 `community_posts` 전용 테이블로 분리하는 편이 장기 유지보수에 더 낫다.

### 4.2 `comments`

현재 정의:

- `post_id`
- `user_id`
- `content`
- `created_at`

장점:

- 기본 댓글 구조는 있다.
- post visibility를 따라 select 정책이 있다.

한계:

- 수정 시간 없음
- soft delete 없음
- moderation 상태 없음
- 대댓글 구조 없음
- 정렬/페이지네이션 확장 여지 약함

평가:

- 단순 1레벨 댓글 MVP는 가능
- 실서비스 운영 댓글 구조로는 부족
- 결론: `부분 재사용 가능, 운영 확장용 컬럼 필요`

### 4.3 `likes`

현재 정의:

- `post_id`
- `user_id`
- unique(post_id, user_id)

장점:

- 가장 단순하고 안전한 구조
- unique 제약으로 중복 좋아요 방지 가능

한계:

- 댓글 좋아요 없음
- post 외 대상 확장 불가
- counter cache 없음

평가:

- 게시글 좋아요 MVP는 재사용 가능
- 댓글 좋아요나 reaction 확장 시 구조 재정의 필요
- 결론: `게시글 좋아요 MVP 한정 재사용 가능`

### 4.4 `reports`

현재 정의:

- `reporter_id`
- `target_type`
- `target_id`
- `reason`
- `status(open/resolved/rejected)`
- `created_at`

장점:

- 신고 원본 저장용 최소 구조는 있다.
- post/comment 대상 구분 가능

한계:

- 처리자 없음
- 처리 시간 없음
- 처리 메모 없음
- 신고 사유 체계 enum화 없음
- 중복 신고 방지 없음
- moderation action 연결 없음
- 신고 누적 집계 전략 없음

평가:

- 서비스 운영용 신고 시스템으로는 부족
- 원본 신고 접수 테이블로만 겨우 가능
- 결론: `부분 재사용 가능, 운영용 확장 필수`

### 4.5 `notifications`

현재 정의:

- `user_id`
- `type`
- `title`
- `body`
- `reference jsonb`
- `is_read`
- `created_at`

장점:

- 기본 사용자 알림 박스 구조는 있다.
- enum에 `community_comment`, `community_like`가 이미 있다.

한계:

- 알림 dedupe 기준 없음
- 발신자 정보 구조 없음
- 읽음/클릭/소비 이력 확장 약함
- bulk fanout 전략 없음
- unread counter 전략 없음

평가:

- MVP 알림 적재 테이블로는 재사용 가능
- 실서비스 대규모 커뮤니티 알림엔 부족
- 결론: `MVP 한정 재사용 가능`

### 4.6 `audit_logs`

현재 정의:

- `user_id`
- `action`
- `payload`
- `created_at`

장점:

- 범용 로그 적재 공간은 있다.

한계:

- actor role 없음
- target entity/type/id 없음
- moderation workflow 관점의 구조화 부족
- 운영자 메모/처리 히스토리용으론 부족
- select own 정책만 존재

평가:

- 개인 감사 로그 또는 일반 이벤트 기록용 최소 구조
- moderation 운영 로그로 그대로 쓰기엔 부족
- 결론: `부분 재사용 가능하나 커뮤니티 moderation 전용 로그 필요`

## 5. 커뮤니티 기능 범위 정의

### 5.1 MVP

- 게시글 목록
- 게시글 상세
- 게시글 작성
- 게시글 수정/삭제
- 1레벨 댓글
- 게시글 좋아요
- 신고
- 내 글 목록
- 이미지 1~4장
- 펫 연결 옵션

### 5.2 운영 안정화 단계

- 댓글 수정/삭제
- soft delete
- 숨김/제재 상태 반영
- 내 댓글 / 내 활동
- 기본 검색
- 카테고리
- 관리자 신고 처리
- 관리자 숨김/복구

### 5.3 확장 단계

- 댓글 좋아요
- 추천/인기글
- 알림 고도화
- 카테고리 운영
- 대댓글 여부 검토
- 웹 moderation 도구

### 5.4 초기 MVP에서 하지 말아야 할 것

- 익명 게시판
- 대댓글 무제한 트리
- 실시간 채팅
- 무제한 이미지 업로드
- 외부 링크 중심 홍보 게시판
- 팔로우 그래프

### 5.5 MVP 포함 / 제외 기능 표

| 구분 | 포함 | 제외 |
| --- | --- | --- |
| 읽기 | 목록, 상세, 카테고리 필터 | 검색, 인기글, 추천 피드 |
| 쓰기 | 게시글 작성, 수정, 삭제 | 익명 글, 다중 이미지 고도화 |
| 상호작용 | 1레벨 댓글, 게시글 좋아요, 신고 | 대댓글, 댓글 좋아요, 팔로우 |

현재 코드와 운영 준비 수준을 기준으로 보면, 목록/상세 읽기와 제한적 쓰기를 먼저 열고 나머지는 뒤로 미루는 것이 안전하다.

## 7. 실서비스급 안정성 요구사항

### 7.1 동시성 / 중복 요청

- 게시글 작성 중복 제출 방지 필요
- 댓글 중복 제출 방지 필요
- 좋아요는 unique 제약 + 클라이언트 버튼 잠금 필요

### 7.2 정합성

- 댓글 수, 좋아요 수는 read-time count만으로 버티기 어렵다.
- 목록 성능과 정합성을 위해 denormalized counter가 필요하다.
- 단, counter는 trigger 또는 함수로 동기화해야 한다.

### 7.3 soft delete / hard delete

권고:

- 사용자 삭제: soft delete
- 운영 영구 삭제: hard delete 또는 별도 purge

이유:

- 신고/감사/복구/이의제기 대응

### 7.4 moderation 정합성

- post status가 바뀌면 목록/상세/댓글/좋아요 노출이 함께 맞아야 한다.
- hidden, removed, blocked 같은 상태를 UI와 DB가 1:1로 공유해야 한다.

### 7.5 pagination

- `created_at` 단독 커서보다 `created_at + id` 복합 커서가 안전하다.
- 목록은 최신순 기준을 고정해야 한다.
- 카테고리/검색과 커서를 섞을 때 질의 규칙을 명확히 해야 한다.

### 7.6 이미지 업로드

- 업로드 실패 시 게시글 생성과 이미지 업로드를 분리할지 같이 묶을지 먼저 정해야 한다.
- 현재 타임라인처럼 queue/복구 구조를 재사용하는 것이 현실적이다.

### 7.7 abuse 대응

- rate limit
- 짧은 시간 다량 게시 방지
- 동일 본문/링크 도배 탐지
- banned words
- 신고 남용 제한

### 7.8 운영 로그

- 신고 처리 이력
- 운영자 액션
- 제재 이력
- 복구 이력

이건 `audit_logs`만으로 부족하다.

### 7.9 신고 사유 체계 권고

초기 신고 사유는 아래 6종으로 고정하는 편이 안전하다.

| 코드 | 표시 문구 |
| --- | --- |
| `spam` | 스팸 / 도배 |
| `hate` | 혐오 발언 / 욕설 |
| `advertising` | 광고 / 홍보 |
| `misinformation` | 잘못된 정보 |
| `personal_info` | 개인정보 노출 |
| `other` | 기타 |

이 구조는 현재 `reports.reason` 자유 텍스트를 대체하는 것이 아니라, 운영 집계용 분류 체계를 별도로 두는 방향으로 이해해야 한다.

## 9. Supabase DB / SQL / 운영 스키마 설계

### 9.1 결론

현재 `posts/comments/likes/reports/notifications/audit_logs`는 “초기 씨앗”으로는 쓸 수 있지만, 실서비스급 커뮤니티 도메인을 안정적으로 운영하기에는 부족하다.

권고는 아래 둘 중 하나다.

- A안: 기존 테이블을 강하게 확장
- B안: `community_*` 전용 테이블 신설

장기 유지보수와 리스크 격리를 생각하면 B안이 더 안전하다.

- posts category + created_at
- posts user_id + created_at

### 9.6 search

초기에는 제목/본문/카테고리 수준의 단순 검색만 허용하고, 필요하면 `tsvector`를 도입한다.

### 9.7 RLS 방향

- posts: public read + own write + moderated state 제외
- comments: visible post에 한해 read
- likes: visible post 기준 read, own insert/delete
- reports: own insert, own read, admin manage
- moderation actions: admin only

### 9.8 admin 정책

현재 프로젝트 role 체계와 맞춰 아래를 권장한다.

- `admin`
- `super_admin`

커뮤니티 moderation은 최소 admin 이상만 허용해야 한다.

### 9.9 권장 상태 모델

현재 SQL에는 이 상태 모델이 구현되어 있지 않다. 다만 실서비스 운영 관점에서는 아래 수준의 상태 모델을 목표로 잡는 편이 안전하다.

| 상태 | 의미 | 현재 구현 여부 |
| --- | --- | --- |
| `active` | 정상 노출 | 미구현 |
| `hidden` | 운영자 수동 숨김 | 미구현 |
| `auto_hidden` | 신고 임계치 기반 자동 숨김 | 미구현 |
| `deleted` | 작성자 soft delete | 미구현 |
| `banned` | 운영자 영구 제재 | 미구현 |

이 항목은 목표 상태 모델이지, 현재 런타임/SQL 완료 상태가 아니다.

## 10. 운영 / 모더레이션 설계

### 10.1 운영자 기능

- 신고 목록
- 신고 상세
- 게시글 숨김
- 댓글 숨김
- 복구
- 제재
- 운영 메모
- 처리 이력

### 10.2 앱 vs 웹

결론:

- 사용자 소비는 앱
- moderation은 장기적으로 웹 CMS가 맞다

이유:

- 신고 처리
- 검색/필터
- 대량 관리
- 운영 메모
- 처리 이력

이 기능은 앱 관리자 화면보다 웹이 훨씬 낫다.

### 10.3 앱 vs 웹 역할 분리

| 채널 | 담당 범위 |
| --- | --- |
| 앱 | 게시글 읽기/쓰기, 댓글, 좋아요, 신고 접수 |
| 앱 관리자 긴급 기능 | 필요 시 제한적 즉시 숨김 정도만 검토 |
| 웹 CMS | 신고 대기열, 숨김/복구, 제재, 운영 메모, 처리 이력 |

현재 NURI 저장소 기준으로는 커뮤니티용 웹 CMS 구현 코드는 없다. 따라서 앱 커뮤니티를 열기 전에 웹 운영 도구 준비 상태를 별도 문서로 다시 정리해야 한다.

### 10.4 `audit_logs`와의 관계

`audit_logs`는 보조 로그로는 가능하지만, moderation workflow의 주 저장소가 되면 안 된다.

권고:

- moderation 원본은 `community_moderation_actions`
- 필요 시 `audit_logs`에 요약 복제

## 11. 보안 / 권한 / 악용 방지

핵심 리스크:

- 게시글 비공개/숨김 상태 누출
- 본인 글/댓글이 아닌데 수정/삭제 가능해지는 RLS 누락
- 신고 남용
- 이미지 업로드 악용
- 광고/링크 도배

최소 방어:

- strong RLS
- own write 정책 명확화
- admin only moderation
- 신고 cooldown
- 게시/댓글 rate limit
- banned words / URL 필터
- 업로드 mime / size 제한

## 12. 현재 NURI에 커뮤니티를 넣을 때의 핵심 리스크

- 앱의 현재 우선순위와 맞지 않아 마감 도메인을 흔들 수 있다.
- Home / More / 탭 구조를 다시 흔들 위험이 있다.
- store/service 복잡도가 급격히 증가한다.
- moderation 운영 비용이 앱 도메인 중 가장 높다.
- 기존 SQL 테이블을 과신하면 출시 후 구조적 부채가 된다.
- 알림 시스템이 급격히 복잡해진다.
- “기능은 열렸는데 운영은 불가능한 상태”가 될 가능성이 높다.

- 목표: soft delete, status, search, 내 활동

- 완료 기준: 삭제/숨김/검색 흐름 정합성 확보

### 4단계: moderation / 관리자

- 목표: 운영자가 실제로 관리 가능
- 포함: 신고 큐, 숨김/복구/제재, 운영 메모

- 선행 조건: admin 도구 선택
- 리스크: 앱 관리자 도구로 억지 구현하면 유지비 급증
- 완료 기준: 신고 처리 가능

### 5단계: 확장

- 목표: 알림, 댓글 좋아요, 추천, 카테고리 운영
- 포함: 점진 확장
- 제외: 실시간 채팅
- 선행 조건: 운영 안정화 완료
- 리스크: 핵심 도메인보다 커뮤니티가 과성장
- 완료 기준: 운영 데이터 기반 확장 가능

- moderation 운영 주체
- 앱 관리자 vs 웹 CMS
- 펫 정보 공개 범위
- soft delete 정책
- 알림 범위

### 14.5 구현 시작 전 체크리스트

| 항목 | 왜 먼저 정해야 하는가 |
| --- | --- |
| 커뮤니티 진입점 | More 유지 여부가 navigation 변경 범위를 결정한다 |
| 이미지 허용 장수 | storage, 업로드 실패 복구, signed URL 비용이 달라진다 |
| 게스트 정책 | 읽기 허용 범위와 SignIn 유도 UX가 달라진다 |
| 자동 숨김 임계값 | 신고 누적 trigger와 운영 부담이 달라진다 |
| 알림 포함 단계 | MVP 범위를 크게 바꾼다 |
| 관리자 화면 위치 | 앱 관리자 vs 웹 CMS 선택이 운영 체계를 결정한다 |
| 익명 허용 여부 | moderation 비용과 abuse 방지 난이도를 결정한다 |

### 14.6 최종 판단

보수적으로 보면, NURI의 커뮤니티는 “지금 바로 만들어도 되는 기능”이 아니라 “운영 기반과 SQL 구조를 먼저 정리한 뒤 제한적 범위로 여는 기능”이다. 가장 현실적인 해법은 기존 SQL 씨앗을 참고하되, 실서비스 기준에서는 커뮤니티 전용 스키마와 moderation 구조를 분리 설계하는 것이다.
