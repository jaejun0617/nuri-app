# memory_images fallback 운영 관찰 체크리스트

이 문서는 `memory_images` 전환 이후, legacy read fallback 축소 여부를 판단하기 전에 운영 환경에서 실제로 관찰해야 하는 항목을 정리한 문서다.

목적은 단순하다.

- fallback이 실제로 얼마나 쓰이는지 본다
- 왜 fallback이 동작했는지 구분한다
- 2~4주 관찰 동안 위험 신호와 단순 노이즈를 나눠 본다
- 최종적으로 `Go / Hold` 판단을 할 수 있게 만든다

이 문서는 운영자가 그대로 복붙해서 사용할 수 있는 템플릿 중심으로 작성한다.

---

## 1. 관찰 대상 이벤트

### 1.1 Sentry Discover에서 볼 이벤트

- `memory_images_read_fallback`
- `memory_images_legacy_read_used`
- `memory_upload_queue_recovered`

### 1.2 이벤트별 의미

#### `memory_images_read_fallback`

`memory_images` 조회 자체가 실패해서 legacy read로 내려간 경우다.

- `reason=schema_fallback`
- `reason=runtime_read_failure`

#### `memory_images_legacy_read_used`

실제 읽기 결과가 `memory_images`가 아니라 legacy image path를 사용한 경우다.

- `reason=schema_fallback`
- `reason=runtime_read_failure`
- `reason=recent_create_gap`
- `reason=legacy_only_row`

#### `memory_upload_queue_recovered`

create/edit 직후 즉시 정합이 끝나지 않았지만, queue 재시도로 복구된 경우다.

## 2. Sentry Discover 확인 절차

### 2.1 기본 필터

아래 조건을 기본으로 둔다.

```text
environment:production
release:nuri@0.0.1
```

릴리즈 버전이 올라가면 `release` 값은 실제 배포 버전에 맞춰 바꾼다.

### 2.2 이벤트별 검색 쿼리 예시

#### memory_images_read_fallback

```text
message:"memory_images_read_fallback" environment:production
```

#### memory_images_legacy_read_used

```text
message:"memory_images_legacy_read_used" environment:production
```

#### memory_upload_queue_recovered

```text
message:"memory_upload_queue_recovered" environment:production
```

### 2.3 Discover에서 같이 볼 컬럼

- `timestamp`
- `message`
- `release`
- `user`
- `tags[reason]`
- `tags[source]`
- `tags[mode]`
- `tags[attempts]`
- `contexts.trace` 또는 기본 trace 정보

### 2.4 확인 순서

1. `memory_images_read_fallback`부터 본다
2. `reason=schema_fallback` 존재 여부를 먼저 확인한다
3. `memory_images_legacy_read_used`로 실제 legacy read 사용량을 본다
4. `reason=legacy_only_row`와 `reason=recent_create_gap` 비중을 나눠 본다
5. `memory_upload_queue_recovered`로 queue 복구가 얼마나 실제로 발생하는지 확인한다

## 3. SQL 점검 절차

아래 SQL은 Supabase SQL Editor 또는 운영 점검용 쿼리에서 사용한다.

### 3.1 legacy-only memories 개수

```sql
select count(*) as legacy_only_count
from memories m
where (coalesce(array_length(m.image_urls, 1), 0) > 0 or m.image_url is not null)
  and not exists (
    select 1
    from memory_images mi
    where mi.memory_id = m.id
  );
```

### 3.2 최근 생성 기록 중 memory_images 없는 건수

```sql
select count(*) as recent_gap_count
from memories m
where m.created_at >= now() - interval '24 hours'
  and (coalesce(array_length(m.image_urls, 1), 0) > 0 or m.image_url is not null)
  and not exists (
    select 1
    from memory_images mi
    where mi.memory_id = m.id
  );
```

### 3.3 memory_images 상태 분포

```sql
select status, count(*)
from memory_images
group by status
order by status;
```

### 3.4 failed 상태가 남아 있는 row

```sql
select memory_id, sort_order, original_path, status
from memory_images
where status = 'failed'
order by memory_id, sort_order;
```

### 3.5 최근 7일 상태 분포

`memory_images.created_at` 컬럼이 실제 운영 스키마에 있을 때만 사용한다.

```sql
select status, count(*)
from memory_images
where created_at >= now() - interval '7 days'
group by status
order by status;
```

## 6. event별 기록 항목

### 6.1 `memory_images_read_fallback`

반드시 기록할 항목:

- 총 건수
- `schema_fallback` 건수
- `runtime_read_failure` 건수
- 특정 release 집중 여부
- 특정 시간대 집중 여부

판단 기준:

- `schema_fallback` 1건이라도 있으면 즉시 `Hold`
- `runtime_read_failure`가 반복 패턴이면 `Hold`

### 6.2 `memory_images_legacy_read_used`

반드시 기록할 항목:

- 총 건수
- `source=fetch_list` / `source=fetch_one`
- `recent_create_gap`
- `legacy_only_row`
- `schema_fallback`
- `runtime_read_failure`

판단 기준:

- `legacy_only_row`가 계속 나오면 fallback 유지
- `recent_create_gap`가 높으면 create 직후 공백 축소가 먼저
- `schema_fallback`, `runtime_read_failure`가 붙어 있으면 fallback 축소 논의 중단

### 6.3 `memory_upload_queue_recovered`

반드시 기록할 항목:

- 총 건수
- create/edit 비율
- attempts 분포
- release별 추세

판단 기준:

- 소수 발생은 허용 가능
- 증가 추세면 구조 안정화가 먼저

## 7. fallback 축소 판단 체크리스트

아래 항목은 실제 판단 직전 마지막으로 체크한다.

```md
# fallback 축소 판단 체크리스트

- [ ] 최근 2~4주 동안 schema_fallback = 0
- [ ] runtime_read_failure가 반복 패턴이 아님
- [ ] legacy_only_row가 사실상 0에 수렴
- [ ] SQL 기준 legacy_only_count도 사실상 0에 수렴
- [ ] recent_create_gap가 낮고 안정적
- [ ] memory_upload_queue_recovered가 증가 추세가 아님
- [ ] 특정 release에서만 급증하는 패턴이 없음
- [ ] 특정 user/pet에 몰린 이상 패턴이 없음
- [ ] failed memory_images row가 장기 잔존하지 않음
- [ ] 운영 관찰 결과, fallback이 실제 안전망으로 거의 작동하지 않는다고 볼 수 있음
```

## 8. Go / Hold 기준

### 8.1 Go

아래 조건이 모두 충족되면, legacy read fallback 축소를 **검토 시작**할 수 있다.

- 최근 2~4주 동안 `schema_fallback = 0`
- `runtime_read_failure`가 사실상 0이거나 반복 패턴이 없음
- `legacy_only_row`가 사실상 0
- `recent_create_gap`가 낮고 안정적
- queue recovered가 증가 추세가 아님
- SQL 기준 legacy-only 데이터가 정리된 상태

### 8.2 Hold

아래 중 하나라도 있으면 fallback 축소는 보류한다.

- `schema_fallback` 1건 이상
- `runtime_read_failure` 반복
- `legacy_only_row` 지속 발생
- `recent_create_gap`가 높음
- `memory_upload_queue_recovered` 증가 추세
- failed `memory_images` backlog 잔존

---

## 9. 운영자가 바로 쓰는 짧은 점검 순서

1. Sentry Discover에서 3개 이벤트가 수집되는지 먼저 확인한다
2. `schema_fallback` 존재 여부를 먼저 본다
3. `legacy_only_row`와 `recent_create_gap`를 분리해서 집계한다
4. SQL로 실제 legacy-only row 개수를 확인한다
5. 일간 기록 템플릿에 숫자를 적는다
6. 주간 리뷰 템플릿으로 추세를 본다
7. 2~4주 누적 후 Go / Hold를 판단한다

---

## 10. 결론

지금 단계에서 중요한 것은 fallback을 빨리 제거하는 것이 아니라,  
fallback이 **왜**, **얼마나**, **어떤 패턴으로** 동작하는지 운영에서 분명히 보는 것이다.

이 문서는 그 판단을 감이 아니라 기록과 수치로 하게 만들기 위한 운영 템플릿이다.
