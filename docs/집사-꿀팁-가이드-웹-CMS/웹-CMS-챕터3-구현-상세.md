# 집사 꿀팁 가이드 웹 CMS 챕터 3 구현 상세

## 1. 문서 목적

이 문서는 `nuri-web` 기준 집사 꿀팁 가이드 웹 CMS 챕터 3의 범위와 구현 구조를 정리한다.

이번 챕터의 목적은 아래 두 가지다.

- 썸네일 / 커버 이미지 업로드와 미리보기 흐름 개방
- 기존 관리자 화면을 더 단순하고 직관적인 실무형 CMS UI로 정리

## 2. 챕터 3 범위

이번 챕터 3 완료 기준:

- `thumbnail_image_url` 업로드 연결
- `cover_image_url` 업로드 연결
- 업로드 후 즉시 미리보기
- 기존 이미지 교체
- 이미지 alt 입력 흐름 정리
- 목록 / 상세 / 등록 / 수정 화면 UI 정리

이번 챕터에서 넘기는 범위:

- 이미지 삭제 전용 UX
- 운영 리포트
- 추천/검색 운영 고도화
- 공개 대표 웹 고도화

## 3. 이미지 업로드 구조

이번 챕터는 Supabase Storage 기반으로 간다.

이유:

- 현재 데이터 저장소가 Supabase 기준이다.
- 웹 CMS와 앱 소비 채널이 같은 source of truth를 써야 한다.
- `thumbnail_image_url`, `cover_image_url` 필드에 업로드 결과를 바로 연결하기 쉽다.

기준:

- bucket: `guide-images`
- bucket 성격: public
- DB 저장값: public URL

## 4. 스토리지 경로 규칙

경로 규칙:

- `guides/{guideId}/thumbnail/{timestamp}-{sanitizedName}`
- `guides/{guideId}/cover/{timestamp}-{sanitizedName}`

의도:

- guide 단위로 파일이 모인다.
- thumbnail / cover 역할이 명확히 분리된다.
- 파일명 충돌을 줄이고, 교체 이력도 추적하기 쉽다.

## 5. 업로드 / 교체 / 미리보기 UX

기본 흐름:

1. 운영자가 파일 선택
2. 클라이언트에서 로컬 미리보기 노출
3. 저장 시 서버 액션에서 Storage 업로드
4. 업로드 성공 후 public URL 생성
5. DB에 `thumbnail_image_url`, `cover_image_url` 저장
6. 상세 화면에서 즉시 반영

교체 기준:

- 기존 이미지가 있으면 새 파일 업로드 시 교체로 본다.
- 같은 버킷의 기존 경로를 파싱할 수 있으면 기존 파일을 제거한다.
- 이번 챕터는 삭제 전용 버튼보다 `교체 중심`으로 간다.

기존 URL 입력과의 관계:

- 기본 운영 흐름은 파일 업로드다.
- URL 입력은 고급 수동 입력 / 긴급 보정용으로만 남긴다.

## 6. UI/UX 정리 포인트

이번 챕터에서 같이 정리한 방향:

- 목록에서 썸네일 유무와 수정 동선이 바로 보이게 정리
- 상세에서 상태 / 액션 / 이미지 / 본문 블록을 더 분리
- 등록/수정 화면에서 콘텐츠 / 정책 / 운영 메타 / 이미지 메타 섹션 위계를 더 명확히 정리
- 저장 버튼과 취소 버튼을 상단/하단 모두에 두지 않고, 주요 동선을 더 분명하게 정리
- 이미지 업로드 블록을 일반 텍스트 필드와 분리해 피로도를 낮춤

## 7. 챕터 2와의 연결

챕터 2가 `쓰기 최소 CMS`를 열었다면, 챕터 3은 그 위에 `이미지와 운영 UI`를 실제 실무 수준으로 붙이는 단계다.

즉:

- 챕터 2: 저장 가능
- 챕터 3: 이미지 포함 저장 가능 + 운영자가 더 쓰기 편한 화면
