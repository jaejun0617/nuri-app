  - 현재 프로젝트는 `.styles.ts` 패턴과 단일 파일 패턴이 혼재해 있는데, 커뮤니티처럼 계속 미세 조정하는 도메인은 스타일 분리만 먼저 해도 유지보수성이 크게 올라간다.
- 현재 최종 선택
  - 1차 대상은 `CommunityDetailScreen`, `CommunityListScreen`, `CommunityPostEditorForm`, `PostCard`
  - 컴포넌트 분해는 리렌더링 범위나 책임 분리가 실제 병목일 때 다음 단계로 진행한다.

## 1. 위치 기반 기능의 조회 정책

- 결정
  - 산책과 펫동반 장소 탐색 모두 `현재 GPS 기반 주변 추천 + 직접 검색`으로 고정한다.
- 이유
  - 복잡한 지역 브라우징보다 현재 위치 기준 탐색이 신뢰도와 사용성에 더 맞다.
- 대안
  - 다른 동 선택
  - 고양시 전체
  - 지역 칩 기반 탐색
- 현재 최종 선택
  - 위 대안들은 제거한다.

## 2. 거리 기준

- 결정
  - 거리 기준은 항상 현재 위치 기준이다.
- 이유
  - anchor가 바뀌면 사용자가 거리 정보를 신뢰하기 어렵다.
- 현재 최종 선택
  - 현재 위치가 없으면 거리값은 null이고 UI는 `거리 확인 중`으로 표시한다.

## 3. 위치 freshness 정책

- 결정
  - 산책/날씨/펫동반 장소 탐색 모두 공통 freshness 정책을 재사용한다.
- 이유
  - stale 위치를 current처럼 보이지 않게 하기 위해서다.
- 현재 최종 선택
  - fresh: `현재 위치 기준`
  - stale: `최근 확인 위치 기준`
  - refreshing: `새 위치 확인 중`

## 4. 펫동반 장소 데이터 전략

- 결정
  - 현재 운영 source of truth는 외부 API(Kakao Local)다.
  - 장기 목표는 `외부 원본 + Supabase 메타 혼합`이다.
- 이유
  - 현재 Supabase에는 자체 장소 DB가 없고, 외부 검색만으로는 운영 메타와 검수 정보를 쌓을 수 없다.
- 대안
  - 외부 API 전용
  - 자체 장소 DB 중심
- 현재 최종 선택
  - 단기: 외부 API 기반 탐색
  - 중기: Supabase 메타/북마크/검수정보 적재
  - 장기: 운영 검수된 장소 DB

## 5. 펫동반 결과 노출 방식

- 결정
  - 결과를 막아두지 않고 실제 탐색 기능으로 노출한다.
  - 대신 출처와 검증 상태를 UI에 명확히 표시한다.
- 이유
  - 이번 턴의 목표는 실제로 동작하는 장소 탐색 구조를 여는 것이기 때문이다.
- 현재 최종 선택
  - 리스트/상세에 `외부 후보` 출처와 `검증 상태`를 함께 노출한다.

## 6. 펫동반 검증 상태 모델

- 결정
  - 펫동반 장소는 `unknown / keyword-inferred / user-reported / admin-verified / rejected` 상태로 분리한다.
- 이유
  - 추정, 제보, 운영 검수를 한 단계로 뭉개면 신뢰도를 잘못 전달하게 되기 때문이다.
- 현재 최종 선택
  - Kakao Local 후보만 있는 경우 `unknown` 또는 `keyword-inferred`까지만 허용한다.
  - `admin-verified`는 향후 Supabase 운영 메타가 붙었을 때만 사용한다.

## 7. 외부 데이터 소스 역할 분리

- 결정
  - Kakao는 후보 수집, Google Places는 보조 속성, 관광공사/공공데이터는 관광/야외 보강, Supabase는 최종 메타로 역할을 나눈다.
- 이유
  - 특정 외부 API 하나를 신뢰 데이터처럼 쓰면 서비스 신뢰도를 통제할 수 없기 때문이다.
- 현재 최종 선택
  - source of truth는 장기적으로 Supabase 메타 레이어가 가져간다.

## 8. 펫동반 메타 스키마 제약 방식

- 결정
  - `verification_status`, `source_type`은 enum보다 `text + check constraint`를 우선 사용한다.
- 이유
  - 초기 운영 단계에서 상태 집합이 바뀔 가능성이 있어 enum 변경 비용이 더 크기 때문이다.
- 현재 최종 선택
  - SQL/앱 타입/UI 문구를 같은 상태 집합으로 맞추고, 제약은 check constraint로 유지한다.

## 9. 외부 원본 매핑 구조

- 결정
  - canonical 메타와 외부 원본 매핑을 분리한다.
- 이유
  - Kakao / Google Places / TourAPI를 한 테이블 1행으로 묶으면 장기적으로 source 충돌과 이관 비용이 커진다.
- 현재 최종 선택
  - `pet_place_service_meta` = canonical truth
  - `pet_place_source_links` = `provider + provider_place_id` unique 매핑
  - `pet_place_external_signals` = Google / 관광공사 / 공공데이터 보조 신호

## 10. 반려동물과 여행 도메인 분리 방향

- 결정
  - `펫동반 장소 탐색`과 `반려동물과 여행`은 장기적으로 별도 메뉴가 맞다.
- 이유
  - 사용자 의도, 데이터 소스, 정렬 기준, 캐시 전략이 다르기 때문이다.
- 현재 최종 선택
  - 일상 탐색은 Kakao 중심
  - 여행 탐색은 관광공사/공공데이터 중심
  - 두 도메인의 최종 운영 메타 source of truth는 Supabase가 공유한다

## 11. 반려동물과 여행 1차 진입 구조

- 결정
  - `반려동물과 여행` 1차는 현재 위치 기반이 아니라 `검색 입력 + 카테고리 선택 + 리스트 + 상세` 구조로 연다.
- 이유
  - 여행 도메인은 사용자가 현재 위치 주변보다 방문 예정 지역, 장소명, 여행 키워드를 먼저 입력하는 흐름이 더 자연스럽기 때문이다.
- 현재 최종 선택
  - 1차 핵심 후보 소스는 한국관광공사 반려동물 동반여행 서비스다.
  - GPS 기반 주변 추천, 거리 표시, 동선 추천은 2차 확장으로 넘긴다.
  - 지역 메타 카탈로그는 UI 칩이 아니라 입력어를 `areaCode/sigunguCode`로 내부 매핑하는 용도로만 유지한다.
  - 공식 Swagger 기준 provider는 `KorPetTourService2`로 고정하고, 검색 입력은 `searchKeyword2`, 기본 목록은 `areaBasedList2`를 사용한다.

## 12. 반려동물과 여행 검색 품질 보정 원칙

- 결정
  - `반려동물과 여행` 검색은 결과를 많이 보여주는 것보다 여행 메뉴답게 믿을 수 있는 결과만 남기는 쪽을 우선한다.
- 이유
  - `searchKeyword2`는 지역명 단독 검색에서 브랜드 매장, 백화점 점포, 일반 상업시설을 넓게 반환할 수 있어 그대로 노출하면 도메인 신뢰도가 크게 떨어지기 때문이다.
- 현재 최종 선택
  - 지역명 단독 검색은 여행지/공원/수목원/야외공간/문화시설 성격 결과를 우선 노출한다.
  - `contentTypeId=38` 쇼핑 결과와 브랜드/매장 패턴은 보수적으로 약화하거나 제외한다.
  - 애매한 결과는 억지로 노출하지 않고 empty 상태로 돌려 precision을 지킨다.
  - 지역명 단독 검색은 일반 `searchKeyword2` 결과를 그대로 쓰지 않고, 내부 region resolver가 잡히면 `areaBasedList2` 기반 narrow mode로 분기한다.

## 13. 반려동물과 여행 placeType / petAllowed 분리 원칙

- 결정
  - 여행 검색 후처리는 `placeType`과 `petAllowed`를 분리해서 계산한다.
- 이유
  - 여행지 성격과 반려동물 동반 가능성은 같은 문제가 아니고, 하나의 패턴 집합으로 뭉개면 상업시설이 여행지처럼 보이거나 숙소가 펫 허용 확정처럼 보일 수 있기 때문이다.
- 현재 최종 선택
  - 점수는 `travelScore / petScore / commercialPenalty` 3축으로 분리한다.
  - 세 점수는 모두 `0~1` 정규화한다.
  - 최종 랭킹은 `travel` 중심 가중치로 계산하고, `petScore`가 `travelScore`를 넘지 못하게 제한한다.
  - `TRAVEL_POSITIVE_PATTERNS`, `PET_POSITIVE_PATTERNS`, `COMMERCIAL_EXCLUDE_PATTERNS`를 분리 유지한다.
  - 정규식만으로 `반려동물 동반 가능` 확정 처리하지 않는다.
  - 숙소/캠핑/목장/농원은 여행 적합도는 줄 수 있지만, 강한 펫 근거가 없으면 `현장 확인 필요`로 유지한다.

## 16. 반려동물과 여행 펫정책 아키텍처

- 결정
  - `반려동물과 여행`의 최종 펫 허용 판정은 `장소 검색 API + TourAPI + 자체 DB` 조합을 전제로 설계한다.
  - 내부 구조는 `Raw Layer -> Candidate Layer -> Trust Layer -> Aggregation Layer` 4단계로 유지한다.
- 이유
  - TourAPI 단독, Kakao 단독, 정규식 단독으로는 실서비스 수준의 신뢰도 통제가 불가능하기 때문이다.
- 현재 최종 선택
  - 현재 런타임에서는 TourAPI raw 기반 `candidate-layer`까지만 계산한다.
  - `confirmed`는 향후 `pet_policies + user_reports + admin review`가 붙기 전까지 사용하지 않는다.
  - 앱 타입은 `petPolicy`, `aggregation`, `places / pet_policies / user_reports` 확장 방향을 먼저 반영한다.

## 17. 반려동물과 여행 지역 intent 일반화 원칙

- 결정
  - 지역 intent는 특정 예시 지역 patch가 아니라 전국 공통 region catalog와 resolver로 처리한다.
- 이유
  - `일산`, `제주`, `강릉`, `강원도` 같은 테스트 케이스를 로직으로 굳히면 다른 지역에서 바로 무너지기 때문이다.
- 현재 최종 선택
  - 전국 시/도 areaCode 메타를 기본 축으로 유지한다.
  - 주요 시/군/구/생활권 별칭은 하나의 resolver catalog로 관리한다.
  - 지역명 단독 검색일 때만 `areaBasedList2` narrow mode를 사용하고, 구체 장소명은 일반 검색 흐름으로 남긴다.

## 18. 반려동물과 여행 Trust Layer DB 시작 단위

- 결정
  - Trust Layer DB는 이번 단계에서 `pet_travel_places / pet_travel_pet_policies / pet_travel_user_reports` 3테이블로 시작한다.
- 이유
  - 지금 필요한 것은 `confirmed`를 열 수 있는 근거 저장소이지, `placeMeta`와 섞인 대형 스키마가 아니기 때문이다.
- 현재 최종 선택
  - `places`는 내부 uuid PK 기반 canonical anchor다.
  - `pet_policies`는 source별 정책, confidence, evidence를 저장한다.
  - `user_reports`는 Trust Layer 집계용 원본 제보다.
  - 외부 원본 매핑 전용 `source_links`는 다음 단계로 미루고, 현재는 `places.primary_source + primary_source_place_id`로 bootstrap한다.

## 19. 반려동물과 여행 confirmed 개방 조건

- 결정
  - `confirmed`는 Trust Layer DB가 생겼다고 바로 열지 않는다.
- 이유
  - 자동 수집 데이터와 텍스트 패턴만으로는 서비스에서 확정 상태를 책임질 수 없기 때문이다.
- 현재 최종 선택
  - TourAPI raw만으로 `confirmed` 금지
  - 정규식만으로 `confirmed` 금지
  - `confirmed`는 `pet_policies + user_reports + admin review` 조합이 붙었을 때만 허용한다.

## 20. 커뮤니티 상세 키보드 대응 기준

- 결정
  - 커뮤니티 상세처럼 `본문 + 댓글 리스트 + 하단 댓글 입력창`이 함께 있는 화면은 일반 폼 화면과 같은 키보드 처리로 보지 않는다.
- 이유
  - 안드로이드 실기기, Fabric, edge-to-edge 조합에서는 단순 `KeyboardAvoidingView`나 수동 `bottom` 계산만으로 입력창이 안정적으로 키보드 바로 위에 붙지 않을 수 있기 때문이다.
- 현재 최종 선택
  - 이 유형은 실기기 기준으로 별도 패턴으로 본다.
  - 일반 폼 화면의 키보드 처리 방식을 커뮤니티 상세에 그대로 복사 적용하지 않는다.
  - 안드로이드에서는 `adjustResize + flex: 1 + 리스트/본문 + 하단 입력 sibling` 구조를 우선 기준으로 본다.
  - `position: absolute` 기반 composer는 이 유형에서 사용하지 않는다.
  - 이후 커뮤니티 상세 키보드 안정화는 `리스트 본문 + 하단 composer` 구조 자체를 기준으로 다시 검토한다.

## 21. 커뮤니티 댓글 depth 정책

- 결정
  - 커뮤니티 댓글은 `depth 1`까지만 허용한다.
- 이유
  - 모바일에서 depth 2 이상은 가독성, 정렬, 삭제/신고/좋아요 처리, RLS 검증 비용이 급격히 커진다.
- 현재 최종 선택
  - 원댓글에는 답글을 여러 개 달 수 있다.
  - 답글에도 `답글쓰기` UX는 줄 수 있지만, 실제 저장은 항상 같은 top-level parent 기준으로 평탄화한다.
  - DB는 `parent_comment_id + depth in (0,1)`로 강제한다.

## 22. 커뮤니티 댓글 초기 노출 정책

- 결정
  - 댓글은 top-level 10개, reply preview 2개 기준으로 노출한다.
- 이유
  - 댓글이 많은 글에서 첫 진입 렌더 비용과 가독성을 동시에 관리하려면 전체 댓글을 한 번에 펼치지 않는 구조가 필요하다.
- 현재 최종 선택
  - `댓글 더보기`는 10개 단위
  - `답글 n개 더보기`는 reply preview 2개 이후 확장
  - reply는 일반 댓글처럼 별도 border를 쓰지 않고 같은 thread 안에서 묶어 보인다

## 23. 커뮤니티 베스트 댓글 기준

- 결정
  - 베스트 댓글은 top-level comment만 대상으로 하고, 좋아요 5개 이상부터 붙인다.
- 이유
  - reply까지 베스트 대상으로 넣으면 리스트 구조가 복잡해지고, 좋아요 기준이 너무 낮으면 신뢰도가 떨어진다.
- 현재 최종 선택
  - `top-level + like_count >= 5 + active`를 초기 베스트 댓글 기준으로 유지한다.

## 24. 커뮤니티 댓글 RLS self-reference 처리 기준

- 결정
  - `comments`처럼 self-reference가 있는 테이블의 insert 검증은 policy 안에서 같은 테이블을 직접 재조회하지 않는다.
- 이유
  - 실제 커뮤니티 댓글 적용 중 `comments_insert_own` policy가 `public.comments`를 다시 읽으면서 `42P17 infinite recursion detected in policy for relation "comments"`가 발생했고, plain 댓글과 답글 모두 저장이 막혔다.
- 현재 최종 선택
  - self-reference 검증은 `security definer` helper function으로 분리한다.
  - 커뮤니티 댓글 기준 helper는 `public.can_insert_community_comment(post_id, parent_comment_id)`를 source of truth로 사용한다.
  - 이후 댓글/답글 정책을 변경할 때는 policy 본문과 helper를 같이 수정한다.

## 25. 메인 하단 탭과 기록 작성 진입 구조

- 결정
  - 하단 탭에는 기록하기를 두지 않고, `CommunityTab`을 고정 탭으로 둔다.
  - 기록 작성은 `RootNavigator.RecordCreate` 독립 스크린으로 연다.
- 이유
  - 기록 작성은 소비 화면이 아니라 작성 플로우라서 상시 탭보다 독립 스크린으로 다루는 편이 자연스럽고, 메인 탭에는 커뮤니티 진입점이 더 적합하기 때문이다.
  - 기존 `returnTo` 복귀 흐름은 유지해야 하므로, 탭 제거와 작성 화면 제거를 같은 개념으로 다루면 안 된다.
- 현재 최종 선택
  - 메인 탭 순서는 `Home / Timeline / Community / Guestbook / More`
  - `RecordCreateTab`은 제거한다.
  - `RecordCreateScreen`은 `RootNavigator.RecordCreate`로 유지한다.
  - 홈/타임라인 등 기존 기록하기 CTA는 전부 `navigation.navigate('RecordCreate', { returnTo: ... })`를 사용한다.

## 26. 커뮤니티 탭 헤더 구조

- 결정
  - 하단 탭의 `CommunityTab`도 직접 화면 연결이 아니라 전용 stack wrapper를 사용한다.
- 이유
  - 커뮤니티는 루트 스택에서 `CommunityStackHeader` 기준으로 정리됐는데, 탭만 직접 `CommunityListScreen`을 물면 헤더가 사라져 UX 일관성이 깨지기 때문이다.
- 현재 최종 선택
  - `CommunityTab`은 `CommunityTabStackNavigator`를 렌더한다.
  - 이 스택의 첫 화면도 `CommunityStackHeader`를 그대로 사용한다.

## 27. 추억 상세 뒤로가기 규칙

- 결정
  - `RecordDetail`은 단순 `goBack()`만 쓰지 않고 `entrySource` 기반 복귀 규칙을 따른다.
- 이유
  - 홈/More/기록 저장 직후 상세처럼 스택 히스토리만으로는 원래 복귀 위치를 복원할 수 없는 진입이 실제로 존재하기 때문이다.
- 현재 최종 선택
  - `TimelineStackParamList.RecordDetail`은 `entrySource`를 받을 수 있다.
  - `home` 진입이면 `AppTabs -> HomeTab`
  - `more` 진입이면 `AppTabs -> HomeTab` 이동 후 More drawer 재오픈
  - 그 외는 `goBack()`, 실패 시 `TimelineMain`

## 28. 주요 소비 화면 배경 톤 기준

- 결정
  - 기록/일정/가이드/탐색/여행 계열 주요 소비 화면의 루트 배경은 `#FFFFFF`를 우선 기준으로 맞춘다.
- 이유
  - 최근 화면 정리에서 회색 루트 배경과 카드 보더가 누적되면서 서비스 전반의 톤이 불필요하게 무거워졌기 때문이다.
- 현재 최종 선택
  - `RecordDetail`, `RecordCreate`, `ScheduleList`, `ScheduleDetail`, `ScheduleCreate`, `GuideList`, `GuideDetail`, `IndoorActivityRecommendations`, `ActivityGuide`, `LocationDiscoveryList`, `PetTravelList`, `Guestbook`은 흰 배경 기준으로 맞춘다.
  - 같은 흐름에서 `RecordDetail` 카드 상하 보더, `Timeline` 리스트 카드 외곽 보더, `ScheduleCreate` 카드 보더는 제거한다.

## 29. Android 시스템 내비게이션 바 색상 기준

- 결정
  - Android 하단 시스템 내비게이션 바 배경은 `#FFFFFF`로 고정한다.
- 이유
  - 소비 화면 전반을 흰 배경 기준으로 맞춘 상태에서 시스템 하단 바만 다른 톤으로 남으면 화면 톤이 끊겨 보일 수 있다.
  - 일부 기기에서는 테마값만으로 일관성이 부족할 수 있어 런타임 재적용 기준도 함께 유지하는 편이 안전하다.
- 현재 최종 선택
  - `styles.xml`
    - `android:navigationBarColor = #FFFFFF`
    - `android:windowLightNavigationBar = true`
  - `values-v35/styles.xml`
    - `android:windowOptOutEdgeToEdgeEnforcement = true`
  - `MainActivity.kt`
    - `window.navigationBarColor = Color.WHITE`
    - `isAppearanceLightNavigationBars = true`
    - `isNavigationBarContrastEnforced = false`
    - `onCreate/onResume/onWindowFocusChanged`에서 시스템 바 스타일을 재적용한다.
  - 실기기에서 어두운 바가 남을 경우, Android 15+ edge-to-edge 강제와 포커스 복귀 타이밍을 먼저 의심한다.

## 30. 전역 타이포그래피 / 접근성 source of truth

- 결정
  - 전역 텍스트 역할은 semantic typography token과 `AppText`를 기준으로 관리한다.
- 이유
  - 화면별 하드코딩으로 작은 텍스트, 부족한 lineHeight, 작은 터치 타겟이 반복되면 이후 Task 4~5에서도 같은 접근성 리스크가 재증식하기 때문이다.
- 현재 최종 선택
  - source of truth는 `src/app/theme/tokens/typography.ts`와 `src/app/ui/AppText.tsx`
  - semantic role은 `display / titleLg / titleMd / titleSm / body / bodySm / button / tab / helper / caption`
  - 한글 기준 letterSpacing은 보수적으로 유지한다.
  - `AppText`는 기본적으로 `allowFontScaling = true`, `maxFontSizeMultiplier = 2`를 따른다.
  - 공통 액션/탭/헤더 버튼은 `44~48` 이상 터치 영역을 우선한다.
