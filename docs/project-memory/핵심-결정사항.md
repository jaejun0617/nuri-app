# 핵심 결정사항

## 문서 운영 원칙

- 더 이상 과거 Task 번호 체계를 source of truth로 쓰지 않는다.
- 현재 메모리는 `기능 이름`, `운영 계약`, `출시 게이트` 기준으로 관리한다.
- 구현 완료와 운영 완료를 같은 말로 쓰지 않는다.
- 출시 판단은 항상 `코드 + linked remote + 실기기/row-level 증적 + release checklist` 조합으로 한다.

## 정책 문서와 가입 흐름

- 정책 문서 source of truth는 `src/services/legal/documents.ts`다.
- 회원가입과 More는 실제 public 문서를 연다.
- 정책 문서는 앱 화면만 정리해도 완료가 아니며, public URL과 앱 복귀 검증이 있어야 닫는다.

## 비밀번호 재설정

- 비밀번호 재설정은 `request -> recovery link -> 앱 복귀 -> 새 비밀번호 설정 -> SignIn 재진입` 계약으로 고정한다.
- recovery 세션은 일반 로그인 세션과 분리해 다룬다.
- reset 완료 후 기존 계정 컨텍스트로 새면 blocker로 본다.

## 계정 탈퇴 7일 유예

- 계정 탈퇴는 `요청 즉시 로그아웃 + 7일 유예 + 복구 가능 + 유예 종료 후 자동 파기` 구조로 고정한다.
- Day 0에는 서버 원본을 보존하고 로컬 세션만 정리한다.
- 유예기간 로그인은 허용하되 앱 내부 진입은 막고 복구 모달만 연다.
- 최종 삭제는 `Scheduled Edge Function + DB finalizer + storage cleanup queue` 구조로 운영한다.
- 자동 파기 worker는 request 단위 fault isolation을 유지하며, object not found는 cleanup 성공으로 수렴한다.

## 커뮤니티 운영 방어선

- 커뮤니티는 moderation 없는 기능 확장을 금지한다.
- write-path는 rate limit, dedupe, blocked-term, stable error contract를 공통으로 사용한다.
- 신고는 self-report 차단, duplicate 차단, reporter flag trace, unique reporter 기준 auto-hide, moderation queue/action log를 기본 계약으로 둔다.
- hidden/auto_hidden/deleted/banned 상태에서는 이미지 비노출과 cleanup pending 추적을 우선한다.

## 공개 신뢰도

- 공개 라벨은 `후보`, `확인 필요`, `검수 반영` 3개만 사용한다.
- `confirmed`는 계속 닫아 둔다.
- personal state는 public trust를 직접 올리지 않는다.
- stale, source conflict, onsite check 필요 상태에서는 더 보수적인 공개 라벨을 우선한다.

## Android UI 기준

- Android 실기기 기준으로 읽기 쉬운 타이포와 안정적인 탭/헤더/모달 구조를 우선한다.
- 한글 장문은 의미 단위 줄바꿈과 충분한 line-height를 유지한다.
- 긴 폼의 키보드 대응은 단일 스크롤 계약으로 유지한다.

## 출시 판단 기준

- 아래가 모두 있어야 `출시 가능`으로 본다.
  - 실기기 최종 QA
  - 운영 row-level 증적
  - cleanup/worker 실제 동작 증적
  - 모니터링/배포 설정 확인
  - 스토어 제출 자산 준비
