### 왜 했는가

### 무엇이 달라졌는가

- `PasswordResetFormScreen`은 성공, 실패, 취소, 뒤로가기 모두 `passwordRecoveryFlow 해제 -> signOut + local auth cleanup -> SignIn reset` 경로를 강제한다.
- `boot.ts`, `HomeScreen`, `MainScreen`은 recovery active 상태에서 `NicknameSetup`, `PetCreate(from: 'auto')`, 홈 자동 진입을 차단하도록 보정됐다.
- 추가로 `pet fetch 실패 + petsCount=0` 상태가 `PetCreate(from: 'auto')`로 새던 기존 auto-route도 같이 차단해 write-path 오분기 방어선을 맞췄다.
- 이번 턴에는 `2026-03-31 02:11:30 UTC` pet row를 삭제하지 않았고, 운영/DB 정리 task로만 남겼다.

### 무엇이 달라졌는가

- 앱 삭제 호출 경로가 `MoreDrawerContent / MoreScreen -> performAccountDeletion -> delete_my_account()` 순서이며, 현재는 RPC 성공 후에만 로컬 세션을 정리한다는 사실을 문서로 고정했다.

- storage(`pet-profiles / memory-images / community-images`)는 현재 RPC 범위 밖이며, 계정 삭제 완료 계약과 분리해서 비동기 cleanup + orphan 확인 대상으로 다뤄야 한다는 설계 방향을 남겼다.
- `pet_care_guide_events`, `audit_logs`, `pet_place_search_logs`, `pet_place_verification_logs`는 delete보다 익명화/분리보관 후보로, `pet_place_bookmarks / pet_place_user_reports / pet_travel_user_reports`는 SQL 문서상 존재하지만 앱 런타임 연결 여부 추가 확인이 필요한 항목으로 분류했다.
- QA 증적은 `요청 전 데이터 존재`, `요청 직후 세션/DB 상태`, `비동기 cleanup 완료`, `orphan 확인`, `부분 실패 재현`으로 나눠야 한다는 구조를 남겼다.
