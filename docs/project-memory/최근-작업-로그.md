# 최근 작업 로그

## 기록 원칙

- 과거 Task 번호 대신 `도메인/기능 이름`으로 기록한다.
- 무엇을 했는가보다 `왜 닫혔는가`, `무엇이 달라졌는가`, `다음에 뭘 해야 하는가`를 먼저 적는다.
- 출시 직전에는 전체 히스토리를 길게 누적하지 않고, 현재 판단에 필요한 핵심 마일스톤만 남긴다.

## 2026-04-10 계정 탈퇴 7일 유예 자동 파기 워커

- 왜 했는가
  - 유예기간 계정 삭제는 요청과 복구까지만 닫혀 있었고, `scheduled_deletion_at` 이후 자동 파기 파이프라인이 없었다.
- 무엇이 달라졌는가
  - `claim_due_account_deletion_requests()`와 `account-deletion-worker`가 remote에 실제 배포됐다.
  - `pg_cron -> Edge Function -> finalizer -> cleanup queue -> storage remove -> completed` 경로가 실제 동작한다.
  - time-travel E2E로 due request 1건이 `completed`, cleanup item 1건이 `completed`, `profiles/pets/auth.users = 0`까지 확인됐다.
- 남은 것
  - 실제 cron 자동 tick 1회와 운영용 캡처 보강

  - Android 실기기에서 recovery 링크 복귀, 비밀번호 변경 성공, `SignIn` 복귀를 확인했다.
  - 기존 계정 컨텍스트로 새던 blocker가 닫혔다.
- 남은 것
  - 출시 후보 빌드 기준 최종 smoke만 남아 있다.

## 2026-03-31 계정 삭제 실기기/row-level 실증

- 왜 했는가
  - 계정 삭제는 요청 row만으로 닫을 수 없고, 실제 데이터와 커뮤니티 콘텐츠 삭제까지 봐야 했다.
- 무엇이 달라졌는가
  - 실기기에서 삭제 완료 토스트와 재로그인 실패를 확인했다.
  - linked remote에서 `auth.users`, `profiles`, `posts`, `comments`, `pets`가 실제 0으로 떨어지는 것을 확인했다.
  - cleanup queue가 분리 추적된다는 점도 확인됐다.
- 남은 것
  - 현재 구조에서는 즉시 삭제가 아니라 7일 유예 + 자동 파기 worker까지 확장되어 운영 증적 패키지만 남았다.
