## 2026-04-13 건강관리 전용 기록하기 1차 전환

- 왜 했는가
  - 건강 입력을 타임라인/생활/건강관리로 분산해 두면 병원, 약, 증상, 체중의 사용자 진입점이 계속 갈라진다.
  - 타임라인 건강 입력 제거 전에 건강관리 안에서 먼저 대체 작성 진입점을 제공해야 기존 사용자 흐름이 끊기지 않는다.
- 무엇이 달라졌는가
  - 전체 메뉴 진입명을 `건강관리`로 바꾸고, 건강관리 화면 타이틀과 탭명을 `기록 / 체중 / 인사이트` 축으로 정리했다.
  - 건강관리 상단에 전용 `기록` 버튼과 `건강 기록하기` 선택 sheet를 추가했다.
  - 병원/약은 기존 `ScheduleCreate` write-path를 쓰고, 증상/컨디션은 기존 `RecordCreate` write-path를 쓰며, 체중은 기존 `WeightLogEntrySheet`를 재사용한다.
  - 건강관리에서 생성한 기록/일정은 저장 후 Health Report 월 데이터 query를 invalidate하고 건강관리 기록 탭으로 돌아오도록 연결했다.
- 남은 것
  - 타임라인 `건강` 신규 작성 선택지는 후속 작업에서 제거됐다.
  - 생활 병원/약 입력 이관도 아직 하지 않았다.
  - 다음 작업은 생활/일정 쪽 병원/약 신규 작성 진입점을 건강관리 전용 기록하기와 충돌하지 않게 이관하는 것이다.

## 2026-04-13 타임라인 건강 신규 작성 선택지 제거

- 왜 했는가
  - 건강관리 안에 증상/컨디션 작성 진입점이 생겼으므로, 타임라인에서 같은 health category를 새로 선택할 수 있게 두면 건강 입력 source가 다시 분산된다.
  - 기존 `memories`의 health category는 Health Report와 legacy 타임라인 읽기에서 계속 필요하므로 저장 계약 자체를 삭제하면 안 된다.
- 무엇이 달라졌는가
  - `RECORD_MAIN_CATEGORIES`는 기존 데이터/표시 source of truth로 유지했다.
  - create/edit 폼에서 선택 가능한 메인 카테고리는 `RECORD_WRITE_MAIN_CATEGORIES`로 분리해 `health`를 숨겼다.
  - 건강관리에서 진입한 증상/컨디션 작성만 route 초기값으로 `health`를 유지할 수 있다.
  - 일반 타임라인 작성 화면에서 persisted draft가 `health`를 되살리는 경로는 기본 작성 카테고리로 정규화했다.
- 남은 것
  - 생활/일정의 병원/약 신규 작성 진입점은 후속 작업에서 건강관리로 이관됐다.
  - 다음 작업은 건강관리/일정 작성 흐름을 Android 실기기에서 연속 smoke로 확인하는 것이다.

## 2026-04-14 건강관리 인사이트 기록밀도와 일정 시간 모달 보정

- 왜 했는가
  - Android 실기기에서 인사이트 탭의 기록밀도 그래프가 적용되지 않는 것처럼 보였다.
  - 일정 추가 화면의 시간 모달은 선택값을 넘기고 있었지만, 화면 콜백이 그 값을 받지 않아 기존 시간이 그대로 저장되는 경로가 있었다.
  - 날짜와 시간을 분리 입력하면 일정 생성 UX가 길어져, 사진 레퍼런스처럼 달력 선택 후 같은 모달에서 시간까지 정하는 흐름이 더 적합했다.
  - 건강관리에서 기록을 저장한 직후에는 오늘이 아니라 방금 저장한 기록일로 포커스되어야 사용자가 저장 결과를 바로 확인할 수 있다.
- 무엇이 달라졌는가
  - 기록밀도 그래프는 건강 이벤트뿐 아니라 체중 로그도 같은 월간 밀도 기준에 포함한다.
  - 인사이트 상단 4개 위젯은 누르면 각 위젯의 상세 리스트가 바텀시트로 열린다.
  - 기록밀도 그래프는 현재 월에 오늘 날짜가 있으면 오늘 기준 위치로 자동 포커스된다.
  - 건강관리에서 증상/병원/약 기록을 저장하면 `focusYmd`를 HealthReport로 넘겨 저장한 날짜의 리스트로 자동 포커스한다.
  - 일정 추가/수정 화면은 `날짜`와 `시간` 필드를 `일시` 하나로 묶고, 날짜 선택 모달 안에서 시간까지 함께 확정한다.
  - 날짜 선택 모달은 월 달력형 전체 화면 UI로 전환했고, 시간 선택은 `오전/오후 + 직접 입력`으로 바꿨다.
  - 직접 입력 시간은 `react-native-keyboard-controller` avoiding view 안에서 처리해 Android 키보드바와 모달 하단 액션이 겹치지 않도록 했다.
- 남은 것
  - Android 실기기에서 저장 후 등록일 포커스, 일시 모달 직접 입력 키보드, 저장 후 실제 `starts_at` 반영을 확인해야 한다.
