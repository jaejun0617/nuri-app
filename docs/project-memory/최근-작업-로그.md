### 왜 했는가

### 무엇이 달라졌는가

- linked project remote `community_blocked_terms`에 `씨발`, `개새끼`, `씹새끼`, `좆같` 4건이, `community_blocked_term_patterns`에 `ㅆ+\\s*ㅂ+` 1건이 실제 입력됐다.

### 검증 결과

### 무엇이 달라졌는가

- 앱 create/edit/comment 실패 UX는 `getCommunityMutationErrorMeta()` 공통 매퍼로 묶였고, raw Korean server message pass-through에 직접 의존하지 않게 됐다.

- community blocked-term source는 false positive 우선 원칙 때문에 빈 source로 만들었고, 운영 term/pattern 입력은 후속 운영 단계로 분리했다.

### 왜 했는가

- 현재 사용자 스트레스의 핵심은 trust 구조 부재 자체보다 `trust 없는 candidate가 너무 많이 먼저 보인다`는 read-path 문제였기 때문이다.
- 특히 place explicit query, walk nearby default, travel region-only/broad query에서 noisy candidate가 service-grade 기준보다 쉽게 살아남고 있었다.

### 무엇이 달라졌는가

- place는 explicit query에서도 trust 없는 후보 상단 점유율과 총량을 더 강하게 제한하도록 exposure guard를 추가했다.
- walk는 nearby 기본 진입을 curated recommendation처럼 보이게 두지 않고, 거리 cap과 item cap을 둔 `근처 후보` 성격으로 더 제한했다.
- travel은 region-only / broad keyword / low-signal candidate에 더 높은 score 컷과 더 작은 cap을 적용했다.
- 리스트 copy도 `추천`보다 `후보`와 `신뢰 우선` 중심으로 보수화해, trust 없는 결과를 추천처럼 읽히지 않게 했다.
- physical-device QA와 Task 4 row-level 운영 캡처는 release blocker registry로 그대로 유지했다.
- 커뮤니티 비속어 SQL 연동은 이번 slice 범위 밖으로 유지하고, 후속 커뮤니티 hardening 스프린트로 남겼다.

- 일부 broad query와 region-only query는 recall이 더 줄 수 있다.

  - 직접 삭제: 개인 콘텐츠와 개인화 데이터
  - 익명화: `reports`, 장소/여행 user reports, guide events, audit/search/verification logs
  - 비식별 보관: `user_consent_history`

- 앱은 계정 삭제 결과를 `completed / completed_with_cleanup_pending / unknown_pending_confirmation` 기준으로 해석하고, 보수적 메시지만 노출하도록 최소 연동됐다.
- release checklist는 계정 삭제 증적 없이는 배포 완료로 보지 않는 Release Gate 기준으로 강화됐다.
