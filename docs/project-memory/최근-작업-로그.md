- 지역 intent resolver를 전국 시/도 + 주요 시/군/구/생활권 별칭 구조로 재정리했다.
- 장기 구조가 `장소 검색 API + TourAPI + 자체 DB` 조합이라는 점을 타입과 문서에 반영했다.
- `places / pet_policies / user_reports` 방향 타입을 추가하고, 관련 아키텍처 문서를 새로 작성했다.

### 왜 바꿨는가

- 기존 구조는 예시 지역 패치와 정규식 기반 펫 허용 판단에 의존해 서비스 배포 수준 신뢰도를 만들기 어려웠기 때문이다.
- 이번 턴의 핵심은 검색량보다 `펫 동반 가능 판정 신뢰도`를 올리는 것이기 때문이다.

### 무엇을 바꿨는가

- 지역명 단독 검색은 일반 `searchKeyword2` 결과를 그대로 쓰지 않고, 내부 region resolver가 잡히면 `areaBasedList2` 기반 narrow mode로 분기하도록 정리했다.
- `강원도`, `제주`, `강릉`, `일산` 같은 검색어를 위한 지역 메타를 실제 region-intent 용도로 정리하고 주소 앵커 기준으로 다시 좁히도록 바꿨다.
- `전체` 카테고리의 region mode에서는 쇼핑·일반 점포·음식점 노이즈를 기본 통과시키지 않도록 더 보수적으로 제한했다.
- 결과 요약 문구를 실제 노출 후보 수 기준으로 바꾸고, empty 문구에 지역명 + 여행지 키워드 예시를 추가했다.

### 왜 바꿨는가

- live 원응답 기준 `제주`, `일산` 검색 상단이 거의 전부 `ct=38` 쇼핑 점포였고, `강원도`는 `searchKeyword2`에서 0건이 나와 지역 여행 탐색 의도를 제대로 처리하지 못했기 때문이다.
- 이번 이슈는 단순 품질 개선이 아니라 `지역명 검색 의도 처리 실패`에 해당하는 핵심 버그이기 때문이다.

### 검증 결과

- direct live 확인
  - `searchKeyword2('제주')` 상단은 `게스 제주점`, `골든구스 ... 제주점`, `골프존마켓 ... 제주점` 등 `ct=38` 위주
  - `searchKeyword2('일산')` 상단은 `롯데백화점 일산점`, `닥스남성 ...`, `삼성스토어 일산` 등 `ct=38` 위주
  - `searchKeyword2('강원도')`는 0건
  - `areaBasedList2(areaCode=39)`는 제주 여행지 중심 응답 확인
- narrow mode 검증

  - `제주` -> 해변, 공원, 목장, 미술관 계열 중심으로 정제
  - `강릉` -> 공원, 해변, 숙소 후보 중심으로 정제
