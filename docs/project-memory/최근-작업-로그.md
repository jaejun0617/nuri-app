### 무엇을 바꿨는가

- 커뮤니티 상세에 `react-native-keyboard-controller` 기반 `KeyboardProvider`를 앱 루트에 연결해, Fabric 환경에서도 키보드 inset 추적이 가능하도록 부팅 구조를 보강했다.
- 커뮤니티 상세 화면은 댓글 입력창 포커스와 키보드 대응을 여러 차례 보수적으로 정리했고, 현재 구조는 `KeyboardAvoidingView` 기준으로 유지하면서 추가 안정화가 가능하도록 정리된 상태다.
- 목록/상세 헤더 액션의 터치 hit area를 키워, 커뮤니티 뒤로가기와 글쓰기 버튼이 실기기에서 더 안정적으로 눌리도록 조정했다.
- 게시글 작성자 스냅샷을 `author_snapshot_*` 기준으로 읽고 쓰도록 보강해, 다른 계정에서 기존 `profiles` 공개 조회가 막히는 환경에서도 작성자명이 안정적으로 보이도록 구조를 확장했다.
- 좋아요 수는 더 이상 `posts.like_count`만 신뢰하지 않고, 실제 `likes` 행 기준으로 읽어 새로고침 뒤 카운터가 다시 내려가는 문제를 줄였다.

### 왜 바꿨는가

- 커뮤니티 2-3을 실제 서비스 흐름으로 열고 나니, 다른 계정 조회, 새로고침 정합성, 헤더 터치성, 댓글 입력창 키보드 대응이 실기기 품질에 직접 영향을 주는 핵심 리스크로 드러났기 때문이다.

- 수정 화면도 이미지 복원 실패 경로를 안전하게 처리해야 실기기에서 예외 상황이 조용히 깨지지 않는다.

- 커뮤니티 구현에 들어가기 전에 현재 `posts/comments/likes/reports/audit_logs` seed 구조를 확장하는 기준 SQL을 `커뮤니티-운영-스키마-최종.sql`로 정리했다.
- 최종본은 seed 확장형 기준으로 `posts/comments/reports` 컬럼 보강, counter trigger, admin helper 함수, RLS 재작성, `community-images` storage bucket 정책까지 포함한다.
- 실제 적용은 한 번에 하지 않도록 `1단계-읽기-MVP-적용.sql`, `2단계-쓰기-상호작용-적용.sql`, `3단계-운영-모더레이션-적용.sql`로 분리했다.
- `3단계`에서는 자동 숨김 trigger 같은 운영 임계값 의존 항목은 제외하고, 현재 문서 기준으로 확정 가능한 관리자 함수와 admin RLS만 넣었다.
- 이후 검토에서 나온 지적을 반영해 1단계에는 `idx_posts_user_created_at`, `idx_posts_status_created_at`를 추가했고, 3단계는 최종본 기준과 맞도록 update/delete/select 정책을 drop/recreate 방식으로 다시 맞췄다.
- `sync_post_comment_count`는 `deleted_at` 단독이 아니라 `deleted_at is null and status = 'active'`를 countable 기준으로 보도록 정리해 soft delete와 status 불일치 가능성을 줄였다.
- 2단계 `reports`에는 운영 처리 컬럼이 아직 없는 것이 의도된 분리라는 점을 주석으로 남겼다.
- 추가로 `posts_admin_all`, `comments_admin_all`처럼 권한이 OR 합산되는 broad policy는 제거하고, 각 operation 정책에 admin 조건을 직접 포함시키는 방식으로 정리했다.
- `deleted_at`를 삭제 source of truth로 고정하기 위해 posts/comments에 deleted consistency check 제약을 넣었고, `repair_post_counts(target_post_id uuid)` 함수를 최종본과 2단계 SQL에 추가했다.
- `report_count` 컬럼을 posts에 추가해 신고 누적 집계 확장 여지를 열었고, rate limiting과 signed URL 만료 전략은 앱/Edge 레이어 의존성이라는 점을 SQL 주석으로 명시했다.
- `is_community_admin()`은 profiles.role 구조 변경 시 조용히 깨지지 않도록 `plpgsql + undefined_column exception fallback` 형태로 보강했다.
- 추가 비교 검토 후 `repair_post_counts`는 단일 게시글뿐 아니라 전체 재집계도 가능한 `target_post_id uuid default null` 시그니처로 바꿨고, `reports`와 `audit_logs`의 select 정책은 `*_own_or_admin` 단일 정책으로 다시 정리했다.
- 최종적으로는 `deleted_at + status('deleted')`를 묶는 consistency check, `sync_post_comment_count`의 `deleted_at + status` 동시 기준, partial `idx_posts_user_created_at`, `plpgsql` admin helper, 통합 select RLS 조합을 기준안으로 확정했다.
- 추가로 seed SQL을 재확인해 `public.reports.target_id`가 실제 `uuid`라는 점을 확인했고, `repair_post_counts`는 캐스팅 없이 `uuid = uuid` 비교를 유지하도록 주석을 보강했다.
- `report_count` 컬럼만 있고 실시간 누적 트리거가 빠져 있던 문제를 막기 위해 `sync_post_report_count()`와 `trg_sync_post_report_count`를 2단계 SQL과 최종본에 함께 추가했다.

- 실내놀이 추천 카드의 아이콘/치브론/강조 영역이 펫 테마를 따르도록 공통 props를 추가했다.

### 왜 바꿨는가

- 주요 CTA만 테마 컬러를 따르고, 리스트 내부 선택 상태와 빈 상태 버튼이 고정 보라로 남아 있으면 화면별 시각 규칙이 다시 분리되기 때문이다.
- 현재 NURI는 선택된 펫의 테마가 서비스 전반의 시각적 source of truth 역할을 하므로, 주요 생활형 화면의 버튼과 상태 강조도 같은 기준을 따라야 한다.

### 무엇을 바꿨는가

- 게시글 작성 화면의 `반려동물 연결` 아래에 선택된 아이의 작은 프로필 이미지, 이름, `견종 · 나이` 미리보기를 추가했다.
- 나이 노출 여부를 작성 화면에서 바로 켜고 끌 수 있도록 `나이 함께 표시` 토글을 추가했다.
- draft 저장/복원 payload에 `showPetAge`를 포함해, 이탈 후 다시 들어와도 연결된 아이 미리보기 상태가 유지되게 했다.

### 왜 바꿨는가

- 기존 작성 화면은 연결된 아이를 칩으로만 고르는 구조라, 실제로 어떤 정보가 게시글에 함께 보일지 사용자가 직관적으로 판단하기 어려웠다.
- 커뮤니티는 타임라인보다 공개 노출 맥락이 강해서, 연결된 아이 메타가 과하거나 부족하지 않은지 작성 시점에 바로 확인할 수 있어야 한다.

### 검증 결과

- `yarn tsc --noEmit` 통과
- `yarn lint` 통과

### 남은 리스크

- 현재 `나이 함께 표시`는 작성 화면 UX와 draft 복원에만 반영된다.
- 실제 게시글에 `나이 표시 여부`를 영구 저장하려면 posts 스키마, 타입, service, detail/list 렌더 규칙을 함께 열어야 한다.
- 연결된 아이 정보는 현재 live pet 데이터 기반이라, 게시글 작성 후 반려동물 프로필이 바뀌면 커뮤니티 글에서 보이는 메타도 함께 달라질 수 있다.
- 이미지 draft는 로컬 URI를 저장하므로, 앱 완전 재시작이나 일부 OS 정리 이후에는 복원이 실패할 수 있다.

### 추가 보강

- 커뮤니티 목록에 `lastFetchedAt` 기준을 추가하고, 화면 재포커스 시 45초 이상 stale하면 자동 새로고침되도록 보강했다.
- 목록 상대시간은 분 단위 tick으로 다시 계산되게 해서, 커뮤니티를 나갔다 다시 들어오지 않아도 `몇 분 전` 표시가 묵지 않게 했다.
- `FlatList`에 `overScrollMode="always"`와 `RefreshControl.progressViewOffset`를 넣어 실기기 pull-to-refresh 체감을 보강했다.

### 추가 남은 리스크

- 실기기에서 pull-to-refresh 제스처 감도는 기기/OS별 차이가 있어, 카드 수가 적을 때도 충분히 당겨지는지는 실제 장치에서 다시 확인해야 한다.
- 현재 자동 새로고침은 포커스 복귀 + stale 기준 45초만 커버한다. 장시간 화면을 열어 둔 상태의 완전 실시간성은 후속 polling 또는 이벤트 구조가 필요하다.

### 추가 고도화

- 커뮤니티 작성 화면에서 draft 이미지 복원 전 파일 존재 여부를 점검하도록 보강했다.
- 로컬 URI가 무효하면 본문/카테고리/반려동물 선택/나이 토글만 복원하고, 이미지는 경고 모달 후 다시 고르게 정리했다.
- 연결된 아이 미리보기 아래에 `현재 프로필 기준 미리보기` 안내를 추가해, 아직 live 참조 구조라는 점을 화면에서 분명히 드러냈다.

- 이 메타는 현재 선택된 반려동물 컨텍스트와 무관하게, 게시글에 연결된 `pet_id` 기준으로 읽어 와서 표시한다.

### 무엇을 바꿨는가

- 지역명 단독 검색은 일반 `searchKeyword2` 결과를 그대로 쓰지 않고, 내부 region resolver가 잡히면 `areaBasedList2` 기반 narrow mode로 분기하도록 정리했다.
- `강원도`, `제주`, `강릉`, `일산` 같은 검색어를 위한 지역 메타를 실제 region-intent 용도로 정리하고 주소 앵커 기준으로 다시 좁히도록 바꿨다.
- `전체` 카테고리의 region mode에서는 쇼핑·일반 점포·음식점 노이즈를 기본 통과시키지 않도록 더 보수적으로 제한했다.
- 결과 요약 문구를 실제 노출 후보 수 기준으로 바꾸고, empty 문구에 지역명 + 여행지 키워드 예시를 추가했다.

### 왜 바꿨는가

- live 원응답 기준 `제주`, `일산` 검색 상단이 거의 전부 `ct=38` 쇼핑 점포였고, `강원도`는 `searchKeyword2`에서 0건이 나와 지역 여행 탐색 의도를 제대로 처리하지 못했기 때문이다.
- 이번 이슈는 단순 품질 개선이 아니라 `지역명 검색 의도 처리 실패`에 해당하는 핵심 버그이기 때문이다.

### 검증 결과

- direct live 확인
  - `searchKeyword2('제주')` 상단은 `게스 제주점`, `골든구스 ... 제주점`, `골프존마켓 ... 제주점` 등 `ct=38` 위주
  - `searchKeyword2('일산')` 상단은 `롯데백화점 일산점`, `닥스남성 ...`, `삼성스토어 일산` 등 `ct=38` 위주
  - `searchKeyword2('강원도')`는 0건
  - `areaBasedList2(areaCode=39)`는 제주 여행지 중심 응답 확인
- narrow mode 검증
  - `강원도` -> 공원, 박물관, 해변, 목장 계열 중심으로 정제
  - `제주` -> 해변, 공원, 목장, 미술관 계열 중심으로 정제
  - `강릉` -> 공원, 해변, 숙소 후보 중심으로 정제
