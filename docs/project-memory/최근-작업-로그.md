### 무엇이 달라졌는가

- remote `comments_status_check`를 `auto_hidden` 허용 형태로 재적용해 댓글 2명 unique report 시 `auto_hidden` 전환과 queue/action log 기록이 다시 동작한다.
- report rate-limit trace는 failed insert 트랜잭션 안에서 남기지 않고, 차단 응답 직후 `bump_community_reporter_rate_limit()` follow-up RPC로 영속화하도록 앱 계약을 보강했다.
- `community-images` guard는 `auth.uid()` 외에 `storage.objects.owner_id / owner` fallback을 보도록 보강했고, 기존 bucket object를 `community_image_assets`에 backfill해 remote 경로와 운영 추적 테이블을 다시 맞췄다.

### 검증 결과

- 1차 원인 분리:
  - linked project에서 `comments_status_check`가 `('active','hidden','deleted')`만 허용한다는 사실을 직접 확인했다.
  - report trigger 안의 `bump_community_reporter_rate_limit()`는 같은 failed insert 트랜잭션에서 rollback된다는 점을 SQL/function 기준으로 확정했다.
  - community image upload 실패는 storage trigger가 `auth.uid()`에만 의존하고 실제 storage insert path의 `owner_id / owner`를 보지 않는 정합성 문제임을 재현했다.
- 2차 최소 수정:
  - SQL source of truth와 remote linked project에 comments constraint 재적용, report trigger 정리, storage actor fallback, `community_image_assets` backfill을 반영했다.
  - 앱은 신고 rate-limit 차단 뒤 reporter flag trace를 남기는 follow-up RPC만 추가했다.
- 3차 remote 재검증:
  - 댓글 auto-hide: comment `e730cc80-40b5-4b12-92fd-0930c3a3b2d3`가 `auto_hidden`, queue `report_count=2`, action `auto_hide`로 남는 것을 확인했다.
  - reporter flag: reporter `6f541ac9-9df9-42a4-8515-eda2d562f17a`의 6번째 신고 차단 뒤 `community_reporter_flags.rate_limit_hit_count=1`이 남는 것을 확인했다.
  - image upload: post `6f965ad9-38d7-424b-979c-7ae951dafbcc`에서 실제 upload 성공, `community_image_assets`의 `attached -> cleanup_pending`, hidden 이후 signed URL 차단(`Object not found`)을 확인했다.

### 남은 리스크

### 무엇이 달라졌는가

- 실DB 확인 결과 `comments.user_id`와 대응 `profiles` row, nickname은 실제로 존재했고, 문제는 댓글 author 조립이 `profiles` 조회에만 의존한다는 점으로 좁혀졌다.

- 3차 품질 점검:
  - `corepack yarn tsc --noEmit` 통과
  - 수정 파일 대상 `corepack yarn eslint src/services/supabase/community.ts src/store/communityStore.ts` 통과
