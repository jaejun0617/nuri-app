# 현재 프로젝트 상태

## 프로젝트 현재 목적

NURI는 반려동물 기록, 일정, 탐색, 정보 소비, 공개 커뮤니티를 통합한 생활형 모바일 앱이다.

현재 5차의 목적은 기능 확장이 아니라, 이미 열려 있는 기능 축을 배포 가능한 서비스 계약과 운영 방어선 기준으로 다시 잠그는 것이다.

이번 단계의 최우선 기준은 아래 4가지다.

- 회원가입과 계정 화면을 실제 정책 문서와 연결한다.
- 인증을 가입 가능 상태에서 복구 가능 상태로 올릴 준비를 한다.
- 공개 도메인보다 운영 방어선을 먼저 닫는다.
- Candidate / Trust / User Layer 경계를 문서와 코드에서 일치시키는 방향을 고정한다.

Task 4 후속 보정 기준 현재 상태는 아래처럼 갱신됐다.

- 정책 문서 source of truth는 `src/services/legal/documents.ts`로 모였다.
- `docs/policies/` 아래에 이용약관/개인정보처리방침/계정 삭제 안내 초안 구조가 추가됐고, 현재 상태는 `앱 연결용 초안`으로 고정됐다.
- 회원가입 화면은 필수/선택 동의 UI, 문서 상태, 초안 요약, consent snapshot 버전 참조가 같은 source of truth를 보도록 맞췄다.
- 회원가입 동의는 이제 하나의 카드 섹션 안에서 accordion 행과 `모두 동의하기` 편의 토글로 정리됐다.
- More 화면은 삭제 실행 로직을 유지한 채, 삭제 요청/정리 완료/안내 문서 미정 상태를 보수적으로 드러내도록 정리했다.
- Task 2 구현으로 `delete_my_account()`는 더 이상 단순 void RPC가 아니라, 삭제 요청 상태와 cleanup queue를 남기는 JSON 응답 계약으로 바뀌었다.
- `account_deletion_requests`, `account_deletion_cleanup_items`가 추가돼 요청/진행 중/cleanup pending/완료/실패/unknown 상태를 추적할 수 있게 됐다.
- `user_consent_history`는 hard delete 대신 `user_id null + anonymized_at + anonymized_subject_id` 기준 비식별 보관으로 전환됐다.
- `reports`, `pet_place_user_reports`, `pet_travel_user_reports`, `pet_care_guide_events`, `audit_logs`, `pet_place_search_logs`, `pet_place_verification_logs`는 user 식별자를 제거하는 익명화 보관 기준이 SQL 계약에 반영됐다.
- storage(`pet-profiles / memory-images / community-images`)는 삭제 RPC 안에서 동기 완료하지 않고, cleanup item queue를 남긴 뒤 `completed_with_cleanup_pending` 또는 `cleanup_pending` 상태로 분리 추적한다.
- More 화면 계정 삭제 메시지는 “즉시 완전 삭제” 대신 요청 처리/cleanup pending/unknown 확인 중 상태를 보수적으로 안내하도록 맞췄다.
- Task 2 적용 후 최종 검증에서 SQL과 앱 계약의 큰 불일치는 발견되지 않았고, 유효 상태값 일부가 앱에서 조용히 빠질 수 있던 분기와 QA gate의 자동/수동 검증 구분만 최소 보정했다.
- Task 3 구현으로 비밀번호 재설정은 `Supabase 기본 password reset email link + custom scheme(nuri://auth/reset) 복귀 + 앱 내 새 비밀번호 설정 화면` 조합으로 연결됐다.
- `App.tsx`에는 React Navigation `linking` 객체가 추가됐고, Supabase recovery redirect의 `#fragment` 토큰을 query 형태로 정규화해 `PasswordResetRecovery` route가 받도록 정리됐다.
- Android `AndroidManifest.xml`, iOS `Info.plist`, `AppDelegate.swift`에 custom scheme 수신 경로가 추가됐다.
- SignIn의 `비밀번호 찾기`는 더 이상 placeholder가 아니고 `PasswordResetRequest`로 진입한다.
- reset 완료 후에는 recovery 세션을 로컬에서 정리하고 SignIn으로 reset 이동한 뒤 `비밀번호가 변경되었습니다. 새 비밀번호로 다시 로그인해 주세요.` 안내를 띄우도록 구현됐다.

- Task 3 구현 완료
  - `PasswordResetRequest`, `PasswordResetRecovery`, `PasswordResetForm` 3개 화면으로 reset 흐름을 최소 구성했다.
  - reset 요청은 enumeration 방지 문구와 60초 재발송 쿨다운을 적용했다.
  - recovery 링크는 `nuri://auth/reset` custom scheme으로 앱에 복귀하고, fragment token을 navigation params로 변환해 recovery 세션을 연다.
  - 만료/오류 링크는 SignIn이 아니라 `PasswordResetRequest`로 복귀시키고 즉시 재요청 가능 흐름을 유지한다.
  - reset 성공 후 자동 로그인은 하지 않고 recovery/local 세션을 정리한 뒤 SignIn 재로그인 유도로 마무리한다.
- Task 4 구현 완료
  - `docs/sql/도메인/커뮤니티/커뮤니티-운영-스키마-최종.sql` 기준으로 게시글/댓글/신고/이미지 업로드의 서버측 guard trigger와 moderation 보조 테이블 계약을 추가했다.

  - 게시글 `제목+본문`, 댓글 `동일 post 내 동일 내용`은 30초 short-window dedupe를 서버 기준으로 막는다.
  - `community_moderation_queue`, `community_moderation_actions`, `community_reporter_flags`, `community_image_assets`를 추가해 신고 triage, auto-hide, action log, 반복 악성 신고 추적, orphan/cleanup 상태를 남기도록 정리했다.
  - auto-hide는 unique reporter 기준으로 게시글 3명, 댓글 2명 이상일 때 `auto_hidden`과 queue/action log를 함께 남기도록 구현했다.

  - repo 기준 운영 검증에서 다중 이미지 업로드 일부 실패 path를 cleanup queue로 회수하도록 보정했고, 신고 직후 auto-hidden 된 게시글이 detail 재조회 후 list 캐시에 active 상태로 남지 않게 정리했다.

## 현재 기준 핵심 도메인 상태

- 산책
  - 공통 `locationDiscovery` 레이어 기반
  - 서비스 추천 정렬 + 거리 정렬 제공
- 펫동반 장소 탐색
  - 메뉴/화면 진입 존재
  - 현재 운영 데이터는 Kakao Local 외부 검색 결과
  - 자체 Supabase 장소 DB는 아직 없음
  - 리스트/상세/지도 미리보기/외부 지도 열기/연관 장소 더보기 구조 반영
  - `외부 후보`와 `서비스 검증 상태`를 분리한 내부 상태 모델 반영
  - `placeMeta.ts`에 `pet_place_source_links -> pet_place_service_meta -> pet_place_external_signals` 실제 조회가 연결됨
  - 현재 앱은 Kakao 후보를 normalize한 뒤, Supabase canonical 메타와 외부 보조 신호를 병합하는 구조까지 반영됨
  - 외부 소스 역할은 `Kakao 후보 수집 / Google Places 보조 속성 / 관광공사·공공데이터 보강 / Supabase 최종 메타`로 정리됨
  - 실제 서비스급 장소 메타 SQL 문서는 `docs/sql/도메인/펫동반-장소-탐색/펫동반-장소-메타-운영-스키마-최종.sql`
  - 신뢰도 보강 API 호출 전략 문서는 `docs/domains/펫동반-장소-탐색/펫동반-장소-탐색-신뢰도-보강-API-전략.md`
  - 장기 확장용 `반려동물과 여행` 분리 설계 문서가 시작됨
- 반려동물과 여행
  - MoreDrawer와 RootNavigator에 별도 메뉴/라우트가 열림
  - 현재 구조는 `검색 입력 -> 카테고리 선택 -> 리스트 -> 상세 -> 외부 지도 열기`
  - 현재 위치/GPS 기반 추천은 아직 넣지 않음
  - 핵심 후보 소스는 한국관광공사 반려동물 동반여행 서비스이며, 리스트/상세 조회가 TourAPI fetch 구조로 연결됨
  - 현재 런타임에는 `TOUR_API_SERVICE_KEY`가 들어가 있음
  - 2026-03-19 공식 Swagger 재확인 결과 유효 provider는 `KorPetTourService2`이며, 리스트/검색/상세 핵심 오퍼레이션은 `areaBasedList2`, `searchKeyword2`, `detailCommon2`, `detailIntro2`, `detailPetTour2`다.
  - 기존 live 실패 원인은 `areaBasedList`, `detailCommon`, `detailIntro` 같은 비공식 오퍼레이션 호출이었다.
  - 현재 mock 리스트 fallback은 없고, 공식 `KorPetTourService2/*2` 호출 기준으로 live 응답이 실제 확인된 상태다.
  - `searchKeyword2`는 지역명 단독 검색에서 쇼핑/브랜드 점포를 넓게 섞어 주거나 `강원도`처럼 0건을 돌려주는 경우가 있어, 지역명 단독 검색은 전국 공통 region resolver + `areaBasedList2` narrow mode로 처리한다.
  - region resolver는 특정 예시 지역 patch가 아니라 전국 시/도 + 주요 시/군/구/생활권 별칭 메타를 기준으로 동작한다.
  - 검색 후처리는 `placeType`과 `petAllowed`를 분리하며, `travelScore / petScore / commercialPenalty`를 `0~1` 정규화 점수로 재구성했다.
  - TourAPI raw 문구만으로 `confirmed`는 만들지 않고, 현재 런타임에서는 `possible / check-required` 중심의 보수적 UI를 유지한다.
  - 장기 source of truth는 `장소 검색 API + TourAPI + 자체 DB` 조합이며, 앱 타입에는 `petPolicy`, `aggregation`, `places / pet_policies / user_reports` 확장 방향이 반영됐다.
  - 이번 턴에 Trust Layer DB 초안이 `pet_travel_places / pet_travel_pet_policies / pet_travel_user_reports` 기준으로 정리됐다.
  - 이번 SQL은 외부 원본 매핑 전용 `source_links`를 아직 만들지 않고, `places.primary_source + primary_source_place_id`로 bootstrap하는 방향이다.

  - 하단 탭 기준 진입점은 `CommunityTab`으로 승격됐고, 메인 탭 구조는 `Home / Timeline / Community / Guestbook / More` 기준이다.
  - 하단 탭의 `CommunityTab`도 별도 `CommunityTabStackNavigator`를 통해 공통 `CommunityStackHeader`를 받도록 정리됐다.
  - 현재 댓글은 `top-level 10개 + 더보기`, `reply preview 2개`, `reply는 depth 1 평탄화` 기준으로 고도화 중이다.
  - 댓글 좋아요는 `comment_likes` 별도 테이블과 optimistic update 기준으로 설계가 넘어갔다.
  - 상세 렌더는 `commentsByPostId` 배열 직접 렌더에서 벗어나 `commentEntitiesById`, `topLevelCommentIdsByPostId`, `replyCommentIdsByParentId` 기반으로 1차 최적화됐다.
  - 리스트 렌더는 `posts` 직접 renderItem 소비에서 벗어나 `postIds + CommunityPostListItem + pure PostCard` 구조로 1차 최적화됐다.
- 기록
  - 기록 작성 화면은 더 이상 하단 탭 스크린이 아니고, `RootNavigator.RecordCreate` 독립 스크린으로 열린다.
  - 홈/타임라인의 기록하기 CTA는 `returnTo`를 유지한 채 root stack `RecordCreate`로 진입한다.
  - `RecordDetail`은 `entrySource` 기반 뒤로가기 복귀 규칙이 복구돼 홈/More 진입 흐름을 다시 보존한다.
  - 기록 상세/작성과 타임라인 리스트는 최근 톤 정리로 흰 배경과 보더 축소 기준을 사용한다.

## 주요 리스크

### 해결됨

- 비밀번호 재설정은 더 이상 미구현 리스크가 아니다. request/recovery/form, custom scheme 복귀, 재발송 쿨다운까지 코드에 반영됐다.
- 계정 삭제는 더 이상 단순 void RPC가 아니다. 요청 상태, cleanup queue, 익명화 보관, 보수적 사용자 메시지 계약이 반영됐다.
- 커뮤니티 abuse 최소 방어선은 repo 기준으로 구현됐다. rate limit, dedupe, auto-hide, moderation queue/action log, image cleanup_pending 계약이 SQL과 앱에 반영됐다.
- 댓글 작성자 unknown 이슈는 DB 유실이 아니라 `profiles` 의존 + RLS 제약 문제로 확정했고, app fallback/store merge 보정까지 반영됐다.

### 부분 해결 / 운영 증적 필요

- 정책 문서 source of truth와 초안 구조는 생겼지만, 실제 약관/개인정보/삭제 안내의 최종 본문과 공식 링크는 아직 미정이다.
- 회원가입은 정책 상태를 숨기지 않지만, 사용자가 실제 전체 본문을 열람하는 최종 연결은 아직 없다.
- SQL 문서는 새 계약으로 갱신됐지만, staging/prod 실제 적용 여부는 아직 사람 확인이 필요하다.
- storage cleanup queue와 상태 추적은 구현됐지만, 실제 worker/service-role 재처리 경로는 아직 운영 환경에서 검증하지 못했다.
- `unknown_pending_confirmation`은 앱과 SQL 양쪽에 반영됐지만, 실제 timeout 재현 QA는 아직 남아 있다.
- `detectSessionInUrl: false` + custom scheme 조합은 의도된 계약이지만, Supabase dashboard redirect 허용과 메일 앱 복귀 실기기 QA는 아직 남아 있다.
- Task 4 SQL/source of truth는 구현됐지만, staging/prod에 trigger/RLS/storage 정책이 실제로 적용됐는지는 아직 사람 확인이 필요하다.
- `community_moderation_queue / community_moderation_actions / community_reporter_flags / community_image_assets`는 구현됐지만, 이를 소비하는 전용 운영 UI와 처리 절차는 아직 없다.
- release checklist에는 커뮤니티 abuse QA gate를 추가했지만, 실DB 증적과 실기기 auto-hide/restore/cleanup_pending 캡처는 아직 남아 있다.
- 댓글 author fallback은 앱 쪽 최소 보정으로 안정화했지만, 공개 post snapshot이 전혀 없는 신규 작성자까지 100% 보장하는 서버 계약은 아직 아니다.
- 커뮤니티 댓글/리스트/공통 header/typography 리팩터링은 1차 반영됐지만, 실기기 성능과 UX QA는 더 필요하다.

### 미해결 / 다음 작업 대상

- Task 4 원격 blocker 3개는 linked project `grmekesqoydylqmyvfke`에서 hotfix 후 재검증됐다. 댓글 auto-hide는 `auto_hidden + queue/action log`까지 정상 전이됐고, 신고 rate-limit 차단 뒤 `community_reporter_flags.rate_limit_hit_count`도 follow-up RPC 경로로 남으며, community image upload는 다시 성공하고 `community_image_assets` backfill/attach/cleanup_pending/hidden 비노출까지 확인됐다.
- 다만 전용 운영 UI 없이 SQL/콘솔 기준으로 queue/action log/cleanup_pending을 확인하는 구조는 그대로라서, release 직전 운영 캡처와 restore/manual cleanup 절차 점검은 남아 있다.
- 장소/여행은 Candidate / Trust / User Layer 원칙은 문서화됐지만 런타임 source of truth는 여전히 candidate 중심이다.
- Kakao Local만으로는 실제 반려동물 동반 가능 여부를 100% 보장할 수 없다.
- TourAPI raw만으로는 `confirmed`를 만들 수 없고, 아직 `pet_policies`/검수 데이터가 없어 최종 신뢰도 레이어가 비어 있다.
- Supabase 장소 메타/운영 검수/북마크/검색 로그의 실제 런타임 승격은 아직 없다.
- 외부 API 키/환경 분리/운영 도구 분리와 같은 장기 운영 인프라 리스크는 여전히 남아 있다.
- 최종 법무 문안, 정책 URL, 위치정보 고지, 운영정책 문서화는 아직 미완성이다.

## 최신 운영 방향

- 5차는 기능 확장보다 계약, 운영, 신뢰도 방어선 우선으로 진행한다.
- 현재 작업 순서는 `Task 4 -> Task 5`로 이어진다.
- Task 4는 repo 기준 운영 검증과 최소 보정에 더해 linked project remote hotfix와 재검증까지 반영됐다.
- Task 4 remote 재검증에서는 `comment auto-hide`, `reporter flag trace`, `community image upload/image_assets`, `hidden 이미지 비노출`, `cleanup_pending` 경로가 다시 확인됐다.
- 최종 정책 문서 링크와 운영 고지가 확정되기 전에는 정책 없는 회원가입 UX 확장, 소셜 로그인 선구현, 결제 선구현, Guestbook 기능 확장, moderation 없는 커뮤니티 기능 확장을 하지 않는다.
- Task 1에서는 정책 본문을 새로 쓰지 않고 정책 연결 구조와 안전한 fallback만 정리했다.
- Task 2.5에서는 최종 법무 문안 확정 대신 초안 구조와 가입 UX를 먼저 정리했다.
- Task 2에서 삭제 계약은 구현까지 반영됐고, 계정 삭제는 `요청 수락`과 `최종 정리 완료`를 분리해 추적한다.
- storage cleanup은 DB와 분리된 queue 계약으로 다루고, pending 상태를 성공으로 포장하지 않는다.
- Task 3는 구현까지 반영됐고, Task 4는 PO lock-in 값 기준 최소 구현까지 반영됐다.
- Task 4 repo 검증에서는 `partial image upload orphan 회수`, `report 후 auto-hidden post 재동기화`, `release checklist의 운영 증적 분리`까지 닫혔다.
- Task 4 후속 점검에서는 댓글 작성자 unknown 이슈의 원인을 `profiles` 의존 + RLS 제약 + refetch overwrite로 확정했고, DB/RLS 변경 없이 앱 author fallback과 store merge만 최소 보정한다.
- Task 5는 Task 4 remote blocker 재검증이 끝난 지금부터 착수 가능하지만, release 직전에는 community moderation/cleanup SQL 콘솔 절차 캡처를 병렬로 보강한다.
- 비밀번호 재설정은 `Supabase recovery link + custom scheme 앱 복귀 + 새 비밀번호 설정 화면 + 성공 후 SignIn 재로그인 유도`로 고정한다.
- reset 요청 단계의 사용자 메시지는 계정 존재 여부를 노출하지 않는 보수 문구를 유지하고, 앱 레벨 재발송 제한은 60초 쿨다운으로 고정했다.
- reset 링크 만료/오류 화면은 SignIn보다 reset request 화면 재진입을 우선 제공한다.
- 전역 typography source of truth는 `src/app/theme/tokens/typography.ts`와 `src/app/ui/AppText.tsx`를 기준으로 `display / titleLg / titleMd / titleSm / body / bodySm / button / tab / helper / caption` 체계로 정리했다.
- 기존 `AppText` preset alias는 호환성을 유지하되 `title1 -> display`, `title2 -> titleLg`, `headline -> titleSm`, `body -> 16/24`, `caption -> helper(13/18)` 기준으로 상향했다.
- 공통 접근성 보정 1차 범위는 하단 탭, 헤더 액션 버튼, Toast, ConfirmDialog, Auth 스타일, GuestHome, Community 카드/리스트/상세다.
- 하단 탭 라벨 `9/11`, 헤더 액션 `34~36`, Community/Auth의 다수 `10~12px` 텍스트를 제거했고, 주요 액션은 `44~48` 터치 영역 기준으로 올렸다.
- 이번 리팩토링은 typography/accessibility 범위로 제한했고 기능 계약, DB/Auth/Supabase 계약 변경은 포함하지 않았다.
- 실기기 기준 추가 확인이 필요한 축은 `200% 글꼴 확대`, `긴 한글 CTA 2줄`, `CommunityDetail 댓글 입력/신고 시트`, `GuestHome 카드형 레이아웃`이다.
- 위치 기반 기능은 복잡한 지역 브라우징 대신 `현재 GPS 기반 주변 추천 + 직접 검색`으로 고정한다.
- 펫동반 장소 탐색은 단기적으로 외부 API 결과를 쓰되, 중기적으로는 `외부 원본 + Supabase 메타 혼합`으로 간다.
- 반려동물과 여행은 GPS 필수 도메인으로 보지 않고 `검색 입력 + 카테고리` 기반 여행 탐색으로 정리한다.
- 펫동반 검증 상태는 `unknown / keyword-inferred / user-reported / admin-verified / rejected`로 분리하고, 카카오 결과만으로 확정 표시하지 않는다.
- Supabase 장소 메타는 `canonical meta + source link + external signal` 구조로 설계하고, 상태 제약은 초기에는 `text + check constraint`를 우선 검토한다.
- 반려동물과 여행 펫정책은 `Raw Layer -> Candidate Layer -> Trust Layer -> Aggregation Layer` 4단계 구조로 정리한다.
- `confirmed`는 향후 `pet_policies + user_reports + admin review` 조합이 붙었을 때만 허용한다.
- 이번 단계의 Trust Layer는 `pet_travel_places / pet_travel_pet_policies / pet_travel_user_reports` 3테이블로 시작하고, `source_links`는 후속 확장으로 미룬다.
