# docs/sql 정리 가이드

`docs/sql`은 사람이 읽는 최종 SQL 문서 세트를 모아 두는 기준 폴더다.

실행 이력은 `supabase/migrations/`에 보존하고, `docs/sql`은 도메인별 최종본, 보조 SQL, legacy 문서를 읽기 쉬운 한글 구조로 유지한다.

## 현재 구조

- `docs/sql/커뮤니티/`
  - 커뮤니티 도메인 최종본과 legacy 단계 SQL
- `docs/sql/인증-계정/`
  - 계정 삭제/동의 이력 final SQL과 닉네임 정책 local mirror, 보조 SQL
- `docs/sql/타임라인-메모리/`
  - 타임라인 메모리 이미지 final SQL과 메모리 호환성/기능 추가 SQL
- `docs/sql/가이드/`
  - 집사 꿀팁 가이드 final SQL, storage 정책, seed, 점검 SQL, legacy 문서 migration
- `docs/sql/펫동반-장소/`
  - 장소 메타 final SQL과 legacy 초안
- `docs/sql/반려동물과-여행/`
  - 여행 Trust Layer final SQL
- `docs/sql/공용/`
  - 전체 초기 구성 bootstrap SQL

## 분류 규칙

- `supabase/migrations/*`
  - 실제 실행 이력이다.
  - 이미 적용된 migration 파일은 삭제/개명/이동하지 않는다.
- `docs/sql/<도메인명>/*-최종.sql`
  - 해당 도메인의 문서용 최종 source of truth다.
  - linked remote drift가 있으면 이 파일을 기준으로 차이를 분석한다.
- `docs/sql/<도메인명>/레거시/*`
  - 과거 단계 적용본, 문서 migration, 초안처럼 현재 신규 적용 기준으로 쓰지 않는 파일이다.
- `docs/sql/<도메인명>/*-보조.sql`, `*-추가.sql`, `*-정책.sql`, `*-seed.sql`
  - 최종본을 보완하는 기능 추가, 호환성, storage, seed, 점검용 SQL이다.
- `docs/sql/공용/*`
  - 신규 프로젝트 bootstrap이나 공용 기능 묶음용 SQL이다.
  - 현재 운영 remote의 exact mirror처럼 읽지 않는다.

## 운용 원칙

- 먼저 linked remote와 `supabase/migrations/*`를 보고 실행 상태를 확인한다.
- 그다음 `docs/sql/<도메인명>/*-최종.sql`로 사람이 읽는 최종 상태를 본다.
- 같은 도메인의 단계별 SQL이 여러 개 있으면 최종본으로 통합하고, 남은 단계 파일은 `레거시/`로 내린다.
- remote와 local이 다르면 최종본이라고 적더라도 운영 반영 완료로 쓰지 않는다.
- 현재 앱이 의존하는 RLS, trigger, function, RPC, Storage 정책은 근거 없이 제거하지 않는다.

## migration baseline 재구성 판단

- 현재 시점에는 `supabase/migrations/*`를 archive/delete 대상으로 올리지 않는다.
- 이유는 현재 `supabase/migrations/*`가 최근 커뮤니티 delta 위주 실행 이력이라 프로젝트 전체 baseline 세트가 아니고, linked remote full schema dump를 확보하지 못했으며, `반려동물과-여행` local-only 최종본과 프로필 닉네임 정책 remote-only drift가 남아 있기 때문이다.
- 새 빈 DB를 baseline만으로 재구축 가능하다고 판단하려면 아래 조건이 먼저 충족돼야 한다.
  - linked remote full schema dump 또는 동등한 catalog 캡처 확보
  - local-only / remote-only drift 해소
  - 빈 DB 기준 재구축 검증
  - 현재 앱 runtime 계약과 baseline SQL의 정합성 재검증
