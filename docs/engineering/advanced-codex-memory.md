# advanced-codex-memory.md

> 이 문서는 Codex가 세션 간 기억이 없다는 전제 아래,
> 프로젝트 내부 문서를 사용해 일관된 판단과 히스토리 추적을 유지하기 위한 운영 규칙이다.

---

## 0. 문서 목적

이 문서의 목적은 다음 3가지를 보장하는 것이다.

1. Codex가 “이전 대화를 기억하는 척” 하지 않도록 한다.
2. 작업의 최신 상태를 문서로 남겨 다음 세션이 그대로 이어받도록 한다.
3. 기억의 기준, 충돌 해소 기준, 갱신 순서를 명확히 하여 문서 오염을 줄인다.

---

## 1. 문서 개선 포인트

### Fix

- 메모리 문서 읽기/쓰기 순서와 우선순위를 명시적으로 통합했다.
- 충돌 해소 기준을 실제 코드 중심으로 재정렬했다.
- 오래된 로그 누적과 최신 상태 덮어쓰기의 역할 차이를 명확히 분리했다.

### Enhance

- 각 메모리 문서의 역할, 필수 섹션, 갱신 트리거, 작성 원칙을 더 세밀하게 정의했다.
- 오염된 문서 복구 절차와 로그 아카이브 절차를 강화했다.
- 도메인 문서, SQL 문서, 운영 문서와 메모리 문서의 연결 방식을 명확히 했다.

### Add

- 문서 ID, 변경 ID, 리스크 ID 규칙을 추가해 추적성을 강화했다.
- 작업 종료 시 “컨텍스트 델타(delta)”를 남기는 규칙을 추가했다.
- `docs/리서치/리서치.md`를 결정 보조 문서로 연결했다.

---

## 2. 핵심 전제

- Codex는 세션 간 기억이 없다.
- 메모리 문서가 없으면 기억이 없는 것으로 취급한다.
- 문서에 없는 것은 “모름”으로 처리한다.
- 기억이 애매하면 추측하지 않고 코드/문서/로그를 다시 확인한다.
- 메모리 문서와 실제 코드가 다르면 코드를 우선한다.
- 다만 행동 규칙은 코드가 아니라 workflow 문서를 우선한다.

## 6. 각 문서의 역할

## 6-1. 현재-프로젝트-상태.md

역할:

- 지금 이 순간의 프로젝트 상태를 가장 빠르게 파악하는 요약본

반드시 포함:

- 프로젝트 목적
- 현재 아키텍처 요약
- 완료된 챕터/단계
- 진행 중 챕터/단계
- 핵심 화면/도메인 상태
- 주요 버그/리스크
- 최신 운영 방향
- source of truth 문서/테이블/구조

작성 원칙:

- 최신 상태를 덮어쓴다.
- 과거 설명보다 현재 기준이 먼저 보이게 한다.
- 장황한 서술보다 운영 요약에 가깝게 쓴다.

## 6-2. 최근-작업-로그.md

역할:

- 최근 작업 이력과 이유를 시간순으로 남기는 누적 로그

반드시 포함:

- 날짜
- 작업 주제
- 변경 ID
- 수정 파일
- 변경 내용
- 변경 이유
- 검증 결과
- 남은 리스크
- 다음 작업 연결 포인트
- 메모리 문서 갱신 내용
- 컨텍스트 델타

작성 원칙:

- 최신 항목을 맨 위에 둔다.
- 사람이 로그만 보고도 맥락을 복원할 수 있어야 한다.
- 로그는 지우지 않되, 오래되면 아카이브한다.

## 6-3. 핵심-결정사항.md

역할:

- 중요한 설계 선택과 그 이유를 남기는 결정 기록

반드시 포함:

- 결정 ID
- 결정 날짜
- 결정 주제
- 현재 선택
- 결정 이유
- 검토 대안
- 변경 시 함께 수정할 항목
- 변경 이력
- 관련 문서 링크

작성 원칙:

- “왜 이렇게 했는가”를 남긴다.
- 현재 선택이 맨 위에서 명확하게 보이게 한다.
- 이전 결정을 지우지 말고 변경 이력으로 남긴다.

## 6-4. 다음-작업-우선순위.md

역할:

- 다음 작업의 우선순위와 선행 조건을 유지하는 실행 대기열

반드시 포함:

- 지금 당장 할 것
- 그 다음 할 것
- 왜 그 순서인지
- 지금 하면 안 되는 것
- 실기기 검증 대기 항목
- 선행 조건
- 차단(blocker) 여부

작성 원칙:

- “언젠가 할 것”과 “지금 당장 할 것”을 분리한다.
- 완료된 항목은 제거하거나 이동한다.
- 우선순위 이유를 반드시 적는다.

- 남은 리스크가 무엇인가
- 다음 우선순위가 무엇인가

## 9. 메모리 문서 갱신 원칙

- 중복 문장을 줄인다.
- 오래된 상태 설명은 최신 상태로 덮어쓴다.
- 로그성 문서는 누적하되 상태 문서는 최신 기준으로 정리한다.
- 초안 문체보다 운영 문서 문체를 쓴다.
- 문서 길이가 길어져도 최신 상태가 먼저 보여야 한다.
- 불확실한 정보는 추정하지 말고 `확인 필요`로 표기한다.
- 코드와 다를 가능성이 있으면 “불일치 후보”로 적고 확인 후 수정한다.

---

## 10. 충돌 해소 기준

여러 메모리 문서가 서로 충돌할 때는 아래 순서로 판단한다.

1. 실제 코드
2. `핵심-결정사항.md`
3. `현재-프로젝트-상태.md`
4. `최근-작업-로그.md`의 최신 항목
5. `다음-작업-우선순위.md`

충돌 발견 시 절차:

1. 실제 코드를 확인한다.
2. 불일치 문서를 식별한다.
3. 사람 확인이 필요한지 판단한다.
4. 확인 후 관련 문서를 한 번에 갱신한다.
5. `최근-작업-로그.md`에 정정 사실을 기록한다.

---

## 11. 메모리 오염 복구 절차

잘못된 문서 갱신이 있었던 것으로 보이면 아래 절차를 따른다.

1. 실제 코드와 현재 동작을 기준으로 올바른 상태를 다시 파악한다.
2. 오염된 항목만 특정한다.
3. 사람 확인이 필요한 항목을 분리한다.
4. 문서 전체를 갈아엎지 말고 잘못된 부분만 수정한다.
5. `최근-작업-로그.md`에 `메모리 문서 오류 수정` 항목으로 남긴다.

---

## 12. 아카이브 규칙

`최근-작업-로그.md`의 작업 항목이 20개를 넘으면:

1. 최신 10개만 남긴다.
2. 그 이전 항목은 `docs/project-memory/아카이브/작업-로그-YYYY-MM.md`로 이동한다.
3. 아카이브 파일은 읽기 전용으로 취급한다.
4. 현재 로그에는 아카이브 파일 경로를 남긴다.

---

## 13. 문서 ID 규칙

추적성을 높이기 위해 아래 ID를 사용한다.

- 결정 ID: `DEC-YYYYMMDD-001`
- 로그 ID: `LOG-YYYYMMDD-001`
- 리스크 ID: `RISK-YYYYMMDD-001`
- 변경 ID: `CHG-YYYYMMDD-001`

원칙:

- 같은 작업의 코드 수정, 문서 수정, 로그 기록은 하나의 변경 ID로 묶을 수 있다.
- 큰 설계 변경은 변경 ID와 결정 ID를 같이 남긴다.

---

## 14. 컨텍스트 델타 규칙

각 작업 로그에는 반드시 “이 작업으로 무엇이 달라졌는가”를 짧게 남긴다.

예:

- 이전: 목록과 상세가 같은 payload를 사용함
- 이후: 목록은 lightweight payload, 상세는 full payload로 분리
- 영향: 타임라인 fetch 비용 감소, 상세 최초 진입 유지
- 후속: 상세 prefetch 전략 검토 필요

이 델타가 다음 세션의 가장 빠른 진입점이 된다.

## 16. 프로젝트 고정 결정사항

아래 항목은 프로젝트 고정 결정사항으로 유지한다.

- 집사 꿀팁 가이드 최종 운영은 웹 관리자 페이지(CMS) 기준
- 현재 앱 관리자 화면은 중간 운영 수단
- SQL은 seed/보정용
- 앱은 사용자 소비 중심

이 항목이 바뀌면 반드시 `핵심-결정사항.md`와 관련 도메인 문서를 함께 수정한다.

## 18. 금지 규칙

- 메모리 문서 없이 지속 작업 진행 금지
- “예전 대화에서 그랬던 것 같다” 식 추정 금지
- 문서 전체를 새로 쓰는 방식의 무차별 덮어쓰기 금지
- 충돌을 발견하고도 로그 없이 지나가는 것 금지
- 현재 상태 문서와 로그 문서의 역할 혼동 금지
