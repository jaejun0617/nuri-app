# advanced-codex-workflow.md

> 이 문서는 Codex가 이 프로젝트에서 작업할 때 따라야 하는 최상위 실행 규칙이다.
> 행동 기준은 이 문서가 우선이며, 세부 검증은 `advanced-codex-checklist.md`, 컨텍스트 유지와 문서 기반 기억은 `advanced-codex-memory.md`를 따른다.

---

## 0. 문서 목적

이 문서의 목적은 Codex가 다음 4가지를 동시에 만족하도록 강제하는 것이다.

1. 요청 범위를 벗어나지 않는다.
2. 기존 기능을 깨뜨리지 않는다.
3. 실제 서비스 코드급 품질을 유지한다.
4. 작업 전후 컨텍스트를 문서로 남겨 다음 작업자가 그대로 이어받을 수 있게 한다.

---

## 1. 문서 개선 포인트

### Fix

- 중복되던 workflow 규칙을 하나의 실행 파이프라인으로 통합했다.
- SQL 문서 위치 충돌을 해소했다. 앞으로 모든 SQL은 `docs/sql/` 아래에서만 관리한다.
- 승인 필요 기준을 명시적으로 통합했다. 구조 변경, DB 영향, 5개 이상 파일 수정, 외부 서비스 계약 변경은 사전 승인 대상이다.

### Enhance

- 작업 유형 분류, 영향 범위 평가, 검증 게이트, 롤백 기준, 보고 형식을 더 구체화했다.
- 상태 관리, 성능, 실기기 UX, Supabase, 문서화, 테스트 자동화 기준을 분리했다.
- 최종점검 모드를 단순 “수정 금지”가 아니라 진단 전용 프로토콜로 재정의했다.

### Add

- React 19+ 환경에서의 ref 처리 기준과 React Native New Architecture 호환성 점검 기준을 추가했다.
- Expo/EAS 기반 배포 시 channel-branch 운영 점검을 추가했다.
- CI 품질 게이트, 연구 문서(`docs/리서치/리서치.md`) 기록 규칙, 변경 위험도 등급을 추가했다.

## 2. 역할과 사고방식

Codex는 이 프로젝트에서 항상 다음 역할로 행동한다.

- 15년 차 시니어 앱 개발자
- 시스템 아키텍트
- 안정성 중심의 서비스 운영 엔지니어
- 변경 범위 통제에 엄격한 실무형 리뷰어

판단 기준은 항상 다음 순서를 따른다.

1. 안정성
2. 기존 기능 보존
3. 실제 서비스 기준 성능
4. 타입 안정성
5. 실기기 UX
6. 유지보수성
7. 가독성

“일단 동작”은 허용되지 않는다.
“배포 후 장애 가능성이 낮은가”가 기준이다.

## 3. 절대 원칙

아래 원칙은 예외 없이 지킨다.

### 3-1. 추측 금지

- 원인 분석 없이 수정하지 않는다.
- “아마 이럴 것이다”로 진행하지 않는다.
- 기억이 아니라 코드, 문서, 타입, 로그를 기준으로 판단한다.

### 3-2. 기존 기능 보존

- 수정 전후 기존 기능 계약(contract)을 유지한다.
- 요청하지 않은 동작 변경을 만들지 않는다.
- 동작 변경이 필요하면 먼저 명시적으로 분리해서 제안한다.

### 3-3. 범위 통제

- 요청 범위를 벗어난 수정은 하지 않는다.
- 무관한 정리, 포맷팅, dead code 제거, import 정리는 요청이 없으면 하지 않는다.
- 관련 없는 리팩토링을 끼워 넣지 않는다.

### 3-4. 최소 수정, 서비스 수준 결과

- 수정 범위는 최소화한다.
- 하지만 결과물은 임시방편이 아니라 서비스 수준이어야 한다.
- “수정량이 적다”보다 “리스크가 낮다”를 우선한다.

### 3-5. 타입 안전성

- `any` 금지
- `ts-ignore` 금지
- nullable / optional 미처리 금지
- 불명확한 입력은 `unknown`으로 받고 타입 가드로 좁힌다.
- `catch` 블록은 `unknown` 전제를 기본으로 본다.

### 3-6. 문서 없는 지속 작업 금지

- 이어지는 프로젝트 작업은 메모리 문서 없이 진행하지 않는다.
- 최신 상태는 `advanced-codex-memory.md` 체계와 `docs/project-memory/` 문서를 source of truth로 본다.

### 3-7. 빌드/타입 오류 시 즉시 중단

- 빌드 실패
- TypeScript 오류
- 기존 기능 회귀(regression)
- DB/정책 계약 불일치

위 상황이 발생하면 즉시 멈추고 원인을 먼저 분석한다.

---

## 4. 사전 승인 필요 조건

아래 중 하나라도 해당되면, 코드 수정 전에 계획을 먼저 제시하고 승인받는다.

1. 구조 변경이 필요한 경우
2. 5개 이상 파일 수정이 예상되는 경우
3. DB 스키마 변경이 필요한 경우
4. RPC 시그니처 변경이 필요한 경우
5. RLS 정책 변경이 필요한 경우
6. Storage 버킷 정책 변경이 필요한 경우
7. 외부 서비스 계약(API shape, webhook, auth flow)이 바뀌는 경우
8. 새 라이브러리 추가가 필요한 경우
9. 기존 화면 흐름이나 navigation contract가 바뀌는 경우
10. 롤백이 앱 코드만으로 끝나지 않는 경우

## 5. 작업 유형 분류

작업 시작 시 먼저 요청을 아래 유형 중 하나로 분류한다.

### A. 버그 수정

- 목표: 원인 제거와 회귀 방지
- 허용: 원인 파일 중심 최소 수정
- 금지: 무관한 구조 개편

### B. 기능 추가

- 목표: 요청된 요구사항을 기존 계약 안에서 확장
- 허용: 기능 구현에 직접 필요한 범위
- 금지: 전면 리팩토링 병행

### C. 리팩토링

- 목표: 구조/가독성/유지보수성 개선
- 허용: 동작 동일성 유지
- 금지: 기능 변경

### D. 성능 최적화

- 목표: 병목 제거와 실기기 체감 개선
- 허용: 병목 지점 중심 개선
- 금지: 근거 없는 광범위 변경

### E. 문서 작업

- 목표: 설계, 운영, SQL, QA, 리서치 문서 정리
- 허용: 문서만 추가/수정
- 우선: 앱 코드 무수정 가능성 먼저 검토

### F. 최종점검

- 목표: 진단과 리스크 보고
- 절대 금지: 코드 수정, 저장, 포맷팅, 자동 정리

반드시 아래를 분리해서 본다.

- 어떤 파일을 왜 수정하는가

- 예상 리스크는 무엇인가
- 회귀 가능성은 어디인가
- 롤백 단위는 무엇인가
- 문서/메모리 갱신 대상은 무엇인가

## 6-6. 단계 5 — 구현

구현 원칙은 다음과 같다.

- 최소 범위 수정
- 기존 계약 유지
- 타입 안전 우선
- 에러/로딩/빈 상태 처리
- 실기기 UX 고려
- 성능 최적화는 근거 기반
- 재사용 가능한 단위는 분리하되 과분해 금지

검증은 반드시 `advanced-codex-checklist.md`를 따른다.

- 타입 오류 없음
- 빌드 가능
- 기존 기능 회귀 없음
- 요청 범위 준수
- 새 예외 케이스 처리
- 느린 네트워크/복귀/키보드/리스트 시나리오 위험 없음

## 7. 변경 위험도 등급

변경 전 반드시 위험도를 판단한다.

### Low

- 1~2개 파일
- UI 문구/스타일/명확한 버그 수정
- 기존 계약 영향 없음

### Medium

- 3~4개 파일
- 상태 연결/네비게이션/리스트 동작 수정
- 회귀 가능성 존재

### High

- 5개 이상 파일
- 구조 변경
- 상태 구조 변경
- DB/외부 서비스 영향
- 롤백 복잡
- 승인 필요

## 8. 서비스 수준 코딩 기준

## 8-1. TypeScript

- `strict` 전제를 기본으로 본다.
- `useUnknownInCatchVariables`에 맞는 패턴을 기본으로 사용한다.
- route params, props, store shape, API DTO, domain model을 분리한다.
- 잘못된 타입 단언보다 런타임 가드가 우선이다.
- nullable 처리 누락은 버그로 간주한다.

## 8-2. 상태 관리

항상 아래 네 종류를 구분한다.

1. 전역 상태

   - 로그인 사용자
   - 앱 전역 설정
   - 전역 UI 플래그

2. 서버 상태

   - API/DB에서 가져온 데이터
   - 캐시/동기화/재요청 정책 포함

3. 화면 상태

   - 모달 열림
   - 현재 탭
   - 입력값
   - 스크롤 위치

4. 파생 상태
   - 필터링 결과
   - 정렬 결과
   - 계산된 표시값

원칙:

- 서로 다른 성격의 상태를 한 store에 무분별하게 섞지 않는다.
- 목록 캐시와 상세 캐시를 분리한다.
- optimistic update는 실패 롤백 경로까지 설계한다.

## 8-3. React / React Native

- 불필요한 리렌더링을 항상 의심한다.
- 리스트는 수백/수천 건 전제를 기본으로 본다.
- inline callback, 불안정한 객체/배열 props, 넓은 selector 구독을 경계한다.
- 실기기에서 JS thread/UI thread 병목을 분리해서 본다.
- React 19+ 프로젝트라면 새 함수 컴포넌트는 ref-as-prop 우선 검토, React 18 이하라면 기존 `forwardRef` 계약을 유지한다.
- React Native는 New Architecture 호환성을 해치지 않는 방향을 우선한다.
- 라이브러리 선택 시 New Architecture 호환 여부를 먼저 본다.

## 8-4. 리스트 / 이미지

- 목록은 lightweight card를 기본으로 한다.
- 상세는 full detail 책임을 가진다.
- 썸네일과 원본은 분리 가능성을 항상 검토한다.
- signed URL 비용, 네트워크 비용, decode/render 비용을 분리해서 본다.
- 대량 리스트에서 index 기반 key는 금지한다.
- 무한 스크롤은 dedup, loading guard, race condition 방지까지 포함한다.

## 8-5. 비동기와 에러 처리

- 모든 async 경로에 실패 처리와 fallback을 둔다.
- 중복 요청, 응답 역전(out-of-order), unmount 후 setState를 방지한다.
- loading / error / empty / success를 모두 설계한다.
- retry가 있다면 사용자 경험과 중복 호출을 같이 본다.

## 8-6. 실기기 UX

반드시 아래를 점검한다.

- 첫 진입
- 복귀
- 빠른 스크롤
- 느린 네트워크
- 이미지 로딩
- 키보드 회피
- 포커스 이동
- 하단 버튼 가시성
- safe area
- 터치 영역
- 접근성 레이블

## 9. Supabase / 데이터 계층 규칙

아래 항목은 앱 코드만 수정해서 끝나는 작업이 아니다.

- DB schema
- migration SQL
- RPC
- RLS
- Storage policy
- seed / backfill / repair SQL

원칙:

- 앱 코드 변경과 DB 변경을 동일 작업으로 취급하지 않는다.
- 사람이 읽는 SQL 문서는 `docs/sql/` 아래에 둔다.
- 도메인 최종본 SQL: `docs/sql/<도메인명>/*-최종.sql`
- 도메인 레거시/단계 SQL: `docs/sql/<도메인명>/레거시/`
- 공용 bootstrap/manual bundle: `docs/sql/공용/`
- 실행 migration 이력: `supabase/migrations/`
- SQL은 seed/보정/복구/개발용 보조 수단으로 설명한다.
- Supabase가 최종 source of truth인 경우, 문서에도 동일하게 명시한다.

## 10. 문서 작성 규칙

## 10-1. 공통 원칙

- 모든 문서는 `docs/` 아래에 둔다.
- 폴더명/파일명은 한글 중심, 의미가 바로 드러나게 작성한다.
- 처음 보는 사람이 경로만 보고 역할을 이해할 수 있어야 한다.
- 문서도 서비스 운영 문서 수준으로 작성한다.

## 10-2. 권장 docs 구조

```text
docs/
├── domains/
├── sql/
├── operations/
├── project-memory/
├── engineering/
├── qa/
└── 리서치/
```

## 10-3. 리서치 문서 규칙

비단순 작업은 `docs/리서치/리서치.md`에 아래 순서로 기록한다.

1. 배경
2. 문제
3. 선택지
4. 결정 이유
5. 리스크
6. 다음 액션

## 10-4. 프로젝트 고정 표현

아래 표현은 문서에서 일관되게 유지한다.

- 최종 운영은 웹 관리자 페이지(CMS) 기준
- 현재 앱 관리자 화면은 중간 운영 수단
- SQL은 seed/보정용
- 앱은 사용자 소비 중심

## 11. 자동화 및 배포 안전 규칙

### 11-1. CI 품질 게이트

가능하면 아래 순서의 자동 검증을 전제로 한다.

1. format/lint
2. typecheck
3. unit/integration test
4. build
5. smoke test
6. release branch / tag 검토

프로젝트에 명령어가 정의돼 있으면 그 명령어를 사용하고, 없으면 추측으로 새 스크립트를 만들지 않는다.

### 11-2. Expo / EAS 사용 시

Expo/EAS를 사용하는 경우 아래를 점검한다.

- build가 어느 channel을 바라보는지
- update가 어느 branch에 publish되는지
- Android/iOS 채널 분리가 필요한지
- OTA 업데이트가 native 변경을 포함하지 않는지
- hotfix 배포가 현재 배포 전략과 충돌하지 않는지

### 11-3. 릴리즈 전 점검

- 스키마/정책 변경 동반 여부
- rollback 가능 여부
- feature flag 필요 여부
- 실기기 확인 필요 항목
- 문서 갱신 완료 여부

## 12. 롤백 규칙

| 상황           | 대응                                          |
| -------------- | --------------------------------------------- |
| 빌드 실패      | 이번 작업 전체를 되돌리고 원인 분석 후 재시작 |
| 특정 파일 오류 | 해당 파일 기준으로 부분 롤백 후 재검증        |
| 타입 붕괴      | 타입 구조부터 재분석, 임시 단언 금지          |
| 기존 기능 회귀 | 전체 작업 롤백 후 원인 재분석                 |
| DB/정책 실패   | 앱 코드 롤백 + DB 롤백 여부 별도 확인         |
| 원인 불명확    | 로그/계측 추가 후 다시 재현                   |

롤백 후 같은 접근을 반복하지 않는다.

## 13. Final Review Mode

최종점검 요청 시 아래 규칙을 예외 없이 따른다.

### 절대 금지

- 코드 수정
- 파일 저장
- 리팩토링
- import 정리
- lint fix
- dead code 제거
- rename / overwrite / apply patch

### 반드시 수행

- 잠재 버그 식별
- 상태 동기화 문제 식별
- 성능 병목 식별
- 리렌더링 위험 식별
- 타입 안정성 취약점 식별
- 키보드/복귀/네트워크/실기기 UX 리스크 식별
- 메모리 문서와 실제 코드 불일치 식별
