# Codex Workflow

이 문서는 이 프로젝트에서 Codex가 작업할 때 반드시 우선 따라야 하는 실무 기준을 정리한 로컬 작업 가이드다.

Codex는 이 프로젝트에서 항상 **시니어 개발자 관점**으로 사고하고 작업해야 한다.  
즉, 단순히 “동작하는 코드”가 아니라, **실제 서비스 운영 환경에서 오래 버틸 수 있는 구조**, **장기 유지보수 가능한 코드**, **안정성과 타입 안전성이 보장된 코드**를 기준으로 판단해야 한다.

모든 수정은 항상 다음 기준을 만족해야 한다.

- 실제 서비스 코드급 품질이어야 한다.
- 임시방편보다 구조적으로 안전한 해법을 우선한다.
- 최소 수정 원칙을 지키되, 결과물은 반드시 서비스 수준이어야 한다.
- “일단 돌아가는 코드”가 아니라 **실제 배포 가능한 코드**를 목표로 한다.

---

## Core Principles

1. 코드를 수정하거나 작업을 진행하기 전에 반드시 전체 코드베이스를 신중하게 분석한다.
2. 프로젝트의 전체 구조와 아키텍처를 충분히 이해한 뒤에만 수정한다.
3. 단계적으로 사고한다. (Think step-by-step)
4. 파일을 무작정 수정하지 않는다.
5. 기존 기능을 절대 깨뜨리지 않는다.
6. 가능한 기존 아키텍처와 설계를 유지한다.
7. 가능한 최소한의 코드만 수정한다.
8. 성능, 안정성, 가독성이 실제로 개선되는 경우에만 수정한다.
9. 실제 모바일 디바이스 환경에서의 UX를 최우선으로 고려한다.
10. TypeScript 타입 안정성을 반드시 보장한다.
11. `any` 사용, `undefined` 접근, 잘못된 타입 단언 등 unsafe 패턴을 피한다.
12. 재사용 가능한 컴포넌트와 구조를 우선적으로 설계한다.
13. 에러가 발생할 수 있는 로직에는 `try-catch`, fallback, guard 처리를 적용한다.
14. 코드 스타일과 프로젝트 컨벤션을 유지한다.
15. 불필요한 라이브러리를 추가하지 않는다.
16. 네트워크 요청은 효율적으로 관리하고 불필요한 API 호출을 줄인다.
17. 항상 실제 서비스 운영 기준으로 생각한다.
18. 단순 동작보다 장기 유지보수성과 확장성을 우선한다.
19. MVP 수준 임시방편보다 실제 서비스 수준의 구조를 우선 검토한다.
20. 성능 이슈는 항상 렌더링 비용, 이미지 비용, 네트워크 비용, 상태 변경 비용을 분리해서 본다.
21. 리렌더링 최적화는 선택이 아니라 필수 점검 항목이다.
22. 리스트 화면은 항상 대량 데이터(수백/수천 건)까지 가정하고 설계한다.
23. 이미지가 있는 화면은 항상 썸네일/원본 분리 가능성을 검토한다.
24. 추측으로 수정하지 않고, 원인 분석 → 최소 수정 → 검증 순서로 진행한다.
25. 실기기 체감 성능이 개선되지 않으면 최적화가 끝난 것이 아니다.
26. 설계 문서, 아키텍처 문서, SQL 문서, 마이그레이션 문서, 운영 문서는 반드시 `docs/` 폴더 아래에 작성한다.
27. 설계 문서는 항상 **해당 작업 이름의 폴더** 아래에 작성한다.
28. SQL 문서가 필요하면 항상 **해당 작업 폴더 안의 `데이터베이스/` 폴더** 아래에 작성한다.
29. 문서는 항상 **한글 폴더명 + 한글 파일명** 형식으로, 한눈에 알아보기 쉽게 작성한다.
30. 문서 생성 작업에서는 기존 앱 코드를 건드리지 않고, 문서만 추가하는 방식이 가능한지 먼저 검토한다.
31. 항상 **시니어 개발자처럼 판단**한다. 즉, 현재 문제만 보는 것이 아니라 이후 운영, 장애 대응, 확장, 유지보수까지 함께 고려한다.
32. 코드를 수정할 때는 항상 **실제 서비스 코드급 품질**로 작성한다.
33. “급한 임시 수정”보다 **구조적으로 안전하고 되돌리기 쉬운 수정**을 우선한다.
34. 수정 후에는 반드시 “이 코드가 실제 배포 환경에서도 안전한가?”를 기준으로 다시 검토한다.
35. 입력창이 있는 화면은 항상 **실기기 키보드 + 스크롤 + 포커스 가시성**까지 함께 점검한다.

## Mandatory Service-Level Requirements

작업 시 아래 조건은 항상 기본 전제로 포함한다.

### 1. 실제 서비스 기준 최적화

- 실제 서비스 수준의 데이터 양을 가정한다.
- 화면이 지금 데이터 수에서는 괜찮아 보여도, 수백/수천 건에서 무너지지 않는지 같이 본다.
- 목록용 데이터와 상세용 데이터는 역할이 다를 수 있음을 항상 고려한다.
- 타임라인/피드 화면은 lightweight card 구조를 기본으로 본다.
- 상세 화면은 단일 콘텐츠 집중 구조가 더 적합한지 항상 검토한다.
- 이미지가 있는 경우:
  - 원본 경로
  - 썸네일 경로
  - 캐시 전략
  - signed URL 비용
  - decode/render 비용
    을 분리해서 본다.
- 최적화는 눈앞의 한 화면만 보지 않고, **실제 서비스 전체 흐름** 안에서 판단한다.
- 모든 코드는 항상 **서비스 코드급 완성도**를 목표로 한다.

### 2. 리렌더링 최적화 필수

- 불필요한 리렌더링을 반드시 점검한다.
- `React.memo`, `useMemo`, `useCallback` 필요 지점을 항상 검토한다.
- inline callback, 불안정한 props, 넓은 selector 구독을 경계한다.
- 카드 리스트에서는 item 1개 변경이 전체 리스트를 흔들지 않는지 확인한다.
- 스크롤 중 state 업데이트가 반복되는 구조를 경계한다.
- 큰 배열/객체를 매 렌더마다 새로 만드는지 반드시 점검한다.
- “조금 덜 느린 정도”가 아니라, **실기기에서 실제로 체감이 좋아졌는지** 기준으로 본다.

### 3. TypeScript 타입 안정성 필수

- `any` 사용 금지
- 잘못된 타입 단언 금지
- nullable / optional 값 안전 처리 필수
- route param / props / store model shape 일치 여부 확인
- API 응답 타입 명확히 유지
- 타입 에러 가능성이 높은 파일은 반드시 우선 점검
- optional chaining과 fallback 값을 적절히 사용한다.
- “일단 돌아가게”보다 “타입상 안전하게”를 우선한다.
- 타입 안정성은 선택이 아니라, **서비스 코드 품질의 기본 조건**으로 본다.

### 4. 실기기 UX 필수

- 실제 모바일 기기에서 손가락으로 쓸 때 자연스러운지 우선 본다.
- 첫 진입, 빠른 스크롤, 복귀, 이미지 로딩, 키보드 대응, 터치 반응을 항상 점검한다.
- placeholder가 오래 남거나, 화면이 흔들리거나, 상단 요소가 sticky처럼 붙는 현상 등을 UX 리스크로 본다.
- 목록/상세 전환 흐름이 자연스러운지 본다.
- 느린 네트워크 / 백그라운드 복귀 / 캐시 만료 상황도 UX 기준으로 점검한다.
- UX는 디자인이 아니라 **실제 사용성 품질**로 본다.

### 5. 키보드 대응 필수

- 검색창, 폼, 편집 화면, 관리자 입력 화면은 항상 키보드가 올라온 상태까지 본다.
- 입력 필드가 키보드에 가려지지 않는지 확인한다.
- 하단 버튼, 저장 버튼, 제출 버튼이 키보드에 묻히지 않는지 확인한다.
- iOS / Android 각각에서 키보드 회피 방식이 자연스러운지 본다.
- `KeyboardAvoidingView` 하나만 넣고 끝내지 않고, 실제 스크롤 가능 여부와 포커스 이동 가시성까지 본다.
- 리스트 + 검색 입력 조합에서는 키보드가 열린 상태에서도 스크롤과 탭 동작이 끊기지 않는지 확인한다.

## Documentation Rules

### 1. 문서 생성 위치

- 설계 문서, SQL 문서, 마이그레이션 문서, 운영 가이드 문서, 아키텍처 문서는 반드시 `docs/` 폴더 아래에 생성한다.
- 설계 문서는 항상 **해당 작업 이름의 폴더**를 먼저 만들고, 그 안에 작성한다.
- 프로젝트 루트나 임의 폴더에 문서를 흩어 놓지 않는다.
- 문서 추가가 목적일 때는 앱 코드 수정 없이 문서만 생성하는 방식을 우선 검토한다.

예시:

- `docs/집사-꿀팁-가이드/집사-꿀팁-가이드-통합-완성본.md`
- `docs/타임라인-메모리-이미지-시스템/타임라인-메모리-이미지-통합-완성본.md`
- `docs/마이그레이션-운영-기준/마이그레이션-운영-기준-통합-완성본.md`
- `docs/공용-릴리즈-묶음-운영-기준/공용-릴리즈-묶음-운영-기준-통합-완성본.md`

### 2. SQL / 데이터베이스 문서 위치 규칙

- 프로젝트의 모든 SQL은 `docs/sql/` 아래에서 한 번에 관리한다.
- 도메인 기준 최종본 SQL은 `docs/sql/도메인/<작업명>/` 아래에 작성한다.
- 실제 릴리즈 이력용 migration 파일은 `docs/sql/마이그레이션/`에 둔다.
- 공용 SQL 중 migration이 아닌 수동 실행용 릴리즈 묶음은 `docs/sql/공용-릴리즈-묶음/` 아래에서 목적별 폴더로 관리한다.
- 즉, 실행 기준으로 바로 참고할 SQL도 `docs/sql` 안에서 경로만 보고 역할을 구분할 수 있어야 한다.

예시:

- `docs/sql/도메인/집사-꿀팁-가이드/집사-꿀팁-가이드-운영-스키마-최종.sql`
- `docs/sql/도메인/타임라인-메모리-이미지-시스템/타임라인-메모리-이미지-운영-스키마-최종.sql`
- `docs/sql/마이그레이션/2026-03-16_집사-꿀팁-가이드-검색-RPC.sql`
- `docs/sql/공용-릴리즈-묶음/전체-초기-구성/누리-전체-초기-구성.sql`

### 3. 문서 네이밍 규칙

- 문서는 항상 **한글 폴더명 + 한글 파일명** 형식으로 작성한다.
- 설계 문서는 해당 작업 이름이 드러나야 한다.
- 파일명은 목적이 바로 드러나게 구체적으로 작성한다.

예시:

- `docs/집사-꿀팁-가이드/집사-꿀팁-가이드-통합-완성본.md`
- `docs/타임라인-메모리-이미지-시스템/타임라인-메모리-이미지-통합-완성본.md`
- `docs/sql/도메인/타임라인-메모리-이미지-시스템/타임라인-메모리-이미지-운영-스키마-최종.sql`

### 4. 문서 작성 원칙

- 문서는 실제 서비스 운영 기준으로 작성한다.
- 임시방편이 아니라 장기 운영/백필/마이그레이션/롤백까지 고려한다.
- 설명은 추상적으로 쓰지 말고, 실제 실행 순서와 적용 범위를 명확히 적는다.
- SQL 문서는 가능한 그대로 실행 가능한 형태로 정리한다.
- 설계 문서는 다음 구현 단계가 바로 이어질 수 있을 정도로 구체적으로 쓴다.
- 문서도 코드와 마찬가지로 **실제 서비스 문서 수준**으로 작성한다.

### 5. 집사 꿀팁 가이드 운영 방향 규칙

- 집사 꿀팁 가이드의 **최종 운영 방식은 웹 관리자 페이지(CMS)** 기준으로 설명한다.
- NURI 모바일 앱은 사용자 조회와 소비 채널로 설명한다.
- 앱 안의 관리자 화면은 현재 단계에서만 쓰는 **임시/중간 운영 수단(브릿지 단계)** 으로 설명한다.
- Supabase는 최종 source of truth로 본다.
- SQL은 기본 운영 입력 수단이 아니라 **초기 세팅 / 개발·테스트 seed / QA 데이터 / 긴급 복구 / 운영 보정용 보조 수단**으로 설명한다.
- 집사 꿀팁 가이드 챕터의 최종 완성 단계에는 반드시 **웹 CMS 구축**이 포함되어야 한다.
- 문서와 설명에서는 아래 표현을 우선 사용한다.
  - `최종 운영은 웹 관리자 페이지(CMS) 기준`
  - `현재 앱 관리자 화면은 중간 운영 수단`
  - `SQL은 seed/보정용`
  - `앱은 사용자 소비 중심`

## Project Analysis Checklist

### 1. 프로젝트 구조 분석

- 디렉토리 구조
- 컴포넌트 계층 구조
- 화면(Screen) 구성
- 공통 컴포넌트 구조
- 도메인별 분리 여부
- 목록 화면 / 상세 화면 / 입력 화면 역할 분리 여부

### 2. 상태 관리 구조 분석

- Zustand 사용 여부 및 구조
- 전역 상태와 로컬 상태 분리 여부
- 불필요한 상태 업데이트 여부
- 상태 변경으로 인한 리렌더링 문제
- selector 범위가 과도하게 넓지 않은지
- entity cache / list cache 분리 여부
- 단건 데이터와 목록 데이터가 뒤엉켜 있지 않은지

### 3. Navigation 구조 분석

- React Navigation 구조
- Stack / Tab / Modal 흐름
- 화면 전환 시 불필요한 리렌더링
- navigation props 전달 구조
- route param 타입 안정성
- 목록 → 상세 → 복귀 흐름 자연스러움
- 포커스 복귀 / reset / replace 시나리오 안정성

### 4. API 및 데이터 흐름 분석

- 데이터 fetch 구조
- API 호출 위치
- 캐싱 여부
- 불필요한 재요청
- 목록용 payload와 상세용 payload 분리 여부
- 이미지 signed URL 발급 구조
- 원본/썸네일 경로 분리 가능성
- 대량 데이터 기준 fetch 전략

### 5. React Native 성능 분석

- `FlatList` / `SectionList` / `FlashList` 사용 여부
- `keyExtractor` 최적화 여부
- renderItem 안정성
- 이미지 렌더링 최적화 여부
- 불필요한 화면/셀 리렌더링 여부
- 스크롤 중 state churn 여부
- placeholder / skeleton / preload 전략 적절성
- viewport 기반 이미지 로딩 여부
- 대량 데이터 기준 리스트 엔진 설정 적절성

## Performance Optimization Checklist

### 1. 리렌더링 최적화

- 불필요한 리렌더링 식별
- `React.memo` 적용 가능한 컴포넌트
- `useMemo` 사용 가능한 계산 로직
- `useCallback` 적용 가능한 함수
- unstable props/callback 존재 여부
- selector로 인한 넓은 리렌더 여부
- 상위 state 하나가 리스트 전체를 흔드는지 여부

### 2. 상태 업데이트 최적화

- 불필요한 `setState` 제거
- Zustand 상태 변경 범위 최소화
- selector 사용 여부 확인
- 스크롤 중 상태 업데이트 여부
- 이미지 로딩 상태가 카드마다 과도하게 퍼지는지 여부
- optimistic update 후 전체 refresh가 과도한지 여부

### 3. 컴포넌트 구조 개선

- 반복되는 UI 패턴을 재사용 컴포넌트로 분리
- 지나치게 큰 컴포넌트 분리
- props 구조 단순화
- 목록 카드와 상세 카드 책임 분리
- lightweight card / full detail card 분리 가능 여부

### 4. React Native 성능 점검

- `FlatList` / `SectionList` / `FlashList` 적용 가능 여부
- `keyExtractor` 안정성 점검
- `renderItem` 안정성
- `estimatedItemSize`, `windowSize`, `maxToRenderPerBatch` 등 리스트 설정 점검
- 이미지 렌더링 최적화
- placeholder / preload / defer 전략 점검
- 불필요한 re-render 방지
- 이미지 decode/render 비용과 signed URL 비용 분리 점검

### 5. 이미지 최적화 점검

- 타임라인은 썸네일 중심인지
- 상세는 원본 중심인지
- 썸네일/원본 경로가 분리 가능한지
- signed URL 발급 비용이 과도하지 않은지
- transform 기반인지, 별도 thumbnail asset이 필요한지
- 캐시 hit 체감이 충분한지
- FastImage 등 이미지 라이브러리 도입 필요성과 호환성 여부

## Standard Working Sequence

### 1단계: 프로젝트 전체 분석

- 구조 분석
- 상태 관리 분석
- 네비게이션 구조 분석
- 데이터 흐름 분석
- 리스트/이미지/상세 구조 분석

### 2단계: 문제점 식별

- 성능 병목
- 불필요한 리렌더링
- 상태 관리 문제
- 타입 안정성 문제
- UI / UX 문제
- 이미지 병목
- 네트워크/캐시 병목

### 3단계: 개선 전략 수립

- 코드 수정 전에 어떤 개선을 할지 먼저 정리한다.
- 원인을 분리해서 본다.
  - 리스트 문제인지
  - 카드 무게 문제인지
  - 이미지 문제인지
  - 상태 업데이트 문제인지
- 필요하면 진단용 로그/토글/프로파일링부터 넣는다.

### 4단계: 코드 수정

- 필요한 부분만 최소 범위로 수정한다.
- 성능 최적화와 코드 가독성을 함께 개선한다.
- 실제 서비스 기준 확장성까지 고려한다.
- 기존 기능과 계약을 최대한 유지한다.
- 항상 **시니어 개발자처럼**, 실제 서비스 코드급 완성도를 목표로 수정한다.

### 5단계: 검증

다음을 반드시 확인한다.

- 프로젝트가 정상적으로 빌드되는지
- TypeScript 오류가 없는지
- 런타임 오류가 발생하지 않는지
- 실제 모바일 디바이스에서 UI가 정상 동작하는지
- 네비게이션 흐름이 깨지지 않는지
- 기존 기능이 그대로 유지되는지
- 스크롤 체감이 실제로 나아졌는지
- placeholder / 이미지 / signed URL / 캐시 체감이 나아졌는지
- create/edit/delete / 복귀 / 상세 진입이 그대로 유지되는지

## Final Review Mode (진단 전용 모드)

최종점검 요청 시 아래 규칙을 따른다.

1. 절대 코드 수정하지 않는다.
2. 절대 파일 저장하지 않는다.
3. 절대 리팩토링하지 않는다.
4. 절대 자동 정리하지 않는다.
5. 절대 import 정리, format, lint fix, dead code 제거를 하지 않는다.
6. 절대 apply_patch, write, overwrite, rename 하지 않는다.
7. 진단 전용으로만 수행한다.

이 모드에서는 아래를 반드시 점검한다.

- 잠재 버그
- 불안정한 흐름
- 상태 동기화 문제
- 실기기 UX 리스크
- 불필요한 리렌더링
- 과한 상태 업데이트
- useMemo / useCallback 필요 지점
- 리스트 렌더링 비효율
- 무거운 계산의 반복 실행
- stale closure / effect 의존성 문제
- any 사용
- nullable 처리 취약점
- route param / props / store model shape 불일치
- 타입 에러 가능성이 높은 파일
- 이미 고친 부분이 다시 깨질 가능성이 있는 지점
