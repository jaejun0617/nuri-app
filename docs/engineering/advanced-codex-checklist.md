# advanced-codex-checklist.md

> 이 문서는 Codex의 작업 전·중·후 검증 기준이다.
> 행동 우선순위와 범위 통제는 `advanced-codex-workflow.md`,
> 컨텍스트 유지와 프로젝트 기억은 `advanced-codex-memory.md`를 따른다.

---

## 0. 사용 방식

이 체크리스트는 아래 4개 모드로 사용한다.

1. 작업 시작 전 구조 분석
2. 작업 중 구현 검증
3. 작업 완료 후 품질 게이트
4. 최종점검 모드(Final Review)

---

## 1. 문서 개선 포인트

### Fix

- 구조 분석, 타입, 성능, UX, 문서, 완료 검증을 하나의 흐름으로 재배치했다.
- “점검만 하고 끝나는 체크리스트”가 아니라 승인/회귀/배포 안전까지 포함하도록 보완했다.
- 중복되던 항목을 통합하고 우선순위를 명확히 했다.

### Enhance

- 상태 관리, API, 리스트, 이미지, 키보드, 접근성, Supabase, 테스트 자동화 기준을 세분화했다.
- 사람이 반드시 확인해야 하는 항목과 Codex가 자체 검증 가능한 항목을 분리했다.
- 최종점검 결과 형식을 더 구체적인 진단 템플릿으로 강화했다.

### Add

- React 19+/React Native New Architecture/EAS Update 관점의 검증 항목을 추가했다.
- `docs/리서치/리서치.md` 기록 여부를 검증 항목에 포함했다.
- 변경 위험도와 승인 필요 여부를 체크리스트에 연결했다.

## 3. 프로젝트 구조 분석 체크리스트

## 3-1. 아키텍처

- [ ] 디렉토리 구조와 도메인 분리 방식이 명확한가
- [ ] 화면, 공통 컴포넌트, 도메인 로직의 경계가 명확한가
- [ ] 목록 / 상세 / 입력 화면 책임이 분리되어 있는가
- [ ] 유틸, 훅, 컴포넌트, 서비스 레이어가 뒤엉켜 있지 않은가

## 3-2. 상태 관리

- [ ] 전역 상태와 화면 상태가 분리되어 있는가
- [ ] 서버 상태와 UI 상태가 섞여 있지 않은가
- [ ] 파생 상태를 원본 상태로 저장하고 있지 않은가
- [ ] 목록 캐시와 상세 캐시가 분리되어 있는가
- [ ] Zustand selector 범위가 과도하게 넓지 않은가
- [ ] store 전체 구독이 있는가
- [ ] optimistic update 롤백 경로가 있는가

## 3-3. Navigation

- [ ] Stack / Tab / Modal 흐름이 명확한가
- [ ] route param 타입이 정의돼 있는가
- [ ] 목록 → 상세 → 복귀 흐름이 자연스러운가
- [ ] reset / replace / back 흐름이 계약과 일치하는가
- [ ] deep link 또는 외부 진입 시나리오가 깨지지 않는가

## 3-4. 데이터 흐름

- [ ] fetch 위치가 적절한가
- [ ] 중복 요청이 발생하지 않는가
- [ ] 목록용 payload와 상세용 payload가 분리되어 있는가
- [ ] signed URL 발급 흐름이 과도하지 않은가
- [ ] 캐시 만료/재요청 정책이 일관적인가
- [ ] race condition 가능성이 없는가

## 4. TypeScript 안전성 체크리스트

## 4-1. 기본 규칙

- [ ] `any` 사용이 없는가
- [ ] `ts-ignore` 사용이 없는가
- [ ] 꼭 필요한 경우 외 타입 단언(`as`)을 쓰지 않았는가
- [ ] `unknown`을 적절히 좁히고 있는가
- [ ] nullable / optional 값을 안전하게 처리하는가
- [ ] `?.`, `??`, guard clause를 적절히 사용하는가

## 4-2. 모델 정합성

- [ ] API 응답 타입이 명확한가
- [ ] route param 타입이 navigation 정의와 일치하는가
- [ ] props 타입이 누락 없이 정의돼 있는가
- [ ] store model shape과 실제 사용 타입이 일치하는가
- [ ] 폼 입력 타입과 저장 타입이 구분되어 있는가
- [ ] 서버 DTO와 화면 VM(ViewModel)이 필요 시 분리되어 있는가

## 4-3. 함수와 비동기

- [ ] 이벤트 핸들러 타입이 정확한가
- [ ] async 함수 반환 타입이 명확한가
- [ ] `catch` 블록에서 `unknown` 기준으로 처리하는가
- [ ] `Promise<void>` / `Promise<Result>` 등 반환 의도가 명확한가
- [ ] unhandled promise가 남지 않는가

## 4-4. useEffect / hooks

- [ ] effect 의존성이 누락되지 않았는가
- [ ] 의존성 과다로 무한 루프가 생기지 않는가
- [ ] async effect에 취소 처리(cancelled flag / AbortController)가 있는가
- [ ] subscription / timer / listener cleanup이 있는가
- [ ] stale closure 위험이 없는가
- [ ] custom hook의 입력/출력 계약이 명확한가

---

## 5. 성능 최적화 체크리스트

## 5-1. 리렌더링

- [ ] `React.memo` 적용 필요 컴포넌트를 확인했는가
- [ ] inline callback을 자식 props로 넘기고 있지 않은가
- [ ] `useCallback`이 필요한 함수가 안정화되어 있는가
- [ ] `useMemo`가 필요한 계산이 메모이제이션되어 있는가
- [ ] 큰 배열/객체를 매 렌더마다 새로 만들고 있지 않은가
- [ ] 상위 state 하나가 리스트 전체를 흔들지 않는가
- [ ] selector가 넓어 필요 없는 리렌더를 유발하지 않는가

## 5-2. 리스트

- [ ] 리스트 성격에 맞는 엔진을 선택했는가
  - [ ] `FlatList`
  - [ ] `SectionList`
  - [ ] `FlashList`
- [ ] `keyExtractor`가 안정적인 고유 키를 쓰는가
- [ ] `renderItem` 경로가 안정적인가
- [ ] estimated size / window / batch 설정이 필요한가
- [ ] 무한 스크롤 중 중복 데이터 dedup을 하는가
- [ ] onEndReached 중복 호출을 막는가
- [ ] 빈 상태 / 로딩 상태 / 에러 상태가 분리돼 있는가

## 5-3. 이미지

- [ ] 목록은 썸네일 중심인가
- [ ] 상세만 원본을 쓰는가
- [ ] signed URL 발급이 반복되지 않는가
- [ ] 캐시 전략이 있는가
- [ ] placeholder / skeleton이 있는가
- [ ] onError fallback이 있는가
- [ ] 이미지 decode 비용과 네트워크 비용을 구분해서 보고 있는가

## 5-4. React Native / 실기기

- [ ] JS thread 병목과 UI thread 병목을 분리해서 봤는가
- [ ] 스크롤 중 state churn이 없는가
- [ ] 애니메이션/스크롤에 무거운 연산이 없는가
- [ ] 느린 Android 기기 기준에서도 무너지지 않는가
- [ ] New Architecture 호환성에 문제 있는 라이브러리를 도입하지 않았는가
- [ ] 성능 개선이 “코드상 좋아 보임”이 아니라 체감상 의미 있는가

## 5-5. React 19+ / 최신 기준

- [ ] 프로젝트가 React 19+인지 확인했는가
- [ ] React 19+면 새 함수 컴포넌트에서 ref-as-prop을 고려했는가
- [ ] React 18 이하 호환성이 필요하면 기존 `forwardRef` 계약을 깨지 않았는가
- [ ] ref 콜백/cleanup 동작 변화로 인한 부작용이 없는가

- [ ] 상태 변경이 스크린리더 사용자에게 혼란스럽지 않은가
