# 타임라인 썸네일 생성 파이프라인 설계

## 목적

이 문서는 `memory_images.original_path` 를 기준으로 서버가 타임라인 썸네일을 생성하고, 생성 결과를 `memory_images.timeline_thumb_path` 와 `status` 에 반영하는 실제 서비스형 파이프라인을 정의한다.

현재 전제:

- `memory_images` 테이블은 이미 생성되어 있다.
- 신규 이미지 row 는 `status = 'pending'` 으로 저장할 수 있다.
- 앱은 아직 `timeline_thumb_path` 전용 조회로 완전히 전환되지 않았다.
- 기존 `memories.image_url / image_urls` fallback 은 당분간 유지된다.

이 문서의 목표는 앱 조회 전환보다 먼저, 썸네일 생성 파이프라인을 안정적으로 설계하는 것이다.

---

## 1. 추천 아키텍처

### 결론

추천 방식은 다음 조합이다.

- `memory_images.status` 기반 상태 관리
- scheduler 가 주기적으로 호출하는 `Edge Function 또는 별도 worker`
- batch polling
- row locking (`for update skip locked`)

즉, 추천 구조는 `queue/cron/worker` 중심이다.

### 왜 이 구조가 적절한가

#### 1) DB trigger 단독 처리 비추천

DB trigger 안에서 이미지 다운로드, 리사이즈, Storage 업로드 같은 무거운 작업을 직접 처리하면 안 된다.

이유:

- 트랜잭션 시간이 길어진다
- DB 부하와 장애 전파 범위가 커진다
- 외부 I/O 실패를 다루기 어렵다
- 재시도와 모니터링이 불편하다

DB trigger 는 썸네일 "생성 작업을 시작하라"는 상태 전환까지만 맡는 게 맞다.

#### 2) Edge Function 단건 즉시 호출만으로 끝내는 구조 비추천

원본 업로드 직후마다 바로 Edge Function 을 동기 호출해 썸네일 생성까지 끝내려 하면 burst 상황에서 불안정해진다.

이유:

- 업로드와 썸네일 생성이 사용자 저장 체감에 직접 붙는다
- timeout, 재시도, 중복 실행 제어가 어려워진다
- 백필과 신규 처리를 같은 파이프라인으로 묶기 어렵다

#### 3) queue/cron/worker 구조 추천

`pending -> ready / failed` 상태 전환을 중심으로 worker 가 배치 처리하면:

- 신규 처리와 백필을 같은 경로로 처리 가능
- 실패 재시도 정책을 일관되게 운영 가능
- 동시성 제어가 명확함
- 사용자 저장 흐름과 썸네일 생성 흐름을 분리 가능

### 추천 구현 형태

가장 현실적인 조합:

1. 앱/서버가 원본 업로드 후 `memory_images` row 생성
2. row 는 `status='pending'`, `timeline_thumb_path=null`
3. `pg_cron` 또는 외부 scheduler 가 주기적으로 Edge Function 호출
4. Edge Function 또는 worker 가 pending row 를 batch 로 처리

요약:

- DB trigger: 상태 전환 보조 용도는 가능
- 실제 생성 책임: Edge Function / worker
- 실행 제어: cron / scheduler

## 2. 처리 흐름

### 신규 업로드 기준 step-by-step

1. 클라이언트가 원본 이미지를 Storage 에 업로드한다.
2. 앱 또는 서버가 `memory_images` row 를 생성한다.
3. row 에 아래 값을 저장한다.
   - `original_path`
   - `timeline_thumb_path = null`
   - `status = 'pending'`
4. scheduler 가 Edge Function 또는 worker 를 실행한다.
5. worker 가 `status in ('pending', 'failed')` 인 row 를 batch 로 가져온다.
6. worker 는 대상 row 를 lock 한다.
7. worker 가 `original_path` 원본 파일을 다운로드한다.
8. worker 가 타임라인용 리사이즈 이미지를 생성한다.
9. worker 가 `timeline-thumb/...` 경로에 썸네일을 업로드한다.
10. worker 가 `timeline_thumb_path` 를 저장한다.
11. worker 가 `status='ready'` 로 업데이트한다.

### 실패 기준 step-by-step

1. 원본이 없거나 다운로드 실패
2. 이미지 디코딩/리사이즈 실패
3. Storage 업로드 실패
4. DB 업데이트 실패

실패 시:

- row 를 `status='failed'` 로 업데이트
- 로그에 원인 기록
- 재시도 대상이면 다음 scheduler 주기에서 다시 처리

## 3. 입력 / 출력 데이터 구조

### 입력 데이터

`public.memory_images`

- `id`
- `memory_id`
- `user_id`
- `pet_id`
- `sort_order`
- `original_path`
- `timeline_thumb_path`
- `width`
- `height`
- `status`
- `created_at`

### worker 내부 입력

- `memory_image_id`
- `original_path`
- `target_variant = timeline-thumb`
- `target_width = 144`
- `target_height = 144`
- `resize_mode = cover`
- `output_format = jpeg 또는 webp`

### 출력 데이터

성공 시:

- `timeline_thumb_path`
- `status = 'ready'`

실패 시:

- `status = 'failed'`

운영상 후속으로 추가 추천하는 메타:

- `retry_count`
- `last_error_code`
- `last_error_message`
- `last_processed_at`

지금 테이블에 없더라도, 장기 운영 기준으로는 후속 migration 으로 추가하는 것이 좋다.

## 4. Supabase Storage 접근 방식

### 권장 경로 구조

- 원본: `memory-images/original/{userId}/{petId}/{memoryId}/{imageId}.jpg`
- 타임라인 썸네일: `memory-images/timeline-thumb/{userId}/{petId}/{memoryId}/{imageId}.jpg`

### worker 접근 방식

worker 는 service role 권한으로 Storage 를 다룬다.

흐름:

1. `original_path` 로 원본 다운로드
2. 서버 측 이미지 라이브러리로 리사이즈
3. deterministic 한 `timeline-thumb/...` path 계산
4. 해당 경로에 업로드
5. `timeline_thumb_path` 저장

### 중요한 운영 원칙

- 앱 사용자 세션으로 처리하지 않는다
- deterministic path 를 사용한다
- 같은 입력에 대해 같은 썸네일 path 가 나오게 한다
- overwrite 전략 또는 upsert 전략은 운영 기준에 맞춰 명확히 정한다

권장:

- `timeline_thumb_path` 는 `memory_image_id` 기준 deterministic path
- 재생성 시 같은 path 재사용

## 5. 실패 처리 / 재시도 전략

### 실패 유형 분리

#### 일시적 실패

- Edge Function timeout
- Storage 응답 지연
- 네트워크 순간 오류
- 일시적 리사이즈 실패

이 경우 재시도 대상이다.

#### 영구 실패

- 원본 파일 없음
- 손상된 파일
- 지원하지 않는 포맷
- 잘못된 path

이 경우 반복 재시도를 제한해야 한다.

### 재시도 전략

권장 방식:

- `pending`, `failed` 둘 다 worker 조회 대상
- `failed` 는 최근 실패 시점과 `retry_count` 기준으로 재시도
- exponential backoff 적용

예:

- 1회 실패: 5분 후
- 2회 실패: 30분 후
- 3회 실패: 6시간 후
- 4회 이상: 운영자 확인 대상

### 운영 추천

후속 migration 으로 아래 필드 추가 검토:

- `retry_count integer not null default 0`
- `last_error_code text`
- `last_error_message text`
- `last_processed_at timestamptz`

## 6. 중복 실행 방지(idempotency)

썸네일 생성 파이프라인은 반드시 idempotent 해야 한다.

### 핵심 원칙

1. 같은 `memory_image_id` 는 항상 같은 썸네일 path 를 사용
2. 이미 `status='ready'` 이고 `timeline_thumb_path` 가 있으면 재처리 스킵
3. 여러 worker 가 떠도 같은 row 를 동시에 처리하지 않게 lock 사용

### 추천 조회 방식

개념 예시:

```sql
select id
from public.memory_images
where status in ('pending', 'failed')
order by created_at asc
for update skip locked
limit 50;
```

### 왜 필요한가

- scheduler 중복 실행
- timeout 이후 재호출
- 운영자가 수동 재실행
- 백필과 신규 처리 동시 실행

이런 상황에서도 결과가 꼬이지 않아야 한다.

## 8. 백필 파이프라인 재사용 가능 여부

재사용 가능해야 한다. 그리고 그래야 한다.

### 권장 원칙

- 신규 생성 row 와 기존 데이터 백필 row 를 같은 worker 가 처리
- 차이는 입력 row 가 신규인지 과거 데이터인지뿐
- 썸네일 생성 본체는 완전히 동일

### 백필 흐름

1. 기존 `memories.image_url / image_urls` 를 `memory_images.original_path` 로 이관
2. 이관된 row 는 `status='pending'`
3. 신규 처리와 같은 worker 가 batch 로 썸네일 생성
4. 완료 시 `timeline_thumb_path`, `status='ready'` 저장

즉, 신규와 백필을 별도 구현으로 나누지 말고 같은 파이프라인을 재사용하는 게 맞다.

## 9. 앱 조회 전환은 왜 이 설계 이후에 해야 하는가

앱이 타임라인 조회를 먼저 `timeline_thumb_path` 우선 구조로 바꾸면, 아직 썸네일이 없는 row 가 많을 경우 fallback 비중이 높아지고 전환 효과도 약해진다.

더 큰 문제는:

- 신규 데이터도 썸네일 생성 지연 시 null 상태가 생김
- 백필 전 기존 데이터는 대부분 ready 가 아님
- 조회 코드는 바뀌었는데 데이터 준비가 덜 되면 운영 중 혼란이 생김

그래서 순서는 아래가 맞다.

1. DB 구조 준비
2. 서버 썸네일 생성 파이프라인 준비
3. 신규 업로드 데이터가 안정적으로 `ready` 로 전환되기 시작
4. 백필 진행
5. 그 다음 앱 타임라인 조회를 `timeline_thumb_path` 우선으로 전환

즉, 조회 전환보다 생성 파이프라인이 먼저여야 한다.

## 10. 추천 구현안 요약

현 시점 추천:

- 실행 주체: Edge Function 또는 별도 worker
- 제어 방식: cron/scheduler 호출
- 대상 조회: `pending`, `failed`
- 동시성 제어: `for update skip locked`
- 결과 상태: `pending -> ready / failed`
- path 규칙: deterministic
- 재시도: backoff 기반
- 신규와 백필: 동일 worker 재사용

## 11. 후속 작업 우선순위

1. worker 상세 구현 명세
2. thumbnail path naming 규칙 확정
3. retry/error 메타 필드 확장 migration 검토
4. 신규 업로드 경로를 `memory_images` 기준으로 정리
5. 앱 타임라인 조회를 `timeline_thumb_path` 우선으로 전환
6. 상세 조회를 `original_path` 기준으로 전환
